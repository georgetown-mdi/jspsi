# Changelog

This changelog records, per release, the changes that affect how PSI-Link is run, deployed, or reviewed: new commands and options, changed defaults and behavior, breaking changes, and security-relevant changes. It is not a log of every commit (that is the git history), and it omits internal refactors and test, CI, and tooling changes that no operator or reviewer acts on. Each entry states the observable change and points to `docs/` for the rationale and `docs/spec/` for wire-level detail. The format loosely follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); PSI-Link uses [semantic versioning](https://semver.org/).

## [Unreleased]

### Added

- `linkage_terms.linkage_strategy`: choose how the agreed linkage keys are run on the wire. The default `cascade` runs one dependent PSI round per key, so round trips grow with the key count; `single-pass` batches every key into one exchange and has the receiver reconstruct the identical association table locally, so the round-trip count is constant in the number of keys -- making a multi-key linkage practical over a high-latency channel such as `filedrop` or `sftp`. Both produce the same result. The strategy is mandatory-consistency: both parties must agree or the exchange aborts. `single-pass` is not a free optimization -- the receiver observes the sender's full per-key duplicate structure, including matches on less precise keys the cascade would have filtered out -- so it is a consented disclosure tradeoff, not a default; the emitted result is unchanged. See `docs/EXCHANGE_REFERENCE.md` and `docs/spec/PROTOCOL.md`.
- `--linkage-strategy cascade|single-pass` on `psilink invite` and the zero-setup exchange (`psilink URL INPUT_FILE`): select the linkage strategy from the CLI when authoring terms, instead of hand-editing `linkage_strategy` into a terms file. The default is `cascade` (behavior unchanged when omitted); an unknown value is a usage error (exit 64); on `invite` the flag is reported as ignored when the linkage terms come from a pre-existing configuration file (that file's `linkage_strategy` stands). Selecting `single-pass` prints the disclosure tradeoff at selection time, and `psilink accept` now shows the chosen strategy -- with the same note for `single-pass` -- on its consent prompt; accepting against a pre-existing configuration whose `linkage_strategy` differs from the invitation's now aborts with that conflict rather than silently reusing the saved strategy, so the strategy you consent to is the one the exchange runs. In the zero-setup form each party infers its own terms, so both must pass the same value or the exchange aborts on the consistency check. See `docs/CLI.md`.
- The web "Advanced options" editor's Expert authoring mode now exposes the linkage strategy as a control, so an inviter can choose `single-pass` without hand-editing the raw terms; the default stays `cascade`, the option surfaces the consented disclosure tradeoff at the point of choice, and import/export round-trips the value. A `single-pass` invitation also carries that disclosure note onto the acceptor's consent screen, always-visible, so the partner sees and consents to the added disclosure before connecting. See `docs/COMMUNICATION.md` and `docs/EXCHANGE_REFERENCE.md`.
- `psilink init [INPUT_FILE]`: write a commented `psilink.yaml` template -- every option documented inline with defaults pre-filled -- to hand-edit before a first exchange, then exit without running an exchange, generating an invitation, or writing a key file. With an `INPUT_FILE` (a CSV, or `-` for stdin), column metadata, linkage fields, and standardizing transformations are inferred from it. A file already at the output path is overwritten only after an interactive confirmation; declining, or any non-interactive run, leaves it untouched. Invalid input exits 64; the command makes no network connection. See `docs/CLI.md`.
- `psilink invite` and `psilink accept`: turn a one-off exchange into a recurring one backed by a shared secret. Each runs offline (write key and config files) or online (connect and run in one step). An expired or malformed invitation is rejected before you are prompted, and a pre-existing config or key at the target path is reconciled rather than clobbered. See `docs/CLI.md`.
- `psilink accept --consent-to-terms`: record advance consent to the invitation's disclosed terms and skip the interactive accept confirmation, so `accept` can run unattended or in a script -- without it, a run with no terminal reads end-of-file at the prompt and declines. It is off by default (the prompt is unchanged when omitted) and bypasses the one human checkpoint before the configuration and linkage key are written from the partner-supplied invitation, so it stays opt-in; it is scoped to this consent and does not affect SSH host-key trust. With `--consent-to-terms` the `INPUT_FILE` may be `-` to read the CSV from stdin, which is otherwise rejected because the prompt claims standard input. See `docs/CLI.md`.
- `psilink exchange` now fails fast, before connecting, when a recurring config can no longer disclose the payload columns it committed to when the invitation was generated: if the current column metadata no longer transmits exactly that promised set, the exchange stops with a local configuration error (exit 64) naming the offending column(s), instead of silently sending a different set and having the partner abort mid-exchange (which surfaced as the *partner's* failure, after data had moved). Every `psilink invite` now records the disclosed set into the config's `disclosed_payload_columns` field (the send-side mirror of `expected_payload_columns`) as it mints the invitation -- a fresh invite writes it with the config, and a re-invite from an existing config refreshes it in place (your comments preserved) -- so the recorded promise stays in step with each new invitation and cannot go stale. The error offers to either restore the column's metadata or re-establish the exchange with the narrower disclosure, so choosing to disclose less is a first-class option. See `docs/EXCHANGE_REFERENCE.md` and `docs/spec/FILE_SYNC.md`.
- Receipt signing identities and `psilink fingerprint`: a long-lived per-party Ed25519 identity and self-signed certificate, pinned by fingerprint exchanged out-of-band. `psilink fingerprint` prints it (and regenerates with `--force`). This is the foundation for signed receipts; receipts themselves are not yet wired up. See `docs/SECURITY_DESIGN.md`.
- Self-attested exchange record: each party writes a local, unsigned audit record (not a non-repudiable receipt) of what it disclosed -- the agreed-terms hash, both identities, records exposed, governance metadata, and salted commitments to the data -- suitable for a HIPAA or FERPA disclosure record, split into a shareable record and a private verification-keys file. The verification keys hold only the per-commitment salts; they are not a copy of the matched data, which is persisted solely in the result file the operator chose to write. The CLI writes the two as separate files (`--record-file`, `--no-record`; the keys go to `<name>.keys.json` alongside the record); the web app offers them as two separate downloads (the shareable record and the private verification keys). Both are written owner-only, and the keys are kept private, not shared with the record. See `docs/spec/EXCHANGE_RECORD.md`.
- The web app now runs invitation-based, backend-free exchanges: "Create an invitation" mints a single-use deep link to deliver out-of-band, the partner opens it to review the terms and consent, and the two browsers rendezvous directly. A web exchange now always carries an invitation secret. See `docs/SECURITY_DESIGN.md` and `docs/COMMUNICATION.md`.
- The web app's "Advanced options" now opens a column-aware editor for the invitation's linkage terms, seeded from the file-derived defaults: the inviter can set its name and the invitation lifetime, choose who receives the matched results (both parties, only the inviter, or only the partner), attach an optional legal agreement, and toggle and reorder which linkage keys are used, with a live preview of exactly what the partner reviews and validation that blocks generating an invalid or unlinkable invitation. The advanced path needs the CSV before generating. The quick (name-only) path is unchanged, and the matching algorithm, deduplication, and fuzzy comparisons are shown read-only and not yet editable. See `docs/COMMUNICATION.md`.
- The web "Advanced options" editor adds an Expert authoring mode: build linkage keys element by element (each element's field chosen from your declared columns, never free-typed), edit a per-element transform pipeline with typed parameter inputs, and set a two-of-N "match in either order" swap from a key's own elements. It also adds a JSON/YAML import/export escape hatch -- export the current terms, or paste a document to load it, round-tripped through the same validation, with invalid input rejected by a readable error. The matching algorithm (psi-c), deduplication, and fuzzy comparisons are now shown as controls but stay disabled until the exchange applies them; importing terms that turn one on is refused. The guided (non-expert) path is unchanged. See `docs/COMMUNICATION.md`.
- The web accept screen shows the inviter's proposed linkage terms and requires explicit consent before connecting; the exchange then runs exactly those terms. Matching method and result-sharing stay always-visible, and the fields the keys match on are summarized always-visible (which data is matched on) above a default-collapsed "Matching strategies" disclosure; inside it each linkage key is its own further disclosure whose header is the key name and a short plain-language one-liner of the fields it matches on -- each field carrying a terse marker when its rule materially changes how it matches (a looseness word such as "partial" for a truncation or "fuzzy" for an approximate-match expansion, or a direct name such as "pattern filter" for a custom partner-authored rule) -- with the per-element transform/swap detail one expand down. That field one-liner is derived from the schema field types, not the partner's wording, so a key's name cannot misrepresent what it matches on. A default-collapsed "Other details" disclosure holds the data constraints, payload columns, and legal agreement. The same terms summary is shown on the shared exchange screen. See `docs/COMMUNICATION.md`.
- The web accept screen now surfaces an always-visible hint in that core, before the acceptor opens "Other details", when an invitation requests extra data columns from the acceptor (shown as a count) or attaches a legal agreement (a fixed flag) -- so neither fact stays hidden in the collapsed detail until expanded. The detail (the column list, the agreement text) stays in the disclosure. See `docs/COMMUNICATION.md`.
- The web accept screen also surfaces an always-visible hint in that core when an invitation will send the acceptor data columns for matched records (a count of the inbound columns), the ingress companion to the egress and legal-agreement hints. The three render together as one labelled "Before you consent" group that the "Other details" disclosure references for assistive technology, and the two column lines read direction-first ("Your partner requests N data columns from you" versus "You will receive N data columns from your partner") so the opposite flows are not confused. The column lists stay in the disclosure; a declared "receive nothing" lock-in raises no ingress hint, and the ingress hint is omitted from the inviter's own preview, which already lists its sent columns. See `docs/COMMUNICATION.md`.
- The web app now has a "Prepare your data" editor on both sides of an exchange: after choosing a file, the operator maps each column to a semantic type and one consequence-labeled disclosure choice (used to match, row identifier, sent to your partner, or ignored), sees a live count of how many of the agreed linkage keys the current mapping can satisfy, remaps columns until the file complies, and confirms exactly which columns will be sent before connecting. The acceptor reaches it where the old "this file cannot be linked" dead-end was (the zero-key block remains, but is now an editable entry rather than a bounce back to the file picker); the inviter edits the same grid under "Advanced options", where changing a column's type updates which matching keys are offered. These per-party settings are local -- never sent to the partner or embedded in the invitation -- so they change only your own match rate and disclosure, not the agreed terms. A column inferred as a row identifier is no longer sent to the partner by default. See `docs/COMMUNICATION.md`.
- The web quick (name-only) invite path now shows, once a file is chosen and before the link is generated, which columns it will send to the partner for matched rows, with a one-click path into Advanced options to change what is disclosed. The displayed set is derived from the same predicate the exchange transmits on, so it cannot drift from what is actually sent; when the quick path would send no columns it says so explicitly rather than going silent. This is awareness only -- the quick path's transmission is unchanged. See `docs/COMMUNICATION.md`.
- The web "Prepare your data" editor now has a per-field standardization workbench: the recommended per-type cleaning is shown as an editable, ordered list of steps (added from a plain-language menu grouped by intent, each step's parameters as typed inputs, not raw text), beside a live before-and-after preview over a sample of your rows that distinguishes a cleaned value, a dropped (empty) value, and a value that fans out into several match candidates. A cleaned value that does not meet a field's declared constraint is flagged with a warning, not blocked. The edited cleaning stays per-party and local. See `docs/COMMUNICATION.md`.
- `psilink exchange` (and the zero-setup form) now warns during data preparation when cleaned values fail a linkage field's declared constraints -- an excluded value, a name character outside the allowed set, an invalid date, or an SSN (or its last four) that breaks a Social Security Administration structural rule -- reporting a count per field rather than the values. It is the same warn-not-enforce check the web "Prepare your data" workbench shows as badges; the exchange still proceeds. See `docs/EXCHANGE_REFERENCE.md`.
- The web "Prepare your data" and "Advanced options" editors let the operator author and edit raw regular-expression cleaning steps (replace, extract, filter, split) for a field, on both sides of an exchange and without an opt-in switch. Each pattern is validated against the supported linear-time dialect and bounded to 1000 characters, and runs under a non-backtracking engine, so a pathological pattern bounds rather than freezing the tab. Each regex step is marked "advanced" and the live preview shows its effect, because a wrong pattern silently changes which of your own rows match; a malformed or incomplete step now blocks generating or continuing on both sides. These steps are per-party and local -- they never change the agreed terms. Raw patterns inside a partner-authored, token-embedded key transform stay read-only (a separate cross-party trust surface). See `docs/COMMUNICATION.md`.
- The web "Prepare your data" editor now lets the operator choose which of its columns feeds each field when more than one column has the field's type, so two linkage fields of one semantic type (e.g. a maiden and a current name) can bind to distinct columns instead of both defaulting to the first. With a single column of the type the binding stays fixed, as before. Per-party and local. See `docs/COMMUNICATION.md`.
- The web inviter's "Advanced options" editor now shows the same data-cleaning by default (no longer behind expert mode): the inviter can author per-field cleaning steps and choose which column feeds each field, and -- as an expert action -- add a second field of one semantic type bound to a distinct column (e.g. a maiden and a current name) and reference both in its linkage keys -- the invite-side counterpart of the acceptor's per-field binding, so two same-typed fields can match end to end from both sides. Exporting such a multi-field terms document and re-importing it reconstructs the second field's column binding against your own file -- choosing only among columns you marked for matching, so an imported (and thus partner-influenceable) document cannot quietly turn a row-identifier or sent-to-partner column into a match field -- so the multi-field capability survives the JSON/YAML import/export round-trip; a key referencing a field your columns cannot supply such a column for is imported turned off -- kept visible and flagged unlinkable, not dropped -- so the keys your columns can satisfy still generate, and an import where no key is satisfiable is refused with a message naming the unsupplyable field rather than the misleading prompt to enable a key (re-establish the binding in the expert workbench, or by marking the column used-to-match). The cleaning is per-party and local: never embedded in the invitation, never changing the agreed terms. See `docs/COMMUNICATION.md`.
- The web "Prepare your data" editor -- and the inviter's "Advanced options" cleaning, which gains the same defense -- now warns when a cleaning step silently collapses a field: for each field it computes, over your whole file (off the main thread for a large one), the share of rows that still produce a value after cleaning, and raises a prominent alarm when a transform has dropped every row to nothing -- a failure the sample preview cannot show and that otherwise looks identical to a genuine non-match. An empty cleaned value still counts as produced (a participating key, distinct from a dropped value), so an all-empty field is not mistaken for this collapse. It warns, it does not block. The editor (metadata grid, standardization preview, and step controls) also gained an accessibility pass: table semantics, a keyboard step-reorder with focus handling on removal, debounced screen-reader announcements, and non-color encoding of each outcome. See `docs/COMMUNICATION.md`.
- `zero-setup` exchange: derive all connection parameters from a URL, with no configuration file, for a one-off exchange. See `docs/CLI.md`.
- `filedrop` channel: exchange through a shared local directory where neither SFTP nor WebRTC is available. See `docs/COMMUNICATION.md`.
- `retain_files` mode (`sftp`/`filedrop`): keep every exchange file as a durable transcript instead of deleting it on consume (`--retain-files`). See `docs/EXCHANGE_REFERENCE.md`.
- Split inbound/outbound directories for the `sftp`/`filedrop` channels: configure `inbound_path`/`outbound_path` (instead of a single `path`/`server.path`) to read the peer's files from one directory and write your own to another, for managed shares or SFTP servers with separate drop and pickup folders. Requires retain mode. See `docs/EXCHANGE_REFERENCE.md`.
- `--outbound-path` configures that split directory from the command line, on the zero-setup, `invite`, `accept`, and `exchange` commands: the server URL or positional path (or, for `exchange`, the configured directory) is the inbound directory and `--outbound-path` is the outbound. Requires retain mode (`--retain-files`, or `retain_files: true` in an `exchange` config). See `docs/EXCHANGE_REFERENCE.md`.
- `--server-private-key-passphrase` supplies the passphrase for an encrypted SSH private key on the zero-setup, `invite`, `accept`, and `exchange` commands, so an encrypted key now works end-to-end on the flag-based flows -- previously the passphrase could be set only through a configuration file's `private_key_passphrase`. Like the sibling credential flags it takes an `@path` file reference (never the literal secret on the command line) and, on `--save`, is persisted as that reference rather than inlined into `psilink.yaml`. Passing it without `--server-private-key` is a usage error (exit 64). See `docs/EXCHANGE_REFERENCE.md`.
- `psilink invite --expires-in DURATION` sets the invitation lifetime (default 1 hour). See `docs/CLI.md`.
- `--log-file <path>` on every command that logs appends diagnostics to an owner-only file for unattended runs; a missing parent directory fails with exit 64. See `docs/CLI.md`.
- Every exchange now logs a one-line runtime banner at the start: the Node version and the memory ceilings the run operates under -- host memory, the V8 heap limit, and the container memory limit when a tighter cgroup limit is set -- so an out-of-memory failure can be read against the ceilings it ran under from the support log alone, without reproducing it. The line is written only to the operator's own log/console and is never sent to the partner. See `docs/SECURITY_DESIGN.md`.
- `--save` on a zero-setup exchange provisions a persistent shared secret in-band when both parties pass it, with no separate invite round; that secret is protected only by the transport. See `docs/SECURITY_DESIGN.md`.
- The `-` input positional reads the CSV from standard input for `exchange`, the zero-setup form, and `invite`; `accept` rejects `-` unless `--consent-to-terms` skips the confirmation prompt that otherwise claims stdin, so a piped `accept` without `--consent-to-terms` fails fast with guidance. See `docs/CLI.md`.
- `--sweep-exchange-files` (and `--force-retain-sweep`) clear protocol files left by a crashed prior run on the `sftp`/`filedrop` channels. See `docs/spec/FILE_SYNC.md`.
- `connection.options.unexpected_files` policy (`error`/`warn`/`ignore`) for a foreign file that appears in the shared directory during an exchange; defaults to `error`, or `warn` under `retain_files`/`lockless_rendezvous`. See `docs/EXCHANGE_REFERENCE.md`.
- `authentication.token_max_age_days`: cap the age of rotated tokens for dormant partnerships. A misspelled key in the `authentication` block is now rejected at parse time. See `docs/EXCHANGE_REFERENCE.md` and `docs/SECURITY_DESIGN.md`.
- Invitations may carry an optional credential-free connection endpoint (a public rendezvous locator); they never carry credentials. For a split-directory `sftp`/`filedrop` exchange the endpoint carries the inviter's `inbound_path`/`outbound_path` pair, which the acceptor mirror-swaps (one party's outbound is the other's inbound) so both sides can keep a fixed mount layout. See `docs/SECURITY_DESIGN.md` and `docs/spec/FILE_SYNC.md`.
- A `psilink:diagnostics` localStorage flag raises web connection-logging verbosity for one browser, without leaking the derived rendezvous ids. See `docs/DEPLOYMENT.md`.
- `psilink invite` (offline) prints how to abandon a pending invitation before it expires: delete its key file. See `docs/CLI.md`.
- Key file schema: a versioned JSON format for persisting the shared token and exchange metadata between sessions.
- Metadata inference (column semantic types and date formats), custom linkage keys via configurable transformations, and data standardization that canonicalizes linkage values by semantic type. See `docs/EXCHANGE_REFERENCE.md`.
- `zip_code` semantic type: a column named `zip`, `zip5`, `zip_5`, `zipcode`, or `zip_code` is inferred as a US ZIP code (canonical form 5 digits, zero-padded). Like `phone_number`/`email_address` it is matchable but in no default linkage key, so it is used for matching only when a key references it. See `docs/EXCHANGE_REFERENCE.md`.
- Metadata `role: ignored`: mark an input column present in your file but excluded from the exchange -- never linked, never an identifier, and never transmitted as payload (even if `is_payload: true`). Opt-in only; inference never assigns it. See `docs/EXCHANGE_REFERENCE.md`.
- The web "Advanced options" editor now authors the invitation's payload columns: the columns you set to "Sent to your partner" (under "Your columns") are listed in the invitation's data dictionary and shown on your partner's consent screen, so your partner sees what extra data it will receive for matched records before consenting. The set is derived from the same disclosure choices that govern transmission, so it cannot over-state what is sent. The columns you receive back are not enumerated in advance -- you receive whatever your partner discloses -- and the editor blocks generating an invitation that sends columns to a partner you have set to receive no result. See `docs/COMMUNICATION.md`.
- The web quick (name-only) invite path now authors the same payload data dictionary for the columns it sends (the columns the inferred metadata discloses -- unrecognized "other" columns and row-identifier columns), so the partner's consent screen lists them before consenting, just as the Advanced editor's does. The declared set is derived from the same predicate the exchange transmits on, so it cannot over- or under-state what is sent; nothing is declared when the quick path would send no columns. This declares what the quick path already transmits -- which columns leave the machine is unchanged. See `docs/COMMUNICATION.md`.

### Changed

- The CLI now launches Node with `--expose-gc` (its Docker entrypoint and the `apps/cli` dev script), which lets `single-pass` linkage release its transient allocation peak at the exchange's phase boundaries and lowers the single-pass receiver's peak resident memory (measured roughly 15-18% at the dataset sizes benched). It is a perceptible-memory and deployment change only -- no command, flag, output, wire format, or result is affected, and every other command is unchanged. The flag is consumed by Node and never reaches the CLI's arguments. The browser has no equivalent, so the web app does not get this relief and runs the unrelieved peak. This relief is the precondition for the raised single-pass dataset ceiling (below). See `docs/spec/PROTOCOL.md`.
- The `single-pass` dataset ceiling (`MAX_SINGLE_PASS_CELLS`) is raised from 2,000,000 to 3,000,000 (key, record) cells per party -- about 214k rows at the 14-key default template, or up to 3M rows at a single key -- re-derived from a fresh near-ceiling memory measurement of the real linkage with the phase-boundary relief above active, with comfortable headroom on a 16 GB host. After the relief the binding constraint is the WebRTC per-frame envelope rather than receiver memory, and the cap is held just below where a single-pass reply would exceed it, so a legitimate reply is never rejected mid-exchange on either transport; an exchange above the ceiling still aborts cleanly on both parties. This widens a denial-of-service bound (the maximum a single exchange will attempt to hold resident before aborting). See `docs/spec/PROTOCOL.md` and `docs/spec/CHANNEL_SECURITY.md`.
- The web app now honors the operating system's reduced-motion preference (`prefers-reduced-motion`): disclosure and overlay transitions, the disclosure chevron rotations, and the status progress bar's looping stripe are skipped for a user who asks for reduced motion. A user with no such preference sees the same animations as before.
- `single-pass` now carries datasets far larger than before in its single reply frame. The reply's accepted size is derived per exchange from the two parties' record counts and the agreed key count -- up to a fixed maximum dataset size set against the receiver's measured peak memory -- instead of a single fixed transport frame cap. An exchange above that ceiling now aborts cleanly on both parties with guidance to reduce the key or record count or split the dataset; the previous "switch to the cascade strategy" guidance is removed, since the strategy is a mandatory-consistency agreed term that cannot change mid-exchange. See `docs/EXCHANGE_REFERENCE.md`, `docs/spec/PROTOCOL.md`, and `docs/spec/CHANNEL_SECURITY.md`.
- The `canonicalize` library that implements the RFC 8785 canonical encoding behind signed receipts and exchange-record commitments is bumped to 3.0.0 (a major release; now ESM-only). The canonical-encoding output is byte-identical and no behavior changes; a security reviewer should note the canonical-encoding dependency advanced a major version. See `docs/spec/CANONICAL_ENCODING.md`.
- Both parties' record counts (raw row counts) now ride the terms-exchange messages rather than a separate two-message round-trip, on every exchange -- both-output and one-sided alike -- so PSI role resolution is computed locally with no dedicated count exchange. This removes one round-trip per exchange (proportionally meaningful on high-latency channels like `filedrop` or `sftp`, where each message handoff is a poll) and puts each party's exact row count on the terms envelope (kept out of the agreed-terms hash, as per-party role metadata). The both-output role assignment is unchanged (the smaller-row party becomes the receiver, the total-work-minimizing choice for both linkage strategies), and the counts still feed the single-pass frame cap. See `docs/spec/PROTOCOL.md`.
- The CLI exchange/accept pre-flight and the web "Prepare your data" editor now warn when a linkage key's own cleaning can never produce a value -- a `parse_date` whose `input_format` omits a component (e.g. `MM/DD`, no year) drops every record regardless of the data -- so a self-defeating key is surfaced before the run instead of yielding a result byte-indistinguishable from an empty intersection. The warning informs (the columns are present, so the column check still passes); the recommended setup is unaffected. See `docs/EXCHANGE_REFERENCE.md` and `docs/COMMUNICATION.md`.
- A column named `zip`, `zip5`, `zip_5`, `zipcode`, or `zip_code` is no longer sent to the partner as payload by default: it previously inferred as `other` (sent) and now infers as the recognized `zip_code` type (linkage, not sent). An operator relying on an inferred ZIP column reaching the partner must mark it `is_payload: true`. See `docs/EXCHANGE_REFERENCE.md`.
- An empty-string linkage key (`""`) is now a matchable key value distinct from a null/missing key, instead of being dropped from matching along with nulls; it participates subject to the existing within-dataset uniqueness rule (a `""` duplicated within a dataset is still dropped from that round). The default per-type standardization maps empty values to null, so default exchanges are unaffected; this matters only for operator-authored standardization that emits `""`. See `docs/spec/PROTOCOL.md`.
- An exchange configuration that authors its own standardization now fails closed when that standardization contradicts its linkage terms -- a transformation output naming no declared linkage field, or an unknown standardization function -- instead of logging a warning and running past the contradiction. The CLI reports it as a configuration error (exit 64), naming the offending output or function, both during data preparation for `psilink exchange` and at mint time for `psilink invite` (so an inconsistent configuration is refused before an invitation is disclosed, never after). A configuration whose standardization is consistent with its terms, and a terms-only configuration that authors no standardization (its cleaning is reconstructed from metadata and terms), are unaffected. See `docs/EXCHANGE_REFERENCE.md`.
- A `metadata` block with duplicate column names is now rejected at parse time; names must be unique (matched case-sensitively). See `docs/EXCHANGE_REFERENCE.md`.
- A `metadata` column name that is empty or longer than 256 characters is now rejected at parse time with a clear error, instead of parsing cleanly and surfacing only later as a downstream failure during the exchange. Ordinary column names are unaffected. See `docs/EXCHANGE_REFERENCE.md`.
- An empty (zero-length) column name in an input CSV -- from a trailing comma, a blank cell, or a leading delimiter in the header row -- is now rejected at intake (when metadata is inferred, not authored) with a clear error, instead of being disclosed and then silently dropping this party's exchange-record audit. The CLI fails such a file with exit 64 during data preparation; the web app rejects it at its file-intake surfaces (the quick and Advanced invite paths and the acceptor's file step), where the quick path previously dead-ended in a generic "please try again" message. Ordinary headers are unaffected. See `docs/EXCHANGE_REFERENCE.md`.
- BREAKING: the CLI authenticates recurring exchanges with the X25519 authenticated key exchange instead of SPAKE2, and the shared credential is renamed `pake_token` -> `shared_secret` in config. Old and new builds do not interoperate; pre-release, no migration shim. `exchange` now requires a `.psilink.key`. See `docs/spec/PROTOCOL.md`.
- BREAKING: the `authentication` block moves from per-channel `connection` config to the top level of the exchange spec, and the WebRTC role moves to `connection.role`. Authentication, rotation, and expiry behavior are unchanged. See `docs/EXCHANGE_REFERENCE.md`.
- BREAKING: the multi-word semantic-type and fuzzy-comparison enum values in config and invitations are now `snake_case` (`first_name`, `last_name`, `date_of_birth`, `phone_number`, `email_address`; `edit_distances`, `adjacent_years`), matching the convention for everything else users write; the old camelCase spellings are rejected. Single-word values (`ssn`, `ssn4`, `identifier`, `other`, `transpositions`) are unchanged. These ride the invitation token, so old and new builds do not interoperate; pre-release, no migration shim. See `docs/EXCHANGE_REFERENCE.md`.
- BREAKING: PSI frames are carried as raw binary on the wire and on disk instead of base64url-in-JSON. The application-layer AEAD envelope is now a self-describing binary frame (`version || IV || ciphertext || tag`) and the file-sync message body a binary envelope (`version || type || seq || payload`), both carrying the frame verbatim with no base64 expansion -- so a frame's on-wire / on-disk size is roughly its raw size, and the frame is no longer capped at Node's maximum string length. Old base64-in-JSON frames are not readable by the new build; this is a clean break, safe because both parties run the same version-coupled release, and the leading version byte turns a format mismatch into a clean error rather than a silent misparse. A security reviewer should note this channel-security wire/on-disk format change. See `docs/spec/CHANNEL_SECURITY.md` and `docs/spec/FILE_SYNC.md`.
- BREAKING: the duration flags `--accept-timeout`, `--connection-timeout`, and `--peer-timeout` now require a unit suffix (`30s`, `2h`); a bare integer is rejected. Update any scripts that pass bare seconds. See `docs/CLI.md`.
- BREAKING (source builds): a source install now requires Node 26, enforced at install rather than advisory. The published Docker image and web deploy are unaffected.
- Renamed the default file-sync rendezvous mode "wave" -> "lock" and its on-disk tiebreaker file; old and new builds do not interoperate at rendezvous, so both ends must run the same build.
- Exit codes: scripts that branch on 64 vs 69 should be reviewed. A class of configuration and usage failures that previously exited 69 now exits 64 (EX_USAGE) -- a dirty exchange directory, concurrent sessions on one path, a send timeout, a malformed or missing config or key file, an unsupported channel or URL scheme, or an invalid connection-option combination -- while transport failures still exit 69. See `docs/CLI.md`.
- The timeout flags now reject a value above a 7-day ceiling with a usage error (exit 64). See `docs/CLI.md`.
- `max_reconnect_attempts` (`--max-reconnect-attempts`) now rejects a value above a sanity ceiling (one retry per second for 7 days) with a usage error (exit 64), instead of accepting an arbitrarily large count that against a fast-failing endpoint would hang the command. Ordinary values, including the default of 3, are unaffected. See `docs/CLI.md` and `docs/spec/CHANNEL_SECURITY.md`.
- A clean file-sync close against a departed peer now fails fast (bounded to about a minute) and logs the wait at default verbosity, instead of blocking up to the full peer-timeout. See `docs/spec/FILE_SYNC.md`.
- Offline `psilink invite` can take its linkage terms from an existing config instead of an input CSV. See `docs/CLI.md`.
- Online `psilink invite` states up front that the printed invitation is acceptable only while the command is waiting. See `docs/CLI.md`.
- CLI diagnostic log output (info and debug included) now goes to stderr, reserving stdout for a command's result data (an exchange's CSV output, the `invite` token, the `fingerprint` value), so a piped or redirected result is no longer corrupted by interleaved diagnostics; a script that read diagnostics off stdout must now read them from stderr. `--log-file` captures every diagnostic, including the data-cleaning and file-handling warnings that previously leaked to the terminal. See `docs/CLI.md`.
- `psilink fingerprint` now prints only the bare fingerprint value to stdout; its action banner, the bound identity, the `--force` regeneration warning, and the out-of-band sharing instructions now go to stderr (or `--log-file`), so `FP=$(psilink fingerprint)` captures a clean value with no banner or instruction text mixed in. Previously stdout carried the whole summary, including the value as a labeled `Fingerprint: <value>` line amid that prose; a script that parsed the old stdout must now take the bare value from stdout and read the surrounding prose from stderr. See `docs/CLI.md`.
- Terminal transport and directory errors (an over-cap frame, a contaminated directory, a stalled server) now end with a concrete next step and suppress the contradictory generic "retry without re-inviting" advisory.
- The CLI and the web accept screen render an invitation or config validation failure as a readable one-liner instead of a raw schema-error dump.
- The web exchange screen shows a fixed, friendly, retry-oriented message when a generic transport failure (a dropped or failed connection) ends an exchange, instead of the raw error text; the detailed error stays in the browser console for diagnosis.
- Documentation is reorganized into `docs/` (overview) and `docs/spec/` (wire formats, byte encodings, constants); some files moved or were renamed, so links to old paths break. See `docs/spec/README.md`.
- The web app now accepts CSV uploads up to 100 MB (was 10 MB), and a file larger than one parser chunk parses to its complete row set instead of risking a silently truncated subset. See `docs/spec/PROTOCOL.md`.
- The web app again parses a large CSV (above a 4 MB threshold) off the main thread, in a Web Worker it bundles itself, so the tab stays interactive while a large invite/accept file is parsed instead of freezing for the whole parse; a typical smaller file still parses inline. The worker streams the parsed rows back in batches, so even a near-100 MB file stays responsive on receipt rather than paying one uninterruptible main-thread step for the whole result. This restores the off-main-thread parse that was dropped when PapaParse's self-hosted worker had to be removed for corrupting the parse under a production build. Parse results and error reporting are unchanged. See `docs/spec/PROTOCOL.md`.

### Removed

- The web app's server-side session backend (`/api/psi`) and its secret-less (zero-setup) web exchange are removed; a web exchange now always carries an invitation secret. The PeerJS signaling server (the transport) is retained.

- Linkage values are now normalized to Unicode NFC before matching, and again after any case-folding step, so the same value in different normalization forms -- the common macOS-NFC vs database-NFD split -- matches instead of silently failing. See `docs/EXCHANGE_REFERENCE.md`.
- Fixed a deadlock in the multi-key matching loop where one party could skip a linkage key the partner still ran, hanging the exchange; it surfaced under one-to-one and many-to-one linkages when the parties' matched sets diverged.
- An exchange now fails with a clear error (bounded by `peer_timeout_ms`) when the partner stops responding, instead of hanging; the timeout message names the likely peer-side causes.
- On `sftp` and `filedrop`, a peer that aborts mid-exchange is signaled so the waiting side fails fast instead of waiting out `--peer-timeout`. See `docs/spec/FILE_SYNC.md`.
- A `filedrop`/local-filesystem connect that times out is now terminal, so a stalled NFS or CIFS mount no longer ties up the process across retries.
- File-sync rendezvous recovers from a peer that fails mid-arrival, and sweeps an orphaned temp file left by a hard-killed prior run, instead of stalling until the peer timeout or aborting the next run. See `docs/spec/FILE_SYNC.md`.

- A `peer_timeout_ms` or `server_connect_timeout_ms` of `0` is now rejected at parse time rather than silently disabling the timeout. `max_reconnect_attempts` still accepts `0`.
- An unset `server_connect_timeout_ms` now applies the documented 30000 ms per-attempt connect deadline on both `sftp` and `filedrop`, instead of letting an `sftp` connect fall back to ssh2's shorter internal default. See `docs/EXCHANGE_REFERENCE.md`.
- The SFTP adapter retries a transient `rename` failure instead of aborting on an intermittent server error. See `docs/spec/FILE_SYNC.md`.
- SFTP exchanges are faster: the client disables Nagle's algorithm on its socket, removing per-round-trip delayed-ACK stalls.
- The SFTP adapter no longer prints ssh2-sftp-client's default `Global ... listener` lines straight to the console (visible during first-use host-key trust setup); the underlying ssh2 client's out-of-band error/end/close events now route through the project logger and respect `--log-level`.
- The CLI percent-decodes a server URL's host, path, and credentials, so `sftp://user@host/my%20drop` targets `my drop`; a malformed percent-escape is a usage error (exit 64). See `docs/CLI.md`.
- `connection.provider_options` keys are passed to the transport library verbatim (for example ssh2's `readyTimeout`) instead of being case-rewritten. See `docs/EXCHANGE_REFERENCE.md`.
- `@path` references resolve only in credential and opaque-options fields; a free-text field beginning with a literal `@` is kept verbatim. See `docs/EXCHANGE_REFERENCE.md`.
- The key-file pre-flight now rejects a parent directory that is writable but not readable, so a key write cannot fail after the handshake has already rotated the token. See `docs/SECURITY_DESIGN.md`.
- Repeating a single-value flag (for example `--server-port`) is now a clean usage error (exit 64) instead of misbehaving. See `docs/CLI.md`.
- A malformed message from a peer is reported as a protocol error instead of crashing, and a failed exchange surfaces its original cause instead of a generic connection error.
- The acceptor at accept time, and a recurring `psilink exchange` run, are warned -- or the run is blocked (exit 64) -- when the CSV cannot satisfy the configured linkage terms, instead of silently producing an empty result. See `docs/CLI.md`.
- `psilink invite` now reconciles an input CSV against a config's explicit `metadata`, resolving columns to linkage fields exactly as the exchange will, instead of falling back to column-name inference. A config that types a non-standard column explicitly is no longer wrongly rejected at invite time, and a CSV the config's metadata cannot satisfy is now caught when the invitation is generated rather than only at the exchange. See `docs/CLI.md`.
- Linkage terms that pair `output.expects_output: false` with `payload.receive` columns are rejected as incoherent at parse time. See `docs/EXCHANGE_REFERENCE.md`.
- A `payload.send` entry that names a column its input metadata does not actually transmit (`is_payload: false`, `role: ignored`, or no such column) is now rejected when the invitation is generated and again at exchange time (exit 64), instead of producing a partner data dictionary, consent screen, and exchange record that over-state what is sent. The mismatch only ever over-declares, so no value is disclosed beyond the dictionary. See `docs/EXCHANGE_REFERENCE.md`.
- The self-attested exchange record's `payloadSent`/`payloadReceived` now list the columns actually committed for matched records -- read from the same metadata disclosure gate that transmits them -- rather than from the optional `payload.send`/`receive` data dictionary, so a record can no longer state that no columns were sent or received while real columns flowed (the case when no dictionary is authored, as on the web path). The cryptographic commitments are unchanged; this corrects only the readable disclosure description. See `docs/spec/EXCHANGE_RECORD.md`.
- Linkage terms that reference an undeclared linkage field, or whose `swap` names no element in its key, are rejected as incoherent at decode, instead of silently collapsing the affected key to an empty result. See `docs/EXCHANGE_REFERENCE.md`.
- The web inviter's "Advanced options" import now refuses a pasted linkage-terms document whose per-field constraints (a custom `exclude` denylist, `validOnly`, `allowedCharacters`, or `affixesAllowed`) the editor cannot represent, instead of silently rebuilding them to the type defaults and generating an invitation under a different cross-party agreement than the document declared. A document carrying only the default constraints imports unchanged, and the acceptor's verbatim adoption of the inviter's terms is unaffected. See `docs/COMMUNICATION.md`.
- The web inviter's "Advanced options" import now re-generates the rest of an imported linkage-terms document faithfully: the linkage-field order and any declared-but-unreferenced linkage field (with its constraints) are preserved rather than silently reordered to a fixed canonical order or dropped, and a benign empty `constraints: {}` on a field whose type has no default constraint is kept rather than refused -- so the regenerated invitation commits the same cross-party agreement the imported document declared. A document carrying a custom constraint on a field a key uses is still refused, and the acceptor's verbatim adoption of the inviter's terms is unaffected. See `docs/COMMUNICATION.md`.
- The web app parses a dropped CSV in a Web Worker, so a large (near-10MB) file no longer freezes the interface -- input and painting stay responsive -- while it is read.
- The invitation consent displays now show the inviter's requested-from-you payload columns consistently on both the web and the CLI, with a declared-empty `payload.receive` rendered "(none) -- any payload column would abort the exchange" (a strict "the partner sends nothing" request, naming the consequence on both the send and receive sides so it does not read as innocuous) rather than hidden: previously the web showed only a non-empty request and the CLI showed this direction not at all, so a declared-empty request -- or, on the CLI, any request -- was invisible and a declared-empty receive read identically to an unstated (lazy) one that has the opposite runtime meaning. This reaches a real cross-tool flow (a hand-authored CLI invitation that declares `payload.receive`). The cross-party compatibility check's mismatch message for an empty receive is self-explaining now too. The empty-vs-absent behavior itself is unchanged (an empty receive was already strict). See `docs/EXCHANGE_REFERENCE.md`.

### Security

- Redirecting a command's matched-records result to a file with a shell `>` (e.g. `psilink exchange data.csv > results.csv`) now surfaces a notice on stderr, kept visible at `--log-level error`: the shell creates that file under its umask, not the owner-only permissions an OUTPUT_FILE path argument gets, so on a shared host the matched records can land group/world-readable. The notice names the exposure and points at the OUTPUT_FILE-path alternative; a TTY, a pipe, and `/dev/null` do not fire, and passing an OUTPUT_FILE path is unchanged (already owner-only, no notice). The check sees only this process's stdout, so a redirect applied outside it (for example on the host across a container boundary) is not detected. See `docs/SECURITY_DESIGN.md`.
- Matching participation now requires `role: linkage`: a column roled `identifier`, `payload`, or `ignored` is no longer hashed into a linkage key merely because its semantic `type` matches a linkage field -- on every linkage path (the dataset builder, the satisfiability pre-flight, and the from-scratch authoring defaults), and an explicit data standardizing transformation naming such a column no longer binds it (the role wins). This closes the overlap where a column the operator marked sent-to-partner, or as a local row identifier, could also be used as a cross-party PSI match key without an explicit matching designation. Transmission is unchanged and stays a separate axis (`is_payload`); a column that should both match and be sent is `role: linkage` with `is_payload: true`. A field left with no `role: linkage` column of its type is reported unsatisfiable before the exchange runs (fail-closed), never silently dropped. The default quick path is unaffected -- inference already roles matchable columns `linkage`. See `docs/spec/PROTOCOL.md` and `docs/EXCHANGE_REFERENCE.md`.
- The web app sets defense-in-depth HTTP response headers on its rendered pages and API routes: `Referrer-Policy: no-referrer` (keeping the fragment-borne invitation token out of `Referer` on older clients too), `X-Frame-Options: DENY` with a Content-Security-Policy `frame-ancestors 'none'` to deny framing (clickjacking), and `X-Content-Type-Options: nosniff`. No active leak is being fixed -- modern browsers already withhold the URL fragment from `Referer`. See `docs/SECURITY_DESIGN.md`.
- One-sided output sharing is now enforced end-to-end: when the agreed terms entitle only one party to the matched result, the exchange returns and writes that result solely to the entitled party, in both the web app and the CLI. The other party contributes its records to find the match but is shown that it receives no result table, rather than (as before) the result being written to both parties regardless of the agreed output terms. The web acceptor now derives its output terms as the mirror of the inviter's (so a one-sided invitation agrees rather than aborting), and the web Advanced-options editor exposes the three-way "who receives the matched results" choice. This enforces what the tool writes and returns, consistent with the exchange-record gating; an honest-but-curious helper can still observe its own match count during the protocol, so this is not a confidentiality guarantee against the helper. See `docs/spec/EXCHANGE_RECORD.md` (the same entitlement gate and helper-observability limit) and `docs/COMMUNICATION.md`.
- Cross-party payload declarations are reconciled lazily, so an invitation's payload can be authored without the inviter knowing the partner's columns: a party that does not declare which columns it expects to receive accepts whatever the partner's own metadata discloses ("take whatever I am given"), while a party that does declare a receive list still requires the partner's send to match it exactly or the exchange aborts before any data moves. A web acceptor derives its payload terms as the mirror of the inviter's, so an asymmetric payload agrees rather than aborting. This relaxes only the declaration cross-check, never what is disclosed -- each party's own input metadata still governs what it sends, so a party only ever receives what the partner's consented metadata transmits, and receiving is not a disclosure. See `docs/EXCHANGE_REFERENCE.md`.
- The invitation now carries the inviter's disclosed-columns subset directly -- the exact set of columns the acceptor will receive for matched records -- so the accepting party's consent screen shows what it will receive even for an invitation that authored no payload data dictionary (a CLI or quick-path invitation), and the displayed set is derived from the same predicate that governs transmission rather than a separately authored list, so it cannot drift from what is actually sent. Only that consent-relevant subset travels; the inviter's other columns do not. The acceptor locks the consented set in -- both an online and an offline accept persist it to the saved config so a later `psilink exchange` enforces it, and an online accept additionally enforces it in memory for its own single run -- and if the payload it later receives carries a different set of columns, the exchange aborts as a protocol error (the partner promised one disclosure and delivered another); an empty consented set is enforced strictly ("receive nothing"). A recurring exchange enforces the same against each side's persisted lock-in; a first-contact exchange with nothing locked in -- a zero-setup run, or the inviter's own open receive side on its first exchange -- reconciles lazily and enforces nothing (a party that then saves a config crystallizes what it observed; see the observe-then-persist entry below). See `docs/EXCHANGE_REFERENCE.md` and `docs/spec/FILE_SYNC.md`.
- A party that learns its received-payload columns only by observing a first exchange now persists that observed set when it saves a config for recurring use, so the recurring exchange fails closed on a later mismatch just as an up-front-locked acceptor does. The two savers are the online inviter (whose received set is whatever the acceptor transmits, unknown until the first run) and a zero-setup party run with `--save`; each records the columns it observed into the saved config's `expected_payload_columns` (distinct from the boolean output flag and from `payload.receive`). A first-contact or no-save exchange still persists nothing and stays lazy, and an observed-empty payload -- an ambiguous zero-match run -- is left lazy rather than frozen to a strict "receive nothing" that would false-abort a later matching run. See `docs/EXCHANGE_REFERENCE.md` and `docs/spec/FILE_SYNC.md`.
- A `payload.send` data dictionary that omits a column its input metadata does transmit is now rejected (exit 64), alongside the existing rejection of one that names a column metadata does not transmit: a present dictionary must name exactly the disclosed set. This keeps the dictionary -- exchanged with the partner, shown for consent, recorded, and mirrored into a recurring partner's received-column lock-in -- from understating what is actually sent, which would otherwise abort an honest recurring exchange when the omitted column arrives. An absent or empty dictionary is unaffected (the guided and default paths author none, and reconcile lazily); the web invite paths derive the dictionary from the disclosure predicate, so they already match. See `docs/EXCHANGE_REFERENCE.md`.
- A party entitled to no result in a one-sided exchange now receives none of the partner's payload columns. The exchange transmits payload only to a partner whose agreed terms entitle it to the result, and a non-receiving party's receive side fails closed if it is sent any. This closes a residual disclosure in which a non-receiving helper -- which contributes its records to compute the match but gets no result -- nonetheless received the receiver's disclosed payload values. The existing rule that a no-output party may not declare `payload.receive` columns is now also enforced at runtime, not only at parse time. See `docs/EXCHANGE_REFERENCE.md`, `docs/spec/FILE_SYNC.md`, and `docs/notes/one-sided-disclosure.md`.
- A malformed operator config (`psilink.yaml`) or credential file (`.psilink.key`, the signing identity) no longer echoes its own contents into CLI error messages or stderr, so an inline SFTP credential, the shared secret, or a private key cannot leak from a 0600 file into logs or bug reports. Parse errors report the file path only. As a backstop, displayed errors also redact any embedded PEM/OpenSSH private-key block. See `docs/SECURITY_DESIGN.md`.
- Recurring CLI exchanges (`sftp`/`filedrop`) are now encrypted end-to-end on the wire with an application-layer AEAD, so a directory or server administrator sees only ciphertext. See `docs/spec/CHANNEL_SECURITY.md`.
- Web exchanges run the X25519 authenticated key exchange before any PSI data, so each party authenticates the peer and an expired or untrusted credential fails closed; a web exchange is single-use and persists no key file. Transport confidentiality on the web path is provided by DTLS -- the application-layer AEAD is not yet applied there, so the session key currently serves only to authenticate. See `docs/SECURITY_DESIGN.md`.
- Invitation tokens carry a bounded lifetime (default 1 hour, now including web-generated ones); rotation tokens carry no expiry unless `token_max_age_days` is set. See `docs/SECURITY_DESIGN.md`.
- The shared token is rotated after every successful handshake. See `docs/SECURITY_DESIGN.md`.
- The file-sync transport bounds untrusted input from a hostile SFTP or file-drop server, closing memory-exhaustion and hang denial-of-service: a roughly 512 MiB inbound frame cap, an 8192-entry / 255-character directory-listing cap, and per-operation read/write liveness deadlines of about a minute. See `docs/spec/CHANNEL_SECURITY.md`.
- A malformed SFTP packet from a hostile server can no longer crash the CLI; it becomes an orderly typed failure with normal cleanup. See `docs/spec/CHANNEL_SECURITY.md`.
- SFTP host-key verification (fail-closed): every SFTP connection verifies the presented host key before authenticating and aborts on a mismatch. Pin it out-of-band via `connection.server.host_key_fingerprint` -- a single fingerprint, or a list to stage a rotated host key alongside the current one for zero-downtime rotation (either is accepted during the rekey window; drop the old entry after the cutover) -- or let the first interactive run establish it on first use: any command that opens the SFTP connection (`exchange`, an online `invite`/`accept`, or a zero-setup exchange) shows the presented fingerprint, prompts for confirmation, and records it (a zero-setup run persists it only with `--save`); later runs verify it silently. A connection with no pin is refused, and a non-interactive (automated) run with no pin fails closed rather than prompting or trusting the key. A key matching none of the pinned fingerprints is never auto-accepted: verify the new fingerprint out-of-band and re-pin deliberately. The unsupported `certificate` and `known_hosts` fields are rejected at parse time. CLI `sftp` only. See `docs/CLI.md` and `docs/SECURITY_DESIGN.md`.
- Cross-party SFTP host-key reconciliation: in a recurring (authenticated) exchange the two parties now compare the host-key fingerprints they each observed, advertised inside the authenticated post-handshake exchange. A divergence -- which a purely local pin cannot catch -- raises a warning naming both observed values, flagging a possible one-sided interception or a server rekey between the two parties' setups for the operators to disambiguate out-of-band; it warns rather than aborts. A party that observes no host key (a file-drop or proxy path) reconciles to no false alarm. See `docs/spec/CHANNEL_SECURITY.md`.
- Untrusted invitation-token fields are bounded at decode (a 64 KiB cap plus per-field limits), closing a denial-of-service and an SSRF-shaped WebRTC-host vector from a crafted but checksum-valid token. See `docs/SECURITY_DESIGN.md`.
- Sensitive files (key file, config, exchange records, signing identity) are written owner-only, flushed to disk for crash durability, and created on a symlink-safe descriptor on POSIX. See `docs/SECURITY_DESIGN.md`.
- The linkage-result CSV is now written owner-only (mode 0600 on Unix, ACL-narrowed on Windows), so the tool's most sensitive output is no longer left world- or group-readable by the umask. See `docs/SECURITY_DESIGN.md`.
- Operator-facing logs, errors, and web console output escape untrusted partner- and server-controlled strings (filenames, rendezvous paths, identities, error messages), neutralizing terminal, bidi-override, zero-width, and homoglyph injection; values used to build paths or to compare are unaffected. See `docs/SECURITY_DESIGN.md`.
- A credential supplied as an `@path` reference is persisted as the reference, never the resolved secret. Loading a config you did not author runs its `@path` references as your own credentials -- a credential-exfiltration vector -- so never run an untrusted config. See `docs/CLI.md` and `docs/SECURITY_DESIGN.md`.
- `connection.provider_options` is applied through a default-deny allowlist, so it cannot override the security-critical connect options (host, credentials, host-key verification). See `docs/EXCHANGE_REFERENCE.md`.
- A peer sending messages out of turn is rejected rather than buffered without limit.
- Untrusted partner-supplied collections in the linkage terms and post-handshake messages are count-bounded, so a counterparty that packs a collection with millions of invalid elements into an otherwise-in-cap frame fails the exchange with a clean protocol error instead of an ungraceful internal parser error (a Zod `RangeError`) or a multi-second CPU burn. See `docs/spec/CHANNEL_SECURITY.md`.
- The PSI decode path now scans every partner-supplied setup/request/response frame and rejects it before deserialization if it declares more encrypted elements than allowed, on both the cascade and single-pass strategies and both transports. This closes a memory-exhaustion denial of service: the protobuf deserializer allocates ~200 bytes per declared element, so a counterparty packing an otherwise-in-cap frame with many minimal repeated entries could drive tens of GiB of allocation inside the deserializer itself; the pre-scan bounds the declared count by the smaller of the authenticated key*record count and a fixed absolute ceiling (no legitimate frame can exceed it), failing the exchange with a clean protocol error instead. Each decoded record count also carries an explicit upper bound at decode, keeping the single-pass cell-count gate's arithmetic exact. See `docs/spec/CHANNEL_SECURITY.md` and `docs/spec/PROTOCOL.md`.
- A partner's payload column names are length-bounded on the wire, and independently in the exchange record, so a malicious or malformed partner cannot write an arbitrarily long column name into this party's local exchange record. See `docs/spec/CHANNEL_SECURITY.md`.
- A partner payload column name is now also required non-empty on the wire (and independently in the exchange record), so a malicious partner cannot suppress this party's exchange record by sending an empty (`""`) column name -- which would otherwise reach the non-fatal record-build guard and skip the audit record while the exchange still completed and disclosed. The floor is safe to enforce now that honest parties no longer emit empty names (rejected at intake, above). See `docs/spec/CHANNEL_SECURITY.md`.
- A crafted invitation whose linkage-terms `transform.params` is nested thousands of levels deep is now rejected with a clean usage error (exit 64) when an `accept` reconciles it against an existing local config, instead of an ungraceful internal error (a stack overflow in the comparison walk, previously surfaced as a generic exit 69). Real invitations are unaffected. See `docs/spec/CHANNEL_SECURITY.md`.
- The web app's bundled PeerJS signaling server caps inbound WebSocket frame size, so a single oversized frame from an unauthenticated client is rejected before parsing, and can no longer crash the rendezvous process (which would drop signaling for every connected peer, not just one session) or pin a large block of its memory. See `docs/spec/CHANNEL_SECURITY.md`.
- The web app's bundled signaling server further hardens that upgrade surface against unauthenticated resource exhaustion: a slow, partial, or idle upgrade handshake is closed server-side, a client that registers but never proves liveness is reaped within seconds instead of holding a slot for 90, the relay's reconnect queues are bounded, and a connection's id, token, and key are length-bounded at the upgrade handshake so a self-chosen id cannot inflate relayed frames or the server's standing memory. These are defense-in-depth -- the server is untrusted by design and relays only opaque setup messages -- while Origin restriction and per-address rate limiting remain the reverse proxy's responsibility. See `docs/DEPLOYMENT.md` and `docs/spec/CHANNEL_SECURITY.md`.
- The web peer-to-peer WebRTC data channel now bounds inbound frames so a hostile or buggy peer can no longer drive the receiving browser tab toward memory exhaustion: an oversized frame (fixed 256 MiB wire cap), a frame that would deserialize to a disproportionately large structure, a flood of tiny or never-completed chunk reassemblies all fail the exchange with a terminal error rather than allocating proportional to what the peer sends. This is the WebRTC transport's own control, since core's application-layer AEAD frame-size cap does not reach this path (it runs under DTLS and declines the AEAD wrap). See `docs/spec/CHANNEL_SECURITY.md`.
- The bundled AWS Elastic Beanstalk nginx reference (`apps/web/deploy/aws_eb/`) now requires TLS 1.2+ (dropping the deprecated TLS 1.0/1.1), sets an explicit forward-secrecy / AEAD-only `ssl_ciphers` list so the TLS 1.2 handshake no longer falls back to the platform default suites (which still permit CBC-mode SHA1 and non-forward-secret plain-RSA key exchange), and applies per-address rate/connection limits on the `/api/peerjs` signaling path by default, with the Origin allowlist and an HSTS (`Strict-Transport-Security`) header shipped as opt-in templates, so a deployment from the reference no longer inherits the documented no-proxy resource-exhaustion residual or a weak default TLS suite list. The limit numbers are illustrative starting points to tune to your load. See `docs/DEPLOYMENT.md`.
- The same AWS Elastic Beanstalk nginx reference now disables TLS session resumption by default (`ssl_session_tickets off`, `ssl_session_cache off`), so every session is a full ECDHE handshake and no resumed session rests on the single, never-rotated per-worker ticket key whose later disclosure would let a passive observer decrypt captured resumed sessions, undercutting the forward secrecy the ECDHE-only cipher list gives full handshakes. Re-enable resumption deliberately (tickets with rotated, cross-instance-shared key file(s), plus a cache) if you need it. See `docs/DEPLOYMENT.md`.
- The same AWS Elastic Beanstalk nginx reference now pins the ECDHE curve list explicitly (`ssl_ecdh_curve X25519:prime256v1`), active by default, so curve selection no longer rests on the silent OpenSSL/platform default -- a choice made marginally more load-bearing now that resumption is off and every connection is a full ECDHE handshake; and it ships an OCSP-stapling opt-in template (commented, with the supporting `resolver` and `ssl_trusted_certificate` directives and inline notes), inert by default because stapling cannot work on the test/self-signed certificate the reference is commonly run with. See `docs/DEPLOYMENT.md`.
- Partner-authored regular expressions in linkage-term element transforms (`replace_regex`, `extract_regex`, `filter_regex`, `split_on`, and the pattern `parse_date` builds from its `input_format`) now execute under a linear-time engine (RE2 semantics, the `re2js` dependency) instead of the JavaScript `RegExp` engine, so a crafted catastrophic-backtracking (ReDoS) pattern can no longer hang the acceptor's browser tab or CLI process -- the blow-up is impossible by construction. Patterns must conform to a restricted dialect (RE2 syntax: no backreferences or lookarounds, and `\x{...}` rather than `\uHHHH`); a pattern outside it is rejected at terms validation before any row runs, with no fallback to the backtracking engine. One behavior note: a pattern that previously backtracked (e.g. `(a+)+`) now simply runs in linear time. This replaces the earlier best-effort static ReDoS screen (and removes the `redos-detector` dependency). See `docs/spec/PROTOCOL.md` for the dialect, and `docs/SECURITY_DESIGN.md`, `docs/spec/CHANNEL_SECURITY.md`, and `docs/EXCHANGE_REFERENCE.md`.
- An over-large `length` on a partner's `pad_left` linkage-term element transform is rejected at terms validation, before any per-row key is built, so a crafted value (such as `1e9`) cannot exhaust the acceptor's memory by allocating a giant padded string for every row -- the memory-allocation sibling of the transform-regex linear-time dialect above. A legitimate left-pad (zero-filling a short identifier) is unaffected. See `docs/SECURITY_DESIGN.md`, `docs/spec/CHANNEL_SECURITY.md`, and `docs/EXCHANGE_REFERENCE.md`.
- A partner's `parse_date` linkage-term element transform has its `input_format`/`output_format` length-capped (256 characters each), bounding the size of the regex it compiles and of the output it assembles per matched row; the raw regex patterns above are likewise length-capped (1000 characters) to bound their compile cost. The shape risk of `parse_date`'s expanded regex (many adjacent variable-width groups from `MM`/`DD` tokens, which would backtrack on the old engine) is closed by running it on the linear-time engine above rather than by a separate screen. Ordinary date layouts (`MM/DD/YYYY`, `YYYY-MM-DD`, `YYYYMMDD`) are unaffected. See `docs/SECURITY_DESIGN.md`, `docs/spec/CHANNEL_SECURITY.md`, and `docs/EXCHANGE_REFERENCE.md`.
- An invitation whose linkage-terms `transform.params` carries `snake_case` parameter keys (which the config and wire paths normalize to `camelCase`, but the invitation decoder left verbatim) is now normalized to `camelCase` at decode, the same as every other path. This closes a class of latent failures with such a token: the per-step length screens being bypassed (a `snake_case` `input_format` evaded the format-length cap, then took effect once camelized downstream), a cross-party agreement-hash divergence in the exchange records, the standardization runtime silently ignoring the parameters (degrading or emptying the match), and a false `linkage_keys` conflict on `psilink accept`. The decoder also now accepts `snake_case` structural linkage-terms keys, matching how a hand-authored config is read. Invitations produced by psilink are unaffected (they are already `camelCase`). See `docs/spec/CANONICAL_ENCODING.md` and `docs/spec/CHANNEL_SECURITY.md`.
