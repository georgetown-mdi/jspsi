# Changelog

All notable changes to PSI-Link are documented here. The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). PSI-Link uses [semantic versioning](https://semver.org/).

## [Unreleased]

### Added

- The web acceptor now runs the inviter's linkage terms instead of a default inferred from its own CSV, closing the gap between the terms shown on the accept screen and the terms the exchange actually runs. Previously the acceptor's exchange called `prepareForExchange` with an empty spec, so it built its own default linkage terms from its uploaded CSV columns -- the acceptor consented to the inviter's proposed terms on the consent screen but then linked on a different, locally-inferred set. The accept route now threads the decoded invitation's `linkageTerms` (the same terms `summarizeInvitation` renders for consent) through to the acceptor's `Exchange`, which adopts them verbatim as the terms governing the PSI run. Only the linkage terms are adopted: metadata, standardization, and payloads still derive from the acceptor's own CSV (`prepareForExchange` resolves those independently and only falls back to CSV inference for linkage terms when none are supplied), so a run where the acceptor's column shape differs from the inviter's still links on the inviter's keys while the acceptor's CSV keeps driving its own standardizations and payloads. The acceptor's own name is substituted as the adopted terms' `identity` so the inviter's identity is not carried into the acceptor's prepared terms -- mirroring the CLI acceptor (`{ ...token.linkageTerms, identity: myIdentity }`), where the inviter is the source of the terms and the acceptor records its own identity. The inviter side is unchanged (it already runs the terms it embedded), and no core change was needed -- the `prepareForExchange` seam already used an explicit `linkageTerms` spec verbatim when present.
- Integration coverage for the online `invite`/`accept` failure paths and an SFTP transport-parity round trip, over the same real online code path (`validateInvite`/`validateAccept` -> `runOnlineBootstrap`, a live authenticated handshake) the happy-path filedrop test already drives. Three failure cases assert the security-relevant no-write guarantee end to end -- a rejected exchange persists neither a config nor a key file: an expired invitation aborts both the acceptor (rejected at decode) and the inviter's handshake (rejected by `runOnlineBootstrap`'s pre-handshake guard before any connection opens); a tampered invitation is rejected at accept validation when its checksum no longer matches; and a shared-secret mismatch aborts the NNpsk0 key confirmation on both sides, so neither saves the rotated key nor fires the config-writing hook. A new SFTP happy-path test mirrors the filedrop round trip's assertions over the real `atmoz/sftp` container, additionally exercising the credential and server-path threading through `connectionFromURL` that the filedrop transport never touches: an inline `sftp://user:pass@host/path` URL is parsed into the live connection and persisted into the config, which keeps inline connection credentials at `0600` while still stripping the shared secret. These behaviors are correct in code today; this is regression-backstop coverage with no runtime behavior change.
- The web key-exchange handshake now threads the invitation's `expires` into the authenticated key exchange, arming the in-handshake expiry guards that were dormant on the web path. The web handshake helper previously passed core's `authenticateConnection` only the shared secret, so its pre- and post-handshake expiry checks (gated on `expires !== undefined`) stayed no-op for web exchanges even though the CLI already ran them; the helper now forwards the invitation's `expires` alongside the secret, from both the inviter (surfaced from `generateInvitation`) and the acceptor (read off the decoded token). An invitation already past `expires` fails the handshake closed before any frame is sent, and one that expires during the key-exchange round-trip fails closed after the handshake completes -- both before any PSI protocol frame and both surfaced as the non-retryable `security`-kind authentication failure ("Could not verify your partner") rather than the generic retryable transport alert. The web acceptor still enforces expiry earlier, at the `/accept` consent boundary, so this is the complementary check that closes the round-trip window that boundary check cannot cover; it changes no wire format and adds no web-specific handshake variant, only populating an existing auth field the CLI already sets. Core's `auth.ts` guards are reused unchanged.
- Invitation expiry enforcement now fails closed on an unparseable `expires`, shared between the CLI and web acceptors through one `isInvitationExpired` helper in `@psilink/core`. Both acceptors previously compared `new Date(token.expires) <= now` inline, which for an unparseable `expires` evaluates to `false` (a `NaN` comparison) and would treat such a token as not-expired. The new helper rejects an unparseable value as well as one at or before the comparison instant, so an acceptor fails closed on its own rather than relying solely on `decodeInvitation`'s schema (`z.iso.datetime()`) to have rejected a malformed `expires` upstream -- defense in depth, since a malformed `expires` does not reach the check through the normal decode path today. Behavior for a well-formed `expires` is unchanged (the boundary remains fail-closed: equal to the instant is expired).
- Web-generated invitations now carry a bounded lifetime. The web inviter stamps an `expires` of one hour ahead of generation onto the `InvitationToken` it mints -- the same default the CLI uses and the one `docs/SECURITY_DESIGN.md` already states for invitation tokens -- so an intercepted web invitation has a finite misuse window instead of an unbounded one. The acceptor's existing fail-closed check (`prepareAcceptedInvitation` rejects a token whose `expires` is at or before the accept instant, before any rendezvous or connect) now consumes that value; both sides read the same ISO-8601 `expires`, so the bound the inviter sets is the bound the acceptor honors. The web app has no lifetime-selection control yet (deferred to the configuration-GUI roadmap work), so every web invitation takes the one-hour default; `generateInvitation` accepts an optional lifetime, bounded to a positive duration up to a one-year hard ceiling so the seam a future selector will use cannot mint an effectively-permanent token. The default and the ceiling are now exported from `@psilink/core` (`INVITATION_LIFETIME_SECONDS`, `MAX_INVITATION_LIFETIME_SECONDS`) and imported by both the CLI's `psilink invite` and the web inviter, which previously each held their own copy, so the shared invitation-lifetime policy has one source of truth and cannot drift between the two. No wire-format or documentation change is needed -- the `expires` field and the "bounded lifetime (default 1 hour)" claim already existed; this makes that claim true for web-generated invitations.
- A documented affordance, and a printed reminder, for abandoning a pending offline invitation before its nominal `expires`. The offline `psilink invite` path persists the pending shared secret to a key file, so unlike the online path -- whose pending secret lives only in the inviter's memory during the wait and lapses on cancel, timeout, or accept-timeout -- it leaves a durable artifact that can be retracted. Deleting that key file invalidates the invitation: the offline key exchange completes only when the inviting party still holds the secret, so once the inviter's copy is gone the secret carried in the forwarded invitation can no longer authenticate a handshake and the partner's copy is inert without a live inviter. The offline `invite` output now prints a reminder naming the key file and noting that only the key file should be deleted, so a configuration a recurring exchange still serves is left intact. No new command or runtime invalidation mechanism is added -- deletion is the existing, now-documented affordance, framed distinctly from the lost/reset/compromised-key recovery that previously was the only documented use of manual `.psilink.key` deletion. Specified in `docs/CLI.md` ("Abandoning a pending offline invitation") and `docs/SECURITY_DESIGN.md` ("Recurring exchange authentication").
- The web accept screen now renders the linkage terms the inviter proposes and requires explicit consent before any exchange begins, replacing the minimal terms preview the backend-free acceptor flow shipped. After an invitation decodes, the `/accept` route shows the decoded `linkageTerms` -- the inviting party, the matching method (`psi` reveals the shared identifiers; `psi-c` reveals only the count), the linkage keys records are matched on, the PII field types used, the inviter's output/result-sharing preferences, and any payload columns and legal agreement -- and gates the exchange behind an affirmative consent checkbox plus the acceptor's name. The exchange UI that dials the rendezvous mounts only after that consent commits, so no rendezvous, key-exchange handshake, or PSI frame can be sent before the user agrees; an expired or otherwise invalid invitation surfaces a distinct error and offers no consent action (the expiry enforcement itself is unchanged, in `prepareAcceptedInvitation`, which fails closed before any connect). All partner-controlled free text the inviter crafts into the token (the self-asserted identity, key names, legal-agreement and payload text) is escaped with the CLI's `sanitizeForDisplay` -- which neutralizes the control, bidi, zero-width, and homoglyph characters JSX escaping alone does not -- at a single `summarizeInvitation` boundary, mirroring the CLI's `displayInvitation`. The consent decision is centralized in one unit-tested `commitAcceptance` gate so the "no frame before consent" invariant is enforced in one place rather than relying on a button's disabled state. The screen frames these as the partner's proposed terms, and the exchange now adopts them as the terms it runs (see the web acceptor terms-adoption entry below).
- Authenticated cross-party abort marker for the `sftp` and `filedrop` channels, turning the sender-side peer-silence hedge into a definitive signal for the failure modes where the failing party's directory is still writable. When an exchange fails and that directory is still writable, the failing party writes a best-effort `<id>-abort.json` so the waiting peer fails fast with a definitive "the peer aborted" error (CLI exit 69) instead of waiting out its full `--peer-timeout` and then printing the generic peer-silence guidance. The marker carries **no cause** -- only the fact of the abort -- since the specific cause is not actionable to the waiting party (the partners coordinate out-of-band and the partner already holds its own full-fidelity error) and a plaintext cause on the shared directory would only worsen disclosure. The body is a 32-byte per-direction token derived from the post-handshake session key (`deriveAbortToken` in `@psilink/core`: `HKDF(sessionKey, "psilink-abort-token-v1:" + role)`, beside `deriveAeadKey`), so the untrusted directory admin or any non-handshake party -- holding no key -- cannot forge a marker the reader accepts, a captured marker never validates in another session (the session key is fresh-ephemeral per handshake), and one renamed to the other party's name fails the per-role binding. The waiting side reads a present `<peerId>-abort.json` size-bounded (its own small cap, never the 512 MB frame cap, refused before `get()` by the listed size), constant-time-compares the token, and on a match fails fast with a `PeerAbortError` whose definitive message also suppresses the generic "retry without re-inviting" advisory; any non-match, absent, unreadable, or oversized marker is ignored and falls back to the unchanged peer-silence guidance, so the marker's absence stays strictly uninformative. It is additive to the filename grammar -- the non-numeric `abort` terminal routes it as a control file, never consumed as a message -- and must persist for the peer to read, so a later exchange's entry guard recognize-and-sweeps a leftover (this party's own and any peer evidenced by a hello at entry, by exact name; delete mode only, retain mode through the existing `--force-retain-sweep` gate). The teardown sequencing parks `close()` on a write-vs-seal decision so the marker write -- short-bounded so a sick directory cannot hang teardown -- lands before the shared transport is ended, while the waiting party's own `PeerAbortError` is gated from echoing a marker back. As a protocol/crypto change it is subject to the explicit security review required by [CONTRIBUTING](CONTRIBUTING.md#dependency-policy). Specified in `docs/FILE_SYNC.md` ("Sender-side peer-silence attribution", file taxonomy).
- Web exchanges now run the X25519 authenticated key exchange before the PSI protocol. After the WebRTC data channel opens and immediately before any PSI frame, both web parties run the same `authenticateConnection` (X25519 NNpsk0) handshake the CLI uses -- keyed from the invitation's shared secret, taking its initiator/responder role from the exchange role the web already assigns -- so each party authenticates the peer and both derive the same 32-byte session key. The crypto is reused from `@psilink/core`, never re-implemented in the web app, so a future CLI WebRTC peer computes the same transcript. A handshake failure aborts the exchange before any PSI frame is sent; a trust failure (a wrong secret, a tampered or malformed frame, an expired or malformed credential) fails closed and is surfaced to the user as a non-retryable authentication failure ("Could not verify your partner") rather than the generic retryable "exchange failed" alert a transport drop produces -- distinguished by a `security`-kind `ConnectionError` that survives the exchange-lifecycle error classifier. The web path declines the application-layer AEAD in the ordinary case (a WebRTC data channel is end-to-end confidential under DTLS), so the derived session key currently serves only to authenticate; web exchanges remain single-use (the rotated secret is discarded, since the web app has no key-file persistence). Wrapping the connection in the application AEAD -- which a DTLS-terminating WebSocket relay would require -- is the remaining step. `docs/SECURITY_DESIGN.md` ("Channel security", "Key-agreement design") is updated to record that the web exchange now authenticates and derives a session key.
- Behavioral coverage pinning that a stalled retain-mode ack-write in the `FileSyncConnection` `poll()` loop is terminal. In retain mode the receiver consumes a message by writing a zero-length acknowledgment marker (`writeAck()`, a `put` plus a `rename`) before advancing its receive state, since the message file is never deleted; if that write surfaces a terminal `TransportOperationStalledError` (a `UsageError` subclass), `poll()` must stop the poller and emit the error once rather than retry into the stall, advance receive state, or re-deliver the message. A new test in `packages/core/test/fileSyncConnection.test.ts` drives that path and asserts the terminal, non-duplicating behavior, completing the consumer-side terminal-on-`UsageError` coverage that `docs/SECURITY_DESIGN.md` ("Channel security") states is pinned by tests -- the read consumers, the rendezvous gate, and the delete-mode consume were already covered. The invariant is correct in code today; this is a regression backstop with no runtime behavior change.
- Backend-free web rendezvous and acceptor flow, completing the web invitation exchange begun with the inviter generation below. Both parties now derive their PeerJS peer ids deterministically from the invitation's shared secret via a new `deriveRendezvousPeerId(sharedSecret, role)` in `@psilink/core` -- HKDF-SHA-256 over the decoded 32-byte secret, zero salt, role-separated info `psilink-webrtc-peerid-v1:inviter|acceptor`, first 16 bytes as lowercase hex (hex, not base64url, because the PeerJS client rejects an id with a leading, trailing, or doubled `-`/`_`). The inviter listens on its derived id and the acceptor dials that id directly, so the parties find each other with no coordination backend. A new `/accept` consent route reads the encoded token from the URL fragment, decodes it, enforces the invitation's `expires` (failing closed before any rendezvous or connect, since `decodeInvitation` validates format and checksum but deliberately does not check expiry), renders the inviter's linkage terms for review, and runs the rendezvous on accept; the homepage accept form takes either a bare encoded invitation or a pasted deep-link URL (peeling the token out of the fragment) and routes to that page. If the acceptor dials before the inviter is listening, PeerJS reports a non-fatal `peer-unavailable` and the dial retries on the same peer at a one-second cadence, bounded by the same ten-minute human-timescale ceiling the inviter's inbound wait uses, so neither side hangs on the other. The server-side session backend is removed: the `/api/psi/create`, `/api/psi/join`, `/api/psi/$uuid`, and `/api/psi/$uuid/wait` (server-sent-events) routes and the `utils/sessions.ts` / `utils/sse.ts` machinery are deleted, and with them the secret-less (zero-setup) web exchange -- a web exchange now always carries an invitation secret, since the rendezvous id is derived from it and no agency runs a standalone PeerJS peer-coordination server the way it runs an SFTP server. The PeerJS signaling server itself (`/api/peerjs/*`) is retained -- it is the transport, not the session backend. The `HKDF(secret, role)` peer-id construction is a shared rendezvous contract the future CLI WebRTC transport must compute identically. `docs/SECURITY_DESIGN.md` ("Channel security") is updated to record that the web app no longer offers a secret-less WebRTC rendezvous.
- Web inviter invitation generation: the web app's "Create an invitation" tab now mints a single-use invitation instead of creating a server-side session. It generates a fresh 32-byte shared secret, builds an `InvitationToken` carrying the inviter's linkage terms and the app's own credential-free PeerJS signaling endpoint, encodes it, and surfaces it two ways that carry the same token -- a deep-link URL (`<origin>/accept#<token>`, with the token in the URL fragment so the confidential value stays out of server logs and Referer headers) and a bare encoded string -- for out-of-band delivery to one partner; regenerating supersedes any prior unsent invitation. The shared-secret generator is extracted into `@psilink/core` as `generateSharedSecret` (`crypto.getRandomValues`-based, shared by the CLI and the web app rather than re-implemented per app). Inviter side only: the acceptor decode, the derived-id rendezvous, and removal of the `/api/psi` backend land with the paired web rendezvous task, so the existing session-based flow is unchanged for now. The invitation's confidentiality model is documented in `docs/SECURITY_DESIGN.md` ("Invitation contents and confidentiality").
- `authentication.token_max_age_days`: an optional operator-policy field that enforces a maximum age on rotated tokens, independent of exchange frequency. Rotation tokens otherwise carry no expiry, on the assumption that active partnerships exchange often; a dormant partnership (a monthly cadence, a holiday gap) could hold a valid token for months. When set to a positive integer, each successful `psilink exchange` stamps `expires` = (the moment of rotation) + that many days into the rotated `.psilink.key`; when omitted, rotated tokens carry no expiry (unchanged). At the next exchange the CLI enforces the stamp at load time, before opening any connection or attempting the key exchange: an already-expired token aborts with an error naming the expired time and directing both parties to re-invite, and a token within `token_max_age_days / 3` days of expiry emits a pre-exchange warning. The warning is suppressed when that exchange succeeds (rotation stamps a fresh, farther-out `expires`) and shown only when no successful rotation refreshed the token, where re-inviting before the lapse is actionable. The `expires` field and its pre/post-handshake enforcement already existed (for invitation tokens); this adds the rotation write path and the load-time check. The top-level `authentication` block is now validated with a strict schema, so a misspelled policy key (for example `token_max_age_dayss`) is rejected at config-parse time rather than silently dropped -- failing closed so a typo cannot quietly disable the control; this departs deliberately from the strip behavior of the sibling spec blocks because `authentication` alone holds an operator security policy. As net-new security behavior, the enforcement is subject to the explicit security review required by [CONTRIBUTING](CONTRIBUTING.md#dependency-policy). Specified in `docs/EXCHANGE_SPEC.md` ("Authentication"), `docs/SECURITY_DESIGN.md` ("Token age and rotation policy"), and `docs/CLI.md` ("Key lifecycle").
- Operator guidance on the sender's peer-silence timeout. When the peer dies mid-exchange (the canonical case being its exchange directory going read-only so it can never write its next message or ack), the receiver names its own cause locally but the remote sender previously saw only a bare "peer went silent" inactivity timeout. The sender's timeout error now carries a file-sync-specific hint (`PEER_SILENCE_GUIDANCE` in `apps/cli/src/protocol.ts`) naming the likely receiver-side causes -- the peer exited, or its directory became unwritable -- pointing the operator at the peer's own logs where the real cause was recorded, and noting the slow-large-dataset case so the timeout is not misread. A definitive cross-party signal was deferred at the time and now ships as the authenticated abort marker above, which upgrades this hedge for the writable-directory modes; the unwritable-directory headline (and a hard kill) stay uncoverable -- exactly what stops the peer from writing any marker the sender could read -- so this guidance remains the floor whenever no valid marker is present. The decision and its rationale are recorded in `docs/FILE_SYNC.md` ("Sender-side peer-silence attribution"). Threaded through a new optional `inactivityHint` on `fromEventConnection`/`QueuedMessageConnection` in `@psilink/core`, which stays transport-agnostic -- the WebRTC transport, which shares the same core inactivity message but supplies no hint, is unchanged.
- Per-operation liveness bounds on the SFTP adapter's write/stat/delete operations (`put`/`rename`/`delete`/`exists`), completing the SFTP liveness-bounds effort so a withheld or stalled server callback fast-fails in ~60 s with a terminal `TransportOperationStalledError` instead of riding the ~1 h whole-exchange budget. The metadata ops `rename`/`delete`/`exists` -- each a single payload-free round-trip -- carry the same 60-second whole-operation deadline as `createExclusive`, with the same negligible false-fail risk. `put` instead carries a 60-second progress idle window: its payload is streamed in 64 KiB chunks and the window resets on each chunk acknowledged under the write stream's ack-driven backpressure, so a withheld write ack trips it while a legitimately large slow-but-progressing upload never is -- the write-path mirror of the capped `get` idle window, and the reason a flat deadline is wrong for `put` specifically. The best-effort `safeDelete` (used on the teardown/cleanup path) carries the same 60-second deadline so it cannot be the soft target left on the coarse budget, but swallows the deadline's stall error to keep its never-reject contract: it always resolves, just within 60 s. The whole-exchange budget (`peerTimeoutMs`) is unchanged and remains the universal backstop beneath these per-op bounds (and the only liveness bound for the local-filesystem adapter). Uses ssh2-sftp-client's public stream interface, so it adds no new raw-`SFTPWrapper` coupling. As net-new transport security behavior it is subject to the explicit security review required by [CONTRIBUTING](CONTRIBUTING.md#dependency-policy). Specified in `docs/SECURITY_DESIGN.md` ("Channel security").
- Behavioral coverage and documentation pinning that a `FileSyncConnection` `poll()` error is terminal on first emit when the connection is composed through the production bridge. In the CLI the connection is always bridged via `fromEventConnection` (`apps/cli/src/protocol.ts`), whose `error` listener calls `QueuedMessageConnection.fail()`, which synchronously tears the connection down (`hooks.close()` -> `conn.close()` -> `conn.stop()`) before `poll()`'s `finally` can reschedule. A new test in `packages/core/test/fileSyncConnection.test.ts` drives a `FileSyncConnection` through the bridge and asserts the first emitted transient `poll()` error fails the awaited `receive()` exactly once -- naming the underlying transport cause rather than a peer-silence timeout -- stops the connection, and is not rescheduled; it lives alongside the unchanged isolation test that exercises the opposite reschedule-on-transient behavior without the bridge. `docs/FILE_SYNC.md` gains a "Production composition note" under invariant I8 recording that I8's reschedule-on-transient behavior (and the isolation tests that drive `poll()` without the bridge) describes the module standalone rather than the composed exchange. The composed behavior is correct in code today; this is a contract backstop with no runtime behavior change.
- Behavioral coverage and documentation for the consumer-side terminal-on-`UsageError` invariant at transport-call retry sites. The typed transport bounds (`FrameSizeExceededError`, `DirectoryListingBoundsError`, `TransportOperationStalledError` -- all `UsageError` subclasses) are terminal by design: a consumer awaiting a transport `list()`/`get()`/`createExclusive()` call must propagate, never retry, a `UsageError`, because retrying loops straight back into the hang or over-allocation the bound exists to prevent. The two consumers that exist today are pinned by tests in `packages/core/test/fileSyncConnection.test.ts` that drive each with a `UsageError` from a transport read and assert it is surfaced once and propagated, not retried: the rendezvous gate (`readControlFileWithGate`, via `synchronize()`) and the background poll loop. `docs/SECURITY_DESIGN.md` ("Channel security") records the consumer-side rule that ties the three bounds together. The invariant is correct in code today; this is a regression backstop with no runtime behavior change.
- `psilink invite --expires-in DURATION` overrides the default one-hour invitation lifetime, for coordination windows longer or shorter than an hour. `DURATION` is a positive integer with a required unit suffix -- `s`, `m`, `h`, or `d` (for example `30m`, `2h`, `1d`); a zero, negative, or malformed value is rejected before any invitation is generated, and the existing `encodeInvitation` future-expiry guard remains the backstop. Online, when the resulting lifetime is shorter than `--accept-timeout` the command warns the token will expire before the wait ends. Parsed by a new shared `parseDuration` helper (`apps/cli/src/util/duration.ts`) intended as the canonical human-readable format for command-line durations. Specified in `docs/CLI.md` ("Offline invitation", "Online invitation").
- Session-start directory hygiene for the `sftp` and `filedrop` channels, relaxing the strict-empty entry guard. Foreign files -- names that do not match the protocol filename grammar, such as a sync tool's conflict copy or partial download -- present in the shared directory at `synchronize()` start are now recorded in an in-memory snapshot and tolerated for the session in both modes (deleting nothing) instead of aborting the exchange; a foreign file that first appears mid-session continues to be handled by `unexpected_files`. A message-shaped `<id>-<digits>.json` still matches the grammar and is treated as a protocol file, so the default stays strict-empty for protocol files. Two new CLI-only, non-persistable recovery flags: `--sweep-exchange-files` deletes every protocol file (this party's and the peer's: hellos, acks, locks, joining sentinels, messages) left by a crashed or bilateral-mode-mismatched prior run and starts a fresh exchange, leaving foreign files untouched. Before deleting it inspects for a retain-mode signal -- local retain mode, a peer hello advertising `retain_files=true` (read through the partial-sync gate, bounded so it cannot stall), or a retain-only message ack -- and refuses, since a retain directory is a durable audit transcript; clearing one requires the additional `--force-retain-sweep`, which permanently discards the prior transcript after a loud warning. `--force-retain-sweep` requires `--sweep-exchange-files` and is rejected on its own. Neither flag is a `connection.options` key, so neither can be set in `psilink.yaml`. The sweep uses a rejecting delete, so a transport that cannot delete surfaces a transport error (exit 69) rather than a false clean slate. Specified in `docs/FILE_SYNC.md` and `docs/EXCHANGE_SPEC.md` ("Directory exclusivity").
- X25519 authenticated key exchange in `@psilink/core` (`runKex`, `KexResult`): an ephemeral X25519 Diffie-Hellman pinned to the Noise NNpsk0 pattern (no static keys, the pre-shared secret mixed at position 0, ephemeral-ephemeral DH) plus an explicit, role-asymmetric mutual key confirmation, yielding a 32-byte session key with forward secrecy and mutual authentication -- the key-agreement primitive that supersedes SPAKE2 as the source of the exchange session key. The session key is derived (NNpsk0 key schedule) over both the X25519 shared secret and the pre-shared secret plus the transcript, so it is neither the pre-shared secret alone (which would have no forward secrecy) nor the raw DH output alone. The X25519 primitive and the ephemeral keygen come from the already-present, audited `@noble/curves` (no new dependency); only the minimal NNpsk0 glue (key-schedule mixing and the two confirmation tags) is hand-written, and the full Noise framework is deliberately not implemented. Carries over SPAKE2's safeguards: strict (`.strict()`) wire schemas, constant-time tag comparison, a single generic non-oracular failure, a best-effort abort to shorten the peer's wait, and an all-zeros contributory check on the X25519 share. Backed by a checked-in known-answer vector and cross-checked against RFC 7748 (X25519) and RFC 5869 (HKDF). Specified in `docs/PROTOCOL.md` ("X25519 authenticated key exchange").
- `--save` on the zero-setup exchange now bootstraps a persistent shared secret in-band, without a separate invitation round. Each party advertises its save intent on the linkage-terms exchange (the one round-trip both sides always perform, so a one-sided save never stalls). When both parties pass `--save`, the initiator generates a fresh base64url 32-byte secret -- the same format a recurring exchange rotates to -- and transmits it to the responder on a dedicated in-band message, and both save config and key files through the provisioning layer (with conflict detection up front, before any network activity). When only one party saves, no secret is generated: that party saves its config alone (no key file) and is told to use `psilink invite`, and the partner is notified that nothing was saved on its end. On a zero-setup exchange there is no application-layer AEAD, so the in-band secret is protected only by the transport (SSH/DTLS/operator access controls) -- the same trust model as the zero-setup exchange that carries it, for that one initial exchange; see `docs/SECURITY_DESIGN.md` ("Bootstrapping a shared secret"). New `exchangeBootstrapSecret` and the `saveIntent`/`bootstrap` plumbing on `runExchange` in `@psilink/core`.
- Receipt signing identities (Phase 2 foundation for signed receipts): a long-lived per-party signing identity -- an Ed25519 keypair and a self-signed certificate carrying the party's `identity` -- generated once, persisted owner-read-only in its own file (default `~/.psilink/signing-identity.json`, separate from the rotating `.psilink.key`), and reused across every exchange and partner. Trust is pinned self-signed (no CA chain, no revocation): the certificate's fingerprint (unpadded base64url SHA-256 over the domain-separated canonical encoding of the certificate body) is exchanged out-of-band and pinned, and a presented partner certificate is accepted only if it self-verifies and its fingerprint matches the pin; a receipt is accepted only if its asserted identity exactly matches the certificate's bound identity (exact match over the canonical identity bytes). The certificate is a canonical-JSON document (no new dependency; reuses `@noble/curves` and the RFC 8785 canonical encoding) with a `version`/`algorithm` seam toward a future authority-backed mode. Exposed in `@psilink/core` (`generateSigningIdentity`, `computeCertificateFingerprint`, `parseCertificate`/`parseSigningIdentity`, `assertPartnerCertificateTrusted`, `verifyPresentedCertificate`, the `signing` block on `ExchangeSpec`) and backed by checked-in cross-implementation vectors. The CLI adds `psilink fingerprint`, which lazily creates the identity on first use, prints the fingerprint to share out-of-band, and regenerates with `--force`. Receipt assembly, signing, and the receipt swap remain to be wired up. Specified in `docs/PROTOCOL.md` ("Signing identity and certificate pinning"), `docs/SECURITY_DESIGN.md` ("Receipt signing identities"), and `docs/EXCHANGE_SPEC.md` ("Signing").
- `psilink invite` and `psilink accept` CLI commands for moving from a zero-setup exchange to a recurring one backed by a shared secret. Each has an offline mode (no network: `invite` generates a shared secret, writes a key file with an `expires`, and prints an invitation string plus usage instructions; `accept` decodes and displays the invitation, prompts for confirmation, and writes config and key files) and an online mode (additionally connect, complete the X25519 key exchange, and run the exchange in one step, persisting the rotated key and config as soon as the handshake succeeds -- before the data exchange begins -- so an exchange that fails after a successful handshake leaves both on disk and is recoverable with `psilink exchange` without re-inviting; a handshake that never succeeds leaves no config behind). The invitation's 4-byte checksum and schema are verified, and an expired token is rejected by name, before the user is prompted; a pre-existing config or key file at a target path is detected before any network activity. An invitation string beginning with `-` is parsed as the positional invitation, not an option flag. The acceptor's config seeds its connection block from the invitation's credential-free endpoint when present (with a `REPLACE_WITH_...` marker on the credential field) and a clearly-marked placeholder otherwise. Some richer behaviors documented in `docs/CLI.md` (seeding the printed online invitation with a connection endpoint) are not yet implemented; see the "Current CLI limitations" note there.
- `psilink accept` and online `psilink invite` now reconcile a pre-existing config or key file at a target path instead of always aborting. `accept` (offline and online) compares an existing config's linkage terms against the invitation -- and, online, its connection block against the connection target (the URL plus any `--server-*` overrides) -- reusing the config unchanged when they agree. The connection comparison distinguishes the rendezvous location (host and path) from how the same drop is reached (the protocol/channel, port, and credentials): a location mismatch fails before any acceptance is sent, showing which values differ and how to resolve them, while an access-detail difference is applied to the exchange with a warning that the saved config is left unchanged rather than aborting. An absent URL field with no matching override is never a conflict, and credential values are never echoed in a warning. Linkage-key strings are compared NFC-normalized so Unicode-equivalent forms are not falsely flagged. A pre-existing key file on the accept path remains a hard error (a stale token must not be reused); online `invite` downgrades a pre-existing key file to a warning (it is overwritten by the rotated token on a successful handshake). The online write path now re-gates the config write immediately before persisting, so a config that appears between the pre-network conflict check and the post-handshake write is not silently overwritten -- matching the offline path. New `diffLinkageTerms`/`diffConnectionAgainstTarget` reconciliation helpers in the CLI. Specified in `docs/CLI.md` ("Offline acceptance", "Online invitation", "Online acceptance").
- Self-attested exchange record (a standalone disclosure-log entry): at the end of every successful exchange each party produces a local audit record of what it disclosed -- the agreed-terms hash, both self-asserted identities, the timestamp, the count of records this party exposed to the exchange (`recordsExposed`, a per-direction figure recorded on every record), the result size (only when both parties are entitled to it), a per-exchange binding nonce, readable governance metadata, and privacy-preserving commitments to the data exchanged (payload columns sent and received, and the association table when held). The two disclosure figures are gated deliberately: `recordsExposed` is the party's own input row count (every contributed record, not only those that resolve to a usable linkage key), known from its own input and recorded regardless of role, while the result size (the intersection cardinality) is recorded only in the both-output case, when both sides are entitled to the result. A single-output helper can observe its match count during the protocol, but the record deliberately does not surface it -- privacy is enforced by what the tool writes down, not by what is theoretically discoverable, and the helper's outbound disclosure is carried by `recordsExposed` instead, a framing that stays meaningful even under a future algorithm that discloses neither the result size nor the partner's set size. The governance metadata makes the record self-standing for a HIPAA accounting of disclosures or a FERPA disclosure record without re-matching the original config: the governing data-sharing agreement (reference, the purpose of the disclosure under it, and expiration -- all three mutually validated, so both parties' records agree on why the disclosure happened), the algorithm (`psi` vs `psi-c`), the categories of data exchanged (payload column names and data-dictionary descriptions sent and received, with the no-payload case explicit), and the matching basis (the standardized name and semantic type of each linkage field the keys reference for matching). It carries names, categories, descriptions, and references only -- never payload values, linkage-field values, or matched identifiers. Commitments are `HMAC-SHA-256` keyed by a fresh per-commitment salt over the canonical encoding, so the record binds to the data without revealing it (a bare hash of a low-entropy result would be brute-forceable). The record is split into a shareable record file and a private opening file (salts plus committed data); both are serialized via the canonical-encoding module and reproduce byte-identically across implementations (checked-in cross-impl vectors, verified on Node and in the browser). It is explicitly unsigned and not a non-repudiable receipt; signing it is separate, deferred work. Each party's record can also carry an optional self-facing retention/disposition pointer -- a free-text note from its own exchange config (`retention_disposition`) recording where it filed its copy of the result and under what retention schedule it is held or disposed of -- written into its own record only, per-party (the two parties' notes are independent and need not match), never exchanged with the partner or folded into the agreed-terms hash, and carrying no protected values. Exposed as `buildExchangeRecord` and returned from `runExchange` in `@psilink/core`; the CLI writes both files atomically to a timestamped default path (`--record-file`, `--no-record`) and the web app offers them as downloads. Specified in `docs/EXCHANGE_RECORD.md` (the dedicated record-format spec) and `docs/EXCHANGE_SPEC.md` ("Retention and disposition").
- Linkage-terms validation now rejects a party that declares it receives no output (`output.expects_output: false`) while listing `payload.receive` columns: a non-receiving party cannot receive payload for matched records it never gets, so the configuration is incoherent and fails at parse/compatibility time rather than producing a record that claims a party received payload it was never entitled to. A non-receiving party may still `send` payload.
- Invitations may now carry an optional credential-free connection endpoint: a public locator (PeerJS signaling URL, SFTP host and port, or file-drop directory) that tells the acceptor where to rendezvous, so the parties need not arrange that detail over a separate channel. Modeled in `@psilink/core` as the `ConnectionEndpoint` discriminated union on `InvitationToken.connectionEndpoint`, discriminated by `channel` like the `connection` block. The endpoint never carries credentials -- it has no field for a password, private key, key file, or PeerJS API key, and an invitation carrying one is rejected when decoded. The field is optional, so existing invitations remain valid. This reverses the prior invariant that an invitation excludes all connection information; the replacement invariant is that it may carry a locator but never credentials.
- Canonical encoding for receipts: a language-independent, deterministic byte serialization (RFC 8785, JSON Canonicalization Scheme) used for everything that is hashed, committed, or signed, so receipt hashes and signatures verify across independent implementations. Exposed as `canonicalString`/`canonicalBytes` (with `safeIntegerSchema` for hashed numeric fields) in `@psilink/core`, specified normatively in `docs/CANONICAL_ENCODING.md`, and backed by checked-in cross-implementation test vectors verified byte-identically on Node and in the browser. Foundation for the self-attested record and the signed non-repudiation receipt.

- `@psilink/core/testing` subpath export: `withSuppressedLogs` and `withCapturedLogs` helpers for controlling loglevel output in tests.
- `EncryptedMessageConnection` in `@psilink/core`: an application-layer AEAD decorator over `MessageConnection` that encrypts every outbound message and decrypts every inbound one with AES-256-GCM, keyed per direction from the key-exchange session key, so an SFTP/filedrop administrator sees only opaque ciphertext. Any integrity, replay, ordering, or format failure surfaces as a terminal `security`-kind `ConnectionError` rather than a silent transport drop. The CLI wires it into recurring (authenticated) exchanges: immediately after authentication `runProtocol` wraps the connection in the decorator and runs the PSI exchange through it, so recurring CLI exchanges are encrypted on the wire end to end. Zero-setup exchanges (no session key) and the web application's application layer are not covered. Of the channel-hardening follow-ups originally deferred here, the inbound frame-size bound, the companion directory-listing bound, and strict mid-stream sequence-gap detection are now enforced; end-of-stream / truncated-tail detection remains an accepted deferral that does not block release on its own. See `docs/SECURITY_DESIGN.md` ("Channel security").
- Key rotation: the shared token is automatically replaced with a new one derived from the key-exchange session key at the end of each successful handshake.
- `filedrop` channel: file-system-based exchange for environments where neither SFTP nor WebRTC is available. Both parties read and write through a shared locally-mounted directory.
- `retain_files` mode for the `sftp` and `filedrop` channels: the receiver writes a zero-length acknowledgment marker after consuming each message instead of deleting it, so no exchange file is deleted as a protocol step and the shared directory becomes a permanent transcript. Requires `lockless_rendezvous` and `timestamp_in_filename`; the `--retain-files` CLI flag implies both. Intended for sync-mediated transports that do not propagate deletions and for audit retention.
- Bilateral mode mismatch detection at rendezvous: each party advertises its `lockless_rendezvous` and `retain_files` settings in its hello, and a mismatch fails fast on both parties with a clear usage error (exit 64) naming each side's setting, instead of stalling until the peer timeout.
- `connection.options.unexpected_files` (`sftp`/`filedrop`): policy for a file that appears in the shared directory during the message loop and is neither part of this exchange nor an in-flight temp write -- `error` (fail with a usage error, exit 64, naming the file and the directory path), `warn` (log once per distinct name and continue), or `ignore` (skip silently, the previous behavior). The setting is local, not bilateral; the two parties may use different values. When unset it defaults to `error` on plain transports and `warn` when `retain_files` or `lockless_rendezvous` is set, where sync tools legitimately produce transient conflict copies and partial downloads. A malformed protocol file (a message-shaped name a correctly configured peer cannot produce) is always reported regardless of this setting.
- `zero-setup` CLI command: derive connection parameters directly from a URL, eliminating the need for a configuration file for one-off exchanges.
- `exchange` CLI command updated to require a `.psilink.key` file and perform the X25519 authenticated key exchange before any data is transmitted.
- Key file schema: versioned JSON format for persisting the pre-shared token and exchange metadata between sessions.
- Meta data inference: guess column semantic types from names and infers date formats.
- Custom linkage keys: linkage terms can be set in config and keys can be derived using transformations.
- Data standardization: transformations can create canonical linkage terms from input columns according to their semantic types.

### Changed

- The partner shared-secret `authentication` block moved out of the per-channel `connection` config and up to the top level of the exchange spec, as a sibling of `signing` -- channel-agnostic across `sftp`/`filedrop`/`webrtc`, and reshaped so it holds both the key-file-injected secret state (`shared_secret`, `expires`) and operator-settable policy fields. The WebRTC `inviter`/`acceptor` `role` -- a peer-addressing concern that derives deterministic PeerJS peer IDs -- relocated into the WebRTC connection config as `connection.role`; the `WebRTCAuthentication` type (which existed only to add `role` to the shared `Authentication`) is dropped. The CLI config loader now strips only the injected fields (`shared_secret`/`expires`) from the top-level block, warning when they are set, and admits any operator-policy field through to schema validation rather than blanket-deleting the whole block; `saveConfig` strips from the top-level block; and `runProtocol` receives authentication on a separate `auth` parameter, collapsing the prior machinery that embedded an `AuthPersist`-or-`null` field into the connection union via a distributive `Omit`. Authentication, rotation, and expiry behavior are unchanged end to end. Breaking config-shape change, accepted because nothing has been released on the current format; no migration shim is provided. Specified in `docs/EXCHANGE_SPEC.md` ("Authentication", "connection.role") and `docs/SECURITY_DESIGN.md` ("Recurring exchange authentication").
- De-duplicated the opaque-key recurse-and-skip traversal shared by the config read and write paths. `@psilink/core` now owns a single walker and produces both directions from it -- `camelizeKeys` (read) and `snakeizeKeys` (write) -- with the `OPAQUE_VALUE_KEYS` skip baked into that one traversal; the CLI config writer (`saveConfig`) imports and delegates to the core `snakeizeKeys` instead of re-implementing the recursion. Opacity is now decided on the canonicalized form of each key in both directions, so the read and write paths provably skip the identical set of subtrees -- a structural guarantee asserted by a new core test driven from `OPAQUE_VALUE_KEYS` (so a second opaque key is covered automatically), replacing the prior arrangement that held two independently-written recursions in agreement by prose and a single round-trip test. It also removes the latent hazard that arrangement left in place: a snake_case key routed through `snakeizeKeys` is now skipped correctly rather than relying on its only caller always passing a typed camelCase `ExchangeSpec`. The `@internal` `OPAQUE_VALUE_KEYS` is no longer re-exported from `@psilink/core` (the CLI now imports the `snakeizeKeys` behavior, not the data); both symbols remain unstable internal API. No runtime behavior change; the `provider_options` write -> read round-trip stays byte-stable.
- `apps/cli` now declares `ssh2` as a direct, exact-pinned dependency (`1.17.0`) with a patch-pinned `@types/ssh2` (`~1.15.5`, the latest published types line), pins `ssh2-sftp-client` to an exact `12.1.1` (from `^12.0.1`), and runs the CLI integration suite in CI on the Docker-capable `ubuntu-latest` runner -- as a dedicated pull-request gate (`cli_build_and_test.yaml`, separate from the web-only Elastic Beanstalk workflow) and again in `release.yaml` as the ship gate before the Docker image publishes. The SFTP adapter deliberately drives `ssh2`'s raw `SFTPWrapper` internals past the public `ssh2-sftp-client` surface, yet `ssh2` entered only transitively and `ssh2-sftp-client` declares its `ssh2` range as `^1.16.0` -- so a transitive re-resolution could swap the very internals the adapter reads with no change to `apps/cli`'s declared deps. Exact-pinning both packages makes every bump -- including a security patch, which becomes a deliberate edit rather than an `npm audit fix` that slips in unreviewed -- a visible change that triggers the SFTP-stack upgrade re-verify; `@types/ssh2` type-checks that raw surface directly rather than via the lagging `ssh2-sftp-client` types; and the integration suite, the only CI coverage that exercises the real wrapper contract end to end (unit tests run against mocks), now fails the pull-request and release CI on a dependency bump that breaks the contract instead of merging or shipping green. The pins are cross-referenced to the re-verify checklist in `docs/SECURITY_DESIGN.md` ("Upgrading the SFTP stack") from `CONTRIBUTING.md`'s dependency policy and the adapter header. Upgrade-safety hardening only; no runtime behavior change.
- Online `psilink invite` now states up front, before it begins waiting, that the printed invitation can be accepted only while this command is waiting: cancelling (Ctrl-C), the connection timing out, or reaching `--accept-timeout` before the partner accepts leaves the rendezvous and discards the one-time setup secret, so the invitation can no longer be accepted afterward and a fresh `psilink invite` is required. This early invalidation was already true by construction for the online path -- the setup secret is held only in memory until a handshake succeeds and is never persisted before acceptance, the rendezvous files are swept on cleanup, and `authenticateConnection` rejects an expired secret before and after the handshake -- so the change is the user-facing notice plus removing the stale "does not yet revoke or expire the token early on timeout or cancellation" entry from the `docs/CLI.md` "Current CLI limitations" note. Early invalidation of a pending offline invitation (whose secret is written to a key file up front, with no waiting process) is a separate, tracked follow-up. Documented in `docs/CLI.md` ("Online invitation").
- The X25519 handshake now negotiates whether to apply the application-layer AEAD, replacing the implicit "encrypt iff authenticated" rule with a per-party signal carried inside the existing handshake. Each party declares a request-encryption flag on its own message (the initiator's on message 1, the responder's on message 2, no extra message or round-trip), the flag is bound into the transcript hash via `MixHash` (so tampering with either flag on the wire aborts the handshake at confirmation rather than yielding a split decision), and both parties derive the same `applyEncryption = own OR peer` decision, now exposed on `KexResult` and `AuthResult` in `@psilink/core`. `runKex` and `authenticateConnection` take a `requestEncryption` argument. The handshake protocol identifier is unchanged (no version bump pre-release): a flag-aware and a flag-unaware build already fail closed at parse -- the newer build rejects the older build's message for the missing required `reqEnc` field, and the older build's strict (`.strict()`) schema rejects the newer build's message for the unrecognized `reqEnc` key -- so the version tag stays reserved for a genuine cryptographic-suite change. The CLI requests encryption unconditionally on its SFTP and file-drop channels and gates the `EncryptedMessageConnection` wrap on the negotiated decision, so observable behavior is unchanged (authenticated file-sync exchanges remain encrypted; zero-setup exchanges remain cleartext); the mechanism unifies the CLI and the future web path, which will request it only when a DTLS-terminating WebSocket relay is forced into the WebRTC path. Known-answer vectors are regenerated to cover all four flag combinations. Specified in `docs/PROTOCOL.md` ("X25519 authenticated key exchange") and `docs/SECURITY_DESIGN.md` ("Channel security").
- Offline `psilink invite` can now source its linkage terms from a pre-existing configuration file instead of always requiring an `INPUT_FILE`. With a config present, invite derives the invitation's linkage terms from it and writes only the key file, leaving the config untouched. When both a config and an `INPUT_FILE` are supplied, the input's columns are checked against the config and invite fails -- naming the unsatisfiable linkage fields -- if the available data standardizations cannot transform those columns into the config's linkage fields. With no config, the previous input-only behavior is unchanged (terms, metadata, and standardizations are inferred from the input and written to a new config). Generating an invitation with neither a usable config nor an `INPUT_FILE` is a usage error. A pre-existing key file is still refused, so an existing shared secret is never clobbered. Documented in `docs/CLI.md` ("Offline invitation").
- The CLI now authenticates recurring exchanges with the X25519 authenticated key exchange (`runKex`) instead of SPAKE2: `authenticateConnection` derives the session key from the new handshake, and the existing application-layer AEAD (`deriveAeadKey`) and shared-secret rotation re-point onto it unchanged in shape (the AEAD wire format and known-answer vectors are untouched; only the session-key input changes). Breaking and intentionally non-interoperable -- an old SPAKE2 proof-of-concept client and a new one fail closed at the handshake with a generic error rather than hanging -- accepted because the project is pre-release. The shared credential is renamed `pake_token` -> `shared_secret` across the YAML field and its config-key alias, the `@psilink/core` identifier `SHARED_SECRET_REGEX`, the CLI-internal `generateSharedSecret`, and the `KeyFile`/`Authentication`/invitation-envelope fields, and the internal HKDF rotation label `psilink-token-rotation-v1` -> `psilink-shared-secret-rotation-v1`; the `.psilink.key` filename is kept. Specified in `docs/PROTOCOL.md` ("X25519 authenticated key exchange") and `docs/SECURITY_DESIGN.md` ("Key-agreement design").
- Renamed the default file-sync rendezvous mode from "wave" to "lock" (the counterpart to the existing "lockless" mode) and its on-disk tiebreaker file from `<id1>-<id2>.wave` to `<peer1>-<peer2>-lock.json`, bringing the tiebreaker under the same control-file filename grammar as every other protocol file. Terminology and on-disk naming only: no protocol behavior or config-schema change, and `lockless_rendezvous: false` still selects this mode. Old and new builds do not interoperate at rendezvous (one writes `.wave`, the other `-lock.json`), so both ends must run the same build.
- Library API: `runExchange` and `authenticateConnection` now take a `MessageConnection` instead of the event-based `Connection`. CLI exchanges are unaffected.
- An exchange now fails with a clear error (bounded by `peer_timeout_ms`) when the partner stops responding, instead of hanging.
- A peer sending messages out of turn is rejected rather than buffered without limit.
- The CLI now exits with code 64 (EX_USAGE) instead of 69 (EX_UNAVAILABLE) when `synchronize()` or `send()` fails due to a configuration or usage problem: a dirty exchange directory (preexisting handshake files), multiple concurrent sessions on the same path, or a send timeout. Transport failures continue to exit with code 69.
- `psilink exchange` now exits 64 (EX_USAGE) for every malformed, unreadable, or missing configuration (`psilink.yaml`) or key (`.psilink.key`) file. Previously only a missing config file exited 64, while a missing or malformed key file and a malformed config exited 69 (EX_UNAVAILABLE), the code reserved for transport failures. A malformed shared secret is now classified the same way whether the key file is read or written.
- The CLI now exits 64 (EX_USAGE) instead of 69 when the requested channel or URL scheme is unsupported -- a `webrtc` config or a `ws://`/`wss://` URL, an unknown URL scheme, or a malformed `file://` authority -- in both `psilink exchange` and the zero-setup command, classifying it as invalid caller input rather than a transport failure.
- Invalid connection-option combinations -- a reserved or unaccompanied `peer_id`, a `retain_files`/`lockless_rendezvous` contradiction, and similar -- now exit 64 (EX_USAGE) instead of 69, whether they originate in `psilink.yaml` or a command-line override.

Source-breaking for a direct TypeScript caller that passed a non-literal label; there is no live caller yet. Rationale under Security.

### Fixed

- A timed-out `filedrop`/local-filesystem connect attempt is now terminal instead of being retried, bounding the libuv thread-pool workers a stalled mount can strand. `LocalFSClient.connect` races each `fs.access` probe against a per-attempt `connectTimeoutMs` deadline; on a stalled NFS/CIFS hard mount the abandoned probe keeps occupying a thread-pool worker until the OS releases the syscall (minutes), and because a timeout previously consumed a reconnect attempt, the default `maxReconnectAttempts: 3` let up to four such workers stack up and exhaust the default four-thread pool, starving all other fs/dns/crypto work in the process. A timeout is now terminal -- the connect fails fast after one timeout window rather than dispatching further doomed probes -- while a genuine fast-returning transient error (`EACCES`/`ENOENT` while a share or its permissions are still settling) still retries up to `maxReconnectAttempts`. This bounds concurrent stalled workers to one; it cannot free that one worker early, because nothing in-process can cancel an already-dispatched blocking syscall (`fs.access` does not honor an `AbortSignal`, and libuv's `uv_cancel` only drops still-queued work). `withTimeout` now rejects with a distinct `TimeoutError` (exported from `@psilink/core`) so the retry predicate keys on the error type rather than a message string; the connect-timeout exit code is unchanged (`TimeoutError` extends `Error`, not `UsageError`, so it stays EX_UNAVAILABLE / exit 69). The `max_reconnect_attempts` description in `docs/EXCHANGE_SPEC.md` is corrected to match the implementation: it retries the initial connection after a transient failure and does not reopen a dropped connection (there is no reconnection after a drop).
- Repeating a single-value CLI option -- a string or number flag such as `--accept-timeout`, `--server-port`, `--connection-timeout`, `--peer-timeout`, `--max-reconnect-attempts`, or `--log-level` -- is now rejected with a clean usage error naming the flag (`--<flag> may be given only once`, exit 64) before the value is used, across every command that exposes it (`invite`, `accept`, `exchange`, `fingerprint`, and the default zero-setup command). yargs collects a repeated flag into an array, which the handlers previously cast to a scalar and used directly: a fat-fingered `psilink invite --accept-timeout 60 --accept-timeout 120` reached the `acceptTimeout <= 0` / `acceptTimeout > lifetimeSeconds` comparisons in `validateInvite` with an array operand (a wrong-but-not-rejected `NaN`/array result), and a repeated string flag like `--log-level` hit `.toLowerCase()` and crashed with a raw `TypeError` that surfaced as a generic exit 69. A shared `singleValue` accessor (`apps/cli/src/util/cli.ts`) now guards each single-value read, generalizing the existing per-handler `--expires-in` guard so a newly added option inherits the rejection rather than re-implementing it. Count flags (`-v`/`--verbose`) and boolean flags (and their `--no-` negations) are unaffected -- a repeat keeps its accumulate / last-one-wins / negation semantics. In the same consistency pass, an unrecognized `--log-level` value on `exchange` and the default zero-setup command now exits 64 with a clean usage message too (it previously threw a plain `Error` that escaped to the top-level handler as a stack dump and an undocumented exit code), matching `invite`/`accept`/`fingerprint`. Documented in `docs/CLI.md` ("Exit codes").
- Standardized linkage-key strings are now normalized to Unicode NFC before they enter PSI. NFC normalization runs unconditionally as the first step of every field's standardization pipeline -- including identity (no-step) and custom pipelines that never strip to ASCII -- so two parties holding the same logical value in different normalization forms (precomposed NFC vs decomposed NFD, the common macOS-filesystem vs Windows/most-database split) produce identical key bytes instead of silently failing to match. Previously the only normalization happened inside `remove_accents`, which is not guaranteed to run. `remove_accents` additionally re-normalizes to NFC after its NFD diacritic strip so no decomposed residue leaks into the key. The same guarantee covers config-supplied literals that are compared against or injected into the value (`null_if` values, a `coalesce` default, a `replace_regex` replacement, a `pad_left` character) and the fully assembled key string, which is normalized once more after its elements are concatenated; regex patterns are matched as authored and are not normalized.
- Mid-pipeline steps that match an authored value, pattern, or delimiter against the intermediate value (`null_if`, `filter_regex`, `extract_regex`, `replace_regex`, `split_on`, `parse_date`) now NFC-normalize the value they inspect before matching. `to_upper_case` can emit non-NFC bytes for six code points (e.g. the Greek `U+0390`), so a comparison authored correctly in NFC but placed after a case-fold could previously miss -- silently letting an excluded value survive into the PSI set, or wrongly dropping a record whose filter, extraction, replacement, or split should have matched. The final assembled-key NFC pass added above only fixes the emitted key, which is produced after these mid-pipeline reads. Output for already-canonical and pure-ASCII inputs is byte-identical (`null_if`/`filter_regex` return the original value; the steps that derive a new value derive it from the normalized form). Follow-up to the Unicode NFC normalization above.
- `connection.provider_options` keys are now passed verbatim, as the spec now documents. Previously the config writer snakeized them to disk and the reader camelized them back, so a key authored in the casing the transport library expects -- e.g. ssh2's `readyTimeout` -- was rewritten (`ready_timeout`) rather than preserved. The map's contents are excluded from key-case transformation on both the write and read paths; all other schema fields, including function `params`, are normalized as before.
- A malformed message from a peer is reported as a protocol error instead of crashing.
- A failed exchange surfaces its original cause instead of a generic connection error.
- Closing a file-sync connection now cancels any in-flight rendezvous or send wait promptly instead of letting the timer fire and resume against a connection that is tearing down. Internal hardening only: cancellation threads a single `AbortController` through every wait site, and a new internal `ConnectionClosedError` may appear in debug logs on a close-during-wait. Not a user-facing exit-code change (a deliberate close under a signal still exits 130/143).
- Wave-path rendezvous: a party whose hello is deleted by a joiner that then fails mid-arrival no longer stalls until the peer timeout. The joiner now signals its arrival with a `<id>-joining.json` sentinel, and the waiting party recovers within a bounded window or aborts with a distinct transport error.
- File-sync rendezvous now sweeps an orphaned in-flight temp file left in the exchange directory by a `send()`/`writeAck()` whose process was hard-killed between the temp write and its atomic rename. The artifact is deleted at rendezvous setup instead of accumulating as litter across crashed exchanges or aborting entry with a spurious usage error. The sweep matches only the protocol's own temp shape -- `temp-<uuidv4()>.tmp`, the exact name `send()`/`writeAck()` write (its stem validated as a v4 UUID) -- never the `*.json` message files (so retain mode's transcript is untouched), and never a foreign `temp-*.tmp` whose stem is not a v4 UUID: such a file (a user or sync-tool `temp-export.tmp` in a namespace collision) is treated as a foreign file and tolerated, consistent with the session-start directory hygiene above, rather than deleted. The mid-exchange recognition checks (`isRecognizedLoopFile` and the protocol-grammar classifier) recognize the same `temp-<uuidv4()>.tmp` shape, so a foreign `temp-*.tmp` that appears during the loop is handled by `unexpected_files` rather than mistaken for an in-flight write.
- The SFTP adapter now retries a transient `rename` failure instead of letting a momentary server `SSH_FX_FAILURE` abort the exchange. The file-sync transport publishes every atomic write as a temp-or-sentinel file renamed to its final name (the rendezvous joiner's `<id>-joining.json` -> `<id>-hello.json`, `send()`'s message publish, `writeAck()`'s ack), so the missing retry surfaced as an intermittent `_rename: Failure` crash in the mixed-connection rendezvous under back-to-back build, unit, and integration load. `rename` was the lone mutating SFTP op without a retry. The retry is bounded the same as `put` (one initial attempt plus up to five re-issues, 100 ms apart) but, because `rename` is not idempotent the way `put` is, fires only on the generic `SSH_FX_FAILURE` (status 4) -- the code that reports the rename did not take effect, so the source still exists and a re-issue is safe. Every other status is terminal and surfaces at once, including `SSH_FX_NO_SUCH_FILE`, which a second attempt would see if the first rename had actually succeeded but its reply was lost; retrying that would manufacture a spurious error from a successful operation.
- `MessageConnection`: a clean half-close that defers behind buffered inbound frames now tears the transport down at half-close time instead of only when the buffer is later drained, so a consumer that abandons the half-closed connection without draining or closing it no longer leaks the transport's listeners/channel. Internal hardening collapses the connection's terminal lifecycle into a single explicit state, so each method's terminal behavior derives from one source of truth rather than from four hand-duplicated flags. No happy-path or API-surface change for a correct caller; two intended changes land on abnormal/teardown paths: teardown now runs at half-close call time, and an already-buffered frame is drained before any abnormal terminal error surfaces (previously a frame buffered behind an abnormal drop was discarded), uniformly across transports.
- An already-expired (or malformed) shared secret now fails before any connection is opened, with the correct "obtain a new invitation" recovery hint on both parties. The pre-handshake secret-precondition check (well-formed base64url 32-byte value; not past `expires`) is knowable from local state alone, but it ran only inside `authenticateConnection`, which executes after the file-sync rendezvous -- so a dead credential first drove the rendezvous, and the losing side of the lock-file race could be shown a misleading `peer appears to have abandoned the handshake ... Retry the exchange` error (retrying never helps an expired token) depending on which side tore down first. `runProtocol` now runs the check in its pre-flight, ahead of `connect()`, so neither party enters the rendezvous, both reject deterministically with the specific re-invite hint, and no hello/lock files are written. The check is extracted into a single shared `assertSharedSecretReadyForHandshake` helper in `@psilink/core` that both `runProtocol` and `authenticateConnection` call, so there is one definition rather than a re-implemented copy; `authenticateConnection` keeps it as the authoritative boundary for library consumers and retains its separate post-handshake expiry check (for a secret that expires during the key-exchange round-trip). This also removes a timing-dependent unit-test flake: the two-party expired-token test asserted both sides reject with `expired`, which the rendezvous race intermittently broke (~1 in 10) by surfacing the abandoned-handshake message on the loser instead.
- `writeFileOwnerOnly` and `writeFileAtomic` (the owner-only and public-artifact atomic writers in `apps/cli/src/fileUtils.ts`) now `fsync` for power-loss durability, through a shared `fsyncParentDir` helper so the two stay consistent rather than diverge. The temp file's data is flushed before the rename and, on POSIX, the parent directory is flushed after it, so a crash cannot leave the rename durable while the file's contents are lost; and because each call flushes its own directory entry before returning, two sequential `writeFileOwnerOnly` calls are crash-ordered -- if the second call's rename survives a power loss, the first call's rename and contents do too. Previously both helpers wrote and renamed with no `fsync`, so filesystem metadata reordering after a power loss could surface a later rename while losing an earlier one -- defeating the self-attested exchange record's crash-ordering invariant (it writes the private opening file before the summary record, see `apps/cli/src/recordFile.ts`, so a crash between the two preserves the proof material) and able to drop a freshly rotated shared-secret token in `saveKeyFile`. The fix lives in the shared writers, so it covers the key file, config, exchange records, signing identity, and exported public certificate alike. The data flush runs on every platform -- `writeFileOwnerOnly`'s Windows branch reopens the ACL-narrowed placeholder to write the content and `FlushFileBuffers` it through a retained fd, matching `writeFileAtomic` -- but the directory entry's durability is left to the OS on Windows: Node's `fs` cannot open a directory handle to `FlushFileBuffers` it, so the cross-call crash-ordering guarantee is POSIX-only and NTFS metadata journaling governs the directory there. On macOS, Node issues `fsync(2)`, not `F_FULLFSYNC`, which flushes the OS to the drive but does not force the drive's volatile cache to media or stop the drive reordering writes, so there the crash-ordering holds against process death but not necessarily a true power loss (recoverable by re-running); Linux, the production target, flushes durably. Documented at the writers and in `docs/SECURITY_DESIGN.md` ("Required permissions"). A durability and auditability fix, not a confidentiality one -- nothing leaks and a lost write is recovered by re-running. Follow-up to the POSIX temp-file symlink-follow hardening of the same writers.
- Empty-linkage-round desync in the multi-key PSI matching loop (`packages/core/src/link.ts`). The loop skipped a key whose local de-duplicated value set was empty (`if (data_j.length === 0) continue;`), but that emptiness is computed per party from each side's own usable values, so on a later key (`j > 0`, where the set is built only from a party's still-unmatched records) one party could skip a round the partner still ran -- the partner then called `identifyIntersection` and blocked indefinitely on a `receive()` the skipping party never answered. Concretely it triggered under one-to-one / many-to-one cardinality when one party was fully matched in an earlier round while the other still held unmatched records. The skip is removed so both parties run `identifyIntersection` for every agreed key (the key set is fixed by the linkage terms, and the PSI library returns an empty intersection without error for an empty input set, so a locally-empty round is a correct no-op), keeping the exchange in lockstep by construction rather than by coincidence of equal data. This pre-existing bug is independent of the AEAD layer; its resolution restores the unconditional lockstep premise that the channel-security truncated-tail deferral reasons from (see `docs/SECURITY_DESIGN.md`, "Channel security").
- The CLI now percent-decodes the host, path, username, and password when building a connection config from a server URL, in both URL-to-config builders (the `invite`/`accept` bootstrap path and the zero-setup `--save` path). The WHATWG `URL` parser keeps these components percent-encoded -- a space stays `%20`, an internationalized-domain host becomes UTF-8 escapes -- but `ssh2`/`ssh2-sftp-client` consume them literally, so a URL path `/my%20drop` previously opened a directory literally named `my%20drop` and a password `p%20w` was sent to SSH as the literal string `p%20w`. Both are now decoded to match every other URL-to-config path (`fileURLToPath` on the filedrop branch, a hand-authored `psilink.yaml`, and the accept-time reconcile comparison), fixing a silent wrong-target/wrong-credential connection that, on an online `accept` reusing a matching pre-existing config, could pass the reconcile "match" check yet fail SSH auth or `opendir` in the live one-time exchange. A malformed percent-escape (e.g. a lone `%`) is now rejected with a `UsageError` (exit 64) routed through the existing credential redaction so a malformed password is not echoed, rather than surfacing as an unhandled `URIError`. The bare-host guard (`sftp://host` and `sftp://host/` leave the remote path unset) now applies to both builders, and the shared decode/redaction helpers live in a new leaf module (`apps/cli/src/util/connectionUrl.ts`) so the two builders stay consistent.

### Security

- `@path` file references are now resolved only in the credential and opaque-options fields that support the convention, not blanket-recursively over every string in a loaded exchange config. `resolveAtSignRefs` rewrote any config value beginning with `@` into the contents of the file at that path, recursively over the whole config, contradicting the documented contract (`docs/EXCHANGE_SPEC.md`, `docs/CLI.md`) that the convention does not apply to free-text fields such as `linkage_terms.identity`, `linkage_terms.legal_agreement.purpose`, and `retention_disposition`. A free-text value that happened to begin with a literal `@` was silently treated as a file path: best case a confusing `ENOENT` aborted config parsing (exit 64) before any exchange ran, but if the path resolved to a real file (a templated or scripted config, or an `@~/...` home-dir expansion) that file's contents were pulled verbatim into the self-attested exchange record -- the artifact intended to be shared with an auditor or partner -- so a stray `@` exfiltrated an unrelated local file into a disclosure artifact the docs affirmatively promised it could not. The CLI now resolves `@`-refs after schema validation through a new `resolveExchangeSpecRefs` (`apps/cli/src/util/atSignRefs.ts`) scoped to the documented `@`-file fields, all under `connection`: the SFTP `server.password`, `private_key`, and `private_key_passphrase`, the HTTP-auth `bearer` / `password` on every provisioning endpoint (`server.provision`, `proxy`, `ice_provision`), each WebRTC `turn[].credential`, and the opaque `provider_options` map. Every other field keeps a literal leading `@` verbatim, including the local-path fields whose consumer opens the path (`signing.identity_file`, `signing.receipt_output`), where resolving to file contents would corrupt the value. Severity: low, pre-existing, and not an external-attacker vector -- self-inflicted, the operator's own config on the operator's own machine -- but the docs promised an exemption the code did not honor. Surfaced by the security review of the retention/disposition pointer work; the persistence-side sibling (keeping an `@path` out of a saved config) is the separate entry below. Specified in `docs/EXCHANGE_SPEC.md` and `docs/CLI.md` ("File references" / "Configuration").
- Empty-id hello and joining sentinels are now rejected in the file-sync rendezvous scans, not only at the entry guard. The `synchronize()` entry guard already classified a bare `-hello.json` (a hello name whose recovered peer id slices to `""`) as an unexpected protocol file, but the in-flight lock and lockless RENDEZVOUS peer-hello scans, and the lock-path joiner-arrival (`joinerRecoveryMs`) handling, recovered the peer id by independently slicing `<name>-hello.json`/`<name>-joining.json` without the non-empty check. So a `-hello.json` injected after entry while rendezvous was mid-flight could be adopted as the peer hello, committing rendezvous to `peerId=""` -- after which `poll()` keys its peer-message scan on `${peerId}-` (= `-`) and the lockless ack barrier waits for an ack no honest peer writes, producing a hang or abort -- and a bare `-joining.json` could separately force the bounded (~30 s) `joinerRecoveryMs` starter stall. A shared `peerIdFromControlName` helper now recovers the id and rejects an empty one at every slicing site, routed through `isPeerHelloName`/`isPeerJoiningName` so the entry guard and the in-flight scans share one definition of a valid peer hello/sentinel; the post-rendezvous prefix-collision guard additionally fails closed on an empty committed `peerId`, and `poll()` already rejected an empty `peerId` before scanning. Severity: medium, a denial-of-service / rendezvous-hijack vector bounded by the file-sync transport being unauthenticated by design -- anyone with shared-directory write access can already disrupt an in-flight exchange, and PSI integrity is enforced at the application layer, not the transport. This is defense-in-depth hardening, not an authentication boundary. Specified in `docs/FILE_SYNC.md` (rendezvous role/collision logic, site 5; invariant I0).
- Operator-facing diagnostics now escape untrusted partner strings. `validateCompatibility` (`packages/core`) builds human-readable mismatch errors that interpolate partner-supplied values -- the legal-agreement `reference`, `purpose`, and `expiration_date`, the payload column names, and the version/algorithm/date fields -- which surface to the local operator over the CLI (`console.error`/logs) and the web error alert. A hostile or misconfigured partner (a mutually-distrusting remote party) can place terminal control characters or ANSI escape sequences (log/terminal spoofing), bidi-override (RLO/LRO), zero-width, or homoglyph/confusable Unicode, or a pathologically long value in any of these free-text fields. A new shared `sanitizeForDisplay` helper in `@psilink/core` rewrites every code point outside printable ASCII to a visible `\xHH`/`\uHHHH` escape (neutralizing controls, ANSI, and deceptive Unicode at once) and truncates to a bounded length with a visible marker; every partner value interpolated into a `validateCompatibility` diagnostic now routes through it (the equality checks still compare the raw bytes, so mismatch detection is unchanged), as do the partner's self-asserted `identity` logged by the CLI, the partner-supplied abort reasons surfaced on a terms-exchange abort, the hostile-filename preview in the directory-listing-bound error, and -- on the invitation accept path -- the inviter-supplied identity and linkage-key names shown in the acceptance summary and the unrecognized-endpoint-key names echoed in the invitation decode error. The schema-validation (Zod) parse-error message is deliberately left raw: it reports the expected type/options and the schema path, never the partner-supplied received value, so it carries no partner-controlled bytes and stays readable for an honestly malformed config. This is diagnostics/log-injection hardening, not a data-disclosure fix -- a mismatch aborts the exchange before any protected data is transmitted. The exchange record stores these fields byte-exact (required for its canonical encoding and the byte-exact cross-party validation); a forward-looking note on `ExchangeRecord` directs any future record viewer/exporter to route the same fields through `sanitizeForDisplay` at the display boundary.
- Invitation tokens carry a bounded lifetime (default 1 hour); rotation tokens carry no expiration by default, making them suitable for recurring scheduled exchanges (an operator may set `token_max_age_days` to impose a maximum age).
- `deriveAeadKey`'s `context` parameter is constrained to a fixed, ASCII-only set of per-direction labels (`AeadContext`/`AEAD_CONTEXTS`, currently `"initiator-to-responder"` and `"responder-to-initiator"`) instead of an open `string`, with a runtime guard for untyped callers. This prevents a future AEAD caller from deriving the per-direction key from a variable, non-ASCII, or non-NFC label, which would make the two parties derive different AES-256-GCM keys and fail AEAD with an opaque auth-tag/decrypt error instead of a clear cause.
- Inbound frame-size bound on the file-sync transport: a read of any inbound file larger than a fixed maximum frame size (536,870,888 bytes, ~512 MiB) is refused with a terminal `FrameSizeExceededError` before the file is loaded into memory, closing a memory-exhaustion denial of service in which a hostile SFTP/filedrop server admin writes an arbitrarily large ciphertext file. Enforced at the transport read layer: the poll loop and rendezvous gate refuse over-cap files before `get()`, and each `FileTransportClient` adapter enforces a hard per-read byte cap (the local-FS adapter `fstat`s the open handle and reads exactly that size; the SFTP adapter aborts the transfer once a counting sink crosses the cap) so a server under-reporting size in its directory listing is still bounded. The maximum is anchored to Node's `MAX_STRING_LENGTH`, the ceiling above which the existing `JSON.parse(buffer.toString())` read cannot process a frame regardless; the AEAD decorator's `{ enc }` envelope schema caps the ciphertext at the same value as documented defense-in-depth. Specified in `docs/SECURITY_DESIGN.md` ("Channel security").
- Directory-listing bounds on the file-sync transport, the directory-enumeration sibling of the frame-size bound: a `list()` of a rendezvous directory with more than 8,192 entries, or any entry whose filename exceeds 255 characters, is refused with a terminal `DirectoryListingBoundsError` (a `UsageError`, exit 64) before the listing is materialized, closing a memory-exhaustion denial of service in which a hostile SFTP/filedrop server admin floods the shared directory with many files or very long names. Enforced at the transport `list()` layer in both adapters: the local-FS adapter streams entries through `fs.opendir` instead of `fs.readdir`, and the SFTP adapter reads one server batch at a time through the low-level `opendir`/`readdir`/`close` protocol instead of ssh2-sftp-client's `list()`, so an oversized or hostile directory is refused before an array proportional to the attacker-chosen entry count is allocated. The 8,192-entry cap is derived from a memory envelope (~5 MiB worst case, two orders of magnitude below the frame-size budget) and the 255-character cap is the POSIX `NAME_MAX`; both are fixed, not configurable. Specified in `docs/SECURITY_DESIGN.md` ("Channel security").
- Transport liveness bounds, the non-termination sibling of the frame-size and directory-listing memory bounds, in two layers plus an observability signal. The per-operation **read fast-fail layer (SFTP)**: every SFTP read awaits a server-driven callback, so a hostile server admin can hang it forever -- a `list()` that returns empty (`count = 0`) non-EOF readdir batches without end or never calls back, a `get()` that withholds data or never ends the stream, or a `createExclusive()` that withholds its open/close callback. Each is bounded and surfaces a terminal `TransportOperationStalledError` (a `UsageError`, exit 64), which both the poll loop and the rendezvous control-file gate treat as terminal (the gate now rethrows any typed `UsageError` from the hello `get()`, not just the over-cap `FrameSizeExceededError`, so a stalled hello read fails in a minute rather than being retried until the one-hour peer budget), instead of awaiting forever (and leaking any handle opened before the stall): `list()` by a 16,384 readdir round-trip cap (twice the 8,192-entry size cap) plus a 60-second wall-clock deadline, closing the directory handle on the way out; the capped `get()` by a 60-second per-chunk idle window, reset on each chunk so a slow-but-progressing large transfer is not failed; and `createExclusive()` and the rarely used uncapped `get()` by a 60-second whole-operation deadline. The memory bounds do not cover this vector (a progress-free stream never grows an allocation). The 60-second budget is roughly 3x the SSH stack's 20-second connection-unresponsiveness threshold and more than an order of magnitude below the one-hour peer-inactivity budget. All fixed, not configurable. The **whole-exchange backstop (universal)**: the per-operation bounds cover only the read path, leaving the always-executed write/stat/delete ops (`put`/`rename`/`delete`/`exists`) and the local-filesystem adapter unbounded, so a server that withheld the callback on a `put` or `delete` still hung the exchange forever -- `synchronize()` and `send()` awaited the call directly so it never returned, and the poll loop's reschedule sits in a `finally` a hung await never reaches, so the poller stopped silently; the peer-inactivity budget was checked only between loop iterations, never over an in-flight await, so a single withheld callback escaped it. The connection now races every transport await -- reads, writes, stat, delete, and the local-filesystem path alike -- against a fresh `peer_timeout_ms` budget at the single consumer seam they all flow through, surfacing the same terminal `TransportOperationStalledError`. Because it wraps the transport interface rather than any one call, it is op-agnostic (covering ops not individually enumerated and any added later) and adapter-agnostic (it also closes the local-filesystem stalled-mount gap, where the earlier "trusted kernel" premise fails for a hung NFS/CIFS mount, and the capped-`get()` slow-drip, where under-cap bytes trickled forever never trip the per-chunk idle window but do cross the budget). It bounds a stalled await against a fresh budget per await, not total exchange duration, so it never caps a healthy long-running exchange. A non-fatal **slow-operation warning** (observability, not a security control) logs once at a fixed 30-second threshold (below the 60-second read fast-fail) naming the operation, elapsed time, and cheap observed progress where available (bytes-so-far and rate for `get`/`put`, entries for `list`, elapsed-only for the atomic ops); it does nothing headless, stays entirely outside the terminal-error paths, and adds no required config. Specified in `docs/SECURITY_DESIGN.md` ("Channel security").
- POSIX temp-file symlink-follow hardening for the shared owner-only and atomic file writers (`apps/cli/src/fileUtils.ts`): `writeFileOwnerOnly` and `writeFileAtomic` now create their sibling temp file on an exclusive, non-following descriptor (`O_CREAT | O_EXCL | O_WRONLY | O_NOFOLLOW`) and set the mode via `fchmod` on that descriptor, mirroring the create-then-restrict discipline the Windows branch already used (an `O_EXCL` placeholder, then ACL restriction). Previously the POSIX path unlinked the temp path and then `writeFileSync`'d it with neither `O_EXCL` nor `O_NOFOLLOW`, so an attacker who planted a symlink at `<destPath>.tmp.<pid>` in the unlink-to-write window could make the write follow the link and write through to its target -- a temp-file TOCTOU / symlink-follow. `O_EXCL` refuses to open through an existing entry at the temp path and `O_NOFOLLOW` refuses a final-component symlink, so the create now fails cleanly rather than following the link, and the explicit `fchmod` preserves the prior umask-correcting intent (exact `0600` for the rotating `.psilink.key` and the signing identity, the explicit mode for the exported certificate) on the descriptor rather than on a resolved path. Real-world exploitability is low -- the temp file is a sibling of the destination, so planting the symlink already requires write access to the directory the `0600` mode assumes against -- and the writer is the shared chokepoint for the key file, config, exchange records, signing identity, and exported certificate, so the fix covers all of them.
- SFTP wrapper-crash bound, the one hostile-server behavior the liveness bounds above did not cover (a crash is neither memory nor time exhaustion): the SFTP adapter drives the raw ssh2 `SFTPWrapper` directly, and after connect that wrapper carries no `'error'` listener -- ssh2's `Client.sftp()` strips its setup-time listener before returning the wrapper, and `ssh2-sftp-client` attaches `'error'` handlers only to the SSH `Client` and to per-operation streams. A server returning a malformed SFTP reply packet (NAME, HANDLE, DATA, an unknown type, or a duplicate VERSION) drives ssh2's fatal-error path to emit `'error'` on a listener-free `EventEmitter`, which Node escalates to an uncaught exception that crashes the CLI -- skipping the lock/temp-file cleanup and typed exit-code mapping an orderly failure runs. The adapter now attaches a single guarded `'error'` listener to the wrapper at connect time (once per wrapper instance: a reconnect's fresh wrapper is re-guarded, a repeated connect does not duplicate it), so a fatal protocol error is handled rather than fatal. The session is left dead but the process alive: ssh2's `doFatalSFTPError` emits, destroys the wrapper, then calls `cleanupRequests`, which fails every request still registered -- but a malformed reply to the in-flight request itself is deleted from ssh2's pending table before the fatal path runs, so `cleanupRequests` does not fail it and that in-flight `list()`/`get()` is bounded by the 60-second wall-clock deadline (it is a fatal error on a different request id, or a connection-level fault, that `cleanupRequests` fails promptly); the captured cause then bounds the NEXT operation, which short-circuits at entry with the same terminal `TransportOperationStalledError` instead of buffering a request the dead channel can never answer. Every server-driven operation consults that guard -- `list`, `get`, `createExclusive`, `put`, `delete`, `rename`, and `exists` reject promptly, and `safeDelete` (which must never reject) resolves promptly as a best-effort no-op -- which matters because after a fatal error the SFTP channel is destroyed while the TCP/SSH socket stays up, so an unguarded write/stat/delete would buffer on the closing channel and hang teardown (`FileSyncConnection.close()` drives `safeDelete`/`delete`) rather than crash. For the distinct case of a server that withholds a callback WITHOUT any protocol error, no fatal `'error'` fires and the entry guard never trips: the read paths stay bounded by their 60-second deadlines, and the write/stat/delete stall (`put`/`delete`/`rename`/`exists`, which carry no per-operation deadline) is bounded by the whole-exchange backstop above. Specified in `docs/SECURITY_DESIGN.md` ("Channel security").
- Operator-facing file-sync transport diagnostics now escape partner- and server-controlled filenames and the peer-derived peer id, extending the `sanitizeForDisplay` hardening from the PSI-protocol/config and invitation surfaces to the transport layer. The directory-listing bound caps a filename's length (255) and the directory entry count but not its charset, and the protocol-filename grammar constrains only the suffix and trailing digit segments, not the id portion -- so a name carrying terminal control characters, ANSI escape sequences, CR/LF (log-line injection), or deceptive Unicode (bidi-override RLO/LRO, zero-width, homoglyph/confusable) passes every existing guard whether it is a foreign file or a grammar-valid protocol hello, and the peer id sliced from a hello filename prefix carries the partner's bytes verbatim. Every interpolation of such a value into a `log.*` line or a thrown `UsageError`/`Error` in `FileSyncConnection` (`@psilink/core`) now routes through `sanitizeForDisplay`: rendezvous (lock and lockless), the background `poll()` loop, the foreign/unexpected-file handler (the highest-priority live vector -- a foreign name passes every guard and was interpolated raw into the terminal error), the `--sweep-exchange-files` sweep, `send()`'s retain-mode ack wait, the close-time cleanup log, the retain-mode signal inspection, the control-file read gate, the whole-exchange peer-inactivity budget error (whose operation target is a transport path that, on a stalled `get`/`delete` of a peer message file, embeds the partner filename -- the core-side twin of the CLI adapter's per-operation stall error), and -- because V8's `JSON.parse` error quotes a span of the offending input -- the JSON-parse and schema-validation error text thrown by `poll()`'s peer-message-body parse, whose body is open peer-controlled JSON (`payload: z.json()`). The CLI SFTP guard error builders are routed at the builder so every caller is covered uniformly: `frameSizeExceededError`, the shared `transportOperationStalledError` that backs the capped-read/write stalls and the dead-session and listing-stall errors, the `withSlowOperationWarning` observability log, and `directoryTooLargeError` (`filenameTooLongError` already escaped its name). Only the display copy is escaped -- values compared, stored, or used to build on-disk paths stay byte-exact, so classification and delete targeting are unchanged. Because both the SFTP and filedrop adapters concatenate the operation path into their own error text, a raw transport error's message can itself carry a peer filename; every place this code interpolates such a message into a log line or thrown error it composes (the cleanup, `--sweep-exchange-files` failure list, `poll()` delete-failure, and superseded-buffered-error paths) is therefore escaped too. Left raw, by contrast, are the local party's own configured id and rendezvous path (operator-controlled, not partner-supplied) and the control-file read gate's schema-validation error (its schema is a closed two-`z.boolean()` `.strip()` `HelloEnvelopeSchema` whose zod error names only expected types and fixed field paths, never a peer value). One residual is deliberately deferred: a raw transport error OBJECT propagated unchanged on the connection's `error` channel or rethrown can carry a partner-controlled path in its message, but neutralizing it cleanly belongs at the operator-facing display boundary (the CLI error printer and the web error alert) rather than by rewrapping every emitted error, which would break the error-type classification the message layer depends on -- tracked as a follow-up. This is diagnostics/log-injection hardening that requires a hostile or broken peer/server and discloses nothing (a mismatch or stall aborts before any protected data is transmitted); it is the transport-layer sibling of the per-field sanitization on the PSI-protocol/config and invitation surfaces and supersedes the earlier CLI-only control/ANSI scope.
- Operator-facing error rendering now escapes untrusted partner/server strings at the display boundary, closing the residual the file-sync transport-diagnostics hardening above deliberately deferred (a raw transport error OBJECT propagated unchanged on the connection's `error` channel or rethrown, whose `.message` -- and, on the filedrop channel, the partner-chosen message-file path it embeds -- reached the operator unescaped). A new shared `sanitizeErrorForDisplay` helper in `@psilink/core` renders an error's message and every chained `cause` message through `sanitizeForDisplay`, reading only `.message`/`.cause` (never `.stack` or arbitrary properties, so no stack frame or credential-bearing field can leak), walking the chain cycle-safe and depth-bounded, suppressing the wrapper/cause duplicate `asConnectionError` produces, and never throwing on a hostile or malformed error (an unreadable link renders as `[unreadable error]`). It is applied at every operator-facing error render: the CLI top-level catch (`apps/cli/src/index.ts`, which previously printed `console.error(err)` -- Node's raw stack and cause dump -- and now renders the sanitized message and cause chain with stack frames dropped), the web exchange-error alert (`apps/web/src/components/Exchange.tsx`, a React-escaped text node so HTML/markup injection was never possible; this closes the deceptive-Unicode and control-byte residual), and -- the common CLI failure path -- every per-command catch (`exchange`, `zeroSetup`, `invite`/`accept` via `runOrExit`, `fingerprint`) plus the `runProtocol` signal-path in-flight log, which render `log.error(...)` then exit before reaching the top-level catch and so are a print path the top-level seam does not cover. In the same pass those per-command catches now also surface the sanitized `cause` chain (previously only the outermost `err.message` was printed), so an operator sees the real underlying reason an exchange failed -- e.g. a read-only filesystem one or more links down -- without re-running. Only the displayed copy is escaped: the error instance that flows through the transport/message layers is untouched, so type-based classification (`ConnectionError.kind`, `instanceof UsageError`) and the per-command exit codes are unchanged. Two adjacent operator-log hygiene gaps are closed alongside: partner-adopted linkage key names rendered in the exchange-time "will link using keys" log and the high-fan-out standardization warning (`packages/core`), and three transport-error debug logs in `FileSyncConnection` that rendered `errMessage(...)` raw while their siblings escaped it. This is diagnostics/log-injection (terminal-spoofing / log-line-injection) hardening, not a data-disclosure fix -- a transport failure aborts the exchange before any protected data is transmitted -- and it completes the operator-facing-render sanitization class begun on the PSI-protocol/config and invitation surfaces and extended to the transport layer. A separate follow-up tracks the remaining lower-severity sinks (devtools-console error logging and the CLI debug/warn paths that pass a raw Error object).
- Untrusted invitation-token fields are now bounded at the shared core decode path, the invitation-surface sibling of the transport frame-size and directory-listing bounds. The invitation checksum detects transcription errors only and is computable by anyone (it carries no authenticity guarantee), so a crafted-but-checksum-valid token could pass `decodeInvitation` (`packages/core/src/config/invitation.ts`) carrying a multi-megabyte `identity`, tens of thousands of `linkageKeys`, or other arbitrarily large partner-controlled free-text, and be accepted and rendered -- briefly janking the victim's browser tab on decode/render. Separately, the WebRTC endpoint `host` was validated only as non-empty, so a malicious invitation could aim the victim's own browser's PeerJS signaling WebSocket at an attacker-controlled host (SSRF-shaped only: the attacker already holds the invitation and is by definition the rendezvous peer, and the endpoint is a credential-free locator by the schema's existing invariant, so no secret is sent there). Both are now closed in the one place the CLI and the web app share: a decode-boundary length cap (`MAX_ENCODED_INVITATION_LENGTH`, 64 KiB) refuses an oversized encoded string before it is base64-decoded, JSON-parsed, or schema-validated -- the single bound that transitively caps every field, and which `encodeInvitation` also enforces on its own output so psilink never emits a token it could not itself decode -- and generous per-field `.max()` bounds back it as defense-in-depth on `identity`, the legal-agreement `purpose`, the payload column `description`, and each constraint `exclude` value (1 KiB), the `linkageFields` / `linkageKeys` counts (256 entries), the linkage key / field / element `name`, element `field`, element-`swap` reference, transform `function` and its `params` keys, payload column `name`, legal-agreement `reference`, the `version` string, and the name-constraint `allowedCharacters` class (256 characters), the WebRTC and SFTP endpoint `host` (256 characters, just above the 253-character DNS limit), and the WebRTC / SFTP / file-drop endpoint `path` (4096 characters, POSIX `PATH_MAX`). The deeper collection counts (per-element `transform` steps, each `exclude` list, the `payload` send/receive arrays, and the `params` record's entries), plus the `params` value content (untyped), are left to the boundary length cap rather than an invented per-field count, since their legitimate sizes vary. The host bound is length-only by design: a strict hostname/IP regex risks rejecting a legitimate but unusual locator (an IPv6 literal, an internal name, a punycode IDN), and the finding asks only for a length/format bound. Every bound is generous enough that no real invitation any current inviter produces is rejected -- legitimate tokens decode unchanged, with no happy-path behavior change -- and no token any prior encoder legitimately produced is rejected either, because every bound sits far above any real value; tightening an existing field this way needs no format-version bump, unlike adding a field to a strict endpoint sub-schema (the contrast the `version` policy in `invitation.ts` draws). Severity: low, defense-in-depth -- the impact is a self-inflicted tab-jank on a victim who opened the attacker's link, bounded by the browser's URL-length limit, not a crash, RCE, or live exposure. Both findings are from the final security review of the web rendezvous work; neither was a merge blocker.
- Persisted CLI configurations now keep an `@path` credential reference instead of inlining the secret. When a credential is supplied as an `@`-file reference (for example `--server-password @secrets/pw` or `--server-private-key @~/.ssh/id_rsa`) and the resulting connection is written to `psilink.yaml`, the saved file records the original `@path` string, not the file contents -- consistently across every persistence site: `--save` on the zero-setup exchange and the `invite`/`accept` provisioning path. Previously the reference was resolved during argument parsing, so every save inlined the raw credential, defeating the indirection the user chose to keep the secret out of process listings and the config file and contradicting the documented promise that the config is safe to commit. A literal (non-`@path`) credential is still persisted as-is. The reference is resolved at the next `psilink exchange`'s config load, before any network activity -- the same point a hand-authored `@path` already resolves -- so a moved, deleted, or unreadable referenced file fails that load with a usage error (exit 64) naming the reference, rather than partway into opening the connection. The string is stored exactly as typed, so a `~`-relative reference stays portable across machines while a relative reference resolves against the working directory of the later exchange (use an absolute or `~/` reference if that exchange will run from a different directory). Implemented by carrying the reference through to the persistence boundary and resolving it only at the live-use boundary (`resolveConnectionCredentials` in `apps/cli`), so the secret is never written to disk; the read side is unchanged. Specified in `docs/CLI.md` ("Configuration").
- The CLI operator-terminal sinks that the error-rendering entry above deferred are now escaped, completing the CLI half of that follow-up. In `apps/cli/src/protocol.ts` the connection log routes the partner-seeded SFTP host (`connection.server.host`) and the filedrop path (`connection.path`) through `sanitizeForDisplay` -- on an offline-accept-seeded config both come from the partner's invitation endpoint and are charset-unconstrained -- and every cleanup/debug/warn catch that previously passed the raw `Error` OBJECT to the logger (the `secure.close()`/`mc.close()`/`conn.close()` cleanup catches, the `onSigint`/`onSigterm` cleanup-threw catches, and the post-open signal-close catch) now renders `sanitizeErrorForDisplay(err)` instead, which both escapes any partner/server bytes the message carries and drops the `.stack` dump `util.inspect` prints for an `Error` argument under `-v`. Only the displayed copy is escaped; the value dialed, compared, or stored stays byte-exact. The operator-facing web accept-page renders this work originally also named are already handled by the accept-consent-screen restructure -- `apps/web/src/psi/invitationSummary.ts` routes the inviter identity, linkage-key names, legal text, and payload column names through `sanitizeForDisplay` at one boundary -- so no web change is needed here. The remaining lowest-severity tier, the web devtools-console logs, stays tracked as a separate follow-up.

- Pinned the `EncryptedMessageConnection` AEAD message-envelope wire format with a checked-in cross-implementation known-answer vector: `packages/core/test/vectors/aead-envelope-vectors.json` fixes a session key, role, sequence number, and plaintext and records the exact serialized `{ enc }` envelope (base64url of IV || ciphertext || 16-byte GCM tag) for both a JSON and a binary payload, so an independent implementation can reproduce the bytes and any change to the nonce construction, byte order, envelope shape, or tagging is caught. Specified in docs/SECURITY_DESIGN.md ("Channel security") and asserted against the decorator in `packages/core/test/encryptedMessageConnection.test.ts`, alongside (and distinct from) the existing `deriveAeadKey` key-derivation vector.
- Documented the transport delivery contract - a send completes on local hand-off rather than peer delivery, and a clean close flushes the final frame before teardown - in docs/COMMUNICATION.md and the `Connection` and `TransportHooks` interfaces, stating the guarantee any future channel must satisfy.
- Recorded two close-teardown decisions in docs/COMMUNICATION.md (cross-referenced from the `ConnectionErrorKind` JSDoc and the web PeerJS clean-close wiring) now that the inbound drain is uniform in the shared message queue: the error-kind decision (a clean remote close stays kind `"transport"` rather than being unified with a local close's `"closed"`, because a remote channel-close event does not convey the peer's intent and the key exchange relies on the `"transport"` tag to report a vanished peer as a handshake timeout) and the narrowed per-transport residue (each transport wires which terminal control its events call - WebRTC maps `close` to `finish()` and `error` to `fail()` - while a shared `Connection.close` event is rejected as the wrong shape because it would reintroduce the no-listener drop window the pull-based queue eliminates).
- Corrected the non-repudiation receipt design in docs/PROTOCOL.md ahead of implementation: the receipt is a post-exchange audit artifact produced after the result already exists on both sides and does not gate or withhold result delivery; the session-derived (symmetric) signing mode is tamper-evident only and does not provide non-repudiation (its MAC is forgeable by either party), so true non-repudiation requires the certificate-backed asymmetric mode; and the receipt swap is best-effort evidence, not a fairness or atomicity guarantee. Clarified in docs/EXCHANGE_SPEC.md that `linkage_terms.identity` is self-asserted and carries evidentiary weight only when bound to a certificate.
