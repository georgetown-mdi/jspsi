# Changelog

This changelog records, per release, the changes that affect how PSI-Link is run, deployed, or reviewed: new commands and options, changed defaults and behavior, breaking changes, and security-relevant changes. It is not a log of every commit (that is the git history), and it omits internal refactors and test, CI, and tooling changes that no operator or reviewer acts on. Each entry states the observable change and points to `docs/` for the rationale and `docs/spec/` for wire-level detail. The format loosely follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); PSI-Link uses [semantic versioning](https://semver.org/).

## [Unreleased]

### Added

- The console-appliance server job API now re-discovers completed jobs after a restart: the console can list prior jobs and download each one's result, exchange record, and verification keys read-only, served through the same gate as a live job. The job table (running state and event history) is still memory-only and no in-flight exchange is resumed -- only completed results become re-listable, reconstructed from their on-disk artifacts (never the shared-secret key file or the connection config). An interrupted run is shown terminated, never resumed or re-run. See `docs/spec/SERVER_JOB_API.md`.
- `--event-stream` on every exchange-running command (the zero-setup exchange, `psilink exchange`, and the online `psilink invite`/`accept`): emit a machine-readable NDJSON event stream on file descriptor 3 for a supervising process -- a stage-list event, per-stage transitions, non-fatal warnings (terms warnings and the SFTP host-key divergence notice), and exactly one terminal result-or-error event whose category (`exchange`, `output`, `security`, `config`) lets a supervisor tell an authentication failure or a local output-write failure from a retryable transport fault. `stdout` (the CSV result) and `stderr` (human logs) are unchanged. Off by default; passing it without fd 3 wired fails fast (exit 64), and it is a no-op on an offline `invite`/`accept`. Also formalizes the CLI exit codes (0, 64, 69, 130, 143, 1). See `docs/CLI.md` and `docs/spec/CLI_EVENTS.md`.
- `--server-host-key-fingerprint` on every SFTP-connecting command: pre-pin the server's host key from the command line (OpenSSH SHA256 fingerprint; `@path` supported), so an unattended run connects without the interactive trust prompt while a server presenting a different key still fails closed. Equivalent to setting `connection.server.host_key_fingerprint` for that run. See `docs/CLI.md` (SFTP host-key trust).
- The web application can run as a single-party console appliance: a server-side job API drives the party's own `psilink` exchange as a subprocess so an operator creates, watches, and downloads a result without invoking the CLI by hand. Off by default and gated by `JOB_DATA_ROOT`; when enabled it requires either a loopback bind or a `JOB_API_TOKEN` bearer token, and the server refuses to start unauthenticated on a non-loopback bind. Job state is memory-only, so a restart cancels in-flight exchanges. See `docs/DEPLOYMENT.md` and `docs/spec/SERVER_JOB_API.md`.
- The console-appliance web build drives a shared-directory (`filedrop`) exchange in place through that server job API instead of only saving an exchange file. Built with `VITE_DEPLOYMENT_PROFILE=console`, the transport chooser offers to run a shared-directory exchange on the appliance, both composing an invitation and accepting one run server-side through the job API, and the file-assurance copy drops the browser-only claim accordingly. On success the browser retrieves the full result set from the appliance -- the result CSV, the shareable exchange record, and the private verification keys -- matching the downloads an in-browser exchange offers; the keys are served over the same gated, no-store, same-origin job API as the result. The default `hosted` build is unchanged: every shared-directory and SFTP exchange still saves an exchange file for the command-line tool. See `docs/DEPLOYMENT.md` and `docs/spec/SERVER_JOB_API.md`.
- The console-appliance web build also drives SFTP exchanges through the server job API, against remotes the operator provisions -- never ones the browser names. A new `JOB_SFTP_REMOTES` variable points at a mounted YAML file of named remotes; every entry must pin its server's host key and reference its credentials as `@path` files (inline secrets and unpinned hosts refuse startup), and the browser's job request carries only a remote's name. The transport chooser offers the provisioned remotes as a picker and authors the invitation's SFTP endpoint from the picked remote; with none configured -- and always on the `hosted` build -- SFTP still saves an exchange file. One exchange runs against a remote at a time, and non-fatal warnings from the run, including the host-key divergence notice, now surface in the exchange UI. See `docs/DEPLOYMENT.md` and `docs/spec/SERVER_JOB_API.md`.
- The published `vdorie/psi-link` image now bundles the web console appliance alongside the CLI: the default `docker run ... vdorie/psi-link <cli-args>` is unchanged, and `docker run -d -p 3000:3000 vdorie/psi-link serve` runs the single-party console (Nitro server on port 3000, console profile baked in; job API still gated by `JOB_DATA_ROOT`/`JOB_API_TOKEN`). See `docs/DEPLOYMENT.md` and `docs/RELEASES.md`.
- The PSI engine selects a native N-API backend when a prebuilt addon is present for the running platform under Node (Linux x64/arm64 glibc and musl, macOS arm64/x64, Windows x64), falling back to the WASM engine otherwise; the browser always uses WASM. It is a performance accelerator that emits byte-identical wire output, so results are unchanged. See `docs/spec/PROTOCOL.md`.
- `--polling-frequency` on the zero-setup exchange (`psilink URL INPUT_FILE`), `psilink exchange`, and the online `psilink invite`/`accept`: override the `sftp`/`filedrop` poll interval (`poll_interval_ms`, default `5s`). It is duration-valued (a unit suffix is required) and uniquely accepts a millisecond unit; a value below `1s` warns it may trip an SFTP server's anti-flood protection. Reported as ignored on an offline `invite`/`accept`. See `docs/CLI.md`.
- `connection.server.keyboard_interactive` (and `--server-keyboard-interactive` on the zero-setup exchange, `psilink exchange`, and the online `invite`/`accept`): answer the SFTP server's `keyboard-interactive` prompts with the configured `password`, for a server that disables the SSH `password` method but accepts the same password over `keyboard-interactive`. It requires a password (rejected without one, exit 64), is off by default, and is reported as ignored on an offline `invite`/`accept`. See `docs/EXCHANGE_REFERENCE.md`.
- `linkage_terms.linkage_strategy`: choose how the agreed linkage keys are run on the wire. The default `cascade` runs one PSI round per key; `single-pass` batches every key into one exchange, keeping the round-trip count constant in the key count for a high-latency `filedrop`/`sftp` channel. Both produce the same result. Both parties must agree or the exchange aborts. `single-pass` is a consented disclosure tradeoff (the receiver observes the sender's full per-key duplicate structure), not a default. See `docs/EXCHANGE_REFERENCE.md` and `docs/spec/PROTOCOL.md`.
- `--linkage-strategy cascade|single-pass` on `psilink invite` and the zero-setup exchange (`psilink URL INPUT_FILE`): select the linkage strategy from the CLI instead of hand-editing the terms file. Default `cascade`; an unknown value is a usage error (exit 64); on `invite` it is reported as ignored when terms come from a pre-existing config. `psilink accept` shows the chosen strategy on its consent prompt and aborts if a pre-existing config's `linkage_strategy` differs from the invitation's. In the zero-setup form both parties must pass the same value. See `docs/CLI.md`.
- The web "Advanced options" Expert authoring mode exposes the linkage strategy as a control (default `cascade`), round-tripped through import/export. A `single-pass` invitation carries the disclosure note onto the acceptor's consent screen, always-visible. See `docs/COMMUNICATION.md` and `docs/EXCHANGE_REFERENCE.md`.
- `psilink init [INPUT_FILE]`: write a commented `psilink.yaml` template to hand-edit before a first exchange, then exit without connecting or writing a key file. With an `INPUT_FILE` (a CSV, or `-` for stdin), column metadata, linkage fields, and standardizing transformations are inferred from it. An existing file at the output path is overwritten only after interactive confirmation. Invalid input exits 64. See `docs/CLI.md`.
- `psilink invite` and `psilink accept`: turn a one-off exchange into a recurring one backed by a shared secret. Each runs offline (write key and config files) or online (connect and run in one step). An expired or malformed invitation is rejected before you are prompted, and a pre-existing config or key at the target path is reconciled rather than clobbered. See `docs/CLI.md`.
- `psilink accept --consent-to-terms`: record advance consent to the invitation's disclosed terms and skip the interactive accept confirmation, so `accept` can run unattended. Off by default; it does not affect SSH host-key trust. With it, `INPUT_FILE` may be `-` to read the CSV from stdin (otherwise rejected because the prompt claims standard input). See `docs/CLI.md`.
- `psilink exchange --invitation CODE`: provision the key file from an invitation code (`@path`-capable, `--invitation @code.txt`) and run the exchange in one command, for the party that composed the exchange in the web app and downloaded a secret-less config. The code is validated (checksum, schema, expiry) and written with its shared secret and expiry before the exchange runs; a malformed or expired code writes nothing. Refused (exit 64) if a key file already exists, since the secret rotates after the first exchange. See `docs/EXCHANGE_REFERENCE.md`.
- `psilink exchange` now fails fast, before connecting, when a recurring config can no longer disclose the payload columns it committed to at invitation time: a local configuration error (exit 64) naming the offending column(s), instead of silently sending a different set and having the partner abort mid-exchange. Every `psilink invite` records the disclosed set into the config's `disclosed_payload_columns` field as it mints the invitation. The error offers to restore the column's metadata or re-establish the exchange with the narrower disclosure. See `docs/EXCHANGE_REFERENCE.md` and `docs/spec/FILE_SYNC.md`.
- Receipt signing identities and `psilink fingerprint`: a long-lived per-party Ed25519 identity and self-signed certificate, pinned by fingerprint exchanged out-of-band. `psilink fingerprint` prints it (and regenerates with `--force`). This is the foundation for signed receipts; receipts themselves are not yet wired up. See `docs/SECURITY_DESIGN.md`.
- Self-attested exchange record: each party writes a local, unsigned audit record of what it disclosed (the agreed-terms hash, both identities, records exposed, governance metadata, and salted commitments to the data), suitable for a HIPAA or FERPA disclosure record, split into a shareable record and a private verification-keys file. Every result file now carries the partner's 0-based row index for each match in a `row_id` column (disambiguated to `their_row_id` on collision) so the pairing is reproducible from the result. The CLI writes both as separate files (`--record-file`, `--no-record`; keys to `<name>.keys.json`); the web app offers two downloads. Both are owner-only. See `docs/spec/EXCHANGE_RECORD.md`.
- `psilink verify-receipt RECORD [INPUT_FILE] [RESULT_FILE]`: read a stored exchange record and report whether it is internally consistent, read-only. Given your retained input and result it opens every commitment; with `--config-file` and `--partner-terms` it re-derives the agreed-terms hash; with neither input nor result it degrades to a structure-and-version check. The verdict rolls up to VERIFIED / INCOMPLETE / VERIFICATION FAILED and exits nonzero only on a definite failure. Partner receipt signatures are not verified yet. See `docs/CLI.md`.
- The web bench's "Verify a receipt" page performs the same read-only consistency check as `psilink verify-receipt`, entirely in the browser: load your exchange record and its verification keys, optionally re-supply your retained input and result and both parties' linkage terms, and get the same VERIFIED / INCOMPLETE / VERIFICATION FAILED verdict. Nothing is uploaded. See `docs/spec/EXCHANGE_RECORD.md`.
- The web app now runs invitation-based, backend-free exchanges: "Create an invitation" mints a single-use deep link to deliver out-of-band, the partner opens it to review the terms and consent, and the two browsers rendezvous directly. A web exchange now always carries an invitation secret. See `docs/SECURITY_DESIGN.md` and `docs/COMMUNICATION.md`.
- The web app's "Advanced options" opens a column-aware editor for the invitation's linkage terms, seeded from the file-derived defaults: the inviter can set its name and the invitation lifetime, choose who receives the matched results (both, only the inviter, or only the partner), attach an optional legal agreement, and toggle and reorder linkage keys, with a live preview and validation that blocks an invalid or unlinkable invitation. It needs the CSV before generating. The quick (name-only) path is unchanged; matching algorithm, deduplication, and fuzzy comparisons are read-only. See `docs/COMMUNICATION.md`.
- The web "Advanced options" editor adds an Expert authoring mode: build linkage keys element by element (fields chosen from your declared columns, never free-typed), edit a per-element transform pipeline, and set a two-of-N "match in either order" swap. It adds a JSON/YAML import/export escape hatch, round-tripped through the same validation. Matching algorithm (psi-c), deduplication, and fuzzy comparisons are shown as controls but stay disabled; importing terms that turn one on is refused. The guided path is unchanged. See `docs/COMMUNICATION.md`.
- The web accept screen shows the inviter's proposed linkage terms and requires explicit consent before connecting; the exchange then runs exactly those terms. Matching method, result-sharing, and the fields the keys match on stay always-visible, above a default-collapsed "Matching strategies" disclosure (each linkage key its own further disclosure). The field summaries are derived from the schema field types, not the partner's wording, so a key's name cannot misrepresent what it matches on. A default-collapsed "Other details" disclosure holds data constraints, payload columns, and the legal agreement. The same summary is shown on the shared exchange screen. See `docs/COMMUNICATION.md`.
- The web accept screen surfaces an always-visible hint (a count) when an invitation requests extra data columns from the acceptor; the column list stays in the "Other details" disclosure. A legal agreement's reference, purpose, and expiry are promoted whole into the always-visible core rather than shown as a bare flag, since the purpose is the compliance-load-bearing field an accounting of disclosures turns on (HIPAA 45 CFR 164.528, FERPA). The promoted fields are partner-controlled text routed through the same display-sanitization boundary as the rest of the terms. See `docs/COMMUNICATION.md` and `docs/COMPLIANCE.md`.
- The web accept screen also surfaces an always-visible hint (a count) when an invitation will send the acceptor data columns for matched records, the ingress companion to the egress hint. The two column lines read direction-first ("Your partner requests N data columns from you" versus "You will receive N data columns from your partner"). A declared "receive nothing" lock-in raises no ingress hint, and it is omitted from the inviter's own preview. See `docs/COMMUNICATION.md`.
- The web accept screen's pre-consent review now forward-references the acceptor's own outbound disclosure (the columns it will send for matched records), previously surfaced only after consent in "Prepare your data". Before a file is chosen a fixed-copy line states a disclosure is coming and the exact columns are confirmed after; once chosen, the actual send list replaces it. See `docs/COMMUNICATION.md`.
- The web accept screen's pre-consent review marks the displayed counterparty identity ("Invitation from &lt;name&gt;") as self-asserted: a fixed-copy note states the partner entered the name and psilink has not verified it. It is informational, on the pre-consent review only. The consent helper also reminds that the invitation should have reached the acceptor over a trusted channel. See `docs/COMMUNICATION.md`.
- The web accept screen's result-sharing block now surfaces the honest-helper membership disclosure when the acceptor's partner is set to receive no result: even a fully honest partner learns which of its own records are in the acceptor's data, an intrinsic property of computing the match. It is fixed copy, kept distinct from the cooperative caveat (a dishonest partner keeping the result table). This corrects a surface that previously framed "partner does not receive" as only a cooperative trust risk. See `docs/COMMUNICATION.md` and `docs/notes/one-sided-disclosure.md`.
- The web app now has a "Prepare your data" editor on both sides: after choosing a file, the operator maps each column to a semantic type and one consequence-labeled disclosure choice (used to match, row identifier, sent to your partner, or ignored), sees a live count of satisfiable linkage keys, and confirms which columns will be sent before connecting. The acceptor reaches it where the old "this file cannot be linked" dead-end was; the inviter edits the same grid under "Advanced options". These settings are per-party and local, never embedded in the invitation. A column inferred as a row identifier is no longer sent to the partner by default. See `docs/COMMUNICATION.md`.
- The web quick (name-only) invite path now shows, once a file is chosen, which columns it will send to the partner for matched rows, with a one-click path into Advanced options. The displayed set is derived from the same predicate the exchange transmits on, and says so explicitly when it would send no columns. Awareness only; transmission is unchanged. See `docs/COMMUNICATION.md`.
- The web "Prepare your data" editor now has a per-field standardization workbench: the recommended per-type cleaning is an editable, ordered list of steps beside a live before-and-after preview over a sample of your rows. A cleaned value that does not meet a field's declared constraint is flagged with a warning, not blocked. The cleaning stays per-party and local. See `docs/COMMUNICATION.md`.
- `psilink exchange` (and the zero-setup form) now warns during data preparation when cleaned values fail a linkage field's declared constraints (an excluded value, a name character outside the allowed set, an invalid date, or an SSN breaking a structural rule), reporting a count per field. The exchange still proceeds. See `docs/EXCHANGE_REFERENCE.md`.
- The web "Prepare your data" and "Advanced options" editors let the operator author raw regular-expression cleaning steps (replace, extract, filter, split) on both sides. Each pattern is validated against the supported linear-time dialect, bounded to 1000 characters, and runs under a non-backtracking engine. A malformed or incomplete step blocks generating or continuing. These steps are per-party and local; a raw pattern inside a partner-authored key transform stays read-only. See `docs/COMMUNICATION.md`.
- The web "Prepare your data" editor now lets the operator choose which column feeds each field when more than one column has the field's type, so two fields of one semantic type (e.g. a maiden and a current name) can bind to distinct columns. Per-party and local. See `docs/COMMUNICATION.md`.
- The web inviter's "Advanced options" editor now shows the same data-cleaning by default (no longer behind expert mode): author per-field cleaning steps, choose which column feeds each field, and -- as an expert action -- add a second field of one semantic type bound to a distinct column (e.g. a maiden and a current name) and reference both in the linkage keys. This survives the JSON/YAML import/export round-trip: the binding is reconstructed against your own file, choosing only among columns you marked for matching (so an imported document cannot turn a row-identifier or sent-to-partner column into a match field); a key your columns cannot supply is imported turned off and flagged unlinkable, and an import where no key is satisfiable is refused. Per-party and local. See `docs/COMMUNICATION.md`.
- The web "Prepare your data" editor and the inviter's "Advanced options" cleaning now warn when a cleaning step silently collapses a field: over your whole file, it raises an alarm when a transform has dropped every row to nothing (a failure the sample preview cannot show). It warns, it does not block. The editor also gained an accessibility pass: table semantics, keyboard step-reorder, debounced screen-reader announcements, and non-color encoding of each outcome. See `docs/COMMUNICATION.md`.
- `zero-setup` exchange: derive all connection parameters from a URL, with no configuration file, for a one-off exchange. See `docs/CLI.md`.
- `filedrop` channel: exchange through a shared local directory where neither SFTP nor WebRTC is available. See `docs/COMMUNICATION.md`.
- `retain_files` mode (`sftp`/`filedrop`): keep every exchange file as a durable transcript instead of deleting it on consume (`--retain-files`). See `docs/EXCHANGE_REFERENCE.md`.
- Split inbound/outbound directories for the `sftp`/`filedrop` channels: configure `inbound_path`/`outbound_path` (instead of a single `path`/`server.path`) to read the peer's files from one directory and write your own to another, for managed shares or SFTP servers with separate drop and pickup folders. Requires retain mode. See `docs/EXCHANGE_REFERENCE.md`.
- `--outbound-path` configures that split directory from the command line, on the zero-setup, `invite`, `accept`, and `exchange` commands: the server URL or positional path (or, for `exchange`, the configured directory) is the inbound directory and `--outbound-path` is the outbound. Requires retain mode (`--retain-files`, or `retain_files: true` in an `exchange` config). See `docs/EXCHANGE_REFERENCE.md`.
- `--server-private-key-passphrase` supplies the passphrase for an encrypted SSH private key on the zero-setup, `invite`, `accept`, and `exchange` commands (previously settable only through a config's `private_key_passphrase`). Like the sibling credential flags it takes an `@path` file reference and, on `--save`, is persisted as that reference. Passing it without `--server-private-key` is a usage error (exit 64). See `docs/EXCHANGE_REFERENCE.md`.
- `psilink invite --expires-in DURATION` sets the invitation lifetime (default 1 hour). See `docs/CLI.md`.
- `--log-file <path>` on every command that logs appends diagnostics to an owner-only file for unattended runs; a missing parent directory fails with exit 64. See `docs/CLI.md`.
- Every exchange now logs a one-line runtime banner at the start: the Node version and the memory ceilings the run operates under (host memory, the V8 heap limit, and the container memory limit when a tighter cgroup limit is set), so an out-of-memory failure can be read against its ceilings from the support log alone. The line is operator-only, never sent to the partner. See `docs/SECURITY_DESIGN.md`.
- `--save` on a zero-setup exchange provisions a persistent shared secret in-band when both parties pass it, with no separate invite round; that secret is protected only by the transport. See `docs/SECURITY_DESIGN.md`.
- The `-` input positional reads the CSV from standard input for `exchange`, the zero-setup form, and `invite`; `accept` rejects `-` unless `--consent-to-terms` skips the confirmation prompt that otherwise claims stdin, so a piped `accept` without `--consent-to-terms` fails fast with guidance. See `docs/CLI.md`.
- `--sweep-exchange-files` (and `--force-retain-sweep`) clear protocol files left by a crashed prior run on the `sftp`/`filedrop` channels. See `docs/spec/FILE_SYNC.md`.
- `connection.options.unexpected_files` policy (`error`/`warn`/`ignore`) for a foreign file that appears in the shared directory during an exchange; defaults to `error`, or `warn` under `retain_files`/`lockless_rendezvous`. See `docs/EXCHANGE_REFERENCE.md`.
- `authentication.token_max_age_days`: cap the age of rotated tokens for dormant partnerships. A misspelled key in the `authentication` block is now rejected at parse time. See `docs/EXCHANGE_REFERENCE.md` and `docs/SECURITY_DESIGN.md`.
- Invitations may carry an optional credential-free connection endpoint (a public rendezvous locator); they never carry credentials. For a split-directory `sftp`/`filedrop` exchange the endpoint carries the inviter's `inbound_path`/`outbound_path` pair, which the acceptor mirror-swaps (one party's outbound is the other's inbound) so both sides can keep a fixed mount layout. See `docs/SECURITY_DESIGN.md` and `docs/spec/FILE_SYNC.md`.
- A `psilink:diagnostics` localStorage flag raises web connection-logging verbosity for one browser, without leaking the derived rendezvous ids. See `docs/DEPLOYMENT.md`.
- `psilink invite` (offline) prints how to abandon a pending invitation before it expires: delete its key file. See `docs/CLI.md`.
- Key file schema: a versioned JSON format for persisting the shared token and exchange metadata between sessions.
- Metadata inference (column semantic types and date formats), custom linkage keys via configurable transformations, and data standardization that canonicalizes linkage values by semantic type. See `docs/EXCHANGE_REFERENCE.md`.
- `zip_code` semantic type: a column named `zip`, `zip5`, `zip_5`, `zipcode`, or `zip_code` is inferred as a US ZIP code (canonical form 5 digits, zero-padded). Like `phone_number`/`email_address` it is matchable but in no default linkage key, so it is used for matching only when a key references it. See `docs/EXCHANGE_REFERENCE.md`.
- Metadata `role: ignored`: mark an input column present in your file but excluded from the exchange -- never linked, never an identifier, and never transmitted as payload (even if `is_payload: true`). Opt-in only; inference never assigns it. See `docs/EXCHANGE_REFERENCE.md`.
- The web "Advanced options" editor now authors the invitation's payload columns: the columns you set to "Sent to your partner" are listed in the invitation's data dictionary and shown on your partner's consent screen. The set is derived from the same disclosure choices that govern transmission, so it cannot over-state what is sent. The editor blocks generating an invitation that sends columns to a partner set to receive no result. See `docs/COMMUNICATION.md`.
- The web quick (name-only) invite path now authors the same payload data dictionary for the columns it sends, so the partner's consent screen lists them, just as the Advanced editor's does. The declared set is derived from the same predicate the exchange transmits on; nothing is declared when it would send no columns. Which columns leave the machine is unchanged. See `docs/COMMUNICATION.md`.

### Changed

- In the web app's exchange and accept flows, the browser Back and Forward buttons now move between the flow's steps in place instead of leaving the page and silently discarding the loaded file and in-progress terms; while a file is loaded and the exchange has not yet been created or saved, closing the tab, reloading, or otherwise navigating away asks for confirmation first. The participant file remains memory-only throughout.
- BREAKING: the redesigned web app now serves the primary routes -- `/` is a lobby offering start an exchange, accept an invitation, or verify a receipt; `/exchange`, `/accept`, and `/verify` are the flows behind them -- and the legacy web interface is removed. The old `/advanced` route is gone (its options moved into the exchange flow), and transition-period `/bench/*` URLs redirect to the primary routes. See `docs/DESIGN.md` and `docs/COMMUNICATION.md`.
- The CLI now rejects an unrecognized option at parse time, before connecting, instead of silently ignoring it (previously a misspelled flag landed unread and the run proceeded with a default or stale value). The zero-setup exchange, `exchange`, `fingerprint`, and `verify-receipt` commands fail fast with a usage error (exit 64) naming the offending option. `invite`, `accept`, and `init` accept a `-`-leading invitation positional, so a mistyped option is caught by the command's own argument validation instead. See `docs/CLI.md`.
- The web accept screen's always-visible consent core is reorganized into four labelled groups by disclosure direction ("What you disclose", "What the exchange produces", "What you receive", "How records are matched"), then the legal agreement last; every previously always-visible fact stays always-visible. A security reviewer should note two result-sharing consent-copy changes: the viewer's own non-receipt is now distinguished as enforced (sent none, fails closed on any it is sent) from the partner's cooperative non-receipt (rests on the agreed terms being honored), so a cooperative "No" no longer reads identically to a cryptographically enforced one; and the partner's "Yes" now notes that the agreement, not the tool, governs the result's use once received. See `docs/COMMUNICATION.md` and `docs/notes/one-sided-disclosure.md`.
- The CLI now runs the PSI elliptic-curve masking in a `worker_threads` worker, off the event-loop-owning thread, and keeps an otherwise-idle SFTP session alive across a long round (a `realPath` every 60 s resets the server's idle timer; kernel TCP keepalive is a transport-layer backstop), so a strict idle-timeout server (Azure Blob SFTP's fixed two minutes) no longer drops the session mid-exchange. No command, flag, output, wire format, or result changes; the browser is unaffected. A security reviewer should note the net-new channel-liveness controls (SFTP heartbeat, TCP keepalive) and that the vendored `@openmined/psi.js` addon was rebuilt with a worker-thread teardown fix. See `docs/spec/PROTOCOL.md` and `docs/spec/CHANNEL_SECURITY.md`.
- The web app now runs the PSI elliptic-curve masking in a Web Worker, off the browser's UI thread, so the tab stays responsive (painting, timers, and WebRTC keepalives keep firing) during a masking round instead of freezing. The worker loads its own WASM engine, and the exchange's secret key is generated inside it and never leaves it; the result is byte-identical to the in-process path. It is the browser sibling of the CLI's worker-thread offload above. A security reviewer should note the PSI crypto now runs in a separate worker isolate with the secret key confined to it. See `docs/spec/PROTOCOL.md`.
- The CLI now launches Node with `--expose-gc` (its Docker entrypoint and the `apps/cli` dev script), letting `single-pass` linkage release its transient allocation peak at phase boundaries and lowering the receiver's peak resident memory (roughly 15-18% at the benched sizes). Perceptible-memory and deployment change only; no command, flag, output, wire format, or result is affected. This relief is the precondition for the raised single-pass dataset ceiling (below). See `docs/spec/PROTOCOL.md`.
- The `single-pass` dataset ceiling (`MAX_SINGLE_PASS_CELLS`) is raised from 2,000,000 to 3,000,000 (key, record) cells per party -- about 214k rows at the 14-key default template, up to 3M rows at a single key. An exchange above the ceiling still aborts cleanly on both parties. This widens a denial-of-service bound (the maximum a single exchange will hold resident before aborting). ### Security

- The web server job API now caps request and intent sizes as defense in depth: `POST /api/jobs` reads its body under a hard byte limit (streamed and counted, never trusting `Content-Length`) and answers `413` on an oversized body before any parse, and the exchange intent bounds its `inputCsv` (anchored to the browser's 100 MiB CSV file gate), its `metadata`/`standardization`/`expectedPayloadColumns` arrays, and their strings. The intent is operator-authored and the API is gated, so the worst case was a single operator exhausting their own appliance; the caps bound it and matter more as the intent widens. Existing well-formed intents are unaffected. See `docs/spec/SERVER_JOB_API.md`.
- The Docker image now builds its dependency tree from the committed `package-lock.json` (`npm ci` in the builder; the final stage copies the production `node_modules` and runs no install), so a rebuild can no longer silently resolve newer dependency versions than CI tested -- in particular `re2js`, whose regex behavior must match between the two parties for PSI keys to agree. A rebuild without a lockfile change yields the same dependency tree. See `docs/spec/DEPENDENCY_PINS.md`.
- The Docker image's `node:26-alpine` base is now digest-pinned in both build stages to its multi-arch index digest, so the Node runtime and Alpine userland beneath the frozen `node_modules` no longer drift between rebuilds. A base security patch is now a deliberate digest bump rather than an automatic float on a rebuild. See `docs/spec/DEPENDENCY_PINS.md`.
- A `psi-c` (count-only) linkage-terms value is now refused before the exchange runs, on every path (a hand-crafted invitation token, a CLI-authored config, or any non-web mint), not only clamped in the web inviter as before. `psi-c` advertises count-only disclosure but has no run path yet (the run always reveals matched identifiers), so an accepted-and-run `psi-c` exchange would have written a record asserting count-only over a run that revealed identifiers -- a false disclosure accounting under HIPAA 45 CFR 164.528 and FERPA. Core fails closed at prepare and at the run boundary, and `psilink invite` refuses to mint one (exit 64). Use `psi` until a count-only run path lands. See `docs/EXCHANGE_REFERENCE.md` and `docs/spec/EXCHANGE_RECORD.md`.
- Redirecting a command's matched-records result to a file with a shell `>` (e.g. `psilink exchange data.csv > results.csv`) now surfaces a notice on stderr (visible at `--log-level error`): the shell creates that file under its umask, not the owner-only permissions an OUTPUT_FILE path argument gets, so on a shared host the records can land group/world-readable. A TTY, a pipe, and `/dev/null` do not fire, and passing an OUTPUT_FILE path is unchanged. A redirect applied outside this process (across a container boundary) is not detected. See `docs/SECURITY_DESIGN.md`.
- Matching participation now requires `role: linkage`: a column roled `identifier`, `payload`, or `ignored` is no longer hashed into a linkage key merely because its semantic `type` matches a linkage field (on every linkage path, and an explicit transformation naming such a column no longer binds it -- the role wins). This closes the overlap where a sent-to-partner or row-identifier column could become a cross-party PSI match key without an explicit designation. Transmission stays a separate axis (`is_payload`). A field with no `role: linkage` column of its type is reported unsatisfiable before the run. The default quick path is unaffected. See `docs/spec/PROTOCOL.md` and `docs/EXCHANGE_REFERENCE.md`.
- The web app sets defense-in-depth HTTP response headers on its rendered pages and API routes: `Referrer-Policy: no-referrer` (keeping the fragment-borne invitation token out of `Referer` on older clients too), `X-Frame-Options: DENY` with a Content-Security-Policy `frame-ancestors 'none'` to deny framing (clickjacking), and `X-Content-Type-Options: nosniff`. No active leak is being fixed -- modern browsers already withhold the URL fragment from `Referer`. See `docs/SECURITY_DESIGN.md`.
- One-sided output sharing is now enforced end-to-end in both the web app and the CLI: when the agreed terms entitle only one party, the result is returned and written solely to the entitled party, rather than (as before) written to both regardless of the terms. The web acceptor derives its output terms as the mirror of the inviter's, and the Advanced-options editor exposes the three-way "who receives the matched results" choice. An honest-but-curious helper can still observe its own match count during the protocol, so this is not a confidentiality guarantee against the helper. See `docs/spec/EXCHANGE_RECORD.md` and `docs/COMMUNICATION.md`.
- Cross-party payload declarations are reconciled lazily, so an invitation's payload can be authored without the inviter knowing the partner's columns: a party that declares no receive list accepts whatever the partner's metadata discloses, while a party that does declare one still requires the partner's send to match it exactly or the exchange aborts before any data moves. A web acceptor mirrors the inviter's payload terms. This relaxes only the declaration cross-check, never what is disclosed. See `docs/EXCHANGE_REFERENCE.md`.
- The invitation now carries the inviter's disclosed-columns subset directly (the exact set the acceptor will receive for matched records), so the consent screen shows what it will receive even for an invitation that authored no payload data dictionary (a CLI or quick-path invitation); the set is derived from the transmission predicate, so it cannot drift. The acceptor locks the consented set into its saved config (and, on an online accept, in memory for that run), and aborts as a protocol error if the received payload carries a different set; an empty set is enforced strictly ("receive nothing"). A first-contact exchange with nothing locked in reconciles lazily. See `docs/EXCHANGE_REFERENCE.md` and `docs/spec/FILE_SYNC.md`.
- A party that learns its received-payload columns only by observing a first exchange now persists that observed set into the saved config's `expected_payload_columns` (distinct from the output flag and from `payload.receive`) when it saves for recurring use, so the recurring exchange fails closed on a later mismatch. The two savers are the online inviter and a zero-setup party run with `--save`. A no-save exchange stays lazy, and an observed-empty payload is left lazy rather than frozen to a strict "receive nothing". See `docs/EXCHANGE_REFERENCE.md` and `docs/spec/FILE_SYNC.md`.
- A `payload.send` data dictionary that omits a column its input metadata does transmit is now rejected (exit 64), alongside the existing rejection of one that names a column metadata does not transmit: a present dictionary must name exactly the disclosed set. This keeps the dictionary from understating what is sent, which would otherwise abort an honest recurring exchange when the omitted column arrives. An absent or empty dictionary is unaffected; the web invite paths derive the dictionary from the disclosure predicate. See `docs/EXCHANGE_REFERENCE.md`.
- A party entitled to no result in a one-sided exchange now receives none of the partner's payload columns: payload is transmitted only to a partner the terms entitle to the result, and a non-receiving party's receive side fails closed if sent any. This closes a residual disclosure in which a non-receiving helper still received the receiver's disclosed payload values. The rule that a no-output party may not declare `payload.receive` is now enforced at runtime too. See `docs/EXCHANGE_REFERENCE.md`, `docs/spec/FILE_SYNC.md`, and `docs/notes/one-sided-disclosure.md`.
- A malformed operator config (`psilink.yaml`) or credential file (`.psilink.key`, the signing identity) no longer echoes its own contents into CLI error messages or stderr, so an inline SFTP credential, the shared secret, or a private key cannot leak from a 0600 file into logs or bug reports. Parse errors report the file path only; as a backstop, displayed errors also redact any embedded PEM/OpenSSH private-key block. See `docs/SECURITY_DESIGN.md`.
- Recurring CLI exchanges (`sftp`/`filedrop`) are now encrypted end-to-end on the wire with an application-layer AEAD, so a directory or server administrator sees only ciphertext. See `docs/spec/CHANNEL_SECURITY.md`.
- Web exchanges run the X25519 authenticated key exchange before any PSI data, so each party authenticates the peer and an expired or untrusted credential fails closed; a web exchange is single-use and persists no key file. Transport confidentiality on the web path is provided by DTLS -- the application-layer AEAD is not yet applied there, so the session key currently serves only to authenticate. See `docs/SECURITY_DESIGN.md`.
- Invitation tokens carry a bounded lifetime (default 1 hour, now including web-generated ones); rotation tokens carry no expiry unless `token_max_age_days` is set. See `docs/SECURITY_DESIGN.md`.
- The shared token is rotated after every successful handshake. See `docs/SECURITY_DESIGN.md`.
- The file-sync transport bounds untrusted input from a hostile SFTP or file-drop server, closing memory-exhaustion and hang denial-of-service: a roughly 512 MiB inbound frame cap, an 8192-entry / 255-character directory-listing cap, and per-operation read/write liveness deadlines of about a minute. See `docs/spec/CHANNEL_SECURITY.md`.
- A malformed SFTP packet from a hostile server can no longer crash the CLI; it becomes an orderly typed failure with normal cleanup. See `docs/spec/CHANNEL_SECURITY.md`.
- SFTP host-key verification (fail-closed): every SFTP connection verifies the presented host key before authenticating and aborts on a mismatch. Pin it out-of-band via `connection.server.host_key_fingerprint` (a single fingerprint, or a list to stage a rotated host key for zero-downtime rotation), or let the first interactive run establish it on first use (shows the fingerprint, prompts, records it; a zero-setup run persists it only with `--save`). A connection with no pin is refused, and a non-interactive run with no pin fails closed rather than trusting the key. A key matching none of the pinned fingerprints is never auto-accepted. The unsupported `certificate` and `known_hosts` fields are rejected at parse time. CLI `sftp` only. See `docs/CLI.md` and `docs/SECURITY_DESIGN.md`.
- Cross-party SFTP host-key reconciliation: in a recurring (authenticated) exchange the two parties now compare the host-key fingerprints they each observed, advertised inside the authenticated post-handshake exchange. A divergence -- which a purely local pin cannot catch -- raises a warning naming both observed values, flagging a possible one-sided interception or a server rekey between the two parties' setups for the operators to disambiguate out-of-band; it warns rather than aborts. A party that observes no host key (a file-drop or proxy path) reconciles to no false alarm. See `docs/spec/CHANNEL_SECURITY.md`.
- Untrusted invitation-token fields are bounded at decode (a 64 KiB cap plus per-field limits), closing a denial-of-service and an SSRF-shaped WebRTC-host vector from a crafted but checksum-valid token. See `docs/SECURITY_DESIGN.md`.
- Sensitive files (key file, config, exchange records, signing identity) are written owner-only, flushed to disk for crash durability, and created on a symlink-safe descriptor on POSIX. See `docs/SECURITY_DESIGN.md`.
- The linkage-result CSV is now written owner-only (mode 0600 on Unix, ACL-narrowed on Windows), so the tool's most sensitive output is no longer left world- or group-readable by the umask. See `docs/SECURITY_DESIGN.md`.
- Operator-facing logs, errors, and web console output escape untrusted partner- and server-controlled strings (filenames, rendezvous paths, identities, error messages), neutralizing terminal, bidi-override, zero-width, and homoglyph injection; values used to build paths or to compare are unaffected. See `docs/SECURITY_DESIGN.md`.
- A credential supplied as an `@path` reference is persisted as the reference, never the resolved secret. Loading a config you did not author runs its `@path` references as your own credentials -- a credential-exfiltration vector -- so never run an untrusted config. See `docs/CLI.md` and `docs/SECURITY_DESIGN.md`.
- `connection.provider_options` is applied through a default-deny allowlist, so it cannot override the security-critical connect options (host, credentials, host-key verification). See `docs/EXCHANGE_REFERENCE.md`.
- A peer sending messages out of turn is rejected rather than buffered without limit.
- Untrusted partner-supplied collections in the linkage terms and post-handshake messages are count-bounded, so a counterparty packing millions of invalid elements into an in-cap frame fails with a clean protocol error instead of a Zod `RangeError` or a multi-second CPU burn. See `docs/spec/CHANNEL_SECURITY.md`.
- The PSI decode path now scans every partner-supplied setup/request/response frame and rejects it before deserialization if it declares more encrypted elements than allowed, on both strategies and transports. This closes a memory-exhaustion denial of service: the protobuf deserializer allocates ~200 bytes per declared element, so a counterparty packing an in-cap frame with many minimal entries could drive tens of GiB of allocation; the pre-scan bounds the declared count by the smaller of the authenticated key*record count and a fixed ceiling, failing with a clean protocol error. See `docs/spec/CHANNEL_SECURITY.md` and `docs/spec/PROTOCOL.md`.
- A partner's payload column names are length-bounded on the wire, and independently in the exchange record, so a malicious or malformed partner cannot write an arbitrarily long column name into this party's local exchange record. See `docs/spec/CHANNEL_SECURITY.md`.
- A partner payload column name is now also required non-empty on the wire (and independently in the exchange record), so a malicious partner cannot suppress this party's exchange record by sending an empty (`""`) column name -- which would otherwise reach the non-fatal record-build guard and skip the audit record while the exchange still completed and disclosed. The floor is safe to enforce now that honest parties no longer emit empty names (rejected at intake, above). See `docs/spec/CHANNEL_SECURITY.md`.
- A crafted invitation whose linkage-terms `transform.params` is nested thousands of levels deep is now rejected with a clean usage error (exit 64) when an `accept` reconciles it against an existing config, instead of a stack overflow in the comparison walk (previously a generic exit 69). Real invitations are unaffected. See `docs/spec/CHANNEL_SECURITY.md`.
- The web app's bundled PeerJS signaling server caps inbound WebSocket frame size, so a single oversized frame from an unauthenticated client is rejected before parsing, and can no longer crash the rendezvous process (which would drop signaling for every connected peer, not just one session) or pin a large block of its memory. See `docs/spec/CHANNEL_SECURITY.md`.
- The web app's bundled signaling server further hardens that upgrade surface against unauthenticated resource exhaustion: a slow, partial, or idle upgrade handshake is closed server-side, a client that registers but never proves liveness is reaped within seconds, the relay's reconnect queues are bounded, and a connection's id, token, and key are length-bounded at the handshake. These are defense-in-depth; Origin restriction and per-address rate limiting remain the reverse proxy's responsibility. See `docs/DEPLOYMENT.md` and `docs/spec/CHANNEL_SECURITY.md`.
- The web peer-to-peer WebRTC data channel now bounds inbound frames so a hostile or buggy peer can no longer drive the receiving tab toward memory exhaustion: an oversized frame (fixed 256 MiB wire cap), a frame that deserializes to a disproportionately large structure, and a flood of tiny or never-completed chunk reassemblies all fail with a terminal error. This is the WebRTC transport's own control, since core's AEAD frame-size cap does not reach this path (it runs under DTLS and declines the AEAD wrap). See `docs/spec/CHANNEL_SECURITY.md`.
- The bundled AWS Elastic Beanstalk nginx reference (`apps/web/deploy/aws_eb/`) now requires TLS 1.2+ (dropping TLS 1.0/1.1), sets an explicit forward-secrecy / AEAD-only `ssl_ciphers` list, and applies per-address rate/connection limits on the `/api/peerjs` signaling path by default, with the Origin allowlist and an HSTS header shipped as opt-in templates, so a deployment from the reference no longer inherits the no-proxy resource-exhaustion residual or a weak default TLS suite list. The limit numbers are illustrative starting points. See `docs/DEPLOYMENT.md`.
- The same AWS Elastic Beanstalk nginx reference now disables TLS session resumption by default (`ssl_session_tickets off`, `ssl_session_cache off`), so every session is a full ECDHE handshake and no resumed session rests on the single, never-rotated per-worker ticket key whose disclosure would undercut the forward secrecy of the ECDHE-only cipher list. Re-enable resumption deliberately if you need it. See `docs/DEPLOYMENT.md`.
- The same AWS Elastic Beanstalk nginx reference now pins the ECDHE curve list explicitly (`ssl_ecdh_curve X25519:prime256v1`) by default, so curve selection no longer rests on the OpenSSL/platform default, and it ships an OCSP-stapling opt-in template (inert by default, since stapling cannot work on the test/self-signed certificate the reference is commonly run with). See `docs/DEPLOYMENT.md`.
- Partner-authored regular expressions in linkage-term element transforms (`replace_regex`, `extract_regex`, `filter_regex`, `split_on`, and the pattern `parse_date` builds) now execute under a linear-time engine (RE2 semantics, the `re2js` dependency) instead of the JavaScript `RegExp` engine, so a crafted catastrophic-backtracking (ReDoS) pattern can no longer hang the acceptor's tab or CLI process. Patterns must conform to the RE2 dialect (no backreferences or lookarounds, `\x{...}` rather than `\uHHHH`); a pattern outside it is rejected at terms validation, with no fallback to the backtracking engine. This replaces the earlier static ReDoS screen (and removes the `redos-detector` dependency). See `docs/spec/PROTOCOL.md` for the dialect, and `docs/SECURITY_DESIGN.md`, `docs/spec/CHANNEL_SECURITY.md`, and `docs/EXCHANGE_REFERENCE.md`.
- An over-large `length` on a partner's `pad_left` linkage-term element transform is rejected at terms validation, before any per-row key is built, so a crafted value (such as `1e9`) cannot exhaust the acceptor's memory by allocating a giant padded string per row. A legitimate left-pad (zero-filling a short identifier) is unaffected. See `docs/SECURITY_DESIGN.md`, `docs/spec/CHANNEL_SECURITY.md`, and `docs/EXCHANGE_REFERENCE.md`.
- A partner's `parse_date` linkage-term element transform has its `input_format`/`output_format` length-capped (256 characters each), and the raw regex patterns above are length-capped (1000 characters), bounding compile cost and per-row output size. Ordinary date layouts (`MM/DD/YYYY`, `YYYY-MM-DD`, `YYYYMMDD`) are unaffected. See `docs/SECURITY_DESIGN.md`, `docs/spec/CHANNEL_SECURITY.md`, and `docs/EXCHANGE_REFERENCE.md`.
- An invitation whose linkage-terms `transform.params` carries `snake_case` parameter keys is now normalized to `camelCase` at decode, the same as the config and wire paths (the invitation decoder previously left them verbatim). This closes a class of latent failures with such a token: the per-step length screens being bypassed, a cross-party agreement-hash divergence, the standardization runtime silently ignoring the parameters, and a false `linkage_keys` conflict on `psilink accept`. The decoder also now accepts `snake_case` structural keys. Invitations produced by psilink are unaffected. See `docs/spec/CANONICAL_ENCODING.md` and `docs/spec/CHANNEL_SECURITY.md`.
