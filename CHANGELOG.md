# Changelog

All notable changes to PSI-Link are documented here. The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). PSI-Link uses [semantic versioning](https://semver.org/).

## [Unreleased]

### Added

- Enforced backstop for the consumer-side terminal-on-`UsageError` invariant at transport-call retry sites. The typed transport bounds (`FrameSizeExceededError`, `DirectoryListingBoundsError`, `TransportOperationStalledError` -- all `UsageError` subclasses) are terminal by design: a consumer awaiting a transport `list()`/`get()`/`createExclusive()` call must propagate, never retry, a `UsageError`, because retrying loops straight back into the hang or over-allocation the bound exists to prevent. This previously held only by convention. A guard test (`packages/core/test/transportUsageErrorTerminal.test.ts`) now AST-scans the file-sync consumer and fails if a retry loop around a transport read swallows a `UsageError` instead of propagating it -- catching a future reintroduction, not just today's call sites -- and `docs/SECURITY_DESIGN.md` ("Channel security") records the consumer-side rule that ties the three bounds together. The invariant is correct in code today; this is a regression backstop with no runtime behavior change.
- Session-start directory hygiene for the `sftp` and `filedrop` channels, relaxing the strict-empty entry guard. Foreign files -- names that do not match the protocol filename grammar, such as a sync tool's conflict copy or partial download -- present in the shared directory at `synchronize()` start are now recorded in an in-memory snapshot and tolerated for the session in both modes (deleting nothing) instead of aborting the exchange; a foreign file that first appears mid-session continues to be handled by `unexpected_files`. A message-shaped `<id>-<digits>.json` still matches the grammar and is treated as a protocol file, so the default stays strict-empty for protocol files. Two new CLI-only, non-persistable recovery flags: `--sweep-exchange-files` deletes every protocol file (this party's and the peer's: hellos, acks, locks, joining sentinels, messages) left by a crashed or bilateral-mode-mismatched prior run and starts a fresh exchange, leaving foreign files untouched. Before deleting it inspects for a retain-mode signal -- local retain mode, a peer hello advertising `retain_files=true` (read through the partial-sync gate, bounded so it cannot stall), or a retain-only message ack -- and refuses, since a retain directory is a durable audit transcript; clearing one requires the additional `--force-retain-sweep`, which permanently discards the prior transcript after a loud warning. `--force-retain-sweep` requires `--sweep-exchange-files` and is rejected on its own. Neither flag is a `connection.options` key, so neither can be set in `psilink.yaml`. The sweep uses a rejecting delete, so a transport that cannot delete surfaces a transport error (exit 69) rather than a false clean slate. Specified in `docs/FILE_SYNC.md` and `docs/EXCHANGE_SPEC.md` ("Directory exclusivity").
- X25519 authenticated key exchange in `@psilink/core` (`runKex`, `KexResult`): an ephemeral X25519 Diffie-Hellman pinned to the Noise NNpsk0 pattern (no static keys, the pre-shared secret mixed at position 0, ephemeral-ephemeral DH) plus an explicit, role-asymmetric mutual key confirmation, yielding a 32-byte session key with forward secrecy and mutual authentication -- the key-agreement primitive that supersedes SPAKE2 as the source of the exchange session key. The session key is derived (NNpsk0 key schedule) over both the X25519 shared secret and the pre-shared secret plus the transcript, so it is neither the pre-shared secret alone (which would have no forward secrecy) nor the raw DH output alone. The X25519 primitive and the ephemeral keygen come from the already-present, audited `@noble/curves` (no new dependency); only the minimal NNpsk0 glue (key-schedule mixing and the two confirmation tags) is hand-written, and the full Noise framework is deliberately not implemented. Carries over SPAKE2's safeguards: strict (`.strict()`) wire schemas, constant-time tag comparison, a single generic non-oracular failure, a best-effort abort to shorten the peer's wait, and an all-zeros contributory check on the X25519 share. Backed by a checked-in known-answer vector and cross-checked against RFC 7748 (X25519) and RFC 5869 (HKDF). Specified in `docs/PROTOCOL.md` ("X25519 authenticated key exchange").
- `--save` on the zero-setup exchange now bootstraps a persistent shared secret in-band, without a separate invitation round. Each party advertises its save intent on the linkage-terms exchange (the one round-trip both sides always perform, so a one-sided save never stalls). When both parties pass `--save`, the initiator generates a fresh base64url 32-byte secret -- the same format a recurring exchange rotates to -- and transmits it to the responder on a dedicated in-band message, and both save config and key files through the provisioning layer (with conflict detection up front, before any network activity). When only one party saves, no secret is generated: that party saves its config alone (no key file) and is told to use `psilink invite`, and the partner is notified that nothing was saved on its end. On a zero-setup exchange there is no application-layer AEAD, so the in-band secret is protected only by the transport (SSH/DTLS/operator access controls) -- the same trust model as the zero-setup exchange that carries it, for that one initial exchange; see `docs/SECURITY_DESIGN.md` ("Bootstrapping a shared secret"). New `exchangeBootstrapSecret` and the `saveIntent`/`bootstrap` plumbing on `runExchange` in `@psilink/core`.
- Receipt signing identities (Phase 2 foundation for signed receipts): a long-lived per-party signing identity -- an Ed25519 keypair and a self-signed certificate carrying the party's `identity` -- generated once, persisted owner-read-only in its own file (default `~/.psilink/signing-identity.json`, separate from the rotating `.psilink.key`), and reused across every exchange and partner. Trust is pinned self-signed (no CA chain, no revocation): the certificate's fingerprint (unpadded base64url SHA-256 over the domain-separated canonical encoding of the certificate body) is exchanged out-of-band and pinned, and a presented partner certificate is accepted only if it self-verifies and its fingerprint matches the pin; a receipt is accepted only if its asserted identity exactly matches the certificate's bound identity (exact match over the canonical identity bytes). The certificate is a canonical-JSON document (no new dependency; reuses `@noble/curves` and the RFC 8785 canonical encoding) with a `version`/`algorithm` seam toward a future authority-backed mode. Exposed in `@psilink/core` (`generateSigningIdentity`, `computeCertificateFingerprint`, `parseCertificate`/`parseSigningIdentity`, `assertPartnerCertificateTrusted`, `verifyPresentedCertificate`, the `signing` block on `ExchangeSpec`) and backed by checked-in cross-implementation vectors. The CLI adds `psilink fingerprint`, which lazily creates the identity on first use, prints the fingerprint to share out-of-band, and regenerates with `--force`. Receipt assembly, signing, and the receipt swap remain to be wired up. Specified in `docs/PROTOCOL.md` ("Signing identity and certificate pinning"), `docs/SECURITY_DESIGN.md` ("Receipt signing identities"), and `docs/EXCHANGE_SPEC.md` ("Signing").
- `psilink invite` and `psilink accept` CLI commands for moving from a zero-setup exchange to a recurring one backed by a shared secret. Each has an offline mode (no network: `invite` generates a shared secret, writes a key file with an `expires`, and prints an invitation string plus usage instructions; `accept` decodes and displays the invitation, prompts for confirmation, and writes config and key files) and an online mode (additionally connect, complete the X25519 key exchange, and run the exchange in one step, persisting the rotated key and config as soon as the handshake succeeds -- before the data exchange begins -- so an exchange that fails after a successful handshake leaves both on disk and is recoverable with `psilink exchange` without re-inviting; a handshake that never succeeds leaves no config behind). The invitation's 4-byte checksum and schema are verified, and an expired token is rejected by name, before the user is prompted; a pre-existing config or key file at a target path is detected before any network activity. An invitation string beginning with `-` is parsed as the positional invitation, not an option flag. The acceptor's config seeds its connection block from the invitation's credential-free endpoint when present (with a `REPLACE_WITH_...` marker on the credential field) and a clearly-marked placeholder otherwise. Some richer behaviors documented in `docs/CLI.md` (seeding the printed online invitation with a connection endpoint) are not yet implemented; see the "Current CLI limitations" note there.
- `psilink accept` and online `psilink invite` now reconcile a pre-existing config or key file at a target path instead of always aborting. `accept` (offline and online) compares an existing config's linkage terms against the invitation -- and, online, its connection block against the connection target (the URL plus any `--server-*` overrides) -- reusing the config unchanged when they agree. The connection comparison distinguishes the rendezvous location (host and path) from how the same drop is reached (the protocol/channel, port, and credentials): a location mismatch fails before any acceptance is sent, showing which values differ and how to resolve them, while an access-detail difference is applied to the exchange with a warning that the saved config is left unchanged rather than aborting. An absent URL field with no matching override is never a conflict, and credential values are never echoed in a warning. Linkage-key strings are compared NFC-normalized so Unicode-equivalent forms are not falsely flagged. A pre-existing key file on the accept path remains a hard error (a stale token must not be reused); online `invite` downgrades a pre-existing key file to a warning (it is overwritten by the rotated token on a successful handshake). The online write path now re-gates the config write immediately before persisting, so a config that appears between the pre-network conflict check and the post-handshake write is not silently overwritten -- matching the offline path. New `diffLinkageTerms`/`diffConnectionAgainstTarget` reconciliation helpers in the CLI. Specified in `docs/CLI.md` ("Offline acceptance", "Online invitation", "Online acceptance").
- Self-attested exchange record (a standalone disclosure-log entry): at the end of every successful exchange each party produces a local audit record of what it disclosed -- the agreed-terms hash, both self-asserted identities, the timestamp, the count of records this party exposed to the exchange (`recordsExposed`, a per-direction figure recorded on every record), the result size (only when both parties are entitled to it), a per-exchange binding nonce, readable governance metadata, and privacy-preserving commitments to the data exchanged (payload columns sent and received, and the association table when held). The two disclosure figures are gated deliberately: `recordsExposed` is the party's own input row count (every contributed record, not only those that resolve to a usable linkage key), known from its own input and recorded regardless of role, while the result size (the intersection cardinality) is recorded only in the both-output case, when both sides are entitled to the result. A single-output helper can observe its match count during the protocol, but the record deliberately does not surface it -- privacy is enforced by what the tool writes down, not by what is theoretically discoverable, and the helper's outbound disclosure is carried by `recordsExposed` instead, a framing that stays meaningful even under a future algorithm that discloses neither the result size nor the partner's set size. The governance metadata makes the record self-standing for a HIPAA accounting of disclosures or a FERPA disclosure record without re-matching the original config: the governing data-sharing agreement (reference, the purpose of the disclosure under it, and expiration -- all three mutually validated, so both parties' records agree on why the disclosure happened), the algorithm (`psi` vs `psi-c`), the categories of data exchanged (payload column names and data-dictionary descriptions sent and received, with the no-payload case explicit), and the matching basis (the standardized name and semantic type of each linkage field the keys reference for matching). It carries names, categories, descriptions, and references only -- never payload values, linkage-field values, or matched identifiers. Commitments are `HMAC-SHA-256` keyed by a fresh per-commitment salt over the canonical encoding, so the record binds to the data without revealing it (a bare hash of a low-entropy result would be brute-forceable). The record is split into a shareable record file and a private opening file (salts plus committed data); both are serialized via the canonical-encoding module and reproduce byte-identically across implementations (checked-in cross-impl vectors, verified on Node and in the browser). It is explicitly unsigned and not a non-repudiable receipt; signing it is separate, deferred work. Each party's record can also carry an optional self-facing retention/disposition pointer -- a free-text note from its own exchange config (`retention_disposition`) recording where it filed its copy of the result and under what retention schedule it is held or disposed of -- written into its own record only, per-party (the two parties' notes are independent and need not match), never exchanged with the partner or folded into the agreed-terms hash, and carrying no protected values. Exposed as `buildExchangeRecord` and returned from `runExchange` in `@psilink/core`; the CLI writes both files atomically to a timestamped default path (`--record-file`, `--no-record`) and the web app offers them as downloads. Specified in `docs/PROTOCOL.md` ("Self-attested record") and `docs/EXCHANGE_SPEC.md` ("Retention and disposition").
- Linkage-terms validation now rejects a party that declares it receives no output (`output.expects_output: false`) while listing `payload.receive` columns: a non-receiving party cannot receive payload for matched records it never gets, so the configuration is incoherent and fails at parse/compatibility time rather than producing a record that claims a party received payload it was never entitled to. A non-receiving party may still `send` payload.
- Invitations may now carry an optional credential-free connection endpoint: a public locator (PeerJS signaling URL, SFTP host and port, or file-drop directory) that tells the acceptor where to rendezvous, so the parties need not arrange that detail over a separate channel. Modeled in `@psilink/core` as the `ConnectionEndpoint` discriminated union on `InvitationToken.connectionEndpoint`, discriminated by `channel` like the `connection` block. The endpoint never carries credentials -- it has no field for a password, private key, key file, or PeerJS API key, and an invitation carrying one is rejected when decoded. The field is optional, so existing invitations remain valid. This reverses the prior invariant that an invitation excludes all connection information; the replacement invariant is that it may carry a locator but never credentials.
- Canonical encoding for receipts: a language-independent, deterministic byte serialization (RFC 8785, JSON Canonicalization Scheme) used for everything that is hashed, committed, or signed, so receipt hashes and signatures verify across independent implementations. Exposed as `canonicalString`/`canonicalBytes` (with `safeIntegerSchema` for hashed numeric fields) in `@psilink/core`, specified normatively in `docs/CANONICAL_ENCODING.md`, and backed by checked-in cross-implementation test vectors verified byte-identically on Node and in the browser. Foundation for the self-attested record and the signed non-repudiation receipt.

- `@psilink/core/testing` subpath export: `withSuppressedLogs` and `withCapturedLogs` helpers for controlling loglevel output in tests.
- `EncryptedMessageConnection` in `@psilink/core`: an application-layer AEAD decorator over `MessageConnection` that encrypts every outbound message and decrypts every inbound one with AES-256-GCM, keyed per direction from the key-exchange session key, so an SFTP/filedrop administrator sees only opaque ciphertext. Any integrity, replay, ordering, or format failure surfaces as a terminal `security`-kind `ConnectionError` rather than a silent transport drop. The CLI wires it into recurring (authenticated) exchanges: immediately after authentication `runProtocol` wraps the connection in the decorator and runs the PSI exchange through it, so recurring CLI exchanges are encrypted on the wire end to end. Zero-setup exchanges (no session key) and the web application's application layer are not covered. Of the channel-hardening follow-ups originally deferred here, the inbound frame-size bound, the companion directory-listing bound, and strict mid-stream sequence-gap detection are now enforced; end-of-stream / truncated-tail detection remains an accepted deferral that does not block release on its own. See `docs/SECURITY_DESIGN.md` ("Channel security").
- Key rotation: the shared token is automatically replaced with a new one derived from the key-exchange session key at the end of each successful handshake.
- `filedrop` channel: file-system-based exchange for environments where neither SFTP nor WebRTC is available. Both parties read and write through a shared locally-mounted directory.
- `retain_files` mode for the `sftp` and `filedrop` channels: the receiver writes a zero-length acknowledgment marker after consuming each message instead of deleting it, so no exchange file is deleted as a protocol step and the shared directory becomes a permanent transcript. Requires `lockless_rendezvous` and `timestamp_in_filename`; the `--retain-files` CLI flag implies both. Intended for sync-mediated transports that do not propagate deletions and for audit retention.
- Bilateral mode mismatch detection at rendezvous: each party advertises its `lockless_rendezvous` and `retain_files` settings in its hello, and a mismatch fails fast on both parties with a clear usage error (exit 64) naming each side's setting, instead of stalling until the peer timeout.
- `connection.options.unexpected_files` (`sftp`/`filedrop`): policy for a file that appears in the shared directory during the message loop and is neither part of this exchange nor an in-flight temp write -- `error` (fail with a usage error, exit 64, naming the file and the directory path), `warn` (log once per distinct name and continue), or `ignore` (skip silently, the previous behavior). The setting is local, not bilateral; the two parties may use different values. When unset it defaults to `error` on plain transports and `warn` when `retain_files` or `lockless_rendezvous` is set, where sync tools legitimately produce transient conflict copies and partial downloads. A malformed protocol file (a message-shaped name a correctly configured peer cannot produce) is always reported regardless of this setting.
- `zero-setup` CLI command: derive connection parameters directly from a URL, eliminating the need for a configuration file for one-off exchanges.
- `exchange` CLI command updated to require a `.psilink.key` file and perform the X25519 authenticated key exchange before any data is transmitted.
- Key file schema: versioned JSON format for persisting the pre-shared token and exchange metadata between sessions.
- Meta data inference: guess column semantic types from names and infers date formats.
- Custom linkage keys: linkage terms can be set in config and keys can be derived using transformations.
- Data standardization: transformations can create canonical linkage terms from input columns according to their semantic types.

### Changed

- `apps/cli` now declares `ssh2` as a direct, exact-pinned dependency (`1.17.0`) with a patch-pinned `@types/ssh2` (`~1.15.5`, the latest published types line), pins `ssh2-sftp-client` to an exact `12.1.1` (from `^12.0.1`), and runs the CLI integration suite in CI on the Docker-capable `ubuntu-latest` runner -- as a dedicated pull-request gate (`cli_build_and_test.yaml`, separate from the web-only Elastic Beanstalk workflow) and again in `release.yaml` as the ship gate before the Docker image publishes. The SFTP adapter deliberately drives `ssh2`'s raw `SFTPWrapper` internals past the public `ssh2-sftp-client` surface, yet `ssh2` entered only transitively and `ssh2-sftp-client` declares its `ssh2` range as `^1.16.0` -- so a transitive re-resolution could swap the very internals the adapter reads with no change to `apps/cli`'s declared deps. Exact-pinning both packages makes every bump -- including a security patch, which becomes a deliberate edit rather than an `npm audit fix` that slips in unreviewed -- a visible change that triggers the SFTP-stack upgrade re-verify; `@types/ssh2` type-checks that raw surface directly rather than via the lagging `ssh2-sftp-client` types; and the integration suite, the only CI coverage that exercises the real wrapper contract end to end (unit tests run against mocks), now fails the pull-request and release CI on a dependency bump that breaks the contract instead of merging or shipping green. The pins are cross-referenced to the re-verify checklist in `docs/SECURITY_DESIGN.md` ("Upgrading the SFTP stack") from `CONTRIBUTING.md`'s dependency policy and the adapter header. Upgrade-safety hardening only; no runtime behavior change.
- Online `psilink invite` now states up front, before it begins waiting, that the printed invitation can be accepted only while this command is waiting: cancelling (Ctrl-C), the connection timing out, or reaching `--accept-timeout` before the partner accepts leaves the rendezvous and discards the one-time setup secret, so the invitation can no longer be accepted afterward and a fresh `psilink invite` is required. This early invalidation was already true by construction for the online path -- the setup secret is held only in memory until a handshake succeeds and is never persisted before acceptance, the rendezvous files are swept on cleanup, and `authenticateConnection` rejects an expired secret before and after the handshake -- so the change is the user-facing notice plus removing the stale "does not yet revoke or expire the token early on timeout or cancellation" entry from the `docs/CLI.md` "Current CLI limitations" note. Early invalidation of a pending offline invitation (whose secret is written to a key file up front, with no waiting process) is a separate, tracked follow-up. Documented in `docs/CLI.md` ("Online invitation").
- The X25519 handshake now negotiates whether to apply the application-layer AEAD, replacing the implicit "encrypt iff authenticated" rule with a per-party signal carried inside the existing handshake. Each party declares a request-encryption flag on its own message (the initiator's on message 1, the responder's on message 2, no extra message or round-trip), the flag is bound into the transcript hash via `MixHash` (so tampering with either flag on the wire aborts the handshake at confirmation rather than yielding a split decision), and both parties derive the same `applyEncryption = own OR peer` decision, now exposed on `KexResult` and `AuthResult` in `@psilink/core`. `runKex` and `authenticateConnection` take a `requestEncryption` argument. The handshake protocol identifier is unchanged (no version bump pre-release): a flag-aware and a flag-unaware build already fail closed at parse -- the newer build rejects the older build's message for the missing required `reqEnc` field, and the older build's strict (`.strict()`) schema rejects the newer build's message for the unrecognized `reqEnc` key -- so the version tag stays reserved for a genuine cryptographic-suite change. The CLI requests encryption unconditionally on its SFTP and file-drop channels and gates the `EncryptedMessageConnection` wrap on the negotiated decision, so observable behavior is unchanged (authenticated file-sync exchanges remain encrypted; zero-setup exchanges remain cleartext); the mechanism unifies the CLI and the future web path, which will request it only when a DTLS-terminating WebSocket relay is forced into the WebRTC path. Known-answer vectors are regenerated to cover all four flag combinations. Specified in `docs/PROTOCOL.md` ("X25519 authenticated key exchange") and `docs/SECURITY_DESIGN.md` ("Channel security").
- Offline `psilink invite` can now source its linkage terms from a pre-existing configuration file instead of always requiring an `INPUT_FILE`. With a config present, invite derives the invitation's linkage terms from it and writes only the key file, leaving the config untouched. When both a config and an `INPUT_FILE` are supplied, the input's columns are checked against the config and invite fails -- naming the unsatisfiable linkage fields -- if the available data standardizations cannot transform those columns into the config's linkage fields. With no config, the previous input-only behavior is unchanged (terms, metadata, and standardizations are inferred from the input and written to a new config). Generating an invitation with neither a usable config nor an `INPUT_FILE` is a usage error. A pre-existing key file is still refused, so an existing shared secret is never clobbered. Documented in `docs/CLI.md` ("Offline invitation").
- The CLI now authenticates recurring exchanges with the X25519 authenticated key exchange (`runKex`) instead of SPAKE2: `authenticateConnection` derives the session key from the new handshake, and the existing application-layer AEAD (`deriveAeadKey`) and shared-secret rotation re-point onto it unchanged in shape (the AEAD wire format and known-answer vectors are untouched; only the session-key input changes). Breaking and intentionally non-interoperable -- an old SPAKE2 proof-of-concept client and a new one fail closed at the handshake with a generic error rather than hanging -- accepted because the project is pre-release. The shared credential is renamed `pake_token` -> `shared_secret` across the YAML field and its config-key alias, the `@psilink/core` identifier `SHARED_SECRET_REGEX`, the CLI-internal `generateSharedSecret`, and the `KeyFile`/`Authentication`/invitation-envelope fields, and the internal HKDF rotation label `psilink-token-rotation-v1` -> `psilink-shared-secret-rotation-v1`; the `.psilink.key` filename is kept. Specified in `docs/PROTOCOL.md` ("X25519 authenticated key exchange") and `docs/SECURITY_DESIGN.md` ("Key-agreement design").
- Renamed the default file-sync rendezvous mode from "wave" to "lock" (the counterpart to the existing "lockless" mode) and its on-disk tiebreaker file from `<id1>-<id2>.wave` to `<peer1>-<peer2>-lock.json`, bringing the tiebreaker under the same control-file filename grammar as every other protocol file. Terminology and on-disk naming only: no protocol behavior or config-schema change, and `lockless_rendezvous: false` still selects this mode. Old and new builds do not interoperate at rendezvous (one writes `.wave`, the other `-lock.json`), so both ends must run the same build.
- Library API: `runExchange` and `authenticateConnection` now take a `MessageConnection` instead of the event-based `Connection`. CLI exchanges are unaffected.
- An exchange now fails with a clear error (bounded by `peer_timeout_ms`) when the partner stops responding, instead of hanging.
- A peer sending messages out of turn is rejected rather than buffered without limit.
- The CLI now exits with code 64 (EX_USAGE) instead of 69 (EX_UNAVAILABLE) when `synchronize()` or `send()` fails due to a configuration or usage problem: a dirty exchange directory (preexisting handshake files), multiple concurrent sessions on the same path, or a send timeout. Transport failures continue to exit with code 69.
- `psilink exchange` now exits 64 (EX_USAGE) for every malformed, unreadable, or missing configuration (`psilink.yaml`) or key (`.psilink.key`) file. Previously only a missing config file exited 64, while a missing or malformed key file and a malformed config exited 69 (EX_UNAVAILABLE), the code reserved for transport failures. A malformed shared secret is now classified the same way whether the key file is read or written.
- The CLI now exits 64 (EX_USAGE) instead of 69 when the requested channel or URL scheme is unsupported -- a `webrtc` config or a `ws://`/`wss://` URL, an unknown URL scheme, or a malformed `file://` authority -- in both `psilink exchange` and the zero-setup command, classifying it as invalid caller input rather than a transport failure.
- Invalid connection-option combinations -- a reserved or unaccompanied `peer_id`, a `retain_files`/`lockless_rendezvous` contradiction, and similar -- now exit 64 (EX_USAGE) instead of 69, whether they originate in `psilink.yaml` or a command-line override.
- Library API: `deriveAeadKey`'s `context` parameter is narrowed from an open `string` to the fixed `AeadContext` union, and `@psilink/core` now exports `AeadContext` and `AEAD_CONTEXTS`. Source-breaking for a direct TypeScript caller that passed a non-literal label; there is no live caller yet. Rationale under Security.

### Removed

- The hand-rolled SPAKE2 implementation (`packages/core/src/pake.ts`) and its `@psilink/core` exports (`runSpake2`, `Spake2Result`), superseded by the X25519 authenticated key exchange. No SPAKE2 coexistence or migration path is provided; this is a pre-release breaking change.

### Fixed

- Standardized linkage-key strings are now normalized to Unicode NFC before they enter PSI. NFC normalization runs unconditionally as the first step of every field's standardization pipeline -- including identity (no-step) and custom pipelines that never strip to ASCII -- so two parties holding the same logical value in different normalization forms (precomposed NFC vs decomposed NFD, the common macOS-filesystem vs Windows/most-database split) produce identical key bytes instead of silently failing to match. Previously the only normalization happened inside `remove_accents`, which is not guaranteed to run. `remove_accents` additionally re-normalizes to NFC after its NFD diacritic strip so no decomposed residue leaks into the key. The same guarantee covers config-supplied literals that are compared against or injected into the value (`null_if` values, a `coalesce` default, a `replace_regex` replacement, a `pad_left` character) and the fully assembled key string, which is normalized once more after its elements are concatenated; regex patterns are matched as authored and are not normalized.
- Mid-pipeline steps that match an authored value, pattern, or delimiter against the intermediate value (`null_if`, `filter_regex`, `extract_regex`, `replace_regex`, `split_on`, `parse_date`) now NFC-normalize the value they inspect before matching. `to_upper_case` can emit non-NFC bytes for six code points (e.g. the Greek `U+0390`), so a comparison authored correctly in NFC but placed after a case-fold could previously miss -- silently letting an excluded value survive into the PSI set, or wrongly dropping a record whose filter, extraction, replacement, or split should have matched. The final assembled-key NFC pass added above only fixes the emitted key, which is produced after these mid-pipeline reads. Output for already-canonical and pure-ASCII inputs is byte-identical (`null_if`/`filter_regex` return the original value; the steps that derive a new value derive it from the normalized form). Follow-up to the Unicode NFC normalization above.
- `connection.provider_options` keys are now passed verbatim, as the spec now documents. Previously the config writer snakeized them to disk and the reader camelized them back, so a key authored in the casing the transport library expects -- e.g. ssh2's `readyTimeout` -- was rewritten (`ready_timeout`) rather than preserved. The map's contents are excluded from key-case transformation on both the write and read paths; all other schema fields, including function `params`, are normalized as before.
- A malformed message from a peer is reported as a protocol error instead of crashing.
- A failed exchange surfaces its original cause instead of a generic connection error.
- Closing a file-sync connection now cancels any in-flight rendezvous or send wait promptly instead of letting the timer fire and resume against a connection that is tearing down. Internal hardening only: cancellation threads a single `AbortController` through every wait site, and a new internal `ConnectionClosedError` may appear in debug logs on a close-during-wait. Not a user-facing exit-code change (a deliberate close under a signal still exits 130/143).
- Wave-path rendezvous: a party whose hello is deleted by a joiner that then fails mid-arrival no longer stalls until the peer timeout. The joiner now signals its arrival with a `<id>-joining.json` sentinel, and the waiting party recovers within a bounded window or aborts with a distinct transport error.
- File-sync rendezvous now sweeps an orphaned in-flight temp file (`temp-*.tmp`) left in the exchange directory by a `send()`/`writeAck()` whose process was hard-killed between the temp write and its atomic rename. The artifact is deleted at rendezvous setup instead of accumulating as litter across crashed exchanges or aborting entry with a spurious usage error. Only `temp-*.tmp` is swept, never the `*.json` message files (so retain mode's transcript is untouched).
- `MessageConnection`: a clean half-close that defers behind buffered inbound frames now tears the transport down at half-close time instead of only when the buffer is later drained, so a consumer that abandons the half-closed connection without draining or closing it no longer leaks the transport's listeners/channel. Internal hardening collapses the connection's terminal lifecycle into a single explicit state, so each method's terminal behavior derives from one source of truth rather than from four hand-duplicated flags. No happy-path or API-surface change for a correct caller; two intended changes land on abnormal/teardown paths: teardown now runs at half-close call time, and an already-buffered frame is drained before any abnormal terminal error surfaces (previously a frame buffered behind an abnormal drop was discarded), uniformly across transports.
- An already-expired (or malformed) shared secret now fails before any connection is opened, with the correct "obtain a new invitation" recovery hint on both parties. The pre-handshake secret-precondition check (well-formed base64url 32-byte value; not past `expires`) is knowable from local state alone, but it ran only inside `authenticateConnection`, which executes after the file-sync rendezvous -- so a dead credential first drove the rendezvous, and the losing side of the lock-file race could be shown a misleading `peer appears to have abandoned the handshake ... Retry the exchange` error (retrying never helps an expired token) depending on which side tore down first. `runProtocol` now runs the check in its pre-flight, ahead of `connect()`, so neither party enters the rendezvous, both reject deterministically with the specific re-invite hint, and no hello/lock files are written. The check is extracted into a single shared `assertSharedSecretReadyForHandshake` helper in `@psilink/core` that both `runProtocol` and `authenticateConnection` call, so there is one definition rather than a re-implemented copy; `authenticateConnection` keeps it as the authoritative boundary for library consumers and retains its separate post-handshake expiry check (for a secret that expires during the key-exchange round-trip). This also removes a timing-dependent unit-test flake: the two-party expired-token test asserted both sides reject with `expired`, which the rendezvous race intermittently broke (~1 in 10) by surfacing the abandoned-handshake message on the loser instead.
- Empty-linkage-round desync in the multi-key PSI matching loop (`packages/core/src/link.ts`). The loop skipped a key whose local de-duplicated value set was empty (`if (data_j.length === 0) continue;`), but that emptiness is computed per party from each side's own usable values, so on a later key (`j > 0`, where the set is built only from a party's still-unmatched records) one party could skip a round the partner still ran -- the partner then called `identifyIntersection` and blocked indefinitely on a `receive()` the skipping party never answered. Concretely it triggered under one-to-one / many-to-one cardinality when one party was fully matched in an earlier round while the other still held unmatched records. The skip is removed so both parties run `identifyIntersection` for every agreed key (the key set is fixed by the linkage terms, and the PSI library returns an empty intersection without error for an empty input set, so a locally-empty round is a correct no-op), keeping the exchange in lockstep by construction rather than by coincidence of equal data. This pre-existing bug is independent of the AEAD layer; its resolution restores the unconditional lockstep premise that the channel-security truncated-tail deferral reasons from (see `docs/SECURITY_DESIGN.md`, "Channel security").

### Security

- Invitation tokens carry a bounded lifetime (default 1 hour); rotation tokens carry no expiration, making them suitable for recurring scheduled exchanges.
- `deriveAeadKey`'s `context` parameter is constrained to a fixed, ASCII-only set of per-direction labels (`AeadContext`/`AEAD_CONTEXTS`, currently `"initiator-to-responder"` and `"responder-to-initiator"`) instead of an open `string`, with a runtime guard for untyped callers. This prevents a future AEAD caller from deriving the per-direction key from a variable, non-ASCII, or non-NFC label, which would make the two parties derive different AES-256-GCM keys and fail AEAD with an opaque auth-tag/decrypt error instead of a clear cause.
- Inbound frame-size bound on the file-sync transport: a read of any inbound file larger than a fixed maximum frame size (536,870,888 bytes, ~512 MiB) is refused with a terminal `FrameSizeExceededError` before the file is loaded into memory, closing a memory-exhaustion denial of service in which a hostile SFTP/filedrop server admin writes an arbitrarily large ciphertext file. Enforced at the transport read layer: the poll loop and rendezvous gate refuse over-cap files before `get()`, and each `FileTransportClient` adapter enforces a hard per-read byte cap (the local-FS adapter `fstat`s the open handle and reads exactly that size; the SFTP adapter aborts the transfer once a counting sink crosses the cap) so a server under-reporting size in its directory listing is still bounded. The maximum is anchored to Node's `MAX_STRING_LENGTH`, the ceiling above which the existing `JSON.parse(buffer.toString())` read cannot process a frame regardless; the AEAD decorator's `{ enc }` envelope schema caps the ciphertext at the same value as documented defense-in-depth. Specified in `docs/SECURITY_DESIGN.md` ("Channel security").
- Directory-listing bounds on the file-sync transport, the directory-enumeration sibling of the frame-size bound: a `list()` of a rendezvous directory with more than 8,192 entries, or any entry whose filename exceeds 255 characters, is refused with a terminal `DirectoryListingBoundsError` (a `UsageError`, exit 64) before the listing is materialized, closing a memory-exhaustion denial of service in which a hostile SFTP/filedrop server admin floods the shared directory with many files or very long names. Enforced at the transport `list()` layer in both adapters: the local-FS adapter streams entries through `fs.opendir` instead of `fs.readdir`, and the SFTP adapter reads one server batch at a time through the low-level `opendir`/`readdir`/`close` protocol instead of ssh2-sftp-client's `list()`, so an oversized or hostile directory is refused before an array proportional to the attacker-chosen entry count is allocated. The 8,192-entry cap is derived from a memory envelope (~5 MiB worst case, two orders of magnitude below the frame-size budget) and the 255-character cap is the POSIX `NAME_MAX`; both are fixed, not configurable. Specified in `docs/SECURITY_DESIGN.md` ("Channel security").
- Transport liveness bounds, the non-termination sibling of the frame-size and directory-listing memory bounds, in two layers plus an observability signal. The per-operation **read fast-fail layer (SFTP)**: every SFTP read awaits a server-driven callback, so a hostile server admin can hang it forever -- a `list()` that returns empty (`count = 0`) non-EOF readdir batches without end or never calls back, a `get()` that withholds data or never ends the stream, or a `createExclusive()` that withholds its open/close callback. Each is bounded and surfaces a terminal `TransportOperationStalledError` (a `UsageError`, exit 64), which both the poll loop and the rendezvous control-file gate treat as terminal (the gate now rethrows any typed `UsageError` from the hello `get()`, not just the over-cap `FrameSizeExceededError`, so a stalled hello read fails in a minute rather than being retried until the one-hour peer budget), instead of awaiting forever (and leaking any handle opened before the stall): `list()` by a 16,384 readdir round-trip cap (twice the 8,192-entry size cap) plus a 60-second wall-clock deadline, closing the directory handle on the way out; the capped `get()` by a 60-second per-chunk idle window, reset on each chunk so a slow-but-progressing large transfer is not failed; and `createExclusive()` and the rarely used uncapped `get()` by a 60-second whole-operation deadline. The memory bounds do not cover this vector (a progress-free stream never grows an allocation). The 60-second budget is roughly 3x the SSH stack's 20-second connection-unresponsiveness threshold and more than an order of magnitude below the one-hour peer-inactivity budget. All fixed, not configurable. The **whole-exchange backstop (universal)**: the per-operation bounds cover only the read path, leaving the always-executed write/stat/delete ops (`put`/`rename`/`delete`/`exists`) and the local-filesystem adapter unbounded, so a server that withheld the callback on a `put` or `delete` still hung the exchange forever -- `synchronize()` and `send()` awaited the call directly so it never returned, and the poll loop's reschedule sits in a `finally` a hung await never reaches, so the poller stopped silently; the peer-inactivity budget was checked only between loop iterations, never over an in-flight await, so a single withheld callback escaped it. The connection now races every transport await -- reads, writes, stat, delete, and the local-filesystem path alike -- against a fresh `peer_timeout_ms` budget at the single consumer seam they all flow through, surfacing the same terminal `TransportOperationStalledError`. Because it wraps the transport interface rather than any one call, it is op-agnostic (covering ops not individually enumerated and any added later) and adapter-agnostic (it also closes the local-filesystem stalled-mount gap, where the earlier "trusted kernel" premise fails for a hung NFS/CIFS mount, and the capped-`get()` slow-drip, where under-cap bytes trickled forever never trip the per-chunk idle window but do cross the budget). It bounds a stalled await against a fresh budget per await, not total exchange duration, so it never caps a healthy long-running exchange. A non-fatal **slow-operation warning** (observability, not a security control) logs once at a fixed 30-second threshold (below the 60-second read fast-fail) naming the operation, elapsed time, and cheap observed progress where available (bytes-so-far and rate for `get`/`put`, entries for `list`, elapsed-only for the atomic ops); it does nothing headless, stays entirely outside the terminal-error paths, and adds no required config. Specified in `docs/SECURITY_DESIGN.md` ("Channel security").
- POSIX temp-file symlink-follow hardening for the shared owner-only and atomic file writers (`apps/cli/src/fileUtils.ts`): `writeFileOwnerOnly` and `writeFileAtomic` now create their sibling temp file on an exclusive, non-following descriptor (`O_CREAT | O_EXCL | O_WRONLY | O_NOFOLLOW`) and set the mode via `fchmod` on that descriptor, mirroring the create-then-restrict discipline the Windows branch already used (an `O_EXCL` placeholder, then ACL restriction). Previously the POSIX path unlinked the temp path and then `writeFileSync`'d it with neither `O_EXCL` nor `O_NOFOLLOW`, so an attacker who planted a symlink at `<destPath>.tmp.<pid>` in the unlink-to-write window could make the write follow the link and write through to its target -- a temp-file TOCTOU / symlink-follow. `O_EXCL` refuses to open through an existing entry at the temp path and `O_NOFOLLOW` refuses a final-component symlink, so the create now fails cleanly rather than following the link, and the explicit `fchmod` preserves the prior umask-correcting intent (exact `0600` for the rotating `.psilink.key` and the signing identity, the explicit mode for the exported certificate) on the descriptor rather than on a resolved path. Real-world exploitability is low -- the temp file is a sibling of the destination, so planting the symlink already requires write access to the directory the `0600` mode assumes against -- and the writer is the shared chokepoint for the key file, config, exchange records, signing identity, and exported certificate, so the fix covers all of them.
- SFTP wrapper-crash bound, the one hostile-server behavior the liveness bounds above did not cover (a crash is neither memory nor time exhaustion): the SFTP adapter drives the raw ssh2 `SFTPWrapper` directly, and after connect that wrapper carries no `'error'` listener -- ssh2's `Client.sftp()` strips its setup-time listener before returning the wrapper, and `ssh2-sftp-client` attaches `'error'` handlers only to the SSH `Client` and to per-operation streams. A server returning a malformed SFTP reply packet (NAME, HANDLE, DATA, an unknown type, or a duplicate VERSION) drives ssh2's fatal-error path to emit `'error'` on a listener-free `EventEmitter`, which Node escalates to an uncaught exception that crashes the CLI -- skipping the lock/temp-file cleanup and typed exit-code mapping an orderly failure runs. The adapter now attaches a single guarded `'error'` listener to the wrapper at connect time (once per wrapper instance: a reconnect's fresh wrapper is re-guarded, a repeated connect does not duplicate it), so a fatal protocol error is handled rather than fatal. The session is left dead but the process alive: ssh2's `doFatalSFTPError` emits, destroys the wrapper, then calls `cleanupRequests`, which fails every request still registered -- but a malformed reply to the in-flight request itself is deleted from ssh2's pending table before the fatal path runs, so `cleanupRequests` does not fail it and that in-flight `list()`/`get()` is bounded by the 60-second wall-clock deadline (it is a fatal error on a different request id, or a connection-level fault, that `cleanupRequests` fails promptly); the captured cause then bounds the NEXT operation, which short-circuits at entry with the same terminal `TransportOperationStalledError` instead of buffering a request the dead channel can never answer. Every server-driven operation consults that guard -- `list`, `get`, `createExclusive`, `put`, `delete`, `rename`, and `exists` reject promptly, and `safeDelete` (which must never reject) resolves promptly as a best-effort no-op -- which matters because after a fatal error the SFTP channel is destroyed while the TCP/SSH socket stays up, so an unguarded write/stat/delete would buffer on the closing channel and hang teardown (`FileSyncConnection.close()` drives `safeDelete`/`delete`) rather than crash. For the distinct case of a server that withholds a callback WITHOUT any protocol error, no fatal `'error'` fires and the entry guard never trips: the read paths stay bounded by their 60-second deadlines, and the write/stat/delete stall (`put`/`delete`/`rename`/`exists`, which carry no per-operation deadline) is bounded by the whole-exchange backstop above. Specified in `docs/SECURITY_DESIGN.md` ("Channel security").

### Documentation

- Pinned the `EncryptedMessageConnection` AEAD message-envelope wire format with a checked-in cross-implementation known-answer vector: `packages/core/test/vectors/aead-envelope-vectors.json` fixes a session key, role, sequence number, and plaintext and records the exact serialized `{ enc }` envelope (base64url of IV || ciphertext || 16-byte GCM tag) for both a JSON and a binary payload, so an independent implementation can reproduce the bytes and any change to the nonce construction, byte order, envelope shape, or tagging is caught. Specified in docs/SECURITY_DESIGN.md ("Channel security") and asserted against the decorator in `packages/core/test/encryptedMessageConnection.test.ts`, alongside (and distinct from) the existing `deriveAeadKey` key-derivation vector.
- Documented the transport delivery contract - a send completes on local hand-off rather than peer delivery, and a clean close flushes the final frame before teardown - in docs/COMMUNICATION.md and the `Connection` and `TransportHooks` interfaces, stating the guarantee any future channel must satisfy.
- Recorded two close-teardown decisions in docs/COMMUNICATION.md (cross-referenced from the `ConnectionErrorKind` JSDoc and the web PeerJS clean-close wiring) now that the inbound drain is uniform in the shared message queue: the error-kind decision (a clean remote close stays kind `"transport"` rather than being unified with a local close's `"closed"`, because a remote channel-close event does not convey the peer's intent and the key exchange relies on the `"transport"` tag to report a vanished peer as a handshake timeout) and the narrowed per-transport residue (each transport wires which terminal control its events call - WebRTC maps `close` to `finish()` and `error` to `fail()` - while a shared `Connection.close` event is rejected as the wrong shape because it would reintroduce the no-listener drop window the pull-based queue eliminates).
- Corrected the non-repudiation receipt design in docs/PROTOCOL.md ahead of implementation: the receipt is a post-exchange audit artifact produced after the result already exists on both sides and does not gate or withhold result delivery; the session-derived (symmetric) signing mode is tamper-evident only and does not provide non-repudiation (its MAC is forgeable by either party), so true non-repudiation requires the certificate-backed asymmetric mode; and the receipt swap is best-effort evidence, not a fairness or atomicity guarantee. Clarified in docs/EXCHANGE_SPEC.md that `linkage_terms.identity` is self-asserted and carries evidentiary weight only when bound to a certificate.
