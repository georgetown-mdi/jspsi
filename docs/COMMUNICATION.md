# PSI-Link communication

This document covers the communication channels available for PSI-Link exchanges, how parties synchronize protocol steps, how errors are handled, and what supporting services are required. Authentication and channel security are not covered here - see [SECURITY_DESIGN.md](SECURITY_DESIGN.md) for those topics. It does not cover the PSI protocol itself (see [PROTOCOL.md](spec/PROTOCOL.md)) or CLI configuration (see [CLI.md](CLI.md)). The implementation-level rationale for the terminal `ConnectionErrorKind` taxonomy is specified in the [transport-contract spec](spec/COMMUNICATION.md). Intended readers are IT staff and developers.

# Communication

The protocol components above define what data parties exchange; this section describes how that exchange is carried over a network. This includes the channels available for transmission, how parties verify each other's identity, and how they coordinate the sequencing of protocol steps. Authentication and channel security are covered in [SECURITY_DESIGN.md](SECURITY_DESIGN.md).

## Channels

If the exchange is to be accomplished without additional infrastructure, it must utilize existing communication channels. Three communication channels are currently supported:

* Peer-to-peer using WebRTC - this is a protocol that is primarily used by browsers to communicate with each other, for example when conducting video calls. Peer-to-peer connections can be difficult to establish when parties are behind corporate firewalls and using Network Address Translation (NAT). To facilitate these connections, a third-party server typically needs to be available to execute to either help establish the connection, or to explicitly route the traffic.
* SFTP - for many exchanges, one partner already runs an SFTP server that is used for secure file transfers. SFTP is less-than-ideal for a communication protocol, as it is a file transfer protocol and not a direct connection. That said, with frequent polling and strategies to resolve synchronization, it can be treated as a message passing channel with slight delays for each message. As the number of messages is independent of the size of the datasets, this represents a fixed, tolerable time cost.
* File-drop - transfers can be made through directories that both parties can access, for example an NFS or SMB network share provisioned by IT, or a folder backed by an SFTP server and mounted locally. The same file-based polling and synchronization protocol used for SFTP is applied directly to the mounted path. This is the simplest deployment option when shared storage already exists. No additional supporting services are required beyond read/write access to the shared directory. It is possible to conduct an exchange where one party utilizes a file-drop that is synchronized to a directory that is served by an SFTP server to which the other party connects.

## Synchronization

The protocol requires both parties to execute it at the same time. For a new exchange where one party may "invite" the other, the inviter can listen for the other partner to respond. For scheduled exchanges, which party shows up first is arbitrary and in order to execute the protocol there needs to be a way of resolving who will "speak" first and who will "listen".

For WebRTC, this is solved by a single-threaded peer-coordination service which ensures only one party can be "first". For SFTP, an implementation has been written that utilizes the uniqueness of file handles and catching server errors to handle rare race conditions.

When using a public coordination server, WebRTC additionally requires both parties to know each other's peer IDs before a connection can be established. These IDs are derived from the shared cryptographic token: each party holds a role of either inviter or acceptor, and each independently computes both IDs from the token and their role. This makes peer discovery self-contained and requires no out-of-band address exchange. The inviter and acceptor roles, the PSI protocol's sender and receiver roles, and the initiator and responder roles from the linkage terms handshake (see [Linkage terms](EXCHANGE_REFERENCE.md#linkage-terms)) are all independent of one another and each operate in a distinct phase of the exchange. For a private peer coordination server, directory listings can be used.

## Web invitation

On the web, the inviting party composes an invitation from its own data file: it chooses the CSV and enters its name, and the browser reads the file locally (it is never uploaded) to derive the invitation's linkage terms from the file's columns -- the default terms, narrowed to the keys those columns can satisfy. Those derived terms are what the acceptor reviews and consents to, and the inviter's own half of the exchange runs on exactly the same terms and the same parsed rows -- so the two parties agree on what to link without the inviter re-deriving the terms or re-selecting the file. Composing fails closed before any setup secret is minted: a file that cannot be read, or whose columns satisfy none of the linkage keys, produces no invitation (the same zero-key condition the acceptor's pre-flight blocks on, naming the field types the file lacks). Once the invitation is generated the inviter moves to the same shared exchange screen, which leads with a share block -- the link and code to send out-of-band, and the invitation's expiry -- weighted above the terms summary it is proposing, and waits there for the acceptor to connect; on connect the share block disappears, since Status then shows the run is underway. Editing the proposed terms is available through an "Advanced options" entry point, which opens a column-aware editor seeded from those same file-derived defaults (never a blank form). There the inviter reviews and overrides the terms -- its name, the invitation lifetime, an optional legal agreement (which the partner must enter identically), a column metadata grid (each column's semantic type and one consequence-labeled disclosure choice -- used to match, row identifier, sent to your partner, or ignored -- where changing a column's type updates which matching keys are offered and the disclosure choice governs what the inviter sends), and which linkage keys are active and in what order (keyboard-operable move controls, with a per-key indicator of whether the inviter's columns can satisfy it) -- against a live preview rendered by the same component the acceptor's consent screen uses, so the inviter authors against the exact wording the partner will see. Validation runs through the same linkage-terms schema the exchange enforces; generating is blocked until the terms are valid and at least one key is satisfiable by the inviter's columns. The advanced path requires the CSV up front and parses it on entry to the editor (so the data-cleaning workbench's before-and-after preview can run over a sample of the rows; generate re-parses the same in-memory file), so only keys the columns can satisfy are offered. The quick path's transmission is unchanged: an inviter who never opens Advanced still gets the file-derived defaults. It does, however, now show -- once a file is chosen and before the link is generated -- which columns it will send to the partner for matched rows, derived from the same predicate the exchange transmits on (so the statement cannot drift from what actually leaves the machine), with a one-click path into Advanced options to change what is disclosed; when the quick path would send no columns it says so explicitly rather than going silent, so that single path into Advanced options stays reachable once a file is chosen. This is awareness only, not a gate: no column the quick path sends today stops being sent because of it. Because the quick path does not normalize disclosure the way the editor does, an inferred row-identifier column it still sends is shown as sent here. Those same disclosed columns are now also declared in the invitation's payload data dictionary, so the partner's consent screen lists them as columns it will receive for matched records -- the partner-facing counterpart of that inviter-side awareness, derived from the same predicate, and likewise changing only what is declared, never which columns are transmitted. The inviter also chooses who receives the matched results -- both parties, only the inviter, or only the partner -- now that the exchange honors one-sided output end-to-end: the acceptor derives its own output terms as the mirror of the inviter's, and a party whose agreed terms entitle it to no result receives none (the forbidden "neither receives" combination is not offered). An Expert authoring mode unlocks building linkage keys element by element -- each element's field chosen from the declared columns (never free-typed), with a per-element transform pipeline whose parameters are typed inputs, and a two-of-N "match in either order" swap drawn from the key's own elements -- plus a JSON/YAML import/export of the whole terms document, round-tripped through the same validation, with a pasted document that turns on an unapplied setting refused and an imported key that references a field the inviter's columns cannot supply loaded disabled but visible, so the keys those columns can satisfy still generate while the unusable key stays on screen to re-enable once a matching column exists. The editor also hosts the same per-field data-cleaning workbench the acceptor's "Prepare your data" editor uses, shown by default rather than behind expert mode (cleaning a field is a normal part of preparing for the exchange, not an expert act): the inviter authors each field's cleaning steps -- raw regular-expression steps included, each marked "advanced" -- and chooses which column feeds it, with the same whole-file silent-empty coverage warning the acceptor has and the same malformed-step gate blocking generation. Adding a second field of one semantic type bound to a distinct column (a maiden and a current name) and referencing both in its keys stays an expert action, since it is a key-authoring decision -- the declared fields the key editor offers are derived from this authored cleaning, so binding a second column declares a second field. Like the column metadata, the cleaning is per-party and local -- threaded into the inviter's own half of the exchange, never embedded in the invitation -- so it changes only the inviter's match rate, not the agreed terms. The matching algorithm (psi-c), deduplication, and fuzzy comparisons are surfaced as controls there but stay disabled until the exchange applies them, gated on the same applied-flags the consent screen reads, so a count-only or duplicate-matching setting cannot be authored while the run would silently not honor it; the built terms are clamped to the applied behavior regardless. Alongside them the inviter chooses the linkage strategy -- but, unlike those three, that control is enabled, because single-pass is honored end to end: it defaults to cascade (so an unchanged draft authors the same terms as before), and choosing single-pass surfaces at the point of choice the consented disclosure tradeoff it carries -- fewer network round trips over a high-latency channel in exchange for disclosing more to the other party -- then mints terms carrying that mandatory-consistency choice (both parties must agree on the strategy or the exchange aborts). The disclosure mechanics and the round-trip accounting are specified in PROTOCOL.md (linkage strategies). The guided (non-expert) path is unchanged. Per-column disclosure (which of the inviter's columns are sent to the partner) is set in the metadata grid; like the acceptor's prepared metadata, the inviter's is per-party and local -- threaded into its own half of the exchange, never embedded in the invitation -- so it changes only what the inviter discloses, not the agreed terms.

## Web acceptance and consent

On the web, the acceptor opens the invitation deep link, which decodes the invitation and enforces its expiry before any rendezvous or connect (failing closed on an expired or invalid token, with no consent action offered). It then renders the inviter's proposed linkage terms, organized for legibility around an always-visible core -- the matching method and the result-sharing directions -- and a default-collapsed "Matching strategies" disclosure for the per-key detail. The fields the keys match on are summarized always-visible above it, so which data is matched on is legible without expanding anything; the per-key breakdown sits inside, one collapsible disclosure per linkage key. Each key's header is its name and a short plain-language one-liner of the fields it matches on, with a terse marker on any field whose rule materially changes how it matches (routine standardization like case-folding or trimming is not flagged): a looseness word where the effect is determinable -- "partial" for a truncation, "fuzzy" for an approximate-match expansion -- or a direct name for a partner-authored rule whose effect is not a fixed direction, such as "pattern filter". Both the always-visible summary and that one-liner are derived from the schema-validated field types, not the partner's key or field names, so a key's name cannot misrepresent what it matches on. The key's own disclosure holds the dense, matching-affecting detail, hidden from assistive technology (not only visually) while collapsed: its ordered elements and, per element, the source field and any value transform -- led by a plain-language statement of what that transform does to matching (a substring on a name field as the literal character slice it takes; a date or other reformatted field is not described by character position, which the consent screen cannot verify), with the function name and parameters following as detail -- plus any fuzzy-comparison expansion or swap. A separate default-collapsed "Other details" disclosure holds each field's constraints, the optional payload columns, the legal agreement, and the deduplicate setting. Two of those buried facts also carry an always-visible presence hint in the core, so neither stays invisible until that disclosure is expanded: when the invitation requests extra data columns from the acceptor a count of them is shown, and when it attaches a legal agreement that is flagged -- the count and the flag only, with the column list and the agreement text staying in the disclosure. A `single-pass` linkage strategy is likewise surfaced always-visible: because it is disclosure-affecting (one party hands the other its full per-key value structure) and the acceptor adopts the inviter's strategy, the acceptor sees the same plain-language disclosure note the inviter read at the point of choice -- viewer-neutral, since which party discloses is settled at exchange time -- so it is consented to here, not just on the authoring side. cascade, the baseline that discloses less, is not flagged. Because a swap has the receiver read each swapped element's value through the other element's field (the field references swap while each element's rules stay with their element), the disclosure presents the swap in the receiver's terms: the always-visible header attributes each swapped field's breadth marker to the field whose value the receiver actually applies that rule to -- its swapped partner -- not the field the rule is declared on, and the per-key detail spells out the receiver's transformed-value cross-application, bidirectionally when both swapped elements carry transforms and one-directionally when only one does. Where a function coerces a declared parameter before applying it, it shows the value that actually executes alongside the declared one, so a displayed parameter cannot misstate the match. On that same review screen the acceptor gives affirmative consent (a checkbox plus their name) and chooses its own CSV -- or, if it already dropped one on the home page (where a single file drop sits below both the invite and accept boxes), finds it carried over and pre-filled here so it need not re-drop; either way both consent and a name are required before the file is parsed, so carrying the file forward never pre-empts the consent gate. The parsed file then opens in a "Prepare your data" editor: a grid maps each column to a semantic type and one consequence-labeled disclosure choice (used to match, row identifier, sent to your partner, or ignored), backed by a running summary of exactly which columns will be sent to the partner; a live verdict reports how many of the agreed linkage keys the current mapping can satisfy, computed over the same metadata and standardization the run will use; and the operator remaps columns until the file complies. Beside the grid, a per-field standardization workbench turns the recommended per-type cleaning from an opaque one-liner into an editable, ordered list of steps: the operator adds, removes, and reorders steps -- chosen from a plain-language menu grouped by intent, each step's parameters rendered as typed inputs rather than raw text -- against a live before-and-after preview over a sample of the operator's own rows, which renders the three pipeline outcomes distinctly: a cleaned value, a dropped (empty) value, and a value that fans out into several match candidates. A cleaned value that does not meet a field's declared constraint is flagged with a warning badge, surfaced but not enforced (the application warns on constraints, it does not block on them). Because that preview runs on a row sample, a separate whole-file check guards the failures the sample cannot show: for each field the editor computes -- off the main thread for a large file -- the share of the operator's rows that still produce a value after cleaning, and raises a prominent per-field alarm when a transform has collapsed the field to no value for any row. That collapse is otherwise invisible: it passes the satisfiability verdict (which checks that a column of the right type is bound, not that cleaning leaves anything behind) and produces a result byte-indistinguishable from a genuine non-match, so the alarm is the operator's only chance to catch a mis-set transform before launching. An empty cleaned value still counts as produced -- it is a participating key, distinct from a dropped value, and a cleaning step can deliberately turn it into a drop -- so an all-empty field reads as covered rather than as this collapse. It informs rather than blocks, like the partial-coverage advisory. A sibling, terms-derived advisory complements that whole-file sweep: a linkage key whose own cleaning can never produce a value -- a `parse_date` whose input format omits a component, so it drops every record regardless of the data -- is flagged from the agreed terms alone, with no file needed, since unlike the operator's own cleaning the fix is the partner's to make (a corrected invitation), not the operator's file. The operator can author and edit the raw regular-expression cleaning steps (replace, extract, filter, and split) for a field directly, with no opt-in switch; each pattern is validated against the supported linear-time dialect and bounded in length, and those steps run under a non-backtracking engine, so even a pathological pattern bounds rather than freezing the tab. Each regex step is marked "advanced" and the live preview shows its effect, because a wrong pattern silently changes which of the operator's own rows match -- a self-directed risk that never reaches the partner or the agreed terms (the standardization is local), so it informs rather than gates. A malformed or incomplete cleaning step does block continuing, the same launch gate the satisfiability and single-identifier checks apply; that gate is also applied on the inviter's side now that raw patterns are authorable there. Raw patterns inside a partner-authored, token-embedded key transform are a separate cross-party trust surface and stay read-only on the consent screen, bounded by the wire dialect (see [PROTOCOL.md](spec/PROTOCOL.md#transform-regular-expression-dialect)). When more than one of the operator's columns has a field's semantic type, the field's input column becomes a selectable choice rather than a fixed note, so two linkage fields of one type (a maiden and a current name) can bind to distinct columns instead of both defaulting to the first. Like the column metadata, the edited cleaning is per-party and local -- it changes only the operator's own match rate, never the agreed terms. The former zero-key dead-end is now this editor's entry rather than a bounce back to the file picker -- the hard block remains, so an exchange that could match nothing still cannot be launched, but the operator fixes the file in place. A column inferred as a row identifier is not sent to the partner by default. The running summary of the columns that will be sent stays beside the editor as the standing last-look, so launching needs no separate confirmation step; it transitions to the shared exchange step, where the same terms summary is shown again as the agreed reference for the run and the exchange dials on arrival. The acceptor reaches that step only after consent commits and its data is prepared, so no rendezvous, key-exchange handshake, or PSI frame is set up before consent. The exchange then runs the inviter's terms with the acceptor's own identity substituted and its output direction mirrored -- the acceptor's "receives output" is the inviter's "shares with partner" and vice versa, so a one-sided invitation (where only one party receives the result) agrees rather than aborting; for the symmetric both-receive case the mirror is identical -- while the metadata and standardization the acceptor prepared -- and the payload disclosure it confirmed -- stay per-party and local: they are never embedded in the invitation or cross-checked, so they change only the acceptor's own match rate and disclosure, never the agreed terms it consented to, which the exchange links on verbatim. When the agreed terms give a party no result, the exchange withholds from that party (in both the web app and the CLI) both the matched result table and the partner's payload columns -- a non-receiving helper learns no matched records, so it is sent none of the partner's payload values and fails closed if it is sent any -- and the web run shows that it contributed to the match but receives no result, rather than an empty result or a failure. The deduplicate and fuzzy-comparison settings are shown as proposed but flagged not-yet-applied, since the run is currently one-to-one and fuzzy expansion is unimplemented. All partner-controlled text in the terms is escaped at display (see [SECURITY_DESIGN.md](SECURITY_DESIGN.md), "Channel security"). The rendezvous peer-id derivation that places the two parties on the connection is specified in [PROTOCOL.md](spec/PROTOCOL.md).

## Message delivery and teardown

Each channel passes messages through a common transport contract whose one subtlety is worth stating explicitly, because it is easy to misread when auditing the code: a send completing does not mean the peer has received the message. There is no end-to-end delivery acknowledgement at this layer. What a completed send does guarantee differs by channel, and that difference is the whole contract. Every channel must guarantee that the final frame of an exchange survives a clean connection close, and it may do so in one of two ways.

The first way is a durable send with a draining close: the send writes the message durably before resolving, and the close waits for the peer to consume the last written message before sweeping it. The file-based channels - SFTP and file-drop - work this way. A send writes the message as a file in the shared directory and does not resolve until that write (and its atomic rename into place) is durable, so the frame is committed to disk before the caller proceeds. Because the sender's close also deletes the files it is responsible for, durability-on-disk alone is not sufficient: the close must drain the last sent file - wait for the peer to consume it - before cleanup runs. This gives the precise guarantee: the terminal frame is durability-until-consumed. Without the drain, the sender's cleanup could delete the file before the peer polls, and the peer would hang until its peer timeout.

The second way is a flushing close: the send only buffers locally, so a message can still be in flight when the sender finishes, and the close is what protects it. WebRTC works this way. A send hands the message to the data channel and returns before it is on the wire, so the final frame is guaranteed not by the send but by the close: a clean (non-error) close must flush the buffered frames before tearing the connection down, and the underlying reliable, ordered data channel delivers them ahead of the close signal the peer acts on. A close caused by an error does the opposite - it discards buffered outbound writes, because an errored link is already unusable.

The WebRTC case is the one that looks like a race when reviewed in isolation: the last operation a party performs is a non-blocking send immediately followed by a close, which appears to drop the frame. It does not, provided the close flushes - so a clean WebRTC close must not be shortcut into an abrupt connection teardown that bypasses the flush. The file-based channels have a different hazard: because the sender is responsible for deleting its own files, a send-then-immediately-close can delete the terminal file before the receiver polls. The drain in close() is what closes that gap.

Any new channel added to the project must satisfy one of these two conditions - a durable send with a draining close, or a flushing clean close. A transport that does neither will silently drop the final frame of an exchange. The same rule is recorded for implementors in the `Connection` and `TransportHooks` interface documentation in the core library.

The rules above govern outbound frames. Inbound delivery has a complementary guarantee that is easy to conflate with the outbound discard-on-error just described: a frame that has already arrived and been buffered is handed to a waiting `receive()` before any terminal error surfaces, whether the connection ended in a clean half-close or an abnormal drop. The two buffers are distinct - an errored close discards pending outbound writes, but never a received frame already sitting in the inbound queue. This is uniform behavior of the shared message queue, which sits above every channel, rather than a per-channel obligation; it is recorded for implementors in the `TransportControls` interface documentation alongside the rule above.

That inbound drain being uniform leaves only a small per-transport choice behind: each transport still wires which terminal control it calls on its own underlying events. Only one channel is close-capable today - WebRTC, in `apps/web/src/psi/peerMessageConnection.ts` - and it maps the data channel's `close` event to `finish()` (the clean half-close) and its `error` event to `fail()` (the abnormal drop); the file-based channels surface only `fail()`. This residue must not be removed by introducing a shared `Connection.close` *event*. A broadcast close event would be delivered only to whatever listener happened to be attached, reintroducing the no-listener drop window that the pull-based queue was built to eliminate - the same window that makes a completed receive, not an event callback, the unit of delivery. Teardown therefore stays driven through the queue's `finish()`/`fail()` controls rather than an event.

## Error handling

In the case of communication channel errors, messages that fail to transmit are retried. The initial connection is retried on a transient failure up to a user-specified limit (`max_reconnect_attempts`); a connection attempt that times out is terminal and is not retried, since a retry cannot recover an unresponsive server or mount, and there is no reconnection after a connection drops mid-exchange. As the communication channels in use guarantee message correctness, any message that fails to validate indicates a deviation from the protocol and results in program termination.

Terminal failures are tagged with a `ConnectionErrorKind` (the kinds and their meanings are documented on that type in the core library) so a consumer can branch on the cause. A deliberate local close surfaces as `"closed"` (nothing went wrong on this side), while a clean remote close, a dropped link, and a peer timeout all surface as `"transport"` (the link ended; a retry is reasonable). The deferred-decision rationale for that taxonomy -- why a clean remote close is not given its own kind, why a non-attributable local crypto/runtime fault is tagged `"transport"` rather than `"security"` or a dedicated `"system"` kind, and the condition that would reopen each -- is in the [transport-contract spec](spec/COMMUNICATION.md#connectionerrorkind-classification).

## Supporting services

The communication channels described above each depend on supporting infrastructure. These services can be deployed in two patterns to minimize resource consumption between exchanges:

**Lifecycle provisioning**: the service has a fixed, permanent address but is started on demand before each exchange and stopped afterward. Both parties know the address statically and may independently trigger startup. This is the preferred model for most supporting services and maps well to serverless compute platforms.

**Address-returning provisioning**: a fresh resource is allocated for each exchange and its address is not known in advance. This is inherently asymmetric - the inviting party allocates the resource during the invitation flow, and the resulting address is communicated to the other party as part of the invitation. By the time either party runs the CLI, both have the static address in their configuration.

Some services support **symmetric provisioning**: both parties call the same provisioning endpoint independently at the start of each exchange and receive their own credentials without needing to coordinate the result with each other. ICE credential services operate this way.

Public fallback instances of each supporting service are provided. When a party uses a public instance, the application issues a warning because connection metadata or encrypted data must flow through a server outside their direct control. Parties are encouraged to operate their own instances when possible; deployment guidance and reference configurations are available in [DEPLOYMENT.md](DEPLOYMENT.md).

### STUN/TURN

For parties behind Network Address Translation (NAT) or corporate firewalls, an Internet Connectivity Establishment (ICE) protocol assists in establishing peer-to-peer WebRTC connections. STUN servers help each party discover their public address; TURN servers relay traffic when no direct path can be found.

STUN and TURN servers can be configured statically or their credentials can be obtained on demand from an ICE credential API. Commercial services such as Twilio Network Traversal Service return time-limited credentials at the start of each exchange. Both parties call the same endpoint independently - this is a symmetric provisioning operation requiring no coordination.

### WebSocket-to-TCP proxy

Browser runtimes cannot open raw TCP connections and must use WebSockets instead. A proxy server translates between the two protocols, allowing a browser to reach services such as an SFTP server that speak TCP natively.

This proxy is a property of the client's runtime environment rather than the server: a browser-based party must configure a proxy while a CLI party connects natively. The two parties' connection configurations may therefore differ here even when connecting to the same underlying server.

### Peer coordination

Establishing a WebRTC connection requires a signaling step in which both parties exchange session descriptions and ICE candidates before a direct channel can be opened. The project uses PeerJS for this: each party registers with a PeerJS-compatible server under a token-derived peer ID and then locates their partner using the complementary ID. PeerJS also provides the data channel implementation used for chunked data transfer over the established connection; signaling and data transport are therefore not independently separable in the current design.

The peer coordination server is well-suited to lifecycle provisioning. It is only needed during the brief signaling phase before the WebRTC data channel is established, after which the two peers communicate directly. Hosting it as a serverless WebSocket function - for example on AWS Lambda with API Gateway or Cloudflare Workers - allows it to cold-start on demand and go idle between exchanges with no standing cost.

### SFTP

A lightweight SFTP server may be operated as a drop zone for exchanges between parties who prefer file-based communication to WebRTC. Like the peer coordination server, an SFTP server is suitable for lifecycle provisioning: its address is fixed and it can be stopped between exchanges. Directory-level access controls on a shared SFTP server can alternatively provide per-exchange isolation without requiring dedicated instances.
