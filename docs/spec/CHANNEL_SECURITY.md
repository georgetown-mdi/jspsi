# Channel security construction

This document specifies the construction and implementation of PSI-Link's
channel-security controls: the application-layer AEAD envelope and key
derivation, the inbound integrity/replay/ordering checks, the memory and
liveness bounds the file-sync transports enforce against a hostile server admin,
the SFTP fatal-packet crash safety, the authenticated abort marker, and the SFTP
host-key verification. It is
the implementation-level complement to the **Channel security** overview in
[SECURITY_DESIGN.md](../SECURITY_DESIGN.md#channel-security), which covers what
each control protects against and why the design is sufficient; this document
covers the wire formats, the derived constant values and their justification, the
enforcement points, and the dependency premises each control rests on. It does
not cover the threat model or authentication design (see
[SECURITY_DESIGN.md](../SECURITY_DESIGN.md)), the PSI and key-exchange wire
protocol (see [PROTOCOL.md](PROTOCOL.md)), or the network channels and transport
contract (see [COMMUNICATION.md](../COMMUNICATION.md)). Intended readers are
security auditors and implementors.

# Application-layer AEAD

Both parties use the HMAC-based Extract-and-Expand Key Derivation Function (HKDF) to derive AES-GCM keys from the 32-byte key-exchange session key - two directional keys, one per direction (HKDF info strings `psilink-aead-v1:initiator-to-responder` and `...responder-to-initiator`) rather than a single common key - so that every key has a single sender, and messages are encrypted using Authenticated Encryption with Associated Data (AEAD). Each message carries a per-direction sequence number as its nonce, preventing replay and reordering. The separate per-direction keys are what keep that nonce construction safe: both directions number their messages from zero, so a single common key would encrypt two different messages under the same key and nonce - a reuse that is catastrophic for AES-GCM - whereas one key per sender keeps every key-nonce pair unique. The server admin sees only opaque ciphertext files, and tampering with a file causes the authentication tag to fail and the exchange to abort. The AEAD context is restricted to this fixed pair of per-direction labels (a closed union, guarded at runtime), not an open string. Both parties must derive identical per-direction keys from identical labels; a variable, non-ASCII, or non-NFC label would make the two sides derive different keys and fail the AEAD with an opaque error rather than a clear authentication failure, so the label set is closed by construction.

The concrete envelope the `EncryptedMessageConnection` decorator emits is `{ enc: base64url(IV || ciphertext || 16-byte GCM tag) }`, where the 12-byte IV is four leading zero bytes (authenticated, but not separately validated) followed by the 8-byte big-endian sequence number, and the encrypted plaintext is a one-byte type tag (`0` for a JSON object, `1` for a `Uint8Array`) ahead of the payload. This wire format is pinned byte-for-byte by a checked-in known-answer vector at [`packages/core/test/vectors/aead-envelope-vectors.json`](../../packages/core/test/vectors/aead-envelope-vectors.json), following the cross-implementation vector precedent of `canonical-vectors.json` and `exchange-record-vectors.json`: each vector fixes a session key, role, sequence number, and plaintext and records the exact serialized `enc` string, so an independent implementation can reproduce it from those inputs and any change to the nonce construction, byte order, envelope shape, or tagging is caught. The vectors are asserted against the decorator in `packages/core/test/encryptedMessageConnection.test.ts`, which separately pins the per-direction HKDF key derivation (`deriveAeadKey`).

## Associated data is deliberately not bound

The envelope binds no Associated Data (AAD), and this is a deliberate, recorded decision rather than an omission. AAD would matter only to stop a frame authenticated under one context (role/direction, session, or transcript) from being spliced or replayed into a different context that shares the same key. Here every such context is already separated below the AAD layer by a distinct key. Direction is bound by the key: each direction uses its own AES-GCM key, derived with a distinct HKDF info string (`psilink-aead-v1:initiator-to-responder` versus `...responder-to-initiator`), so a frame presented to the opposite direction decrypts under the wrong key and the tag fails. Session is bound by the key: both keys derive from the per-session key-exchange session key, which is transcript-bound and mixes in each run's random ephemeral scalars, so it is distinct even across two runs of the same shared secret - a frame from one session decrypts under the wrong key in another. Ordering within a single direction of a single session is bound by the sequence-number nonce and the strict-increasing replay guard. The only set of frames that share one AEAD key is those one sender emits in one direction of one session, and the decorator carries exactly one logical stream there - it does not multiplex sub-channels or protocol phases, and the one-byte payload type tag that distinguishes binary from JSON lives inside the authenticated plaintext. There is therefore no second logical context under a shared key for AAD to separate, so binding the direction (or a session id) as AAD would only restate what the key already enforces. The sequence number is the GCM nonce, not a substitute for AAD: the reason AAD is unnecessary is the absence of a second context under one key, not that the nonce stands in for AAD. Practically, the decorator holds only its inner connection, the session key, and the role - no session or transcript id is in scope to bind without new plumbing that belongs to the not-yet-wired caller integration.

Revisit this decision if a future change multiplexes more than one logical stream or protocol phase over a single direction's key, or collapses the per-direction key split into one shared key. In either case a context or sub-stream id bound as AAD would stop being redundant: it would still block cross-direction splicing, though not the key-nonce reuse that collapsing the key split would itself make catastrophic (AAD does not protect against nonce reuse). Until then, the per-direction key plus the sequence nonce are sufficient and no AAD is bound.

## Inbound integrity, replay, and ordering checks

Now that this layer is wired in (recurring CLI exchanges), its AEAD/MAC and sequence-number (replay and ordering) checks run at the protocol layer, on the output of a `receive()` call: the decorator validates and decrypts each inbound envelope inside `receive()` and rejects with a `security` `ConnectionError` on any integrity, replay, or ordering failure. These checks are deliberately not latched into the inner connection as a transport control (the `fail`/`finish` teardown signals) behind a buffered inbound frame. The message queue drains an already-received frame to the consumer ahead of any terminal error, whether the close was clean or abnormal (see [Message delivery and teardown](../COMMUNICATION.md#message-delivery-and-teardown)), so a check wired as a transport-level latch could hand a suspect frame to the consumer before the integrity alarm fires. The decorator honors this - it raises the failure from `receive()` itself, and the inner transports raise only `transport` and `protocol` terminal errors with no integrity check latched behind a frame - so the constraint holds and remains one to preserve.

Two channel-hardening concerns attach to this layer once it is wired into live recurring CLI exchanges, where each becomes live residual risk against a transport-level attacker (for SFTP, the server admin). One - the inbound frame-size bound - is now enforced (described below, alongside the companion directory-listing bound that closes the related directory-enumeration vector). The other - detection of dropped, withheld, and truncated frames - is now split: strict mid-stream gap detection is enforced (described next), and end-of-stream / truncated-tail detection is a deliberate accepted deferral on the reasoning recorded below. The gap-detection half was the release-blocking concern and is resolved (pending the explicit security review); the truncated-tail deferral does not block release on its own; the reasoning below records why it stands on the matching loop's self-driven lockstep - a premise the empty-round desync (board item 197674102) had made conditional and which that item has since restored.

Mid-stream gap detection is now enforced. The sequence-number check previously rejected only a number that was not strictly greater than the last accepted, which stopped replay and reordering but not omission: a gap (a sequence number that skips ahead) was accepted. The decorator now rejects any inbound sequence number that is not exactly the last accepted plus one, so a frame dropped or withheld between two delivered frames is caught here as a `security` failure on the output of `receive()` - the same protocol-layer surfacing as the replay and tag checks, never a transport-level latch. This did not require the once-feared send-path change ("advance the counter only on a fully successful send"): a send that consumes a counter value but then fails to transmit (an encrypt fault or an inner transport failure) already latches the wrapper terminal, so no later frame is ever sent, and a consumed-but-unsent value can only truncate the tail, never leave a gap between two delivered frames. Strict gap detection therefore raises no false alarm while the synchronous `sendSeq++` before the encrypt is retained - which is what keeps two concurrent sends from peeking the same counter and reusing an AES-GCM nonce. As net-new security behavior it is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## End-of-stream and truncated-tail detection

End-of-stream / truncated-tail detection is a deliberate accepted deferral, not a coverage gap to be closed with a marker. A truncated tail - a stream that simply stops, including a dropped final frame - is indistinguishable at the AEAD layer from a clean end, and no AEAD tag can authenticate the *absence* of a frame: an omitted frame can be detected only by expecting one and not receiving it. The PSI protocol is self-driven lockstep (see [PROTOCOL.md](PROTOCOL.md)): each party drives a deterministic script of matched send/receive calls, so the receiver always knows from its own script whether another frame is due. In the symmetric case a truncated tail therefore surfaces - as a stalled `receive()` that hits the inactivity timeout - and an explicit, integrity-protected end-of-stream marker could only reclassify that timeout from `transport` to `security`, not detect the omission any sooner. Reclassification alone does not justify net-new, security-reviewed cryptographic surface, so the marker is not built. Earlier framing gated this on a single hypothetical reopen condition - whether the two parties' lockstep loops could legitimately stop at different points. That condition was confirmed reachable and is now closed. The empty-linkage-round skip in the matching loop (`packages/core/src/link.ts`, `if (data_j.length === 0) continue;`) was computed per party from each side's own usable values, with nothing forcing the two sides to skip the same key: for `j > 0`, `data_j` is built only from a party's still-unmatched records, so under one-to-one or many-to-one cardinality a party fully matched in an earlier round had zero unidentified values for every later key and skipped it while a partner still holding unmatched records did not - the partner called `identifyIntersection` and blocked on a `receive()` the first party never answered. Board item 197674102 removed that skip: both parties now run `identifyIntersection` for every agreed key (the key set is fixed by the linkage terms, and the PSI library returns an empty intersection without error for an empty input set, so a locally-empty round is a correct no-op), keeping the exchange in lockstep by construction rather than by coincidence of equal data. The lockstep premise is therefore restored - unconditional within the implemented one-to-one and many-to-one cardinalities - and the truncated-tail deferral stands on the original symmetric-lockstep reasoning; any future cardinality loop must avoid reintroducing a per-party-data-gated skip. The desync was in any case a protocol-loop fault, not an absent-frame one: no AEAD-layer marker could have detected or repaired it, so its resolution neither revived nor depended on the end-of-stream marker.

## Inbound frame-size bound

The enforced control is the inbound frame-size bound. Left unbounded, the transport read path loads an entire frame into memory and JSON-parses it before any integrity check runs, so a hostile filedrop/SFTP server admin - an adversary under the threat model (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#threat-model)) - could write an arbitrarily large file and exhaust memory: a denial of service that allocates a string and byte array proportional to the attacker-chosen file size. The transport now refuses any inbound file larger than a maximum frame size of **536,870,888 bytes (~512 MiB)** before it is read into memory, surfacing a clear terminal error (a `FrameSizeExceededError`, exit 64) rather than allocating proportionally to the attacker-chosen size.

The value is deliberately chosen, not a round number: it is the exact value of Node's maximum string length (`buffer.constants.MAX_STRING_LENGTH` on 64-bit). That length is the hard ceiling above which the existing JSON-text read cannot process a frame at all - both read sites do `JSON.parse(buffer.toString())`, and `Buffer.prototype.toString()` throws above MAX_STRING_LENGTH - so anchoring the cap there (a) never rejects a frame the transport could otherwise decode, since anything larger already fails at `.toString()` regardless of memory, and (b) converts that late, opaque failure (raised only after the full attacker-sized buffer is already resident) into an early, clean refusal before the allocation. As a headroom check against the realistic worst-case legitimate frame: the largest PSI frame is one party's full encrypted set sent as raw elliptic-curve points (~64 bytes per element; see [PROTOCOL.md](PROTOCOL.md#psi)), base64url-expanded by 4/3 on the wire, so 512 MiB of wire text decodes to ~384 MiB raw, on the order of 6 million elements - already more than this single-frame JSON-text transport can carry, since a larger set would itself exceed MAX_STRING_LENGTH and require chunking before the cap could ever bind it. The bound is a fixed constant rather than a configurable option: a configurable bound risks an operator raising it high enough to reintroduce the DoS.

The bound is enforced at the transport read layer, which is the meaningful enforcement point: the poll loop and rendezvous gate refuse a file whose listed/declared size exceeds the cap before calling `get()`, and - so a server that under-reports a file's size in its directory listing cannot slip past that check - each `FileTransportClient` adapter also enforces a hard per-read byte cap (the local-filesystem adapter `fstat`s the open handle and reads exactly that many bytes, closing the append-after-stat TOCTOU; the SFTP adapter streams into a counting sink that aborts the transfer once the running total crosses the cap, buffering at most one stream chunk past it). A `.max()` on the AEAD decorator's `{ enc }` envelope schema caps the base64 ciphertext at the same single value, but that is documented defense-in-depth only, not the primary control: by the time the decorator runs, the inner transport has already read and parsed the frame, so the dominant allocation has already happened. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Directory-listing bound

A companion transport-read control bounds the directory listing itself. This one is independent of the AEAD layer and of file contents: the two parties rendezvous through a shared directory the adversary controls, and the transport's `list()` call enumerates it. Left unbounded, a hostile filedrop/SFTP server admin can write a very large number of files, or files with very long names, and drive the listing to allocate an array and per-entry metadata proportional to the attacker-chosen directory contents - the same memory-exhaustion impact class as the frame-size DoS, reached through a different vector (directory enumeration - entry count and name length - rather than a per-file body read). The transport now refuses a listing that breaches either of two bounds - more than **8,192 directory entries**, or any single entry whose filename exceeds **255 characters** - with the same kind of terminal, typed error as the frame-size bound (a `DirectoryListingBoundsError`, a `UsageError` subclass, exit 64). The refusal fails the exchange rather than silently truncating the listing to the cap and processing it: against a hostile rendezvous directory, refusal is the safer default, since truncation risks dropping a legitimate exchange file an attacker buried under filler.

Both bounds are derived, not round numbers, and both comfortably exceed what a legitimate exchange produces. The **8,192-entry** cap is set from a memory envelope: the worst-case at-cap allocation is the retained entries, each carrying a name of up to the 255-character limit plus per-entry object overhead (~600 bytes), so the listing allocation is bounded to roughly 5 MiB - about two orders of magnitude below the 512 MiB single-frame budget the frame-size bound already governs, so directory enumeration cannot become the dominant memory vector - while still exceeding the order-of-ten files a legitimate exchange produces (the two `-hello.json` files, at most one `-lock.json` or the `-ack.json` markers, transient `-joining.json` sentinels and `temp-*.tmp` writes, and the bounded set of PSI message frames) by roughly three orders of magnitude, with generous headroom for a retain-mode directory that accumulates message and ack files across many sequential exchanges before the operator rotates it (retention is an out-of-band operator responsibility). The **255-character** filename cap is the POSIX `NAME_MAX`, the per-path-component limit every mainstream filesystem enforces (ext4, XFS, APFS, NTFS): the longest filename a legitimate exchange writes - the ack marker of a timestamped message - is on the order of 120 characters, and a name longer than 255 cannot exist on a conformant filesystem, so it is necessarily synthetic (the SFTP protocol imposes no name-length limit, so a hostile server can return such a name in a READDIR response even though its own filesystem could not hold it). Both bounds are fixed constants rather than configurable options, for the same reason as the frame-size bound: a configurable cap risks an operator raising it high enough to reintroduce the DoS.

The bound is enforced at the transport `list()` layer in both adapters - the point before the listing is materialized - not at a higher layer that has already consumed it. The local-filesystem adapter streams entries through `fs.opendir` (which yields entries in bounded batches) rather than `fs.readdir` (which materializes the whole directory into one array, on top of the per-file `stat` fan-out that follows), so the entry-count check stops the walk - capping both the retained array and the stat fan-out - at the bound regardless of how many entries the directory holds. The SFTP adapter does not delegate to ssh2-sftp-client's `list()` (which passes the directory path to `readdir`, accumulating the entire remote listing into one array before returning); it opens a directory handle and reads one server batch at a time through the low-level `opendir`/`readdir`/`close` protocol, applying the checks as entries arrive, so a single READDIR response - itself bounded by the SSH transport's maximum packet size - bounds the allocation to at most the cap plus one batch. Guarding only the local adapter would leave the SFTP path - the one with the explicitly in-scope adversary, the server admin - exposed, so both adapters are bounded. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Per-operation liveness bounds

A third transport control bounds these operations for liveness rather than memory. It is the **per-operation fast-fail layer** - covering both the reads and, as of the write-path bounds described with them below, the always-executed write/stat/delete operations - beneath the whole-exchange budget described next (which is the universal backstop over every transport await). The size bounds above stop a hostile directory or file from exhausting memory, but they do not stop a hostile server admin from hanging the operation itself. Every SFTP read awaits a callback the server drives, so the server can make one never terminate: a `list()` can return valid but empty (`count = 0`) non-EOF READDIR responses forever (advancing neither size bound, since no entry accumulates and the end-of-directory status never comes, so the batch loop recurses without end) or simply never invoke a readdir/close callback; a `get()` can hold the read stream open and withhold data, or trickle under-cap bytes without ever ending it, so the transfer never completes; a `createExclusive()` can withhold its open or close callback. Because `list()` and `get()` run on the poll loop - which awaits them with no per-call timeout - and `createExclusive()` drives the rendezvous lock path, any of these is a liveness / non-termination denial of service: the exchange stalls forever (and a directory or file handle opened before the stall is leaked) instead of failing. Each read is now bounded by the mechanism its shape allows and surfaces a single terminal, typed `TransportOperationStalledError` - a `UsageError` subclass, exit 64, the liveness sibling of the memory-bound `DirectoryListingBoundsError` and `FrameSizeExceededError` - so the consumers that await these reads, the poll loop and the rendezvous control-file gate (`readControlFileWithGate`, which reads the peer hello over `get()`) alike, treat it as terminal and fail the exchange rather than retrying into the same hang. The gate keys this on the typed `UsageError` rather than a single error class, so a stalled hello read is terminal there exactly as an over-cap `FrameSizeExceededError` is - without it, a server that stalls the hello `get()` would be retried at the polling cadence until the one-hour peer budget instead of failing in a minute. The streamed `list()` carries a **16,384 readdir round-trip** cap for the empty-batch flood plus a **60-second wall-clock deadline** for the withheld-callback case, and closes the open directory handle on the bounded-failure path. The capped `get()` bounds the **60-second idle gap between chunks** (reset on each chunk), not the total transfer, so a legitimately large but slow-and-progressing read is never failed while a withheld or stalled one is. `createExclusive()` and the rarely used uncapped `get()` - neither of which has a progress loop - carry the **60-second whole-operation deadline**. The always-executed write/stat/delete operations are bounded by this same layer, each in the mode its shape allows. The metadata ops `rename()`, `delete()`, and `exists()` - each a single server round-trip with no payload - carry the same **60-second whole-operation deadline** as `createExclusive()`, with the same negligible false-fail profile, since a metadata round-trip completes in well under a second on any honest server. `put()` instead carries a **60-second progress idle window**, reset on each uploaded chunk: a tight flat deadline is wrong for `put` specifically, because a legitimately large ciphertext upload over a slow link can exceed it while still progressing, so - exactly as the capped `get()` bounds its idle gap rather than its total transfer - `put` bounds the gap between upload-progress chunks. The payload is streamed in bounded chunks so a withheld write acknowledgement stalls the source and trips the window while a slow-but-progressing one keeps resetting it; this uses ssh2-sftp-client's stream interface rather than driving the raw `SFTPWrapper`, so it adds no new internal-API coupling.

The bounds are derived, not round numbers, and all comfortably exceed any legitimate operation. The **16,384 round-trip** cap is twice the 8,192-entry size cap, so the two move together: a compliant server sends at least one name per non-EOF READDIR response (an empty non-EOF response is not protocol-conformant - a server out of names sends end-of-directory instead), so a legitimate listing makes at most 8,192 batches before the entry-count bound would already refuse it, and only in the pathological-but-legal case of one entry per batch; an honest server packs many entries per packet and finishes a normal rendezvous directory in a single batch, so the cap only ever fails a progress-free flood. The **60-second** budget is a single coarse "the server has gone silent" threshold, applied as a whole-operation deadline for `list()`, `createExclusive()`, the metadata write/stat/delete ops `rename()`/`delete()`/`exists()`, and uncapped `get()`, and as a progress-reset idle window for the capped streaming `get()` and for `put()`. It sits well above any legitimate operation - a normal listing, a lock-file create, and each chunk of a healthy transfer all complete in well under a second - at roughly three times the SSH stack's 20-second connection-establishment unresponsiveness threshold (ssh2's default `readyTimeout`, also this project's `serverConnectTimeoutMs`), so a transiently slow but live server is not cut off, yet more than an order of magnitude below the one-hour (3,600-second) peer-inactivity budget, so a withheld response fails the exchange in a minute rather than after an hour; applied as an idle window it never rejects a slow-but-progressing large transfer, only one that stops sending. All bounds are fixed constants, not configurable, for the same reason as the other bounds: a configurable budget risks an operator raising it high enough to reintroduce the denial of service. These per-operation bounds are on the SFTP adapter, whose every call crosses the network to the in-scope server admin; they cover its reads and its write/stat/delete operations alike. They do not cover the local-filesystem adapter (an earlier framing held its reads to face only a trusted kernel, `fs.opendir`/`fs.stat`, that exhibits none of these failure modes - but a stalled NFS/CIFS hard mount breaks that premise). That remaining gap is closed not here but by the whole-exchange budget described next. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Whole-exchange budget

A fourth liveness control is the **whole-exchange budget**: the universal backstop beneath the per-operation bounds above, and the *only* per-operation-free liveness bound for the local-filesystem adapter. The per-operation bounds fast-fail a stalled SFTP read or write in 60 seconds; the budget is the layer they were added *beneath*, not in place of. The always-executed critical-path write/stat/delete operations - `put` (every message and every hello/sentinel write), `rename` (every durable publish), `delete` (message consumption and rendezvous cleanup), and `exists` (the lock-path race check) - now each carry their own 60-second per-operation bound (above), but before those bounds existed this budget was all that stood between a hostile or dead server that withheld any of their callbacks and an exchange hung for an hour: `synchronize()` and `send()` awaited the call directly so it never returned, and the poll loop's reschedule sits in a `finally` a hung await never reaches, so the poller stopped silently with no error. The peer-inactivity budget (`peerTimeoutMs`, default one hour) was checked only between loop iterations, never over an in-flight await, so a single withheld callback escaped it entirely. The connection now races **every** transport await - reads, writes, stat, delete, and the local-filesystem path alike - against that budget at the single consumer seam they all flow through, surfacing the same terminal, typed `TransportOperationStalledError` on expiry. Because it wraps the transport interface rather than any one operation, it is op-agnostic (it covers operations not individually enumerated and any added later) and adapter-agnostic (it bounds the SFTP adapter and the local-filesystem `LocalFSClient` identically, closing the stalled-mount gap). It bounds a stalled await against a *fresh* budget per await (silence from the peer), not the total exchange duration, so it never caps a healthy long-running exchange; the budget is the same single coarse knob an operator already tunes, and it stays deliberately coarse as the backstop: the tight, false-fail-safe per-operation bound a slow `put` needs is the progress idle window above, not a flat budget timeout, which is exactly why the budget itself can remain one coarse knob rather than being split per operation. The capped `get()` slow-drip (under-cap bytes trickled forever, which resets the per-chunk idle window and so never trips the 60-second read bound) is caught here too, since total elapsed eventually crosses the budget - as is the symmetric `put` slow-drip, whose progress idle window the budget likewise backstops. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Connect-probe bound

A fifth liveness control bounds the connect probe itself, the one transport call deliberately outside the whole-exchange budget's seam. `connect()` runs before the exchange begins and is wrapped by neither the per-operation bounds nor the whole-exchange budget - it passes through `boundTransport` unwrapped - so each adapter bounds its own connect attempt instead: the SFTP adapter by ssh2's `readyTimeout`, which tears down the socket on expiry and strands nothing, and the local-filesystem adapter by a per-attempt `withTimeout` deadline around its `fs.access` reachability probe. On the local-filesystem adapter a stalled NFS/CIFS hard mount turns that probe into the same liveness hazard the post-connect bounds address, with a sharper resource cost: `fs.access` blocks a libuv thread-pool worker (not the event loop), and nothing in-process can cancel an already-dispatched blocking syscall - `fs.access` does not honor an `AbortSignal`, and libuv's `uv_cancel` only removes still-queued work, never work a pool thread is already executing - so the worker stays pinned until the OS releases the syscall, minutes on a hard mount. The per-attempt timeout fires on the event loop and fails the attempt, but retrying it (the prior behavior) launched a fresh probe each time, so under the default `maxReconnectAttempts` of 3 up to four such probes could stack, exhausting the default four-thread pool and starving every other `fs`/DNS/crypto operation in the process - a process-wide denial of service induced by a single unreachable mount. The connect now treats a per-attempt timeout as terminal: it fails fast after one timeout window, surfacing a `TimeoutError` (which extends `Error`, not `UsageError`, so it classifies exit 69 / `EX_UNAVAILABLE` like any transport-unavailable failure), rather than dispatching further probes; a fast-returning transient error - `EACCES`/`ENOENT` while a share or its permissions are still settling - still retries under `max_reconnect_attempts`, which governs only this initial connect, not any mid-exchange reconnection, of which there is none. This bounds concurrent stalled workers to one and the caller-visible latency to roughly one `serverConnectTimeoutMs` window; it does not bound the lifetime of that one worker, which only the OS reclaims, because no in-process mechanism can free a thread wedged in a single blocking syscall - only a killable worker thread or child process could, which is disproportionate for a connect pre-check. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Consumer-side terminal-error rule

These transport bounds share one consumer-side rule that ties them together, stated here because the bounds above each describe their enforcement in isolation. Every one surfaces as a `UsageError` subclass - the memory bounds as `FrameSizeExceededError` and `DirectoryListingBoundsError`, the liveness bounds as `TransportOperationStalledError` - and any consumer that awaits a transport call must treat a `UsageError` as terminal: it propagates the rejection and never retries it, swallows it, or proceeds as if the call had succeeded. This binds the read consumers (a `list()`/`get()`/`createExclusive()` in the poll loop or the rendezvous gate) and, now that the write/stat/delete ops carry their own per-operation bounds, the write consumers too - the `put`/`rename` publish in `send()`, the rendezvous lock-path `createExclusive`/`exists`, the consume-`delete` that signals a message consumed, and the retain-mode ack-write (`writeAck`'s `put`/`rename`) that signals it on a transport that never deletes. Retrying loops straight back into the hang or the over-allocation the bound exists to prevent, and swallowing-then-proceeding turns a stalled consume into a re-emitted duplicate - a swallowed bound is no bound - so the only effective response is to fail the exchange, which is why the poll loop stops on a `UsageError` (including from the consume-`delete`, which rethrows it rather than swallowing it as it does a transient delete failure), the rendezvous control-file gate (`readControlFileWithGate`) rethrows it, and the `send()`/joiner write paths propagate it. The terminal behavior keys off the `UsageError` base class rather than each subclass, so a future bound added to the family inherits it. This consumer rule is what makes the transport-layer bounds effective end to end. The consumers are pinned by behavioral tests in `packages/core/test/fileSyncConnection.test.ts` that drive each with a `UsageError` from a transport call - a stalled read in the poll loop and the gate, a stalled consume-`delete` in the poll loop, and a stalled retain-mode ack-write in the poll loop - and assert it is surfaced once and propagated (the exchange fails), never retried at the poll/gate cadence and never swallowed into a duplicate delivery. A new transport-call consumer must uphold the same rule; it is not enforced automatically, so adding one is a change to review against this rule.

## Slow-operation warning

Distinct from both controls is a non-fatal **slow-operation warning**, which is observability, not a security control. At a fixed, generous elapsed-time threshold (30 seconds, below the 60-second read fast-fail) the SFTP adapter logs one warning naming the operation, the elapsed time, and - where a cheap progress signal already exists - the observed progress (bytes-so-far and rate for a `get`, entries-so-far for a `list`, the payload size for a `put`; elapsed-only for the atomic operations), then lets the operation continue unchanged. It needs no duration estimate: a slow-but-honest transfer and a deliberately slow or withholding server are observationally identical (intent is not on the wire), so a false warning is cheap and the line reports observed signal for a human to judge. It does nothing on a headless run with no human watching - the whole-exchange budget defends that run - and it stays entirely outside the terminal-error paths, so it can never affect correctness or the liveness gate. It introduces no new required configuration.

## Display sanitization escape format

The display sanitizer escapes a string for operator-facing output by replacing every byte or code point that is not printable ASCII - C0 and C1 control characters, DEL, and any non-ASCII code point - with a hex escape: `\xHH` for a code point <= 0xFF, `\uHHHH` for a code point <= 0xFFFF, and `\u{HHHHH}` for an astral code point above 0xFFFF; a literal backslash is doubled (`\\`), and printable ASCII (0x20 to 0x7E) is otherwise left unchanged. The output is truncated to a bounded maximum length with a visible truncation marker. It is a one-way display transform: the escaped form is never parsed back, never used to build a filesystem path, and never used in an equality or hash comparison - those always use the original bytes - so escaping changes only what an operator sees, not how a value matches or routes.

## SFTP fatal-packet crash safety

The size and liveness bounds above turn a hostile server's behavior into a bounded failure, but they leave one behavior unbounded: a process crash, which is neither a memory nor a time exhaustion. The SFTP adapter drives the raw ssh2 `SFTPWrapper` directly (for `list()`, `createExclusive()`, and the streamed `get()`), and after the session connects that wrapper carries no `'error'` listener of its own - ssh2's `Client.sftp()` attaches a setup-time listener but strips it before handing the wrapper back, and `ssh2-sftp-client` attaches `'error'` handlers only to the SSH `Client` object and to per-operation read/write streams, never to the wrapper. So when a server returns a malformed SFTP reply packet (a malformed NAME, HANDLE, or DATA packet, an unknown packet type, or a duplicate VERSION packet after the session is already ready - a malformed *first* VERSION instead rejects `connect()`, whose setup-time listener is still attached then, so it never reaches this listener-free state) - or faults the framing layer itself with an over-long packet, which trips the length-prefix check before any request id is consumed - ssh2's fatal-error path emits `'error'` on a listener-free `EventEmitter`, which Node escalates to an uncaught exception - crashing the CLI outright, before the lock-file and temp-file cleanup and the typed exit-code mapping that an orderly failure runs. The adapter closes this by attaching a single guarded `'error'` listener to the wrapper at connect time (once per wrapper instance, so a reconnect's fresh wrapper is re-guarded and a repeated connect does not duplicate the listener), so a fatal protocol error is handled rather than fatal.

A handled fatal error leaves the session dead but the process alive. ssh2's `doFatalSFTPError` emits the `'error'`, destroys the wrapper, then calls `cleanupRequests`, which fails the callback of every request still registered. What bounds an *in-flight* `list()` or `get()` depends on which request carried the fatal packet, and the realistic case is not `cleanupRequests`: when the malformed packet is the reply to the in-flight request itself, ssh2 has already deleted that request from its pending table by the time `doFatalSFTPError` runs. The `NAME` and `DATA` handlers delete it unconditionally up front, *before* the parse/check that triggers `doFatalSFTPError`; the `HANDLE` handler deletes it inside its own malformed branch (on a defined request id), immediately before triggering it. The placement differs but the net effect is the same: `cleanupRequests` finds nothing left to fail for that request and the in-flight callback never fires - the operation is bounded by the adapter's 60-second wall-clock deadline (`list()`'s deadline, the capped `get()`'s idle window, or the uncapped `get()`'s whole-operation deadline), not by `cleanupRequests`. `cleanupRequests` does fail an in-flight callback promptly only in the other cases: a fatal error that arrives on a *different* request id, or a connection-level fault (such as an over-long packet, which faults the framing/length-prefix layer before any request id is consumed), leaves the in-flight request registered for it to fail. Either way the in-flight operation is bounded - the 60-second deadline is the bound that covers the realistic in-flight case, so it is load-bearing and must not be removed on the assumption that `cleanupRequests` covers in-flight operations. The listener also captures the cause, which bounds the *next* operation: every server-driven method consults this captured error at entry and short-circuits before issuing a request the dead channel can never answer. `list`, `get`, `createExclusive`, `put`, `delete`, `rename`, and `exists` reject promptly with the same terminal `TransportOperationStalledError`; `safeDelete`, whose contract forbids rejecting (callers run it in `catch` blocks), instead resolves promptly as a best-effort no-op. This entry guard matters beyond saving a deadline wait, because after a fatal error ssh2 destroys the SFTP channel but the underlying TCP/SSH socket stays up (a hostile server keeps it alive), so an unguarded write, stat, or delete would buffer its request on the closing channel and hang until its own per-operation bound fires (60 s) rather than failing at once - turning the crash, once fixed, into a teardown that stalls for the full bound, since `FileSyncConnection.close()` drives `safeDelete`/`delete` during cleanup. The guard converts that stall into a prompt, orderly failure.

This bounds the malformed/fatal-packet crash and, once a fatal error is known, every server-driven operation: the entry guard short-circuits the next read or write rather than buffering a request the dead channel can never answer. The one behavior the guard cannot catch - a hostile server that withholds a write, stat, or delete callback *without* any protocol error, so no fatal `'error'` fires and `put`/`delete`/`rename`/`exists` never short-circuit at entry - is not left unbounded: each of those operations now carries its own per-operation bound (the 60-second deadline for `rename`/`delete`/`exists`, the 60-second progress idle window for `put`) that fails the withheld callback terminally, with the whole-exchange budget backstopping beneath. The best-effort `safeDelete` carries the same 60-second deadline so a teardown delete cannot be the soft target left on the coarse budget while every other write op fast-fails, but - because its contract forbids rejecting - it swallows the deadline's stall error and resolves rather than surfacing it, bounding the wait without breaking the never-reject guarantee its `catch`-block callers rely on. The entry guard's role, now that those bounds exist, is to spare the 60-second wait once a fatal error is already known, not to be the sole thing standing between the call and an indefinite hang. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

The bounds above reach past the public `ssh2-sftp-client` API and drive ssh2 internals directly, so they rest on premises about ssh2's internal behavior that an upgrade can silently break. Before raising either `ssh2` or `ssh2-sftp-client`, follow the upgrade checklist in [CONTRIBUTING.md](../../CONTRIBUTING.md#upgrading-the-sftp-stack-ssh2--ssh2-sftp-client), which names the internal premises, the source files to re-read, and the contract-assertion test that fails red on a lifecycle change.

## Authenticated abort marker

A distinct channel control is the **authenticated cross-party abort marker**, the one mechanism here that is not a bound on a hostile operation but an authenticated signal between the two honest parties. On the file-sync transports a failed exchange leaves the waiting peer with no in-band notice that its partner has given up: it can only wait out the full peer-inactivity budget (default one hour) and then print a generic peer-silence hedge. When a party fails terminally with its directory still writable, it now writes a best-effort `<id>-abort.json` carrying a 32-byte token `HKDF(sessionKey, "psilink-abort-token-v1:<role>")`; a waiting peer that reads a `<peerId>-abort.json` whose token verifies against its own locally-derived peer token fails fast with a definitive `PeerAbortError` (exit 69) instead. The token is the authentication: a hostile filedrop/SFTP admin can write or rename any file in the rendezvous directory, but cannot produce a token that verifies without the per-session session key, so cannot forge an abort against a live peer. The label is domain-separated from the other session-key derivations - exact-distinct and prefix-free from the AEAD `psilink-aead-v1:<context>` and the `psilink-shared-secret-rotation-v1` labels - so an abort token can never collide with or substitute for an AEAD key or the rotation secret. The comparison is constant-time and length-safe, and the read fails closed on every absent, oversized, malformed, wrong-version, or non-matching input. The per-direction (role) binding makes a captured marker renamed to the other party's name verify under the wrong token and fail; the per-session ephemeral key makes a marker captured from one exchange inert in another.

The marker is best-effort and carries no cause, both deliberate. Its absence is strictly uninformative: a deleted, withheld, or forged marker simply degrades to the pre-existing peer-silence timeout and hedge - it never reads as success - so a hostile admin can at most *suppress* the fast-fail optimization (the behavior before this feature existed), never forge an abort or turn a failure into an apparent success. It carries no cause because the cause is not actionable to the waiting party - which holds no local error and must contact its partner regardless - and a plaintext cause on the untrusted channel would only widen disclosure; the token itself is a one-way HKDF output that reveals nothing about the session key or any other derived key. Because the marker is re-read every poll cycle, it is capped small - a pre-`get()` listed-size refusal and a bounded `get()`, both at 1 KiB rather than the ~512 MiB frame maximum - so an admin-plantable file cannot become an availability vector; the write is short-budgeted on both its `put` and `rename` (a few seconds each) so a sick directory cannot hang teardown. The mechanism is file-sync only: the connectionless file transport is the only one with a parked-waiter problem, since a WebRTC peer's death drops the DTLS data channel and surfaces promptly; a future cross-transport abort would instead carry a protocol-layer error kind rather than an on-disk marker. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## SFTP host-key verification

The controls above defend against a hostile administrator of the server a party has correctly reached; a distinct control defends against reaching the wrong server. On the SFTP channel an operator may pin the server's host key by setting `connection.server.host_key_fingerprint` to its OpenSSH SHA256 fingerprint, and every connection then verifies the presented host key against that pin before authentication, aborting on a mismatch. This is the SFTP analogue of the certificate pinning the signing identity uses (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#receipt-signing-identities)): trust comes from a fingerprint rather than a certificate authority, and it replaces SSH's *silent* trust-on-first-use with an explicit, enforced pin -- supplied out-of-band, or established by a confirmed first-use prompt (below) and then verified on every subsequent run -- so a man-in-the-middle or a substituted server presenting its own host key is detected rather than silently trusted.

The fingerprint is `SHA256:` followed by the unpadded *standard* base64 (alphabet `[A-Za-z0-9+/]`, not base64url) of the SHA-256 digest of the raw SSH host-key wire blob -- byte-identical to what `ssh-keygen -lf` and OpenSSH clients display, so an operator pins exactly the value those tools print. It is validated at config-parse time by `HOST_KEY_FINGERPRINT_REGEX` (`/^SHA256:[A-Za-z0-9+/]{42}[AEIMQUYcgkosw048]$/`): the final character's class is restricted because the last 6-bit base64 symbol of a 32-byte (256-bit) value carries two low bits that are zero in the canonical encoding, so only those sixteen symbols are valid there. A bare 43-character base64url value -- the shape of a signing `partner_fingerprint` -- is detected separately and rejected with a message naming that confusion, since the two fingerprints use different alphabets and one is easily pasted for the other.

Verification runs in ssh2's `hostVerifier` callback, which the SSH handshake invokes with the raw host-key blob before authentication. psilink always installs a callback (in one of three forms below) and installs it *after* applying `provider_options`, so a provider option can neither override nor disable it; the `provider_options` allowlist is in any case default-deny and drops `host_verifier`/`hostVerifier` and `host_hash`/`hostHash`. `hostHash` is never set: setting it would make ssh2 pre-hash the key before the callback, destroying the raw bytes the fingerprint is computed over. The three forms: with a pin set, an *enforcing* verifier computes `SHA256:` + base64(SHA-256(blob)) and compares it against the pin with a constant-time byte comparison over the decoded digests (nothing secret is compared -- a host key and its fingerprint are both public -- so the constant-time comparison is house-style hygiene for digest comparisons, not a timing-oracle defense); with no pin, a *fail-closed* verifier computes the presented fingerprint into the error and refuses (see the no-pin default below); and on the deliberate first-use probe, a *capture* verifier records the presented fingerprint and refuses, so the probe learns the key without authenticating (see first-use trust below).

The control is fail-closed on every path. A mismatch, a malformed or empty blob, or any error inside the callback results in `verify(false)`, which ssh2 raises as a fatal handshake error whose message contains `Host denied (verification failed)` -- before authentication, so no credential is presented to an unverified server. The CLI adapter treats that rejection as terminal and does not retry it: retrying would only re-run the key exchange against the same untrusted host. The `Host denied` message fragment is an ssh2 internal premise recorded in the [SFTP-stack upgrade checklist](../../CONTRIBUTING.md#upgrading-the-sftp-stack-ssh2--ssh2-sftp-client); even if a future version renamed it, core re-wraps the same rejection into an `SFTP host-key verification failed:` message and the outcome is unchanged, only slower. The mismatch message names the presented fingerprint and key type so an operator can distinguish a legitimate key rotation from an attack; the key type is decoded from the server-controlled blob, so it is routed through `sanitizeForDisplay` and quoted before it reaches the operator's terminal or log. A changed key is never auto-accepted -- there is no standing always-accept flag (which would be permanent trust-on-first-use disguised as protection). The recovery for a legitimate rotation is a deliberate one-shot re-pin, modeled on ssh's `known_hosts` flow: verify the new fingerprint out-of-band, then either set `connection.server.host_key_fingerprint` to it or remove that field and re-run interactively to re-establish trust on first use (which overwrites the stored pin).

The pin may be supplied as an `@`-file reference (`@/path/to/fingerprint`) for operators who keep it in a read-only secrets mount. Because `@`-file resolution runs after parsing, the literal `@path` -- which cannot match the `SHA256:` format -- is exempt from the parse-time format check, and the resolved file contents are re-validated against `HOST_KEY_FINGERPRINT_REGEX` at resolution: a malformed or empty secrets file fails as a `UsageError` (exit 64) naming the reference, rather than as a confusing host-key mismatch at connect time.

Verification applies to the CLI `sftp` channel only. A file-drop makes no SSH connection, and the browser/proxy SFTP path does not run ssh2's `hostVerifier`. An attacker cannot induce the unpinned state to inject their own key, because the pin is configured out-of-band (or established on first use, below) and the invitation endpoint schema rejects a `host_key_fingerprint` field outright (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#transport-layer-authentication)). As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

**No-pin default: fail-closed.** When no fingerprint is pinned, `FileSyncConnection.open` installs the fail-closed verifier, which refuses the connection before authentication (so no credential reaches an unverified server) with an error naming the presented fingerprint to pin. This replaces the earlier warn-and-proceed default, which trusted any presented key. The flip is deliberately gated on first-use existing (below): a pin can only be reliably obtained once first use exists, so flipping the default earlier would have stranded every unpinned config with no recovery path.

**First-use trust (ssh-style).** An operator who cannot obtain the fingerprint out-of-band can still reach the verified-on-every-subsequent-run state by establishing the pin on a deliberate first connection. The CLI `exchange` command, before opening an unpinned `sftp` connection, runs this flow:

- On a TTY (interactive), it *probes* the server: a throwaway connection whose capture verifier records the presented host key and immediately refuses, so the key is learned without presenting a credential and without waiting on a user prompt inside the handshake (which would race ssh2's `readyTimeout`). It then surfaces the fingerprint and key type, prompts for confirmation, and on a yes pins the fingerprint into `connection.server.host_key_fingerprint` (persisted to `psilink.yaml` in place, preserving comments) and proceeds with a real, now-enforcing connection. Because that real connection re-verifies the just-pinned key, a key swapped between the probe and the connect is still caught (the same brief TOCTOU window as `ssh-keyscan`-then-`ssh`).
- Non-interactively (no TTY -- an automated run, or one piping its CSV through stdin), it does not prompt and does not auto-accept: it fails closed with an actionable error naming the recovery (run once interactively to pin, or set `host_key_fingerprint` out-of-band). This is the same fail-closed posture as the no-pin default, surfaced earlier and with CLI-specific guidance.

The probe reuses the exact connect options (algorithms included) the real connection uses, so both negotiate the same host-key type and the pinned key is the one later runs verify.
