# Channel security construction

This document specifies the construction and implementation of PSI-Link's
channel-security controls: the application-layer AEAD envelope and key
derivation, the inbound integrity/replay/ordering checks, the memory and
liveness bounds the file-sync transports enforce against a hostile server admin,
the SFTP fatal-packet crash safety, the authenticated abort marker, the SFTP
host-key verification, and -- on a separate surface -- the defense-in-depth
bounds on the web application's signaling upgrade surface. It is
the implementation-level complement to the **Channel security** overview in
[SECURITY_DESIGN.md](../SECURITY_DESIGN.md#channel-security), which covers what
each control protects against and why the design is sufficient; this document
covers the wire formats, the derived constant values and their justification, the
enforcement points, and the dependency premises each control rests on. It does
not cover the threat model or authentication design (see
[SECURITY_DESIGN.md](../SECURITY_DESIGN.md)), the PSI and key-exchange wire
protocol (see [PROTOCOL.md](PROTOCOL.md)), or the network channels and transport
contract (see [COMMUNICATION.md](../COMMUNICATION.md)). Intended readers are
security auditors and implementors.

# Application-layer AEAD

Both parties use the HMAC-based Extract-and-Expand Key Derivation Function (HKDF) to derive AES-GCM keys from the 32-byte key-exchange session key - two directional keys, one per direction (HKDF info strings `psilink-aead-v1:initiator-to-responder` and `...responder-to-initiator`) rather than a single common key - so that every key has a single sender, and messages are encrypted using Authenticated Encryption with Associated Data (AEAD). Each message carries a per-direction sequence number as its nonce, preventing replay and reordering. The separate per-direction keys are what keep that nonce construction safe: both directions number their messages from zero, so a single common key would encrypt two different messages under the same key and nonce - a reuse that is catastrophic for AES-GCM - whereas one key per sender keeps every key-nonce pair unique. The server admin sees only opaque ciphertext files, and tampering with a file causes the authentication tag to fail and the exchange to abort. The AEAD context is restricted to this fixed pair of per-direction labels (a closed union, guarded at runtime), not an open string. Both parties must derive identical per-direction keys from identical labels; a variable, non-ASCII, or non-NFC label would make the two sides derive different keys and fail the AEAD with an opaque error rather than a clear authentication failure, so the label set is closed by construction.

The concrete envelope the `EncryptedMessageConnection` decorator emits is `{ enc: base64url(IV || ciphertext || 16-byte GCM tag) }`, where the 12-byte IV is four leading zero bytes (authenticated, but not separately validated) followed by the 8-byte big-endian sequence number, and the encrypted plaintext is a one-byte type tag (`0` for a JSON object, `1` for a `Uint8Array`) ahead of the payload. This wire format is pinned byte-for-byte by a checked-in known-answer vector at [`packages/core/test/vectors/aead-envelope-vectors.json`](../../packages/core/test/vectors/aead-envelope-vectors.json), following the cross-implementation vector precedent of `canonical-vectors.json` and `exchange-record-vectors.json`: each vector fixes a session key, role, sequence number, and plaintext and records the exact serialized `enc` string, so an independent implementation can reproduce it from those inputs and any change to the nonce construction, byte order, envelope shape, or tagging is caught. The vectors are asserted against the decorator in `packages/core/test/encryptedMessageConnection.test.ts`, which separately pins the per-direction HKDF key derivation (`deriveAeadKey`).

## Associated data is deliberately not bound

The envelope binds no Associated Data (AAD), and this is a deliberate, recorded decision rather than an omission. AAD would matter only to stop a frame authenticated under one context (role/direction, session, or transcript) from being spliced or replayed into a different context that shares the same key. Here every such context is already separated below the AAD layer by a distinct key. Direction is bound by the key: each direction uses its own AES-GCM key, derived with a distinct HKDF info string (`psilink-aead-v1:initiator-to-responder` versus `...responder-to-initiator`), so a frame presented to the opposite direction decrypts under the wrong key and the tag fails. Session is bound by the key: both keys derive from the per-session key-exchange session key, which is transcript-bound and mixes in each run's random ephemeral scalars, so it is distinct even across two runs of the same shared secret - a frame from one session decrypts under the wrong key in another. Ordering within a single direction of a single session is bound by the sequence-number nonce and the strict-increasing replay guard. The only set of frames that share one AEAD key is those one sender emits in one direction of one session, and the decorator carries exactly one logical stream there - it does not multiplex sub-channels or protocol phases, and the one-byte payload type tag that distinguishes binary from JSON lives inside the authenticated plaintext. There is therefore no second logical context under a shared key for AAD to separate, so binding the direction (or a session id) as AAD would only restate what the key already enforces. The sequence number is the GCM nonce, not a substitute for AAD: the reason AAD is unnecessary is the absence of a second context under one key, not that the nonce stands in for AAD. Practically, the decorator holds only its inner connection, the session key, and the role - no session or transcript id is in scope to bind without new plumbing that belongs to the not-yet-wired caller integration.

Revisit this decision if a future change multiplexes more than one logical stream or protocol phase over a single direction's key, or collapses the per-direction key split into one shared key. In either case a context or sub-stream id bound as AAD would stop being redundant: it would still block cross-direction splicing, though not the key-nonce reuse that collapsing the key split would itself make catastrophic (AAD does not protect against nonce reuse). Until then, the per-direction key plus the sequence nonce are sufficient and no AAD is bound.

## Inbound integrity, replay, and ordering checks

Now that this layer is wired in (recurring CLI exchanges), its AEAD/MAC and sequence-number (replay and ordering) checks run at the protocol layer, on the output of a `receive()` call: the decorator validates and decrypts each inbound envelope inside `receive()` and rejects with a `security` `ConnectionError` on any integrity, replay, or ordering failure. These checks are deliberately not latched into the inner connection as a transport control (the `fail`/`finish` teardown signals) behind a buffered inbound frame. The message queue drains an already-received frame to the consumer ahead of any terminal error, whether the close was clean or abnormal (see [Message delivery and teardown](../COMMUNICATION.md#message-delivery-and-teardown)), so a check wired as a transport-level latch could hand a suspect frame to the consumer before the integrity alarm fires. The decorator honors this - it raises the failure from `receive()` itself, and the inner transports raise only `transport` and `protocol` terminal errors with no integrity check latched behind a frame - so the constraint holds and remains one to preserve.

Two channel-hardening concerns attach to this layer once it is wired into live recurring CLI exchanges, where each becomes live residual risk against a transport-level attacker (for SFTP, the server admin). One - the inbound frame-size bound - is now enforced (described below, alongside the companion directory-listing bound that closes the related directory-enumeration vector). The other - detection of dropped, withheld, and truncated frames - is now split: strict mid-stream gap detection is enforced (described next), and end-of-stream / truncated-tail detection is a deliberate accepted deferral on the reasoning recorded below. The gap-detection half was the release-blocking concern and is resolved (pending the explicit security review); the truncated-tail deferral does not block release on its own; the reasoning below records why it stands on the matching loop's self-driven lockstep - a premise the empty-round desync (board item 197674102) had made conditional and which that item has since restored.

Mid-stream gap detection is now enforced. The sequence-number check previously rejected only a number that was not strictly greater than the last accepted, which stopped replay and reordering but not omission: a gap (a sequence number that skips ahead) was accepted. The decorator now rejects any inbound sequence number that is not exactly the last accepted plus one, so a frame dropped or withheld between two delivered frames is caught here as a `security` failure on the output of `receive()` - the same protocol-layer surfacing as the replay and tag checks, never a transport-level latch. This did not require the once-feared send-path change ("advance the counter only on a fully successful send"): a send that consumes a counter value but then fails to transmit (an encrypt fault or an inner transport failure) already latches the wrapper terminal, so no later frame is ever sent, and a consumed-but-unsent value can only truncate the tail, never leave a gap between two delivered frames. Strict gap detection therefore raises no false alarm while the synchronous `sendSeq++` before the encrypt is retained - which is what keeps two concurrent sends from peeking the same counter and reusing an AES-GCM nonce. As net-new security behavior it is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## End-of-stream and truncated-tail detection

End-of-stream / truncated-tail detection is a deliberate accepted deferral, not a coverage gap to be closed with a marker. A truncated tail - a stream that simply stops, including a dropped final frame - is indistinguishable at the AEAD layer from a clean end, and no AEAD tag can authenticate the *absence* of a frame: an omitted frame can be detected only by expecting one and not receiving it. The PSI protocol is self-driven lockstep (see [PROTOCOL.md](PROTOCOL.md)): each party drives a deterministic script of matched send/receive calls, so the receiver always knows from its own script whether another frame is due. In the symmetric case a truncated tail therefore surfaces - as a stalled `receive()` that hits the inactivity timeout - and an explicit, integrity-protected end-of-stream marker could only reclassify that timeout from `transport` to `security`, not detect the omission any sooner. Reclassification alone does not justify net-new, security-reviewed cryptographic surface, so the marker is not built. Earlier framing gated this on a single hypothetical reopen condition - whether the two parties' lockstep loops could legitimately stop at different points. That condition was confirmed reachable and is now closed. The empty-linkage-round skip in the matching loop (`packages/core/src/link.ts`, `if (data_j.length === 0) continue;`) was computed per party from each side's own usable values, with nothing forcing the two sides to skip the same key: for `j > 0`, `data_j` is built only from a party's still-unmatched records, so under one-to-one or many-to-one cardinality a party fully matched in an earlier round had zero unidentified values for every later key and skipped it while a partner still holding unmatched records did not - the partner called `identifyIntersection` and blocked on a `receive()` the first party never answered. Board item 197674102 removed that skip: both parties now run `identifyIntersection` for every agreed key (the key set is fixed by the linkage terms, and the PSI library returns an empty intersection without error for an empty input set, so a locally-empty round is a correct no-op), keeping the exchange in lockstep by construction rather than by coincidence of equal data. The lockstep premise is therefore restored - unconditional within the implemented one-to-one and many-to-one cardinalities - and the truncated-tail deferral stands on the original symmetric-lockstep reasoning; any future cardinality loop must avoid reintroducing a per-party-data-gated skip. The desync was in any case a protocol-loop fault, not an absent-frame one: no AEAD-layer marker could have detected or repaired it, so its resolution neither revived nor depended on the end-of-stream marker.

## Inbound frame-size bound

The enforced control is the inbound frame-size bound. Left unbounded, the transport read path loads an entire frame into memory and JSON-parses it before any integrity check runs, so a hostile filedrop/SFTP server admin - an adversary under the threat model (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#threat-model)) - could write an arbitrarily large file and exhaust memory: a denial of service that allocates a string and byte array proportional to the attacker-chosen file size. The transport now refuses any inbound file larger than a maximum frame size of **536,870,888 bytes (~512 MiB)** before it is read into memory, surfacing a clear terminal error (a `FrameSizeExceededError`, exit 64) rather than allocating proportionally to the attacker-chosen size.

The value is deliberately chosen, not a round number: it is the exact value of Node's maximum string length (`buffer.constants.MAX_STRING_LENGTH` on 64-bit). That length is the hard ceiling above which the existing JSON-text read cannot process a frame at all - both read sites `.toString()` the buffer before parsing it (through `parseBoundedJson`), and `Buffer.prototype.toString()` throws above MAX_STRING_LENGTH - so anchoring the cap there (a) never rejects a frame the transport could otherwise decode, since anything larger already fails at `.toString()` regardless of memory, and (b) converts that late, opaque failure (raised only after the full attacker-sized buffer is already resident) into an early, clean refusal before the allocation. As a headroom check against the realistic worst-case legitimate frame: the largest PSI frame is one party's full encrypted set sent as raw elliptic-curve points (~64 bytes per element; see [PROTOCOL.md](PROTOCOL.md#psi)), base64url-expanded by 4/3 on the wire, so 512 MiB of wire text decodes to ~384 MiB raw, on the order of 6 million elements - already more than this single-frame JSON-text transport can carry, since a larger set would itself exceed MAX_STRING_LENGTH and require chunking before the cap could ever bind it. The bound is a fixed constant rather than a configurable option: a configurable bound risks an operator raising it high enough to reintroduce the DoS.

The bound is enforced at the transport read layer, which is the meaningful enforcement point: the poll loop and rendezvous gate refuse a file whose listed/declared size exceeds the cap before calling `get()`, and - so a server that under-reports a file's size in its directory listing cannot slip past that check - each `FileTransportClient` adapter also enforces a hard per-read byte cap (the local-filesystem adapter `fstat`s the open handle and reads exactly that many bytes, closing the append-after-stat TOCTOU; the SFTP adapter streams into a counting sink that aborts the transfer once the running total crosses the cap, buffering at most one stream chunk past it). A `.max()` on the AEAD decorator's `{ enc }` envelope schema caps the base64 ciphertext at the same single value, but that is documented defense-in-depth only, not the primary control: by the time the decorator runs, the inner transport has already read and parsed the frame, so the dominant allocation has already happened. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Directory-listing bound

A companion transport-read control bounds the directory listing itself. This one is independent of the AEAD layer and of file contents: the two parties rendezvous through a shared directory the adversary controls, and the transport's `list()` call enumerates it. Left unbounded, a hostile filedrop/SFTP server admin can write a very large number of files, or files with very long names, and drive the listing to allocate an array and per-entry metadata proportional to the attacker-chosen directory contents - the same memory-exhaustion impact class as the frame-size DoS, reached through a different vector (directory enumeration - entry count and name length - rather than a per-file body read). The transport now refuses a listing that breaches either of two bounds - more than **8,192 directory entries**, or any single entry whose filename exceeds **255 characters** - with the same kind of terminal, typed error as the frame-size bound (a `DirectoryListingBoundsError`, a `UsageError` subclass, exit 64). The refusal fails the exchange rather than silently truncating the listing to the cap and processing it: against a hostile rendezvous directory, refusal is the safer default, since truncation risks dropping a legitimate exchange file an attacker buried under filler.

Both bounds are derived, not round numbers, and both comfortably exceed what a legitimate exchange produces. The **8,192-entry** cap is set from a memory envelope: the worst-case at-cap allocation is the retained entries, each carrying a name of up to the 255-character limit plus per-entry object overhead (~600 bytes), so the listing allocation is bounded to roughly 5 MiB - about two orders of magnitude below the 512 MiB single-frame budget the frame-size bound already governs, so directory enumeration cannot become the dominant memory vector - while still exceeding the order-of-ten files a legitimate exchange produces (the two `-hello.json` files, at most one `-lock.json` or the `-ack.json` markers, transient `-joining.json` sentinels and `temp-*.tmp` writes, and the bounded set of PSI message frames) by roughly three orders of magnitude, with generous headroom for a retain-mode directory that accumulates message and ack files across many sequential exchanges before the operator rotates it (retention is an out-of-band operator responsibility). The **255-character** filename cap is the POSIX `NAME_MAX`, the per-path-component limit every mainstream filesystem enforces (ext4, XFS, APFS, NTFS): the longest filename a legitimate exchange writes - the ack marker of a timestamped message - is on the order of 120 characters, and a name longer than 255 cannot exist on a conformant filesystem, so it is necessarily synthetic (the SFTP protocol imposes no name-length limit, so a hostile server can return such a name in a READDIR response even though its own filesystem could not hold it). Both bounds are fixed constants rather than configurable options, for the same reason as the frame-size bound: a configurable cap risks an operator raising it high enough to reintroduce the DoS.

The bound is enforced at the transport `list()` layer in both adapters - the point before the listing is materialized - not at a higher layer that has already consumed it. The local-filesystem adapter streams entries through `fs.opendir` (which yields entries in bounded batches) rather than `fs.readdir` (which materializes the whole directory into one array, on top of the per-file `stat` fan-out that follows), so the entry-count check stops the walk - capping both the retained array and the stat fan-out - at the bound regardless of how many entries the directory holds. The SFTP adapter does not delegate to ssh2-sftp-client's `list()` (which passes the directory path to `readdir`, accumulating the entire remote listing into one array before returning); it opens a directory handle and reads one server batch at a time through the low-level `opendir`/`readdir`/`close` protocol, applying the checks as entries arrive, so a single READDIR response - itself bounded by the SSH transport's maximum packet size - bounds the allocation to at most the cap plus one batch. Guarding only the local adapter would leave the SFTP path - the one with the explicitly in-scope adversary, the server admin - exposed, so both adapters are bounded. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Per-operation liveness bounds

A third transport control bounds these operations for liveness rather than memory. It is the **per-operation fast-fail layer** - covering both the reads and, as of the write-path bounds described with them below, the always-executed write/stat/delete operations - beneath the whole-exchange budget described next (which is the universal backstop over every transport await). The size bounds above stop a hostile directory or file from exhausting memory, but they do not stop a hostile server admin from hanging the operation itself. Every SFTP read awaits a callback the server drives, so the server can make one never terminate: a `list()` can return valid but empty (`count = 0`) non-EOF READDIR responses forever (advancing neither size bound, since no entry accumulates and the end-of-directory status never comes, so the batch loop recurses without end) or simply never invoke a readdir/close callback; a `get()` can hold the read stream open and withhold data, or trickle under-cap bytes without ever ending it, so the transfer never completes; a `createExclusive()` can withhold its open or close callback. Because `list()` and `get()` run on the poll loop - which awaits them with no per-call timeout - and `createExclusive()` drives the rendezvous lock path, any of these is a liveness / non-termination denial of service: the exchange stalls forever (and a directory or file handle opened before the stall is leaked) instead of failing. Each read is now bounded by the mechanism its shape allows and surfaces a single terminal, typed `TransportOperationStalledError` - a `UsageError` subclass, exit 64, the liveness sibling of the memory-bound `DirectoryListingBoundsError` and `FrameSizeExceededError` - so the consumers that await these reads, the poll loop and the rendezvous control-file gate (`readControlFileWithGate`, which reads the peer hello over `get()`) alike, treat it as terminal and fail the exchange rather than retrying into the same hang. The gate keys this on the typed `UsageError` rather than a single error class, so a stalled hello read is terminal there exactly as an over-cap `FrameSizeExceededError` is - without it, a server that stalls the hello `get()` would be retried at the polling cadence until the one-hour peer budget instead of failing in a minute. The streamed `list()` carries a **16,384 readdir round-trip** cap for the empty-batch flood plus a **60-second wall-clock deadline** for the withheld-callback case, and closes the open directory handle on the bounded-failure path. The capped `get()` bounds the **60-second idle gap between chunks** (reset on each chunk), not the total transfer, so a legitimately large but slow-and-progressing read is never failed while a withheld or stalled one is. `createExclusive()` and the rarely used uncapped `get()` - neither of which has a progress loop - carry the **60-second whole-operation deadline**. The always-executed write/stat/delete operations are bounded by this same layer, each in the mode its shape allows. The metadata ops `rename()`, `delete()`, and `exists()` - each a single server round-trip with no payload - carry the same **60-second whole-operation deadline** as `createExclusive()`, with the same negligible false-fail profile, since a metadata round-trip completes in well under a second on any honest server. `put()` instead carries a **60-second progress idle window**, reset on each uploaded chunk: a tight flat deadline is wrong for `put` specifically, because a legitimately large ciphertext upload over a slow link can exceed it while still progressing, so - exactly as the capped `get()` bounds its idle gap rather than its total transfer - `put` bounds the gap between upload-progress chunks. The payload is streamed in bounded chunks so a withheld write acknowledgement stalls the source and trips the window while a slow-but-progressing one keeps resetting it; this uses ssh2-sftp-client's stream interface rather than driving the raw `SFTPWrapper`, so it adds no new internal-API coupling.

The bounds are derived, not round numbers, and all comfortably exceed any legitimate operation. The **16,384 round-trip** cap is twice the 8,192-entry size cap, so the two move together: a compliant server sends at least one name per non-EOF READDIR response (an empty non-EOF response is not protocol-conformant - a server out of names sends end-of-directory instead), so a legitimate listing makes at most 8,192 batches before the entry-count bound would already refuse it, and only in the pathological-but-legal case of one entry per batch; an honest server packs many entries per packet and finishes a normal rendezvous directory in a single batch, so the cap only ever fails a progress-free flood. The **60-second** budget is a single coarse "the server has gone silent" threshold, applied as a whole-operation deadline for `list()`, `createExclusive()`, the metadata write/stat/delete ops `rename()`/`delete()`/`exists()`, and uncapped `get()`, and as a progress-reset idle window for the capped streaming `get()` and for `put()`. It sits well above any legitimate operation - a normal listing, a lock-file create, and each chunk of a healthy transfer all complete in well under a second - at roughly twice this project's 30-second per-attempt connect bound (`serverConnectTimeoutMs`, applied as ssh2's `readyTimeout` even when the operator leaves it unset; ssh2's own default `readyTimeout` is 20 seconds), so a transiently slow but live server is not cut off, yet more than an order of magnitude below the one-hour (3,600-second) peer-inactivity budget, so a withheld response fails the exchange in a minute rather than after an hour; applied as an idle window it never rejects a slow-but-progressing large transfer, only one that stops sending. All bounds are fixed constants, not configurable, for the same reason as the other bounds: a configurable budget risks an operator raising it high enough to reintroduce the denial of service. These per-operation bounds are on the SFTP adapter, whose every call crosses the network to the in-scope server admin; they cover its reads and its write/stat/delete operations alike. They do not cover the local-filesystem adapter (an earlier framing held its reads to face only a trusted kernel, `fs.opendir`/`fs.stat`, that exhibits none of these failure modes - but a stalled NFS/CIFS hard mount breaks that premise). That remaining gap is closed not here but by the whole-exchange budget described next. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Whole-exchange budget

A fourth liveness control is the **whole-exchange budget**: the universal backstop beneath the per-operation bounds above, and the *only* per-operation-free liveness bound for the local-filesystem adapter. The per-operation bounds fast-fail a stalled SFTP read or write in 60 seconds; the budget is the layer they were added *beneath*, not in place of. The always-executed critical-path write/stat/delete operations - `put` (every message and every hello/sentinel write), `rename` (every durable publish), `delete` (message consumption and rendezvous cleanup), and `exists` (the lock-path race check) - now each carry their own 60-second per-operation bound (above), but before those bounds existed this budget was all that stood between a hostile or dead server that withheld any of their callbacks and an exchange hung for an hour: `synchronize()` and `send()` awaited the call directly so it never returned, and the poll loop's reschedule sits in a `finally` a hung await never reaches, so the poller stopped silently with no error. The peer-inactivity budget (`peerTimeoutMs`, default one hour) was checked only between loop iterations, never over an in-flight await, so a single withheld callback escaped it entirely. The connection now races **every** transport await - reads, writes, stat, delete, and the local-filesystem path alike - against that budget at the single consumer seam they all flow through, surfacing the same terminal, typed `TransportOperationStalledError` on expiry. Because it wraps the transport interface rather than any one operation, it is op-agnostic (it covers operations not individually enumerated and any added later) and adapter-agnostic (it bounds the SFTP adapter and the local-filesystem `LocalFSClient` identically, closing the stalled-mount gap). It bounds a stalled await against a *fresh* budget per await (silence from the peer), not the total exchange duration, so it never caps a healthy long-running exchange; the budget is the same single coarse knob an operator already tunes, and it stays deliberately coarse as the backstop: the tight, false-fail-safe per-operation bound a slow `put` needs is the progress idle window above, not a flat budget timeout, which is exactly why the budget itself can remain one coarse knob rather than being split per operation. The capped `get()` slow-drip (under-cap bytes trickled forever, which resets the per-chunk idle window and so never trips the 60-second read bound) is caught here too, since total elapsed eventually crosses the budget - as is the symmetric `put` slow-drip, whose progress idle window the budget likewise backstops. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Connect-probe bound

A fifth liveness control bounds the connect probe itself, the one transport call deliberately outside the whole-exchange budget's seam. `connect()` runs before the exchange begins and is wrapped by neither the per-operation bounds nor the whole-exchange budget - it passes through `boundTransport` unwrapped - so each adapter bounds its own connect attempt instead: the SFTP adapter by ssh2's `readyTimeout`, which tears down the socket on expiry and strands nothing (an unresponsive SFTP connect is retried, so its total caller-visible latency stacks to roughly `maxReconnectAttempts + 1` such per-attempt windows - bounded wait, not accumulating resource, since each expiry strands nothing - unlike the local-filesystem timeout below, which is terminal at one window), and the local-filesystem adapter by a per-attempt `withTimeout` deadline around its `fs.access` reachability probe. On the local-filesystem adapter a stalled NFS/CIFS hard mount turns that probe into the same liveness hazard the post-connect bounds address, with a sharper resource cost: `fs.access` blocks a libuv thread-pool worker (not the event loop), and nothing in-process can cancel an already-dispatched blocking syscall - `fs.access` does not honor an `AbortSignal`, and libuv's `uv_cancel` only removes still-queued work, never work a pool thread is already executing - so the worker stays pinned until the OS releases the syscall, minutes on a hard mount. The per-attempt timeout fires on the event loop and fails the attempt, but retrying it (the prior behavior) launched a fresh probe each time, so under the default `maxReconnectAttempts` of 3 up to four such probes could stack, exhausting the default four-thread pool and starving every other `fs`/DNS/crypto operation in the process - a process-wide denial of service induced by a single unreachable mount. The connect now treats a per-attempt timeout as terminal: it fails fast after one timeout window, surfacing a `TimeoutError` (which extends `Error`, not `UsageError`, so it classifies exit 69 / `EX_UNAVAILABLE` like any transport-unavailable failure), rather than dispatching further probes; a fast-returning transient error - `EACCES`/`ENOENT` while a share or its permissions are still settling - still retries under `max_reconnect_attempts`, which governs only this initial connect, not any mid-exchange reconnection, of which there is none. This bounds concurrent stalled workers to one and the caller-visible latency to roughly one `serverConnectTimeoutMs` window; it does not bound the lifetime of that one worker, which only the OS reclaims, because no in-process mechanism can free a thread wedged in a single blocking syscall - only a killable worker thread or child process could, which is disproportionate for a connect pre-check. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

The per-attempt windows above bound each connect attempt, but the *count* of attempts had no ceiling: `max_reconnect_attempts` was validated only as a non-negative integer (up to `Number.MAX_SAFE_INTEGER`). Against an endpoint that refuses fast - `ECONNREFUSED` on `sftp`, `EACCES`/`ENOENT` on `filedrop`, exactly the fast transients the retry budget exists to ride out - the retry loop spaces attempts with a fixed one-second inter-attempt delay and the fast-fail attempts themselves are near-instant, so the wall clock is essentially the delay total (the `maxReconnectAttempts` delays across the `maxReconnectAttempts + 1` attempts above) - about that many seconds, a linear self-inflicted hang. The count is therefore capped at `MAX_RECONNECT_ATTEMPTS = 604800` (defined in `packages/core/src/config/connection.ts`), derived as the CLI's seven-day timeout ceiling `MAX_TIMEOUT_SECONDS` divided by the one-second delay (`604800 s / 1 s = 604800`): the largest count whose fast-fail delay total stays within the same seven days the duration flags already cap. A value past it is rejected with a flag-named `UsageError` (exit 64) at both validation boundaries that already agree on this field - the connection schema's `.max()` and the CLI's `nonNegativeIntFlag` parse guard, which imports the constant - so an over-ceiling value is refused whether it arrives from `psilink.yaml` or `--max-reconnect-attempts`. This bounds a proxy: the count equals wall clock only at the one-second floor (the fast-fail case), so it does not tightly bound a slow-but-answering endpoint, whose attempts each run up to `serverConnectTimeoutMs` and are already bounded per attempt by it; a direct wall-clock deadline on the whole connect phase is the stronger control left for if that case ever proves to matter. This is a usability/self-DoS sanity bound, not a cross-party control - the operator only hangs their own command - in the same class as the `MAX_TIMEOUT_SECONDS` duration-flag cap.

## Consumer-side terminal-error rule

These transport bounds share one consumer-side rule that ties them together, stated here because the bounds above each describe their enforcement in isolation. Every one surfaces as a `UsageError` subclass - the memory bounds as `FrameSizeExceededError` and `DirectoryListingBoundsError`, the liveness bounds as `TransportOperationStalledError` - and any consumer that awaits a transport call must treat a `UsageError` as terminal: it propagates the rejection and never retries it, swallows it, or proceeds as if the call had succeeded. This binds the read consumers (a `list()`/`get()`/`createExclusive()` in the poll loop or the rendezvous gate) and, now that the write/stat/delete ops carry their own per-operation bounds, the write consumers too - the `put`/`rename` publish in `send()`, the rendezvous lock-path `createExclusive`/`exists`, the consume-`delete` that signals a message consumed, and the retain-mode ack-write (`writeAck`'s `put`/`rename`) that signals it on a transport that never deletes. Retrying loops straight back into the hang or the over-allocation the bound exists to prevent, and swallowing-then-proceeding turns a stalled consume into a re-emitted duplicate - a swallowed bound is no bound - so the only effective response is to fail the exchange, which is why the poll loop stops on a `UsageError` (including from the consume-`delete`, which rethrows it rather than swallowing it as it does a transient delete failure), the rendezvous control-file gate (`readControlFileWithGate`) rethrows it, and the `send()`/joiner write paths propagate it. The terminal behavior keys off the `UsageError` base class rather than each subclass, so a future bound added to the family inherits it. This consumer rule is what makes the transport-layer bounds effective end to end. The consumers are pinned by behavioral tests in `packages/core/test/fileSyncConnection.test.ts` that drive each with a `UsageError` from a transport call - a stalled read in the poll loop and the gate, a stalled consume-`delete` in the poll loop, and a stalled retain-mode ack-write in the poll loop - and assert it is surfaced once and propagated (the exchange fails), never retried at the poll/gate cadence and never swallowed into a duplicate delivery. A new transport-call consumer must uphold the same rule; it is not enforced automatically, so adding one is a change to review against this rule.

## Slow-operation warning

Distinct from both controls is a non-fatal **slow-operation warning**, which is observability, not a security control. At a fixed, generous elapsed-time threshold (30 seconds, below the 60-second read fast-fail) the SFTP adapter logs one warning naming the operation, the elapsed time, and - where a cheap progress signal already exists - the observed progress (bytes-so-far and rate for a `get`, entries-so-far for a `list`, the payload size for a `put`; elapsed-only for the atomic operations), then lets the operation continue unchanged. It needs no duration estimate: a slow-but-honest transfer and a deliberately slow or withholding server are observationally identical (intent is not on the wire), so a false warning is cheap and the line reports observed signal for a human to judge. It does nothing on a headless run with no human watching - the whole-exchange budget defends that run - and it stays entirely outside the terminal-error paths, so it can never affect correctness or the liveness gate. It introduces no new required configuration.

## Application-layer parsed-input bounds

The transport bounds above stop a hostile *server admin* from exhausting memory or hanging the link; a distinct control stops the authenticated *exchange counterparty* - a mutually-distrusting peer (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#threat-model)) - from exhausting CPU or crashing the parser with a pathological-count message. The linkage terms (carried in an invitation token and re-parsed off the post-handshake wire by `protocolSetup`) and every post-handshake wire message are validated with Zod, and a frame within the ~512 MiB frame-size cap above can still pack millions of *invalid* elements into a partner-controlled collection. On Zod 4.4.3 that drives one of two `RangeError`s as Zod accumulates one issue per invalid element: a collection nested two or more array/record/tuple frames deep overflows the call stack while Zod spreads that issue array up through the frames (`Maximum call stack size exceeded`, at ~130k elements); a flat collection with no such frame instead makes Zod throw `Invalid string length` while it builds the error string (the `ZodError` message is `JSON.stringify`-d eagerly in the constructor, exceeding V8's maximum string length at ~3.5M issues, after a multi-second CPU burn). A plain `z.array(element).max(N)` closes neither: Zod v4 validates every element *before* the length check, so the issues accumulate before `.max()` can fire.

Two strategies bound the complete set of partner-parsed collections, chosen per field by whether a legitimate value is large:

- A collection legitimately in the millions - the PSI association-table indices, the payload rows / row indices, a partner's original-index list - carries a single-issue validator (`packages/core/src/utils/singleIssueArray.ts`): it validates the element type in one `every` pass that emits at most ONE issue regardless of count, so an arbitrarily large *valid* message still parses while a hostile one fails as a clean, bounded rejection. A count cap is unusable here - any bound low enough to forestall the `RangeError` would reject a real result.
- A collection whose legitimate count is small - the linkage fields and keys, each constraint's `exclude` denylist, a transform step list, a key's elements, the payload `send`/`receive` column lists, the abort-reason list - carries a count gate (`packages/core/src/utils/boundedArray.ts`): a permissive `z.array(z.unknown())` first stage accepts the array unvalidated, a count refine rejects an over-count array with one issue *before* per-element validation runs, and a `.pipe` re-validates the now count-capped array against the real element schema, preserving every per-element check and its issue path for an in-range array. The refine is `abort`, so an over-count value stops rather than letting the still-raw `unknown[]` flow on to a cross-field refine (which would dereference an unvalidated element). The `transform.params` record's key count is bounded by the same permissive-stage + count-refine + pipe shape, but with a bare **key count** (`packages/core/src/utils/objectKeyCount.ts`, `MAX_PARAMS_ENTRIES`) in place of the array length check. That count is still O(n) in the record's key count - a materialized `JSON.parse` object offers no sub-linear own-key count, and `for...in` enumerates eagerly - but it is the *cheapest* O(n) pass, and `for...in` avoids the result array `Object.keys(rec).length` would allocate and stops its body once the bound is passed. For this record the gate also closes a distinct LINEAR event-loop burn, not the `RangeError` class above: a millions-key `params` record otherwise burns multiple seconds *twice* before its count is even checked, once in the snake->camel `camelizeKeys` pre-pass that runs ahead of Zod (`parseLinkageTerms`) and once in the permissive record stage. Each of those is an EXPENSIVE per-key pass (a snake->camel rewrite, a per-key Zod parse); the gate replaces both with the single cheap key count. `parseLinkageTerms` hands the same bound to the camelize pre-pass, which leaves an over-count `params` value verbatim (handing it to the schema's refine, which rejects it) instead of rewriting every key, so the over-count record is counted but neither rewritten by the camelize pass nor per-key-validated by the record stage - cutting the multi-second burn to roughly one key enumeration rather than eliminating it (it stays O(n), bounded above by the frame cap). The `camelizeKeys` pre-pass walks arrays too, so the count-gated arrays above would incur an analogous pre-pass walk this per-record skip does not address; a global node budget in the walker (next paragraph) closes that, bounding the TOTAL the pre-pass rewrites rather than only this one record's width. The per-field count constants - 256 to 4096, each far above any real config and far below the `RangeError` thresholds - live beside their schemas in `packages/core/src/config/linkageTerms.ts` and `packages/core/src/protocolSetup.ts`.

The `camelizeKeys` pre-pass carries its own GLOBAL width bound, the running-total analogue of the per-record `params` skip and of the pre-pass's existing `MAX_NESTING_DEPTH` = 256 depth bound. The snake->camel rewrite is O(total nodes) and runs ahead of all Zod validation, so a partner payload wide enough - within the frame and decode-layer caps - would burn multiple seconds before any count gate fires; and for a wide object placed under an UNKNOWN top-level key the non-strict schema then SILENTLY strips it, so the parse SUCCEEDS after the burn rather than rejecting (worse than a rejection, since nothing is surfaced). A single node counter threaded across the WHOLE walk - `MAX_NODE_COUNT` = 262144, one object member or one array element per node - rejects such a payload with a clean, terminal `NodeCountExceededError` (a `UsageError` like the depth bound, fixed message carrying no input bytes) before the rewrite completes: an over-wide array is refused by its O(1) length before `.map`, and an over-wide object by a streaming own-key count before `Object.entries` materializes it, so neither the burn nor the materialization runs. A running total, not a per-container cap, is required to catch BOTH a single wide object (path a) and width spread across many small ones - the count-gated arrays of objects, each walked in full before its `boundedArray` refine could fire (path b). 262144 (2^18) sits far above any realistic operator `ExchangeSpec` - low tens of thousands of nodes at most (the single widest object is a `transform.params` record at `MAX_PARAMS_ENTRIES` = 256, collections cap at 256 to 4096 entries) - and far below the burn, which reaches a multi-second cost only past ~1M nodes (measured); rejecting at this budget caps the wasted walk well under a second. Like the rest of this family it is defense-in-depth, not a semantic limit: the per-collection caps DO compose to a schema-valid maximum above the budget (256 `linkageFields`, each with a 4096-entry `exclude`, is ~1M nodes), so a pathological-but-schema-valid config could trip it - but no real operator config comes within an order of magnitude. It composes with the per-record `params` skip WITHOUT double-counting - a skipped subtree (an opaque value, or an over-count width-bounded value) is not recursed into and so never counts toward the budget - so a massively over-count partner collection is now refused by this budget BEFORE Zod (the boundedArray count gate still rejects a moderate over-count that stays under the budget), and the throw is caught by the same `parseLinkageTerms` parse-error path that already handles the depth bound.

The `MAX_NESTING_DEPTH` = 256 depth bound the `camelizeKeys` pre-pass applies is enforced on all three linkage-terms parse paths, the invitation decode path included. `decodeInvitation` normalizes `transform.params` to camelCase through the bounded `camelizeKeys` chokepoint (the camelize pre-pass in `packages/core/src/config/invitation.ts`) as it validates the token against `InvitationTokenSchema` -- folding before the per-step length and dialect screens so they run on the normalized form -- so a partner-controlled one-key-per-level `params` - bounded otherwise only by the 64 KiB `MAX_ENCODED_INVITATION_LENGTH` cap and `parseBoundedJson`'s `MAX_JSON_NESTING_DEPTH` = 4096 decode ceiling, both ABOVE the ~2000-level native call-stack limit - is rejected at decode with a terminal `NestingDepthExceededError` (a `UsageError`, CLI exit 64 / a clean web decode error via `describeDecodeError`) before it reaches any downstream consumer. The normalization MUST use the bounded `camelizeKeys` walk for exactly this reason: a raw snake->camel recursion over the schema-opaque (`z.unknown()`, decode-unbounded) `params` value would itself overflow the stack. Because the field is now normalized at the decode chokepoint, every consumer of a decoded invitation's terms - the canonical comparison, the reconcile, the agreed-terms hash, the standardization runtime - inherits both the camelCase form and this depth bound, rather than each having to re-impose it. The CLI's invitation-vs-config reconcile (`apps/cli/src/config.ts`, `nfcDeep` reached via `diffLinkageTerms`) nonetheless threads its own depth counter and rejects at the same 256 as a backstop: `nfcDeep` is an independent recursive NFC fold that must not couple its safety to its callers having pre-bounded the input, and an unguarded deep value would otherwise overflow it with a `RangeError` the CLI command boundary maps to a generic internal-error exit (69) - catchable and non-fatal, but ungraceful, and distinct from the uncatchable `JSON.parse` aborts the decode-layer bound below guards. With decode and config load now bounding both sides, that backstop is not normally reachable; it stays as defense-in-depth. The bound fires far below the overflow and far above any real config (the deepest schema path is under a dozen levels, and `params` legitimately holds shallow scalars).

A further bound sits *upstream* of the entire Zod/`camelizeKeys` family above, at the JSON-decode layer, and is enforced as a single chokepoint: every parse of untrusted JSON in the core routes through `parseBoundedJson` (`packages/core/src/utils/boundedJson.ts`), which checks the body's structure against a per-object key ceiling, a per-array element ceiling, and a nesting-depth ceiling (`MAX_JSON_OBJECT_KEYS` = 65536, `MAX_JSON_ARRAY_ELEMENTS` = 16777216, `MAX_JSON_NESTING_DEPTH` = 4096) *before* handing it to `JSON.parse`. A lint rule forbids a raw `JSON.parse` elsewhere in the package, so a new parse site cannot silently reopen the hole. Unlike the rest of the family this is not a CPU-burn or `RangeError` guard: a single object made wide enough, or a single array made long enough, drives `JSON.parse` into a process-terminating internal limit of the JSON engine - a per-object property ceiling, or the array backing-store length limit - that is *not* a thrown exception, so a `try`/`catch` around the parse cannot intercept it. The only defense is to refuse the body before the parse can reach either limit, not to catch its failure. The check is a single O(n) pass over the raw UTF-8 bytes (or the already-decoded string a transport hands over) - the JSON structural markers are unambiguous ASCII in either representation (a single byte in UTF-8, a single non-surrogate code unit in UTF-16) - counting each object's members (by colon) and each array's elements (by comma) against its own ceiling, with early exit; on the byte path an over-budget body is refused without even allocating the decoded string.

This chokepoint covers both threat-model adversaries, not only the counterparty. The post-handshake `TYPE_JSON` frame is decoded through it after AEAD decryption (the *counterparty*'s reach); but the file-sync transport's own reads - the peer-hello control file and every message body - are JSON-parsed off the raw inbound file *before* decryption, so they are shaped by the hostile *server admin*, and the invitation token is parsed from attacker-influenced bytes a victim opens. All route through the same chokepoint, so a wide-object or long-array body from any source is refused rather than crashing the process. Each caller maps the refusal to its own terminal error - a `ConnectionError("security")` on the wire, a `UsageError` on a transport file, a rejected invitation - carrying no attacker bytes.

The unit is per *container*, not per byte or per total count, because that is what fits the data *at this layer*: a decoded-byte or total-count budget here would have to sit below the largest legitimate message and so would reject it, yet no single legitimate container is pathological. (The `camelizeKeys` total-node budget above *can* be a running total precisely because it sees only the small parsed-config family - linkage terms and the like - never these large set frames, which are parsed by other schemas the pre-pass does not touch.) The widest legitimate object is the `transform.params` record at `MAX_PARAMS_ENTRIES` = 256; the longest legitimate array is an association-index, payload-row, or mapped-pair list, one entry per matched record and so transport-bounded to a few million (a larger set cannot reach this layer - the binary set frame itself exceeds the transport's string limit first). So `MAX_JSON_OBJECT_KEYS` sits far above any real object and far below the per-object engine limit, and `MAX_JSON_ARRAY_ELEMENTS` sits above any real array and well below the array-length limit - each fires only on a pathological container and never pre-empts a schema-layer rejection or a legitimate large message. The depth ceiling sits far above any real nesting (the parsed-config ceiling is `camelizeKeys`' `MAX_NESTING_DEPTH` = 256) and exists chiefly to bound the scan's own per-container stack against a degenerate all-`{`/`[` body, which would otherwise exhaust memory in the pre-pass. This bound composes with rather than overlaps the post-parse `camelizeKeys` bounds: it caps any single CONTAINER's width before the walker ever sees it, while the walker's own `MAX_NODE_COUNT` budget (above) caps the TOTAL it then rewrites - a width the per-container ceiling does not bound (a body within every per-container limit can still hold far more nodes in aggregate than the walker should rewrite).

The severity is defense-in-depth, not a live vulnerability: the counterparty is authenticated and can already abort the exchange trivially, and most of these sites already caught the `RangeError` harmlessly - a `receiveParsed` / `parseOrProtocolError` wrap surfaces a `ConnectionError("protocol")`, and the linkage-terms parse-error catch aborts cleanly - so the bound turns an ungraceful internal exception and a one-shot CPU burn into a clean, bounded rejection rather than closing a memory or correctness break. The decode-layer chokepoint is the sharper of the set - it forestalls an uncatchable process crash, an availability loss the others do not reach, rather than a recoverable burn - but rests on the same reasoning: the crash kills only a session its instigator already controls - the counterparty it is party to, or the transport a hostile server admin already mediates and can disrupt at will - so it too is defense-in-depth against a self-limited adversary, not a live vulnerability. The set of accepted messages is unchanged: each single-issue predicate mirrors the element schema it replaces, and each count gate's `.pipe` re-runs the original element schema, so a legitimate exchange parses to a byte-identical structure (the cross-party receipt hash is preserved). As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Signaling-server inbound frame bound

The bound above lives in `@psilink/core` and covers the AEAD application channel and the file-sync transport - a self-limited adversary (the authenticated counterparty, or a hostile server admin). The web app's bundled PeerJS signaling server is a separate surface with a broader adversary: it runs in production under `/api/` to broker the WebRTC rendezvous, and its only authentication gate is the well-known default key `peerjs` (the id and token are free strings), so its inbound parse is reachable by ANY unauthenticated internet client. Its `socket.on("message")` handler in `apps/web/src/contrib/peerjs-server/services/webSocketServer/index.ts` decodes each frame with `JSON.parse(data.toString())`, the same uncatchable-V8-abort class as the core bound above - a single wide-enough object or long-enough array drives `JSON.parse` into a process-terminating internal engine limit that no `try`/`catch` can intercept. Because the broker process is shared by every peer, that abort is an availability loss for all of them, not one self-controlled session; and even short of the crash ceiling, a single oversized frame is independently a memory pin.

This is vendored third-party code, distinct from core and not routed through the `parseBoundedJson` chokepoint, so the bound is applied at the idiomatic `ws` layer instead: the `ws` `Server` is constructed with an explicit `maxPayload` (`MAX_SIGNALING_PAYLOAD_BYTES` = 262144, i.e. 256 KiB) rather than the 100 MiB `ws` default. `ws` enforces it in the receiver as each frame's length header is read, refusing an over-cap frame with a 1009 close (`WS_ERR_UNSUPPORTED_MESSAGE_LENGTH`) before the payload is buffered and before the message handler's `JSON.parse` ever runs - the error surfaces as a handled `error` event the server already routes, not a crash. The cap sits far above any legitimate signaling frame: this server brokers only small control messages - the PeerJS OPEN / OFFER / ANSWER / CANDIDATE / HEARTBEAT family, SDP and ICE that are KB-scale - while the PSI payload itself flows peer-to-peer over the WebRTC data channel and never crosses this socket. It sits far below both the 100 MiB default and the ~109 MB a single 2^23-key object needs to reach the per-object ceiling, so it closes the single-frame crash outright rather than only shrinking the window, and caps each inbound frame against the single-frame memory pin. Like the file-sync frame-size cap above it is a fixed constant, not a configurable option: a configurable bound risks an operator raising it high enough to reintroduce the DoS. It would only need revisiting if the signaling protocol began carrying a legitimately large payload through this socket - it does not, and a redesign that routed bulk data through the broker rather than the data channel would be the change to re-evaluate against. As net-new security behavior on an internet-facing surface, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Transform-regex linear-time dialect

A second control against the same mutually-distrusting counterparty bounds the *complexity*, not the count, of one partner-supplied field: the regular-expression pattern in a linkage-key element transform. The four regex step functions (`replace_regex`, `extract_regex`, `filter_regex`, `split_on`) compile their pattern and run it per row over the acceptor's full dataset (`packages/core/src/standardization.ts`); `parse_date` likewise *expands* its partner-controlled `inputFormat` into a per-row regex whose adjacent `(\d{1,2})` groups (one per `MM`/`DD` token) are a catastrophic-backtracking construct - a ~48-character format reaches a hang. On the JavaScript `RegExp` engine a catastrophic-backtracking (ReDoS) pattern would hang the single, non-interruptible thread. The pattern cannot be rewritten - PSI requires byte-identical keys and the terms are canonically hashed into the exchange-agreement receipt, so a unilateral rewrite both breaks matching and surfaces as a terms mismatch.

The control is by construction: these patterns execute under a **linear-time regex engine** (re2js, an RE2-semantics Thompson NFA / lazy DFA - `packages/core/src/utils/linearRegex.ts`) instead of `new RegExp`. Matching is linear in the input length with no backtracking, so the catastrophic-backtracking blow-up is *impossible*, not merely screened for. This includes `parse_date`'s expanded regex, which moves onto the same engine, so its adjacent-group construct is bounded by construction rather than by a separate analysis. The engine is pure JavaScript, so the SAME build runs in the CLI (Node) and the web (browser): both parties and both build targets derive byte-identical keys, with no second engine build whose semantics could diverge on an adversarial pattern. The permitted pattern syntax - the **dialect** - is pinned normatively in [PROTOCOL.md](PROTOCOL.md#transform-regular-expression-dialect); both parties MUST evaluate patterns under it.

What remains at validation is a dialect-conformance gate: a refine on `LinkageTermsSchema` (`packages/core/src/config/transformRegexDialect.ts`), so it covers every path that parses partner terms at once - the initiator/joiner `parseLinkageTerms`, the invitation-token decode (`InvitationTokenSchema`), and `ExchangeSpecSchema`. The refine walks every raw-pattern step (the function-to-pattern-param map `REGEX_STEP_PATTERN_PARAM`, `split_on` keying on `delimiter` rather than `pattern`, kept in parity with the `tier: "regex"` function descriptors by a test), coerces the param to a string exactly as the engine would (`coerceToPatternString`, so a non-string JSON value that stringifies to a pattern is screened as the literal it executes as), and rejects - fail closed, before any row runs - a pattern the engine cannot compile: a backreference, a lookaround, or an unsupported escape, exactly the features RE2 drops. `parse_date` is not screened here: it is not a raw-pattern step and its library-generated regex is always in-dialect; its safety is the engine, not the gate. There is deliberately **no fallback to `new RegExp`** - a pattern outside the dialect is refused, never silently re-run on the backtracking engine, which would reopen the hole. A total wall-clock budget across one validation (`REGEX_DIALECT_TOTAL_BUDGET_MS`, 2000 ms) bounds the gate's own cost so a terms set packed with patterns - the per-collection count caps above still permit a large product - cannot turn validation itself into a denial of service; on exhaustion the remaining patterns reject closed. A separate per-step length cap on the raw pattern (`MAX_TRANSFORM_PATTERN_LENGTH`, 1000) caps the source-parse and one-time compile cost (`applyElementTransform` compiles each step once per distinct transform array, memoized, rather than per row). These constants live beside the code in `transformRegexDialect.ts` and `linkageTerms.ts`.

The bundled default standardization patterns - including the email filter `^[^\s@]+@[^\s@]+\.[^\s@]+$` - are all in-dialect and need no allowlist; a checked-in cross-build vector file (`packages/core/test/vectors/transform-regex-vectors.json`) pins each default pattern's output, on the inputs it covers, as byte-identical to the previous `new RegExp` engine, and asserts the Node and browser builds agree. The few code points where RE2 and JavaScript regex semantics diverge - `\s` is ASCII-only under RE2 but Unicode-broad in JavaScript, `.` excludes only `\n` under RE2 but also `\r` and Unicode line separators in JavaScript, and `.` spans a code point rather than a UTF-16 code unit - are documented in [PROTOCOL.md](PROTOCOL.md#transform-regular-expression-dialect) and are outside that vector domain; they are not cross-party divergences (both parties run the same engine) and, in the bundled defaults, the email filter is preceded by `remove_non_ascii`, so no such code point reaches it.

The severity addressed is a denial of service bounded to the availability of the acceptor's own session - its browser tab or CLI process, on the acceptor's own (typically isolated, periodically-run) hardware - with no data disclosure, no code execution, and no effect on the counterparty or the privacy guarantee: standardization is local key-building, before and independent of the PSI rounds, so a hang neither leaks a non-intersection member nor reaches the peer, and the worst outcome is that the run produces no result and is re-run. The realistic trigger is a foot-gun - a catastrophic-backtracking pattern an operator carries over from a JavaScript engine or mistakenly authors, which a human reviewing the agreed terms cannot reliably vet for cost. The linear-time engine prevents that class by construction, and is the load-bearing control here; because the linkage terms are agreed once and reused, with any later modification surfaced for manual review, it also covers the residual hostile-or-tampered pattern (an attacker would have to subvert the one-time terms agreement for a non-escalating local hang). A deliberately-crafted in-dialect-but-expensive pattern remains possible but is bounded - its source length is capped (`MAX_TRANSFORM_PATTERN_LENGTH`, applied to the coerced source the factory compiles), RE2 caps any single and nested repeat count, and matching stays linear in the input - and non-escalating, so it is accepted under this threat model rather than further bounded with complexity caps. This control is the by-construction successor to, and replaces, an interim best-effort static screen (the `redos-detector` analyzer, board item 199539401): the linear-time engine eliminates the class the heuristic could only estimate, and the heuristic, its allowlist, and that dependency are removed. As net-new security behavior that adds a security-relevant dependency (re2js) and pins a wire-level dialect, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Name-constraint character class

One more partner-controlled regular expression reaches the local party outside the transform pipeline: a name field's `allowedCharacters` constraint (`NameConstraints`, `packages/core/src/config/linkageTerms.ts`), a regex character class. It differs from the transform patterns above in two ways that lower its stakes but not its ReDoS exposure: it is **not** key-building and **not** hashed into the agreement receipt, so it carries no cross-party determinism requirement; and the value-level check that evaluates it is a **local, warn-not-enforce advisory** - the web "Prepare your data" editor flags a cleaned value whose characters fall outside the class, before any exchange round and with no effect on what is sent. It is still partner-authored (it arrives in the invitation token), so a crafted class is the same ReDoS foot-gun and takes the same by-construction control: the editor compiles `^[allowedCharacters]$` under the linear-time engine (`compileLinearRegex`) and tests one code point at a time, so a class that breaks out of the brackets can neither backtrack catastrophically nor match a multi-character construct to silently suppress the warning. `NameConstraintsSchema` validates the class under that **same engine** at parse time - not `new RegExp` - on every path that decodes partner terms, so a class it accepts is guaranteed to compile at check time and the advisory cannot fail open on a native-versus-RE2 dialect gap; the degenerate empty class, which the engine cannot compile, is rejected there with the existing character-class message. The value-level check is currently web-local (core's `validateStandardizationAgainstTerms` checks names only); promoting it to core for CLI reuse (board item 202994324) must keep it on this engine. Because this path builds no keys and feeds no receipt, it needs no cross-build vector and is exempt from the dialect's byte-identical requirement - here the linear-time engine is purely the ReDoS control, not a determinism guarantee.

## Unbounded transform-parameter rejection

A third control against the same mutually-distrusting counterparty bounds the *magnitude* of the partner-supplied transform-param values whose size drives unbounded per-row work - the `transform.params` values typed `z.unknown()` above, which the count gates do not reach into. The first is the `length` of a `pad_left` linkage-key element transform. `pad_left` runs per row over the acceptor's full dataset (`packages/core/src/standardization.ts`, `applyElementTransform` driven by `buildKeyStrings`) and allocates `String.prototype.padStart(length, char)` for each row, so an unbounded `length` - a crafted `1e9` - makes every row allocate a ~1 GB string and exhausts memory, hanging or crashing the acceptor (the web acceptor's browser tab, the CLI acceptor's process). It is the memory-allocation analogue of the ReDoS regex above: the same fail-open denial of service reachable from the authenticated counterparty, in the same partner-influenced-terms-executed-at-the-local-party model, driven by an unbounded numeric param rather than a pattern. The value cannot be rewritten - PSI requires byte-identical keys and the terms are canonically hashed into the exchange-agreement receipt, so a unilateral clamp both breaks matching and surfaces as a terms mismatch - so the control rejects an out-of-range value at validation, before any row runs.

Enforcement is a per-step refine on `TransformStepSchema` within `LinkageTermsSchema` (`packages/core/src/config/linkageTerms.ts`), so - like the dialect-conformance refine - it covers every path that parses partner terms at once: `parseLinkageTerms`, the invitation-token decode (`InvitationTokenSchema`), and `ExchangeSpecSchema`. The cap must live on this wire schema rather than on the editor-facing `STANDARDIZATION_FUNCTION_DESCRIPTORS`, which an attacker-authored token never passes through: the partner-supplied `transform.params` values are typed `z.unknown()` and carry no content bound (only their entry count is gated, above). A `pad_left` step whose `length` is a positive integer above `MAX_PAD_LEFT_LENGTH` (**256** - far above any legitimate left-pad, which is tens of characters for a zero-padded SSN or phone, and far below an allocation that matters) is rejected. Only a positive-integer `length` ever reaches `padStart` - `padLeftFactory` throws on a non-number, non-integer, or non-positive value before allocating, and that runtime check remains as the backstop for the operator-local `standardization` path, which never reaches this schema - so bounding positive integers above the ceiling closes the whole allocation vector; a malformed value is left to that runtime check, whose clean-abort path is unchanged. A legitimate `pad_left` parses and runs unchanged, preserving the cross-party receipt hash.

The second is `parse_date`'s `inputFormat` and `outputFormat`. `parse_date` builds a regex from `inputFormat` (compiled once per distinct transform array, memoized like the other steps) and assembles its output from `outputFormat` once per matched row. The size of those values still drives cost: a long `inputFormat` compiles a larger regex, and a long `outputFormat` allocates a larger output per matched row, so a per-step refine on `TransformStepSchema` caps each format string at `MAX_DATE_FORMAT_LENGTH` (**256** - far above any real date layout like `MM/DD/YYYY` or `YYYY-MM-DD`, yet small enough that the compile and per-row output stay cheap). Only a string value drives this cost; the factory treats a non-string as an empty/absent format, so a non-string is left to it unchanged, as a malformed `pad_left` length is. The format's SHAPE - the adjacent `(\d{1,2})` groups its `MM`/`DD` tokens expand into, which catastrophically backtrack on `new RegExp` at a length well under this cap - is no longer a separate control: `parse_date` compiles its regex under the linear-time engine (see [Transform-regex linear-time dialect](#transform-regex-linear-time-dialect)), which bounds the shape by construction. `parse_date` is not screened by the dialect gate (its library-generated regex is always in-dialect); its safety is the linear-time engine plus the 256-character format cap, which keeps the generated regex small and low-ambiguity by construction (only bounded `\d` quantifiers, never an unbounded quantifier over a sub-expression), so its per-row cost stays sub-millisecond on realistic input. This length cap is therefore now a pure work-SIZE bound, not a backstop against a backtracking blow-up.

The severity mirrors the ReDoS sibling: a fail-open denial of service reachable from the authenticated counterparty, bounded to DoS (no data, no code execution), and not meaningfully self-inflicted - a human reviewing the consent screen cannot vet a numeric `length` or a date-format string for an allocation or compile blow-up, and the handshake authenticates identity, not that the authored terms are benign. The bound is fixed defense-in-depth, fail-closed, not a semantic limit. The sibling numeric and expansion vectors audited alongside it are latent, not live, and carry their bound into their own implementation: `substring` reads a partner `length` but slices with `String.slice`, which cannot exceed its input; the `split_on` fan-out cross-product and the `generateFuzzyComparisons` expansion (`transpositions` / `edit_distances` / `adjacent_years`, board item 200737599) are gated behind unimplemented features and must be bounded when that fan-out ships. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## Display sanitization escape format

The display sanitizer escapes a string for operator-facing output by replacing every byte or code point that is not printable ASCII - C0 and C1 control characters, DEL, and any non-ASCII code point - with a hex escape: `\xHH` for a code point <= 0xFF, `\uHHHH` for a code point <= 0xFFFF, and `\u{HHHHH}` for an astral code point above 0xFFFF; a literal backslash is doubled (`\\`), and printable ASCII (0x20 to 0x7E) is otherwise left unchanged. The output is truncated to a bounded maximum length with a visible truncation marker. It is a one-way display transform: the escaped form is never parsed back, never used to build a filesystem path, and never used in an equality or hash comparison - those always use the original bytes - so escaping changes only what an operator sees, not how a value matches or routes.

## SFTP fatal-packet crash safety

The size and liveness bounds above turn a hostile server's behavior into a bounded failure, but they leave one behavior unbounded: a process crash, which is neither a memory nor a time exhaustion. The SFTP adapter drives the raw ssh2 `SFTPWrapper` directly (for `list()`, `createExclusive()`, and the streamed `get()`), and after the session connects that wrapper carries no `'error'` listener of its own - ssh2's `Client.sftp()` attaches a setup-time listener but strips it before handing the wrapper back, and `ssh2-sftp-client` attaches `'error'` handlers only to the SSH `Client` object and to per-operation read/write streams, never to the wrapper. So when a server returns a malformed SFTP reply packet (a malformed NAME, HANDLE, or DATA packet, an unknown packet type, or a duplicate VERSION packet after the session is already ready - a malformed *first* VERSION instead rejects `connect()`, whose setup-time listener is still attached then, so it never reaches this listener-free state) - or faults the framing layer itself with an over-long packet, which trips the length-prefix check before any request id is consumed - ssh2's fatal-error path emits `'error'` on a listener-free `EventEmitter`, which Node escalates to an uncaught exception - crashing the CLI outright, before the lock-file and temp-file cleanup and the typed exit-code mapping that an orderly failure runs. The adapter closes this by attaching a single guarded `'error'` listener to the wrapper at connect time (once per wrapper instance, so a reconnect's fresh wrapper is re-guarded and a repeated connect does not duplicate the listener), so a fatal protocol error is handled rather than fatal.

A handled fatal error leaves the session dead but the process alive. ssh2's `doFatalSFTPError` emits the `'error'`, destroys the wrapper, then calls `cleanupRequests`, which fails the callback of every request still registered. What bounds an *in-flight* `list()` or `get()` depends on which request carried the fatal packet, and the realistic case is not `cleanupRequests`: when the malformed packet is the reply to the in-flight request itself, ssh2 has already deleted that request from its pending table by the time `doFatalSFTPError` runs. The `NAME` and `DATA` handlers delete it unconditionally up front, *before* the parse/check that triggers `doFatalSFTPError`; the `HANDLE` handler deletes it inside its own malformed branch (on a defined request id), immediately before triggering it. The placement differs but the net effect is the same: `cleanupRequests` finds nothing left to fail for that request and the in-flight callback never fires - the operation is bounded by the adapter's 60-second wall-clock deadline (`list()`'s deadline, the capped `get()`'s idle window, or the uncapped `get()`'s whole-operation deadline), not by `cleanupRequests`. `cleanupRequests` does fail an in-flight callback promptly only in the other cases: a fatal error that arrives on a *different* request id, or a connection-level fault (such as an over-long packet, which faults the framing/length-prefix layer before any request id is consumed), leaves the in-flight request registered for it to fail. Either way the in-flight operation is bounded - the 60-second deadline is the bound that covers the realistic in-flight case, so it is load-bearing and must not be removed on the assumption that `cleanupRequests` covers in-flight operations. The listener also captures the cause, which bounds the *next* operation: every server-driven method consults this captured error at entry and short-circuits before issuing a request the dead channel can never answer. `list`, `get`, `createExclusive`, `put`, `delete`, `rename`, and `exists` reject promptly with the same terminal `TransportOperationStalledError`; `safeDelete`, whose contract forbids rejecting (callers run it in `catch` blocks), instead resolves promptly as a best-effort no-op. This entry guard matters beyond saving a deadline wait, because after a fatal error ssh2 destroys the SFTP channel but the underlying TCP/SSH socket stays up (a hostile server keeps it alive), so an unguarded write, stat, or delete would buffer its request on the closing channel and hang until its own per-operation bound fires (60 s) rather than failing at once - turning the crash, once fixed, into a teardown that stalls for the full bound, since `FileSyncConnection.close()` drives `safeDelete`/`delete` during cleanup. The guard converts that stall into a prompt, orderly failure.

This bounds the malformed/fatal-packet crash and, once a fatal error is known, every server-driven operation: the entry guard short-circuits the next read or write rather than buffering a request the dead channel can never answer. The one behavior the guard cannot catch - a hostile server that withholds a write, stat, or delete callback *without* any protocol error, so no fatal `'error'` fires and `put`/`delete`/`rename`/`exists` never short-circuit at entry - is not left unbounded: each of those operations now carries its own per-operation bound (the 60-second deadline for `rename`/`delete`/`exists`, the 60-second progress idle window for `put`) that fails the withheld callback terminally, with the whole-exchange budget backstopping beneath. The best-effort `safeDelete` carries the same 60-second deadline so a teardown delete cannot be the soft target left on the coarse budget while every other write op fast-fails, but - because its contract forbids rejecting - it swallows the deadline's stall error and resolves rather than surfacing it, bounding the wait without breaking the never-reject guarantee its `catch`-block callers rely on. The entry guard's role, now that those bounds exist, is to spare the 60-second wait once a fatal error is already known, not to be the sole thing standing between the call and an indefinite hang. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

The bounds above reach past the public `ssh2-sftp-client` API and drive ssh2 internals directly, so they rest on premises about ssh2's internal behavior that an upgrade can silently break. Before raising either `ssh2` or `ssh2-sftp-client`, follow the upgrade checklist in [CONTRIBUTING.md](../../CONTRIBUTING.md#upgrading-the-sftp-stack-ssh2--ssh2-sftp-client), which names the internal premises, the source files to re-read, and the contract-assertion test that fails red on a lifecycle change.

## Authenticated abort marker

A distinct channel control is the **authenticated cross-party abort marker**, the one mechanism here that is not a bound on a hostile operation but an authenticated signal between the two honest parties. On the file-sync transports a failed exchange leaves the waiting peer with no in-band notice that its partner has given up: it can only wait out the full peer-inactivity budget (default one hour) and then print a generic peer-silence hedge. When a party fails terminally with its directory still writable, it now writes a best-effort `<id>-abort.json` carrying a 32-byte token `HKDF(sessionKey, "psilink-abort-token-v1:<role>")`; a waiting peer that reads a `<peerId>-abort.json` whose token verifies against its own locally-derived peer token fails fast with a definitive `PeerAbortError` (exit 69) instead. The token is the authentication: a hostile filedrop/SFTP admin can write or rename any file in the rendezvous directory, but cannot produce a token that verifies without the per-session session key, so cannot forge an abort against a live peer. The label is domain-separated from the other session-key derivations - exact-distinct and prefix-free from the AEAD `psilink-aead-v1:<context>` and the `psilink-shared-secret-rotation-v1` labels - so an abort token can never collide with or substitute for an AEAD key or the rotation secret. The comparison is constant-time and length-safe, and the read fails closed on every absent, oversized, malformed, wrong-version, or non-matching input. The per-direction (role) binding makes a captured marker renamed to the other party's name verify under the wrong token and fail; the per-session ephemeral key makes a marker captured from one exchange inert in another.

The marker is best-effort and carries no cause, both deliberate. Its absence is strictly uninformative: a deleted, withheld, or forged marker simply degrades to the pre-existing peer-silence timeout and hedge - it never reads as success - so a hostile admin can at most *suppress* the fast-fail optimization (the behavior before this feature existed), never forge an abort or turn a failure into an apparent success. It carries no cause because the cause is not actionable to the waiting party - which holds no local error and must contact its partner regardless - and a plaintext cause on the untrusted channel would only widen disclosure; the token itself is a one-way HKDF output that reveals nothing about the session key or any other derived key. Because the marker is re-read every poll cycle, it is capped small - a pre-`get()` listed-size refusal and a bounded `get()`, both at 1 KiB rather than the ~512 MiB frame maximum - so an admin-plantable file cannot become an availability vector; the write is short-budgeted on both its `put` and `rename` (a few seconds each) so a sick directory cannot hang teardown. The mechanism is file-sync only: the connectionless file transport is the only one with a parked-waiter problem, since a WebRTC peer's death drops the DTLS data channel and surfaces promptly; a future cross-transport abort would instead carry a protocol-layer error kind rather than an on-disk marker. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

## SFTP host-key verification

The controls above defend against a hostile administrator of the server a party has correctly reached; a distinct control defends against reaching the wrong server. On the SFTP channel an operator may pin the server's host key by setting `connection.server.host_key_fingerprint` to its OpenSSH SHA256 fingerprint -- or to a non-empty list of them (see *Multiple pinned fingerprints* below) -- and every connection then verifies the presented host key against the pinned fingerprint(s) before authentication, aborting unless it matches one. This is the SFTP analogue of the certificate pinning the signing identity uses (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#receipt-signing-identities)): trust comes from a fingerprint rather than a certificate authority, and it replaces SSH's *silent* trust-on-first-use with an explicit, enforced pin -- supplied out-of-band, or established by a confirmed first-use prompt (below) and then verified on every subsequent run -- so a man-in-the-middle or a substituted server presenting its own host key is detected rather than silently trusted.

The fingerprint is `SHA256:` followed by the unpadded *standard* base64 (alphabet `[A-Za-z0-9+/]`, not base64url) of the SHA-256 digest of the raw SSH host-key wire blob -- byte-identical to what `ssh-keygen -lf` and OpenSSH clients display, so an operator pins exactly the value those tools print. It is validated at config-parse time by `HOST_KEY_FINGERPRINT_REGEX` (`/^SHA256:[A-Za-z0-9+/]{42}[AEIMQUYcgkosw048]$/`): the final character's class is restricted because the last 6-bit base64 symbol of a 32-byte (256-bit) value carries two low bits that are zero in the canonical encoding, so only those sixteen symbols are valid there. A bare 43-character base64url value -- the shape of a signing `partner_fingerprint` -- is detected separately and rejected with a message naming that confusion, since the two fingerprints use different alphabets and one is easily pasted for the other.

Verification runs in ssh2's `hostVerifier` callback, which the SSH handshake invokes with the raw host-key blob before authentication. psilink always installs a callback (in one of three forms below) and installs it *after* applying `provider_options`, so a provider option can neither override nor disable it; the `provider_options` allowlist is in any case default-deny and drops `host_verifier`/`hostVerifier` and `host_hash`/`hostHash`. `hostHash` is never set: setting it would make ssh2 pre-hash the key before the callback, destroying the raw bytes the fingerprint is computed over. The three forms: with a pin set, an *enforcing* verifier computes `SHA256:` + base64(SHA-256(blob)) once and compares it against each pinned fingerprint with a constant-time byte comparison over the decoded digests, accepting on the first match (nothing secret is compared -- a host key and its fingerprint are both public -- so the constant-time comparison is house-style hygiene for digest comparisons, not a timing-oracle defense, and the short-circuit on the first match is likewise not a timing concern); with no pin, a *fail-closed* verifier computes the presented fingerprint into the error and refuses (see the no-pin default below); and on the deliberate first-use probe, a *capture* verifier records the presented fingerprint and refuses, so the probe learns the key without authenticating (see first-use trust below).

**Multiple pinned fingerprints.** `host_key_fingerprint` accepts either a single fingerprint or a non-empty list, validated entry by entry at config parse (each against `HOST_KEY_FINGERPRINT_REGEX`, with the signing-`partner_fingerprint` confusion detected per entry; an empty list is rejected, since it would pin no key while reading as "pinned"). The enforcing verifier accepts a presented key matching *any* listed fingerprint, which gives zero-downtime host-key rotation: an operator stages the incoming key alongside the current one for the rekey window so either is accepted with no failed exchange in between, then drops the old entry after the cutover. The single-fingerprint form is the one-element case and behaves identically. The verifier records the *matched* pin as the observed host key (`FileSyncConnection.observedHostKey`) -- the one fingerprint the server actually presented, not the whole set -- so the cross-party reconciliation below still compares one observed key per party even when each party pins several. The set membership lives in one tested primitive (`matchHostKeyFingerprint`, the set extension of `verifyHostKeyFingerprint`, which delegates to it), so there is a single decode-and-compare implementation. On a mismatch the error names the presented fingerprint and the entire pinned set.

The control is fail-closed on every path. A mismatch, a malformed or empty blob, or any error inside the callback results in `verify(false)`, which ssh2 raises as a fatal handshake error whose message contains `Host denied (verification failed)` -- before authentication, so no credential is presented to an unverified server. The CLI adapter treats that rejection as terminal and does not retry it: retrying would only re-run the key exchange against the same untrusted host. The `Host denied` message fragment is an ssh2 internal premise recorded in the [SFTP-stack upgrade checklist](../../CONTRIBUTING.md#upgrading-the-sftp-stack-ssh2--ssh2-sftp-client); even if a future version renamed it, core re-wraps the same rejection into an `SFTP host-key verification failed:` message and the outcome is unchanged, only slower. The mismatch message names the presented fingerprint and key type so an operator can distinguish a legitimate key rotation from an attack; the key type is decoded from the server-controlled blob, so it is routed through `sanitizeForDisplay` and quoted before it reaches the operator's terminal or log. A changed key is never auto-accepted -- there is no standing always-accept flag (which would be permanent trust-on-first-use disguised as protection). The recovery for a legitimate rotation is a deliberate re-pin, modeled on ssh's `known_hosts` flow: verify the new fingerprint out-of-band, then add it to `connection.server.host_key_fingerprint` (alongside the current pin for a staged rotation, or in place of it) or remove that field and re-run interactively to re-establish trust on first use (which overwrites the stored pin).

Each pinned fingerprint may be supplied as an `@`-file reference (`@/path/to/fingerprint`) for operators who keep it in a read-only secrets mount; in a list, entries may freely mix literals and `@`-file references. Because `@`-file resolution runs after parsing, a literal `@path` -- which cannot match the `SHA256:` format -- is exempt from the parse-time format check, and each resolved file's contents are re-validated against `HOST_KEY_FINGERPRINT_REGEX` at resolution: a malformed or empty secrets file fails as a `UsageError` (exit 64) naming the offending reference, rather than as a confusing host-key mismatch at connect time.

Verification applies to the CLI `sftp` channel only. A file-drop makes no SSH connection, and the browser/proxy SFTP path does not run ssh2's `hostVerifier`. An attacker cannot induce the unpinned state to inject their own key, because the pin is configured out-of-band (or established on first use, below) and the invitation endpoint schema rejects a `host_key_fingerprint` field outright (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#transport-layer-authentication)). As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

**No-pin default: fail-closed.** When no fingerprint is pinned, `FileSyncConnection.open` installs the fail-closed verifier, which refuses the connection before authentication (so no credential reaches an unverified server) with an error naming the presented fingerprint to pin. This replaces the earlier warn-and-proceed default, which trusted any presented key. The flip is deliberately gated on first-use existing (below): a pin can only be reliably obtained once first use exists, so flipping the default earlier would have stranded every unpinned config with no recovery path.

**First-use trust (ssh-style).** An operator who cannot obtain the fingerprint out-of-band can still reach the verified-on-every-subsequent-run state by establishing the pin on a deliberate first connection. Every CLI command that opens an `sftp` connection -- `exchange`, the online `invite`/`accept`, and a zero-setup exchange -- runs this flow before connecting (a shared `establishHostKeyTrust` step):

- On a TTY (interactive), it *probes* the server: a throwaway connection whose capture verifier records the presented host key and immediately refuses, so the key is learned without presenting a credential and without waiting on a user prompt inside the handshake (which would race ssh2's `readyTimeout`). It then surfaces the fingerprint and key type, prompts for confirmation, and on a yes pins the fingerprint into `connection.server.host_key_fingerprint` in memory and proceeds with a real, now-enforcing connection. Because that real connection re-verifies the just-pinned key, a key swapped between the probe and the connect is still caught (the same brief TOCTOU window as `ssh-keyscan`-then-`ssh`).
- Non-interactively (no TTY -- an automated run, or one piping its CSV through stdin), it does not prompt and does not auto-accept: it fails closed with an actionable error naming the recovery (run once interactively to pin, or set `host_key_fingerprint` out-of-band). This is the same fail-closed posture as the no-pin default, surfaced earlier and with CLI-specific guidance.

How the confirmed pin reaches disk depends on the command: `exchange` reads an existing `psilink.yaml` and does not re-write it, so the pin is written there in place (preserving comments); the online `invite`/`accept` and a zero-setup run with `--save` carry the in-memory pin into the configuration they write after the handshake; a zero-setup run without `--save` trusts the key for that one exchange and persists nothing. The probe reuses the exact connect options (algorithms included) the real connection uses, so both negotiate the same host-key type and the pinned key is the one later runs verify.

**Cross-party fingerprint reconciliation.** Pinning above is each party's *local* decision: the two parties pin the same server independently, at different times on different machines, and nothing otherwise compares their views. A one-sided interception -- one party reaching an attacker's server while the other reaches the real one, each having pinned the key it was shown -- is therefore invisible to both, as is a server host-key rotation that happens between the two parties' setups. To close this, each party advertises the fingerprint and key type it observed (`FileSyncConnection.observedHostKey`, recorded by the enforcing verifier on the pin-matched success path) inside the authenticated post-handshake terms exchange, and the two compare. The advertisement rides the terms exchange -- the one bidirectional round-trip both parties always perform, the same carrier as the `--save` intent -- as an optional `hostKey: { fingerprint, keyType }` field on the terms message; a party that observed no host key omits it. Reconciliation lives in `reconcileHostKeyFingerprints` and is wired through `runExchange`.

Placement is load-bearing: the field is consumed only from the terms exchange, which on the authenticated path runs inside the per-direction AEAD (it is keyed from the X25519 session key), so the advertised value is authenticated and an unauthenticated party cannot forge or rewrite it. The CLI threads its observed key into `runExchange` only when that AEAD wrap is in force (`secure` set); a zero-setup (unauthenticated) exchange advertises nothing, because over an unauthenticated channel an active man-in-the-middle could simply rewrite the advertised value to match and suppress the divergence. This is the post-authentication, in-band complement to the endpoint decision in [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#invitation-contents-and-confidentiality): the invitation endpoint still carries no server-identity material, and reconciliation adds its check strictly after the authenticated handshake instead.

The comparison is a verbatim string equality on the fingerprint (a host key and its fingerprint are both public, and the result drives only a warning, not a trust decision, so no constant-time compare). It surfaces a divergence -- never an abort, in keeping with the honest-but-curious model -- as a warning naming *both* observed fingerprints, with wording that distinguishes the two cases the difference can represent: when the key *types* match, the difference is narrowed to a server rekey between the two setups or a one-sided interception; when the key types differ, the message additionally surfaces the benign multiple-host-key case, because a server that presents more than one host key may legitimately show each party a different one, so a type difference alone is not evidence of an attack. A party that observed no host key (a file-drop mount or the browser/proxy SFTP path) advertises nothing, and a one-sided absence reconciles to no divergence rather than a false positive. The advertised field is fail-soft: a malformed or over-bound `hostKey` (reachable only from a non-conforming or future-versioned peer, since the field rides the AEAD) is read as absent rather than aborting the linkage, so this advisory check can never turn a valid exchange into a hard failure. That drop is silent toward the protocol, but `exchangeTerms` still distinguishes a present-but-malformed advertisement (a `hostKey` on the wire that fails validation) from a genuine absence and reports it out as `partnerHostKeyMalformed`; the CLI logs the malformed case at debug, so an operator can tell a non-conforming peer from one that simply observed no host key (a file-drop or proxy path), while the benign no-host-key path stays quiet and the level stays low enough that a version-skew does not warn on every exchange. The dropped bytes are deliberately not logged -- they are unusable and partner-controlled, so echoing them would widen disclosure and risk log injection. Both parties' key types are server-controlled and are routed through `sanitizeForDisplay` before they enter the warning. As net-new security behavior layered on the host-key control, this reconciliation is in scope for the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

# Web signaling surface bounds

The bounds above harden the CLI's file-sync transport against a hostile server admin. The web application's bundled PeerJS-compatible signaling server is a separate surface with a different model: it is untrusted by design (see the **Channel security** and **Third parties** overviews in [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#channel-security)), relaying only opaque rendezvous-setup messages between two browsers that authenticate each other directly, so the residual exposure on its WebSocket upgrade surface is resource exhaustion, not access to any party's data. The guards below are defense-in-depth against that nuisance, enforced in the application unconditionally; the deployment-dependent Origin and per-address controls are the reverse proxy's responsibility and are described operationally in [DEPLOYMENT.md](../DEPLOYMENT.md#hardening-the-signaling-surface), not here. All values are fixed constants, not configurable, for the same reason as the file-sync bounds: a configurable bound risks an operator raising it enough to reintroduce the denial of service.

**Upgrade-handshake timeout.** The shared HTTP server bounds a pre-101 upgrade handshake with a **10-second** header timeout (`SIGNALING_HEADERS_TIMEOUT_MS`) and a **15-second** request timeout (`SIGNALING_REQUEST_TIMEOUT_MS`), set in `apps/web/server/upgradeHardening.ts`. A client that opens a connection and dribbles -- or never finishes -- its request headers is closed server-side rather than held open. The entry attaches its own `clientError` handler to perform the close rather than relying on Node's built-in default close-on-`clientError`, because that default is silently disabled the moment any other `clientError` listener is attached (an embedding framework, or under test a loaded `ws`, may attach one). Node checks the header and request timeouts on a periodic sweep (default 30 seconds), so the effective close is the bound plus up to one sweep interval -- coarse, which is sufficient for a resource-exhaustion guard.

Those two timeouts only arm once HTTP request parsing has begun, so they do not cover the cheapest hold of all: a peer that completes the TCP handshake and then sends nothing, which has no request for them to bound and would otherwise sit until the OS reaps the idle socket. A **10-second** per-socket idle timeout (`SIGNALING_PREHANDSHAKE_IDLE_MS`, applied on each new connection) closes that case, on its own precise timer rather than the periodic sweep. It covers only a socket that reads no bytes: a peer that dribbles actual request bytes resets this idle timer and is instead bounded by the header and request timeouts above (whose deadlines are absolute from the start of the request, not reset by each byte). `ws` resets the socket timeout to `0` the instant a socket completes the 101, so an established WebSocket -- governed by the `ws` close timer and the liveness reaper below -- is never cut by it; only the pre-handshake window is bounded. This guard would need revisiting only if a real client could legitimately take longer than the bound to send its upgrade request, which a single small WebSocket handshake never does, or if `ws` stopped resetting the socket timeout on upgrade (a regression test pins that an established connection outlives the bound).

**Per-message size cap.** Each inbound frame is capped at the `ws` layer so an over-cap frame is refused (close code 1009) before the message handler parses it. This control and its constant (`MAX_SIGNALING_PAYLOAD_BYTES`, in the vendored `services/webSocketServer/index.ts`) are specified above under [Signaling-server inbound frame bound](#signaling-server-inbound-frame-bound), alongside the file-sync frame-size bound; it is one of this surface's bounds, closing the single-oversized-frame crash and memory pin from an unauthenticated peer.

**Two-tier liveness reaping.** A registered client that has not yet sent any inbound frame is reaped after **20 seconds** (`unconfirmed_timeout` in the vendored `config/index.ts`); once it sends its first frame it is marked confirmed and governed by the existing **90-second** `alive_timeout`, refreshed by each heartbeat. The split ties reaping to liveness rather than a flat wall-clock: a socket that registers under the public realm key and then goes silent -- an abandoned or junk registration -- is cleared in 20 seconds instead of squatting a slot for 90, while a real peer is never cut short however slow the human-paced exchange. The reap is keyed on the WebSocket heartbeat, which is independent of and far faster than WebRTC pairing: the PeerJS client sends its first heartbeat one cadence interval after the socket opens and then every interval, so even an inviter that waits minutes for its partner has long since graduated to the generous window. The cadence is psilink-owned, not a transitive dependency default: the web client sets `pingInterval` explicitly to **5 seconds** (`PEER_PING_INTERVAL_MS` in `apps/web/src/psi/rendezvous.ts`) rather than relying on the `peerjs` default, a default parameter inside a bundled file that an upstream version change could shift (now an exact-pinned, review-gated `peerjs` bump rather than a silent caret drift -- see [CONTRIBUTING](../../CONTRIBUTING.md#upgrading-the-peerjs-stack-peerjs--peerjs-js-binarypack)). The 20-second value is four times that 5-second first-heartbeat cadence, leaving margin for a slow socket open and one missed heartbeat. The unconfirmed window must stay comfortably above the first-heartbeat delay so a real peer always graduates before it fires; that `unconfirmed_timeout >= 4x PEER_PING_INTERVAL_MS` relationship is pinned by a unit check (`test/unit/signalingReaping.test.ts`), so a change to either value -- the reap window or the cadence -- that narrows the margin fails CI rather than only the weaker `unconfirmed_timeout < alive_timeout` invariant. Because the cadence is now set explicitly, a `peerjs` minor bump no longer shifts it out from under the window at all; tracking a future upstream default change is a deliberate edit to `PEER_PING_INTERVAL_MS`, which the same check then re-validates against the window. The check guards the numeric margin but not the *meaning* of the cadence, so -- as with the SFTP-stack internal premises -- a `peerjs` bump must still re-confirm two behavioral premises this control rests on: that `pingInterval` remains the sole client-originated liveness signal (the reaper keys only on observed inbound frames, and on an otherwise-idle registered peer the heartbeat is what produces them; a field rename instead surfaces as a TypeScript error on the `PeerOptions` literal), and that no upstream WebSocket-layer keepalive was introduced that would hold a socket open independently of that heartbeat -- either change would decouple liveness from the cadence the window is sized against, which neither the type check nor the margin check would catch. The check runs on the existing 300 ms sweep (`checkBrokenConnections`), so the effective reap is within ~20.3 seconds. This bounds the cheap fire-and-forget / register-and-idle flood; a flood that keeps each socket alive with heartbeats is bounded instead at the reverse proxy (per-address limiting, see [DEPLOYMENT.md](../DEPLOYMENT.md#hardening-the-signaling-surface)), since the application cannot distinguish it from a live peer. The only in-application ceiling on such a maintained flood is the global `concurrent_limit` (default 5,000 registered clients), which is shared across all clients, so on a deployment with no proxy a maintained flood degrades to global connection exhaustion rather than per-address throttling -- the reason per-address limiting is delegated to the proxy rather than attempted here.

**Relay queue bounds.** When a client addresses a signaling message to a destination id that is not currently registered, the relay holds it briefly in a per-destination queue for a reconnect. Left unbounded, a registered client could spray messages to arbitrarily many made-up destination ids -- each allocating a queue -- or grow one queue without limit. The realm caps the number of distinct queued destinations at **1,000** (`MAX_OUTSTANDING_QUEUES`) and the depth of any one queue at **100** messages (`MAX_MESSAGES_PER_QUEUE`), both in `models/realm.ts`; a message past either bound is dropped. Both sit far above any legitimate rendezvous, which queues at most a handful of frames for a momentarily-absent peer, so a drop only ever affects a spammer or a peer already far past needing a reconnect hold.

The message-count cap alone, though, leaves a queue's resident ceiling high: a frame is capped at 256 KiB on the wire (`MAX_SIGNALING_PAYLOAD_BYTES`), but V8 stores a JavaScript string as two bytes per character whenever it holds any non-Latin1 character (>= U+0100), so one buffered frame can occupy roughly **512 KiB** of heap -- about **50 MiB** per queue (100 frames), **50 GiB** across the full 1,000 -- since a flood of max-size frames is bounded only in count, not bytes. A third bound, **524,288 bytes** (512 KiB, `MAX_QUEUE_BYTES` in `models/realm.ts`), caps each queue's total buffered bytes directly: before a frame is queued, its resident size is added to the queue's running total, and the frame is dropped if that would exceed the cap. The size (`messageByteSize`) is measured as UTF-16 code units times two, not UTF-8 -- a UTF-8 measure would undercount a non-Latin1 payload by up to 2x and let it occupy roughly twice its measured size, so counting the worst-case two-byte residency bounds the real heap regardless of payload charset. This holds the per-queue ceiling at 512 KiB and the global ceiling at about **512 MiB** (1,000 queues x `MAX_QUEUE_BYTES`). The cap is twice the wire frame cap, so any single legal frame is always holdable, and real signaling frames are KB-scale, so a queue still holds dozens; a queue drained by a reconnecting peer releases the read frames' bytes from the running total, so it accepts fresh frames again rather than staying wedged at the cap. The ~5-second read-cold expiry (the `MessagesExpire` prune) still clears an unread queue regardless, so these caps bound the transient peak within a window rather than a permanent residency.

**Peer-id length cap.** The per-queue byte cap above sizes a queued frame by its string fields' resident bytes, `src` included -- and the server overwrites `src` with the connecting client's own id (the `message.src = client.getId()` stamp in `services/webSocketServer/index.ts`). That id arrives as the `id` query parameter of the WebSocket upgrade request and, left to itself, is bounded only incidentally by Node's ~16 KiB HTTP header-size limit on the upgrade URL, not by any application bound. So a peer that registers a multi-KiB id and then addresses a near-maximum-payload frame to an absent destination produces a single frame whose `src` alone pushes it past `MAX_QUEUE_BYTES` -- the "any single legal frame is always holdable" property above rests on a queued frame's residency staying within twice its wire size, which the `src` overwrite is the one thing that can break. The same unbounded id is also retained in the realm `clients` map, a standing per-client memory surface (up to the `concurrent_limit` default of 5,000 ids x the incidental ~16 KiB ~= ~80 MiB). Each upgrade-handshake parameter -- `id`, `token`, and `key` -- is therefore length-bounded to **256 characters** (`MAX_HANDSHAKE_PARAM_LENGTH`, in the vendored `services/webSocketServer/index.ts`) where it is parsed in `_onSocketConnection`, before the id is stored in `clients` or used as `src`; an over-length parameter is refused with the same error-frame-and-close (`WS_PARAMETER_TOO_LONG`) as a missing one. The cap sits far above any legitimate id -- psilink's rendezvous ids are 32 hex characters (`deriveRendezvousPeerId`) and a PeerJS default id is a UUID (~36), so 256 is ~7x any real id and refuses no legitimate peer -- and well over an order of magnitude below the incidental header limit (~64x).

The `id` bound is the load-bearing one: it holds the `src` contribution to a queued frame at most `2 x MAX_HANDSHAKE_PARAM_LENGTH` = 512 resident bytes, a negligible fraction of the 512 KiB queue cap, so a real KB-scale frame from any accepted id stays holdable. (The byte cap's drop of an *adversarial* near-256-KiB frame is unaffected and stays benign: it loses only that peer's own reconnect-hold frame, never desyncing the accounting or amplifying memory.) `token` and `key` are bounded at the same parse point for a uniform "no handshake parameter is unbounded" invariant: `token` is likewise retained per-client in the `clients` map (the same standing-memory surface the id presents, which the cap cuts to ~5,000 x 512 bytes ~= ~2.5 MiB), while `key` is only compared against the realm key and never stored, bounded for symmetry. The addressed destination id (`dst`) is deliberately left unbounded here, not an overlooked sibling: unlike `src` -- which the server appends after the wire-frame cap is checked -- `dst` rides inside the client's own frame and is therefore already within the 256 KiB wire cap, and its residency both as a queue-map key and as a counted frame field is already governed by `MAX_OUTSTANDING_QUEUES` and `MAX_QUEUE_BYTES` with the read-cold expiry, so a separate length bound would be redundant. As net-new security behavior on an internet-facing surface, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.

# WebRTC data-channel inbound bound

The bounds above cover the signaling *socket* -- the broker the two browsers rendezvous through. The PSI payload itself never crosses that socket: it flows peer-to-peer over the WebRTC *data channel*, which is the surface this bound covers. The adversary here is the exchange counterparty, a mutually-distrusting peer (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#threat-model)) -- authenticated by the X25519 handshake, but free to be hostile or buggy thereafter. The data channel is end-to-end confidential under DTLS, so neither the broker nor a TURN relay is in scope; the peer is.

No existing control reaches this transport. The web path discards the rotated key and declines the application-layer AEAD wrap under DTLS (`apps/web/src/psi/authenticateExchange.ts`, `requestEncryption: false`), so core's `EncryptedMessageConnection` and its ~512 MiB `MAX_FRAME_SIZE_BYTES` envelope -- a file-sync transport control -- never bind on the web path. (If the deferred web-encryption work, board item 195802633, ever forces the AEAD wrap on, that envelope would come into scope and could subsume part of this need; until then the wrap is declined, so this control stands on its own and does not wait on the encryption rework.) The only inbound limiters that do apply are core's count/structure bounds on the *parsed* payload-exchange message (the web wire is BinaryPack, not JSON, but those bounds act on the deserialized structure either way) and the `QueuedMessageConnection` message-count capacity -- none a byte-size bound. The large raw PSI set frame -- one party's full encrypted set as raw elliptic-curve points (~64 bytes/element; see [PROTOCOL.md](PROTOCOL.md#psi)) -- is carried as a single binary `Uint8Array` that, left unbounded, PeerJS reassembles with no byte cap.

The control lives in `apps/web/src/psi/boundedReassembly.ts` and bounds the receive path on several axes, each a fixed constant (not a configurable option, since a configurable bound risks being raised to reintroduce the denial of service) and each fail-closed with a terminal `protocol` `ConnectionError` -- the same class as core's inbound-buffer overflow, since every bound sits far above any legitimate frame so an over-bound frame is never benign. PeerJS chunks a binary frame larger than its ~16 KiB MTU into slices and reassembles them in `_handleChunk` (accumulating slices keyed by message id, `_chunkedData`), and `unpack`s every frame -- an unchunked one directly, or the reassembled buffer on completion -- into a JS structure in `_handleDataMessage`, with no cap on any of total wire bytes, deserialized structure size, retained chunk count, or concurrent reassemblies, and no eviction of a never-completed partial. `boundChunkReassembly` wraps *both* methods to add all of these *before* delegating to the original, so the offending chunk is never stored and the offending frame is never unpacked. Wrapping `_handleDataMessage` -- the sole unpack point, which both an unchunked frame and the reassembled-completion path flow through -- rather than only `_handleChunk` is load-bearing: the deserialized-structure attack below needs only a tiny frame, which is sent unchunked and never reaches `_handleChunk`.

**Wire-byte cap.** A running total of bytes across all in-flight reassemblies is bounded by **`MAX_WEBRTC_FRAME_BYTES` = 268,435,456 (256 MiB)**; a slice that would push it over fails closed rather than allocating proportional to the peer-chosen size. The total spans every in-flight reassembly, so this single bound caps a single oversized frame, a single big never-completed partial, and the aggregate of concurrent partials alike. Unlike the file-sync frame-size cap -- a *decode* ceiling anchored to Node's `MAX_STRING_LENGTH` -- this is a browser-tab *memory envelope*, in the spirit of the directory-listing cap: above the realistic largest legitimate frame, below a tab-crashing allocation. The largest legitimate inbound frame is the raw set frame above, carried as binary with no base64url 4/3 inflation (the data channel is binary, unlike the file-sync JSON-text wire); the 100 MiB CSV upload cap (`MAX_CSV_FILE_BYTES`) bounds a set to at most a few million unique key values, tens to low-hundreds of MiB, so 256 MiB (~4 million elements) sits above any realistic set while half the file-sync 512 MiB cap in deference to the tab's smaller heap. This counts *wire* bytes; the deserialized structure those bytes unpack to is bounded separately, next.

**Deserialized retained-byte budget.** The wire-byte cap alone is insufficient, because BinaryPack `unpack`s a frame into a JS structure *synchronously, before delivery and before any schema validation*, and that structure can be far larger than the wire. BinaryPack encodes an empty object or array in one byte but `unpack` allocates a real JS value per element, and the *retained* cost of that value varies by kind: ~64 bytes for an empty object, ~40 for an empty array, ~8 for an integer (a tagged SMI in its parent's backing slot), and ~`16 + 2 x declared-bytes` for a string (a SeqString header plus its UTF-16 characters); worse, an `array32` header eagerly allocates `new Array(N)` for its declared count *even when the elements are absent*, because `unpack` reads past the end of the buffer as zero rather than throwing. So a tiny wire frame of object/array headers -- an in-protocol shape, since the association-table and mapped-element frames are arrays of numbers/objects -- could deserialize to many GiB, and a flat per-value count (the prior value-count budget) charged the heaviest kind no more than the lightest. Before delegating to `unpack`, `structureOverBudget` scans the frame's BinaryPack bytes and **sums each declared value's per-kind weight** (`WEBRTC_VALUE_WEIGHTS`: object 64, array 40, scalar 8, string `16 + 2/byte`; a `bin`/`raw` value is charged only the scalar slot, its ~1x-wire payload already bounded by the wire-byte cap), rejecting if the running cost would exceed **`MAX_WEBRTC_FRAME_STRUCTURE_BYTES` = 1,073,741,824 (2^30, 1 GiB)** or the structure nests deeper than **`MAX_WEBRTC_REASSEMBLY_DEPTH` = 256**. The scan reads only container headers and scalar length prefixes -- never materializing the payload, so it skips a large binary set frame in O(1) -- and enforces, as it walks, both the running weighted cost (the root plus all containers' children, so a value already committed by an ancestor or earlier sibling is counted, not just the current container's) and that no container declares more elements than the bytes that follow it can encode (each element needs at least one byte, so this ties the value count to the wire size and refuses a `new Array(N)`-from-a-tiny-header). An over-budget frame is caught at the offending header before `unpack` allocates; a read past the end (a truncated frame that passed both checks) is delegated, since its committed structure is already bounded and PeerJS's unpack errors on it.

The budget is derived from the largest legitimate frame's *retained* cost. That is the mapped-element frame (`Array<{theirIndex, iteration}>`, one entry per matched record), which `unpack`s per record to one object (64) + two key strings ("theirIndex" 16+20, "iteration" 16+18) + two integer values (8 each) ~= 150 bytes under the weights above. At the ~4-million-element (2^22) set ceiling the wire-byte cap implies, that is ~4.19M records ~= 629 MiB; 2^30 leaves ~1.6x headroom (matching the prior count budget's ~21M-value-vs-2^25 ratio), so no exchange the wire cap admits is rejected on a downstream frame -- a property a unit test pins against the real frame shape. The weights are a deliberately *conservative* upper bound -- e.g. every object key string is charged in full, though V8 internalizes repeated property keys to one shared string, so a key-heavy frame's real retained peak is lower -- because a true memory envelope is simpler to audit than one resting on V8 interning/representation choices an engine update could change. Residual: the per-frame worst case for the kinds the budget charges in full is now this budget rather than the ~2 GiB an all-empty-objects frame reached under the flat count. A frame of the heaviest such kind (~16.7M empty objects) reaches ~1 GiB and is rejected there, and reaching even that requires ~16 MiB of proportional wire (the per-container byte check ties cost to wire, so a pathological frame is no cheaper than a legitimate one of the same retained size) and is freed once the schema layer rejects it. One kind is charged below its retained cost: a `float`/`double`/wide-integer value charged the 8-byte scalar slot retains ~24 bytes when a peer forces per-element HeapNumber boxing (an array made general-elements kind by mixing in one non-number), so a number-heavy frame can reach ~1.2 GiB -- slightly above this budget, bounded there by the wire-byte cap rather than by the structure budget, and likewise transient and freed on rejection. (Charging those markers their boxed cost would close that at the structure budget; it is not done because they never appear in a legitimate frame, so it is part of the how-aggressive-the-weights judgment below rather than a correctness fix.) A tighter budget overall would require less-conservative weights (crediting key-string internalization); how aggressive to make them is a security-review judgment, weighed with the queue residual below.

**String-size cap.** The structural budget charges a string its *resident* weight (`16 + 2/byte`), but `unpack_string` builds that string via a per-code-point concatenation whose transient cons-string tree is several times larger again (a ~250 MiB-wire `str32`, whose resident weight stays within the structure budget, measured into the multiple-GiB range *during the build*). The scan therefore refuses any string whose declared length exceeds **`MAX_WEBRTC_STRING_BYTES` = 1,048,576 (1 MiB)**. Binary set frames are `bin` (whose `unpack_raw` copies ~1x its wire size, bounded by the wire-byte cap, not a string), and every legitimate string a PSI frame carries -- the `{theirIndex, iteration}` keys, a `status` value, a payload cell -- is far shorter, so the cap rejects no legitimate frame; the value is orders of magnitude above any of them yet keeps the worst-case build transient in the tens of MiB.

**Retained chunk-count cap.** PeerJS retains each chunk as its own `Uint8Array` (measured ~232 bytes resident even for a one-byte slice), an overhead the wire-byte cap -- which counts only payload bytes -- undercounts, so a flood of tiny chunks could exhaust memory while staying under the byte cap. Each chunk is therefore charged at least **`MIN_CHUNK_RESIDENT_BYTES` = 256** against the byte cap, and the chunks per reassembly are bounded by **`MAX_CHUNKS_PER_REASSEMBLY` = 131,072 (2^17)** -- ~8x the ~16,500 chunks a 256 MiB frame produces at the ~16 KiB (16,300-byte) MTU, so a legitimate frame is never rejected while a tiny-chunk flood is bounded. Legitimate chunks (~16 KiB) far exceed the 256-byte floor, so it never affects them.

**Concurrent-reassembly cap.** The number of concurrent incomplete reassemblies is bounded by **`MAX_CONCURRENT_REASSEMBLIES` = 8**; a new message id beyond the cap evicts the oldest incomplete partial (deleting its `_chunkedData` entry and releasing its bytes). The PSI protocol is strictly lockstep (see [PROTOCOL.md](PROTOCOL.md)) and the data channel is reliable and ordered, so at most one frame is ever mid-reassembly on an honest exchange; 8 is generous headroom, and the cap bounds a flood of partials from many distinct ids -- the metadata-overhead case the byte running-total does not catch, since many tiny partials stay under it. Eviction is silent and non-fatal: a legitimate exchange never has a second partial, so eviction only drops adversarial data, and logging per eviction would itself be a spray-amplified log-flood vector.

**Delivered-frame backstop.** Finally, at the stable `data` event (`checkDeliveredFrameBound`): a fully delivered binary frame (a `Uint8Array`/`ArrayBuffer`) over the byte cap is refused there too, regardless of how -- or whether -- PeerJS chunked it, so the bound holds at the public layer even if the internal reassembly premise below ever breaks. A parsed object/array is not byte-measured here (the reassembly bounds above govern it before delivery, and core's count/structure bounds after); only binary frames are, which is exactly the raw set frame this is sized for.

The bounds above are *per frame*. A delivered frame is then queued in core's `QueuedMessageConnection`, whose capacity is a message *count* (`DEFAULT_CAPACITY` = 1024), so the aggregate retained across queued-but-unconsumed frames is in principle the per-frame budget times that depth. This queue residual was weighed when the per-frame budget was made byte-aware, and the queue was **deliberately left a count bound**: the lockstep protocol drains one frame before sending the next, so the queue sits at depth ~1 in practice, and a peer that floods frames out of turn trips the count capacity (a `protocol` failure) rather than accumulating -- so the theoretical per-frame x depth aggregate requires both proportional wire bytes per frame *and* the consumer to stall, and is the count capacity's concern rather than this control's. A byte-aware queue bound (or a smaller web-transport capacity) would close the theoretical multiplier but buys little against a practical depth of ~1, and -- unlike the per-frame scan, which is web-side because core's transports never `unpack` BinaryPack -- would force a web-vs-core placement choice (a core bound would let a future CLI WebRTC transport inherit it) better made when a transport actually needs it. The per-frame budget is therefore the byte envelope; the queue stays bounded by count.

The wrap reaches past the public `DataConnection` API into PeerJS internals, so it rests on two dependency premises a `peerjs`/`peerjs-js-binarypack` upgrade could silently break. The first is the reassembly/unpack shape on the binary/chunked connection class (the one `peer.connect` and an incoming connection use by default): `_handleChunk` reassembling slices keyed by `__peerData` into `_chunkedData` and deleting the entry on completion, and `_handleDataMessage` as the sole point each frame is unpacked (an unchunked frame directly, the reassembled buffer via the completion recursion). This is encoded as a runtime check, not a comment: `assertChunkReassemblySupported` (called before any listener is attached) throws at install time if any of those internals are absent, so the live browser exchange test -- which installs the guard on every web exchange -- fails loud rather than running with no inbound bound. The second is the BinaryPack wire format the structural scan parses (the marker dispatch in `peerjs-js-binarypack`'s `Unpacker.unpack`: fixint/fixraw/fixstr/fixarray/fixmap and the `0xc0`-`0xdf` markers, with maps declaring two child values per pair). A format change there would not silently disable the bound -- a misparse either over-charges (rejecting early, fail-closed) or runs the cursor off the end (treated as a malformed frame and delegated to PeerJS, which would itself error) -- but it would weaken the scan's precision, so re-verify the marker table on a `peerjs-js-binarypack` bump. On a `peerjs` bump, re-verify both premises. As net-new security behavior, this control is subject to the explicit security review required by [CONTRIBUTING](../../CONTRIBUTING.md#dependency-policy) before release.
