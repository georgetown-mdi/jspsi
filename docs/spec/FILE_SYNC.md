# File-sync transport state model

This is the single source of truth for the *intended* state mechanics of `FileSyncConnection` (the `sftp` and `filedrop` channels). It exists because the rendezvous and message-loop behavior is an implicit state machine spread across several enforcement sites in one file, and because a cluster of planned changes each add a new file type or transition to it. Without one model to point at, each change re-derives the mechanics from prose and the enforcement sites drift apart. This document is the shared target; the planned work is tracked against it in [How the planned work converges on this model](#how-the-planned-work-converges-on-this-model).

This is a developer/design document. For the user-facing configuration reference and the canonical filename grammar see [EXCHANGE_REFERENCE.md](../EXCHANGE_REFERENCE.md); for the high-level channel and synchronization overview see [COMMUNICATION.md](../COMMUNICATION.md); for the cryptographic protocol see [PROTOCOL.md](PROTOCOL.md). Where this document and EXCHANGE_REFERENCE overlap, EXCHANGE_REFERENCE owns the filename grammar and this document owns the state machine. This is a deliberate tier inversion: the filename grammar -- this document's single load-bearing invariant -- stays in the overview-tier [EXCHANGE_REFERENCE.md](../EXCHANGE_REFERENCE.md) because that is the document an operator authoring a `psilink.yaml` opens, so this spec-tier document references *up* into the overview tier for it. Do not "tidy" the grammar down into `docs/spec/`; doing so would silently break the grammar references throughout this document. (The seam is recorded in [docs/spec/README.md](README.md).)

## Core principle: the directory is the state machine

Two parties share one directory (an SFTP path or a local filedrop path) and never hold a lock on the directory itself. Each party mutates the shared state only by `put`, `delete`, and `rename`, and observes the peer only by `list`. The protocol state is therefore exactly **the set of filenames present in the directory**. Everything else -- roles, modes, who has consumed what -- is inferred from that set.

Two consequences follow, and they are the reason this document exists:

1. Correctness reduces to agreement on two things: which filenames are legal at each phase, and how each file is routed (treated as a message, a control signal, or ignored). A bug is almost always a disagreement about one of those two between the two parties, or between two of this code's own enforcement sites.
2. Every new file type is a new symbol in the state machine's alphabet, and must be accounted for at *every* site that reads the directory. Adding one without updating all of them is the standard failure mode.

## The single load-bearing invariant: the filename grammar

Routing safety rests entirely on one rule, defined in full under [EXCHANGE_REFERENCE.md "Filename grammar"](../EXCHANGE_REFERENCE.md#filename-grammar) and restated here as the discriminant the state machine depends on:

> A protocol file is named `<id>-...-<token>.json`. If `<token>` is all digits the file is a **message** and `<token>` is its declared byte count. Otherwise `<token>` is a type word (`hello`, `ack`, `joining`, `lock`) and the file is a **control file** that is never routed as a message.

In code this discriminant is `parseMessageByteCount(name) !== undefined` (in [fileSyncConnection.ts](../../packages/core/src/connection/fileSyncConnection.ts)). Only the protocol's own in-flight temp write `temp-<uuidv4()>.tmp` is excluded earlier, matched by `isProtocolTempName` (its `temp-` prefix and a v4-UUID stem, the exact shape `send()`/`writeAck()` produce) -- a *foreign* `temp-*.tmp` whose stem is not a UUID is not excluded here; it fails the grammar and is treated as a foreign file (198451188). Every `.json` control file -- `hello`, `ack`, `joining`, and the `lock` tiebreaker -- is excluded by the discriminant instead, because its terminal token is non-numeric. The grammar is what lets the message scan ignore a renamed hello, an ack, or a lock file without an explicit per-type exclusion -- so every new control file gets a non-numeric terminal token *for free*, provided the site that reads the directory uses the discriminant rather than a bare prefix glob.

**Foreign and snapshotted files (a classification, not a new file type).** A name that FAILS the grammar above -- a conflict copy, a partial download, an unrelated file -- is *foreign*: it has no `<id>-...-<token>` shape, sits outside the NNN alphabet, and is never a protocol file. Foreign names present at `synchronize()` entry are recorded in an in-memory snapshot and tolerated for the session (195255994); a foreign name that first appears *after* entry is a newcomer handled by `unexpected_files` (194800733). Neither is prefix-screened at `poll()`: a foreign name that coincidentally matches `<peerId>-<digits>.json` MATCHES the message grammar, so it is a protocol file (a message), not foreign -- rejected at the entry guard with no flag, or swept under `--sweep-exchange-files`, and never snapshotted. Two existing rows are also read at *entry* now by the pre-sweep retain-signal inspection (site 1): the hello row's payload supplies `retain_files`, and a retain-only message ack's mere presence (an `-ack.json` whose two trailing dash segments -- the NNN and the byte count of a timestamped message -- are both all-digit) is itself a retain signal.

## The five enforcement sites

The state machine is enforced -- and therefore must be kept consistent -- at exactly these five places in [fileSyncConnection.ts](../../packages/core/src/connection/fileSyncConnection.ts). Adding or changing a file type is a change to *all applicable rows of the taxonomy at all five sites*; this is the anti-drift checklist.

1. **Preexisting-file guard** (`synchronize()` start). Three distinct mechanisms, kept separate. **(a) Foreign snapshot (tolerate, both modes, always).** A name that fails the grammar is foreign; foreign names present at entry are recorded in an in-memory snapshot and tolerated, deleting nothing (safe in retain mode, where sync-mediated conflict copies are expected noise). The snapshot feeds the recognized-for-the-loop set at site 3 so a pre-existing foreign name is not re-flagged mid-loop. **(b) Default strict-empty guard (no flag).** With no sweep flag the directory must be empty except for at most one **peer** hello -- a `-hello.json` whose recovered id is non-empty and not this party's own (a bare `-hello.json` is an unexpected protocol file, not a peer hello; see site 5), the only file kind that can legitimately predate entry -- plus the snapshotted foreign files. Any other protocol file (a second or self hello, a lock, a `joining` sentinel, an ack, or a message -- including a message-shaped `<id>-<digits>.json`, which matches the grammar and is therefore a protocol file, not foreign) is a terminal `UsageError` whose text now also points at `--sweep-exchange-files` (the full rejection list is [I0](#invariants)). This code-enforces retain mode's fresh-directory precondition (see [Fresh directory in retain mode](#preconditions-for-a-correct-exchange)). **(c) Opt-in sweep (`--sweep-exchange-files`).** Before any delete, a *pre-sweep retain-signal inspection* checks whether the directory belongs to a retain exchange: local retain mode, a retain-only message ack (a grammar-aware filename scan -- an `-ack.json` whose two trailing dash segments are both numeric, the NNN and byte count of a timestamped message, vs a `-hello-ack.json` rendezvous ack a lockless-delete exchange also writes), or a peer hello advertising `retain_files=true` (read through the I5a gate, bounded to a near-future deadline so a non-resolving body cannot stall; an unresolved or unparseable body is retain-uncertain). On any signal the bare flag refuses (exit 64) and `--force-retain-sweep` is required; otherwise the sweep deletes every protocol-grammar file (this party's and the peer's) via `client.delete` (which rejects -- NOT `safeDelete`, so a transport that cannot delete surfaces exit 69) and entry proceeds against a clean slate. Foreign files are never swept. Finally, the `ignored` set remains the extension point for the orphaned temp sweep (193792285): a crashed in-flight write is `safeDelete`d and added to `ignored` rather than rejected. That sweep matches only the protocol's own `temp-<uuidv4()>.tmp` shape (`isProtocolTempName`); a foreign `temp-*.tmp` whose stem is not a v4 UUID fails the grammar and is handled by the foreign snapshot in (a), tolerated like any other foreign name rather than deleted (198451188). The flag's protocol-file deletion and the temp sweep are kept separate; the temp logic is not forked. **(d) Leftover abort-marker recognize-and-sweep (delete mode, always).** Because every authenticated terminal failure leaves a `<writerId>-abort.json` that must persist for the peer to read, a directory reused for a later exchange would otherwise reject "directory not clean". This guard `safeDelete`s a leftover marker and adds it to `ignored` -- like the orphaned temp -- but only by *exact name*: this party's own id, plus any peer id evidenced by a peer hello present at entry (the only notion of peer identity before rendezvous commits `peerId`). A foreign `<other>-abort.json` with no matching hello is therefore not swept here; it stays an ordinary unexpected protocol file under the normal policy. The sweep runs in delete mode only: in retain mode the marker sits beside a durable audit transcript, so it is left for the strict-empty guard to refuse (exit 64) and the existing `--force-retain-sweep` gate to clear -- reusing that machinery rather than a parallel retain check so the two cannot drift. This is a scoped, documented loosening of the entry guard: a malformed or garbage token in one of the two expected names is swept/ignored rather than surfaced -- acceptable under the threat model.
2. **`responsibleFiles` + `cleanup()`**. The set of files this party created and must remove on `close()`/`cleanup()`. Pre-track before writing so a crash mid-write is still swept. Retain mode is the sole exception: exchange files are intentionally not swept. No interaction with the entry sweep (site 1): the files it deletes predate the session and were never tracked here, and `synchronize()` prunes `responsibleFiles` against the live listing before the guard runs.
3. **`poll()` scan**. Reads the peer's messages: `<peerId>-` prefix, `.json`, numeric terminal, and `size >= declaredSize`. In retain mode the scan additionally selects only the message whose `<NNN>` equals the in-memory `recvSeq` counter, so an already-consumed message still on disk (retain mode never removes it) is skipped rather than re-delivered; a `<peerId>-...-<byteCount>.json` whose NNN segment is non-numeric is a malformed protocol file -- a correctly configured retain peer cannot produce it once both sides have agreed on retain/timestamp at rendezvous -- and is a terminal `UsageError` raised *before* the `recvSeq` selection, reported regardless of the `unexpected_files` policy (and deliberately *not* a `BilateralModeMismatchError`, since by this point the bilateral agreement is settled). Beyond message selection, the scan now classifies *every* file in the listing against a baseline of files legal during the loop -- the **recognized-for-the-loop** set: both hellos, both acks, the lock, both parties' messages and message-acks, our own writes, and an in-flight `temp-<uuidv4()>.tmp` (only the protocol's own temp shape; a foreign `temp-*.tmp` with a non-UUID stem is not recognized and falls to the unexpected-file policy unless snapshotted at entry -- 198451188). A file that is not recognized for the loop is an **unexpected** file, handled per `unexpected_files`: `error` (terminal `UsageError`, the default on plain delete-mode transports), `warn` (logged once per distinct name, the default under `retain_files`/`lockless_rendezvous`), or `ignore` (the former unconditional skip). This is the post-entry revision of the I2 "non-numeric terminals are ignored, not errors" rule (see [I2](#invariants)); the policy is local and unilateral, not bilateral, so the two parties may run different values. A fully-synced message that fails to parse or validate is terminal and stops the poller (see [I5b](#invariants)); transient transport and ack-write failures reschedule (an `emit`-handler throw would too, but the production consumer cannot throw -- see [I8](#invariants)). 195255994 supplies this site's *entry foreign snapshot*: pre-existing foreign (grammar-failing) names are added to the recognized-for-the-loop set so they are tolerated rather than warned as newcomers. It does NOT suppress a grammar-matching `<peerId>-<digits>.json`: that name is a message by the discriminant, so it is selected and terminally rejected by the message scan (see [I8](#invariants)), never snapshotted -- the foreign snapshot and the message scan are disjoint by construction.
4. **`hasOutstandingMessage`** (in `send()`). The delete-mode sender waits here for the peer to consume (delete) its last message. Must use the grammar discriminant (it does: `parseMessageByteCount(name) !== undefined`) so the party's own hello and ack markers are not mistaken for an outstanding message. One residual the foreign snapshot does NOT close: a foreign `<thisId>-<digits>.json` (this party's own id as prefix, numeric terminal) matches `startsWith(thisId-) && endsWith(.json) && parseMessageByteCount !== undefined`, so the sender would treat it as an unconsumed message and spin. Low severity -- it requires a foreign file carrying the party's own id as prefix -- but it is a real finding, not benign: the snapshot tolerates such a name for the loop's recognized set, yet `hasOutstandingMessage` keys on the grammar, not the snapshot, so it is unaffected by the tolerate.
5. **Rendezvous role and collision logic** (`synchronize()`: lock path, lockless path, and role commit). Assigns `role`/`handshakeRole`/`peerId`, detects the two-hello collision, and recovers the peer id by fixed-suffix slicing (never by parsing). The recovered id must be **non-empty**: a bare `-hello.json` slices to `""`, which is never a usable peer identity, so it is not a peer hello -- and the sibling bare `-joining.json` is not a joiner arrival. Both the entry guard and the in-flight lock/lockless peer-hello scans (and the lock-path joining-sentinel handling) route their slicing through one helper (`peerIdFromControlName`, via `isPeerHelloName`/`isPeerJoiningName`) so the non-empty check holds at every site, not just at entry; the post-rendezvous prefix-collision guard additionally fails closed on an empty committed `peerId` as a last gate. This closes a gap where the entry guard rejected a bare `-hello.json` but the in-flight scans did not, so one injected after entry (mid-rendezvous, the directory being unauthenticated) could be adopted as `peerId=""` -- after which `poll()` keys its peer scan on `${peerId}-` = `-` and the lockless ack barrier waits on an ack no honest peer writes -- or a bare `-joining.json` could force the bounded `joinerRecoveryMs` starter stall. Unaffected by 195255994: the foreign snapshot, the pre-sweep inspection, and the sweep all run *before* role logic, and the snapshot is read-only. After a sweep the directory is clean, so this site sees no pre-existing peer hello and the party proceeds as if it arrived first.

## Two orthogonal mode axes

The transport has two independent binary modes. They are bilateral: both parties must choose the same value, and there is no negotiation (see [Bilateral configuration](#bilateral-configuration-detect-and-fail-never-negotiate)).

- **Rendezvous mode** -- how the two parties meet:
  - *Lock* (default, `lockless_rendezvous: false`): atomic exclusive-create plus a `-lock.json` tiebreaker; the joiner deletes the peer hello and writes its own. Requires deletion visibility and exclusive-create.
  - *Lockless* (`lockless_rendezvous: true`): an ack-handshake barrier that uses neither exclusive-create nor delete; both hellos and both acks coexist. For sync-mediated transports where both sides "win" a local create or deletions do not propagate.
- **Message-loop mode** -- how send/consume is signaled:
  - *Delete-as-signal* (default): the receiver deletes the sender's message file; the sender waits for that deletion before sending the next.
  - *Retain* (`retain_files: true`): no exchange file is deleted; the receiver writes a zero-length `ack` marker the sender waits for instead. For transports that do not propagate deletions, and for audit transcripts.

Not all combinations are valid. Retain mode requires lockless rendezvous: lock rendezvous is delete-based (the joiner deletes the peer hello as a role-assignment signal) and cannot produce the whole-directory no-delete transcript retain mode guarantees. The three valid combinations are lock + delete-as-signal (default), lockless + delete-as-signal, and lockless + retain. Lock + retain is rejected by schema validation.

## Split inbound/outbound directories

By default one directory is both read and written. A *split* configuration names two: this party reads the peer's files from the **inbound** directory and writes its own to the **outbound** directory, with no single shared path. It is for deployments where inbound and outbound folders are genuinely separate -- a managed share, or an SFTP server with distinct drop and pickup directories -- bridged so one party's outbound is the other's inbound. Configured as `inbound_path`/`outbound_path` (filedrop top-level, or under `server` for sftp), the pair mutually exclusive with the single `path`/`server.path`; see [EXCHANGE_REFERENCE.md](../EXCHANGE_REFERENCE.md#connectioninbound_path--connectionoutbound_path).

**Routing rule.** Every *peer-file read* -- the hello, message, ack, and abort-marker listings and reads -- comes from the inbound directory; every *self-write* -- this party's hello, messages, ack markers, abort marker, in-flight `temp-*.tmp` files, and the lockless self-bookkeeping scan -- goes to the outbound directory. A message's temp write and its atomic rename both occur within the outbound directory, so **no rename ever crosses the two directories** (cross-filesystem rename is not atomic). In shared mode inbound and outbound are the same directory, so this reduces to the single-directory behavior exactly. `this.path` is the inbound directory throughout the code; the outbound directory is resolved via `outboundPath` (which falls back to `this.path` in shared mode).

**Split mode requires retain mode** -- a deliberate constraint, not a mechanical necessity in every topology. On a shared store a delete-mode sender could poll its outbound (which is the peer's inbound) and observe the peer's delete; but that only holds when deletions propagate back, which fails across the one-way or delete-suppressing bridge that motivates separate folders. Retain mode signals with ack/abort markers that are always carried forward, and keeping the files directly serves the separate-folders goal of an auditable record of what flowed each way. Lock rendezvous is excluded for the same reason it is excluded from retain mode generally: its lock/tiebreaker file must exist in one shared location for the delete-based handshake. Because retain mode already implies lockless rendezvous and timestamped filenames, **only the lockless+retain code paths ever observe split directories** -- the lock and delete-mode paths are unreachable with an outbound directory configured (rejected at config validation), so they stay on `this.path` unchanged.

**Both directories must be fresh.** Retain mode's [fresh-directory precondition](#preconditions-for-a-correct-exchange) is enforced on the inbound *and* the outbound directory at `synchronize()` entry: peer files never land in outbound (the peer writes to its own outbound, which is this party's inbound), so any protocol-grammar file there is this party's own leftover from a crashed prior session and fails the clean-start guard exactly as on the inbound side. An orphaned `temp-*.tmp` is swept from both directories; a foreign file in either is snapshotted and tolerated. The two directories must differ, checked by one shared textual-normalization rule (`pathsResolveToSameDir`) at both config validation and `open()`, so the schema and the live connection give the same verdict (the `open()` check additionally guards a caller that constructs a connection directly, bypassing the schema). On both channels it catches redundant separators, `.` segments, trailing slashes, and -- filedrop -- backslashes, so e.g. `/x` vs `/x/`, `/x//y` vs `/x/y`, and `/x/./y` vs `/x/y` are all caught. The normalization is textual only: `..` segments (which may cross a symlink), case (Windows is case-insensitive), and a relative-vs-absolute SFTP path under the unknown login home cannot be resolved client-side and remain the operator's responsibility, not a guaranteed reject.

**Invitation-borne role swap.** Split directories are a per-party choice the deployment bridges (one party's outbound is the other's inbound), so an invitation cannot carry a single shared path -- but it can carry the *role mapping*. The invitation connection-endpoint wire format therefore carries the split pair: the sftp and filedrop endpoint sub-schemas (`packages/core/src/config/invitation.ts`) gained `inbound_path`/`outbound_path` beside the single `path`, under the same directory-mode rules the connection config uses (both halves together, mutually exclusive with `path`, the two halves distinct, and filedrop must name a directory in one form). They remain credential-free locators. Distinctness is enforced on the endpoint itself -- it survives the mirror swap (equal inviter halves yield equal acceptor halves), unlike the absolute-path rule, which is a per-party property the acceptor remaps and so stays deferred to the connection config's own validation. The token records the **inviter's own** pair verbatim; the cross-party swap happens at exactly one consumer, `connectionFromEndpoint` in apps/cli, which sets the acceptor's `inbound_path` from the inviter's `outbound_path` and its `outbound_path` from the inviter's `inbound_path`, so the two parties start as mirror images. Centralizing the swap at that single seeding site keeps the token a faithful record of the inviter's config rather than a pre-swapped acceptor view, and forecloses a double-swap -- the offline invite path calls the same function with no endpoint, so it never reaches a seeded branch. The seeded block also carries the retain-mode trio a split exchange requires (so it is runnable once credentials are filled in); the inviter's literal paths are a starting point the operator reconciles, since the swap fixes the *roles*, not the literal names -- in an asymmetric topology the paths, host, and even channel differ between parties. An SFTP login-home (unset) path has no concrete value to mirror, so such an endpoint conveys roles with no suggested path rather than erroring. Two accept paths consume that swap, both through this one function so the single-swap-site invariant holds across them: the **offline** accept seeds its whole connection block from `connectionFromEndpoint` directly, and the **online** accept (`applyEndpointSplitDirectories` in apps/cli) calls the same function only to lift out the swapped pair, then grafts that pair onto the connection it built from the acceptor's own URL. The online merge keeps the channel, host, port, and credentials from the URL (the endpoint is credential-free, and a bridged acceptor may reach the rendezvous over a different host or even channel), takes the split `inbound_path`/`outbound_path` from the endpoint -- fully replacing the URL's single path, which cannot express a role mapping and whose inbound leg, if it won, would break the mirror -- and layers the retain-mode trio over any URL-derived options. An explicit `--outbound-path` (which instead routes through `applyConnectionOverrides`) is the override and wins, so the merge is skipped whenever it is set. The grafted split connection is re-validated through the connection schema before any network activity, so a degenerate endpoint (e.g. relative filedrop paths the endpoint schema defers) fails with the schema's own message rather than at connect time. The accept-time reconcile of a pre-existing config diffs against this merged connection, not a URL-only target, so a location disagreement still aborts before any acceptance is sent.

**No token-version bump.** Adding `inbound_path`/`outbound_path` to the `z.strictObject` endpoint sub-schemas is in principle an incompatible wire-format change -- an older decoder rejects the unknown keys rather than ignoring them, unlike a non-strict top-level addition. The invitation-token `version` was nonetheless left at `"1"`: the change landed while psilink is pre-release with no decoder deployed in the field, so no reader can observe the incompatibility and a bump would only churn the version. This is a one-time pre-release allowance, recorded in the `version` policy comment in `invitation.ts`; a strict-endpoint addition made *after* a release ships must bump the version (or otherwise stage compat). What is newly disclosed on the wire is the inviter's own inbound/outbound directory paths -- public locators carrying no credential or server-identity material, exactly as the single `path` already did.

## Disclosed-columns subset on the token

**The `disclosedPayloadColumns` top-level field.** The invitation token carries an optional top-level `disclosedPayloadColumns: string[]` (camelCase on the wire, beside `connectionEndpoint`): the inviter's disclosed-columns subset -- exactly the column names `disclosedColumnNames(inviterMetadata)` / `isDisclosedToPartner` selects and `preparePayload` transmits for matched records. It is the set the acceptor will RECEIVE, in the **inviter's** column namespace. Only that consent-relevant subset is carried; the inviter's linkage/identifier/ignored columns are not transmitted post-match, so they are not in the subset and do not leave the inviter's machine. Each name is bounded to `MAX_NAME_LENGTH` and the count to `MAX_PAYLOAD_ENTRIES` (the same bounds a `payload.send`/`receive` list carries), atop the whole-token structural bound `parseBoundedJson` already applies at decode.

This closes a structural decoupling: the acceptor's consent display and its runtime lock-in derive from the **wire's own transmission predicate** rather than from a separately-authored `terms.payload.send` dictionary each mint path had to remember to write. The displayed/consented set therefore cannot drift from the bytes that flow -- both are `isDisclosedToPartner` over the same metadata. (`disclosedPayloadColumns` is distinct from the acceptor's own `payload.send`, which is the inviter's `receive` mirrored into the acceptor's namespace; the carried subset is the inviter's *send* set, consumed by the acceptor as a receive-side declaration.) The field is omitted -- and the acceptor reconciles lazily -- when the mint path does not know its metadata (a config-as-source invite whose config carries no explicit `metadata` block) or when nothing is disclosed (the acceptor receives no payload regardless, so a declared empty set could never differ from what it gets).

**No token-version bump (top-level, unlike the split pair above).** `disclosedPayloadColumns` is an optional field at the *top* level, which the non-strict `z.object` token schema simply ignores in an older decoder -- the backward-compatible case the `version` policy comment in `invitation.ts` describes, the same path `connectionEndpoint` took. So it does not bump the version on the documented rule (not the one-time strict-endpoint allowance the split pair needed). What is newly disclosed on the wire is the names of the columns the inviter would already transmit post-match -- already shown for consent and written into the [exchange record](EXCHANGE_RECORD.md) -- not any new category of data.

**Runtime lock-in (`reconcileReceivedPayload`).** A party that has LOCKED IN an expected received-column set verifies, after the payload exchange and before the result or audit record is built, that the partner transmitted exactly that set; a mismatch aborts the exchange as a `ConnectionError` of kind `protocol` (the party promised one disclosure on the invitation and delivered another). The locked-in set is the acceptor's carried `disclosedPayloadColumns` (consented at review time, threaded to `runExchange` as `prepared.expectedPayloadColumns`) or a recurring party's persisted `payload.receive`. The three exchange modes:

- **Invite/accept:** the inviter carries its disclosed subset on the invitation and leaves its own receive side blank (it does not know the acceptor's schema), filling lazily from the acceptor's first transmission. The acceptor locks in the carried subset and throws if the payload it later receives diverges.
- **Zero-setup:** neither party holds the other's metadata in advance; both reconcile lazily (no `expectedPayloadColumns`), and neither throws on the first transmission.
- **Recurring:** both parties' persisted configs carry the agreed payload, so each locks in its own `payload.receive` and throws on a mismatch -- the runtime (actual-bytes) counterpart to `validateCompatibility`'s terms-level send/receive mirror.

Two cases are deliberately not a mismatch: an **absent or empty** expected set (lazy -- "take whatever I am given"; an empty set collapses to the lazy path rather than reading as "expect exactly zero columns, abort on any," since an honest mint omits the field when nothing is disclosed and a no-output recurring helper's `payload.receive` is empty by schema), and an **empty received set** (the partner sent no payload -- no transmittable columns or no matched rows -- which can never exceed any consent). Laziness never widens disclosure: transmission stays governed by each sender's own `isDisclosedToPartner` metadata and the forward-only `assertPayloadSendDisclosed`, so a lazy receiver still only ever receives what the sender's consented metadata transmits. `reconcileReceivedPayload` is the receive-side (over-DELIVERY) runtime counterpart to `assertPayloadSendDisclosed`'s send-side (over-DECLARATION) mint-boundary guard.

## Phases and legal directory contents

Read each phase as "given this mode, the directory may legitimately contain exactly this set; anything else is an error or a not-yet-synced transient."

### Phase 0 -- clean start

Legal contents at entry: nothing this protocol owns, except at most one **peer** hello (the other party may have arrived first), plus any foreign files, which are snapshotted and tolerated. The preexisting-file guard (site 1) aborts in both modes, with one `UsageError`, on any other protocol file (full list in [I0](#invariants)). Tolerating the peer hello is what lets whichever party arrives second -- it always sees the first party's hello -- proceed instead of failing. With `--sweep-exchange-files`, the legal-contents rule is replaced by a clean slate: once the pre-sweep retain-signal inspection clears (or `--force-retain-sweep` is supplied), every protocol file is deleted and entry proceeds as if the directory held only the untouched foreign files.

### Phase 1 -- rendezvous, lock path

```
A put A-hello.json
B list -> sees A-hello.json            (B is the joiner)
B put B-joining.json ; B delete A-hello.json ; B rename B-joining.json -> B-hello.json
A list -> sees B-hello.json ; A writes the lock
both reconstruct <first>-<second>-lock.json and assign roles by filename order
```

Legal transient contents: one self hello, optionally one peer hello, optionally one lock file, optionally one joiner sentinel. The joiner closes the one observable inconsistency window -- peer hello gone, joiner hello not yet written -- with a `<id>-joining.json` sentinel: it publishes the sentinel (carrying its hello body), deletes the peer hello, then renames the sentinel to its own hello. The rename is atomic at the SFTP layer, so on a delete-visible transport the sentinel spans exactly that window; a sync-mediated transport may briefly expose both the sentinel and the renamed hello, which the peer tolerates because the lingering sentinel's id matches the peer hello it is completing against (a sentinel whose id matches *no* peer hello is third-party contamination and is rejected as a usage error, like a second peer hello or lock file). The peer's wait loop treats a `<peerId>-joining.json` as "joiner mid-arrival" and waits a bounded recovery window (`joinerRecoveryMs`, default 30s) for the rename to land; if the sentinel persists past the window the joiner failed mid-arrival -- after writing the sentinel but before publishing its hello, on either side of the delete -- and the peer aborts with a distinct transport error (CLI exit 69) instead of polling to the full `peerTimeoutMs`. The joiner releases the sentinel from `responsibleFiles` the moment it deletes the peer hello, so a failure after that point leaves the sentinel on disk as the peer's recovery signal (and, if the joiner process dies, for the next run's Phase 0 guard to reject); a failure before that point is swept by `cleanup()` with the peer hello still intact. The peer never re-creates its own deleted hello on recovery -- that would race the joiner's rename and trip the two-hello collision check ([I1](#invariants)). Role is derived from hello *filename order* (`thisFile.name < otherFile.name`), which is the source of truth the lock producer encodes -- never from a fresh id comparison, because for ids where one is a prefix of the other the two can diverge.

### Phase 1 -- rendezvous, lockless path

```
A put A-hello.json ; B put B-hello.json   (neither deletes the other)
each, on seeing the peer hello, writes a zero-length ack named after it
  (A writes A-B-hello-ack.json; B writes B-A-hello-ack.json)
each completes when the ack of its OWN hello exists in the listing
  (A waits for B-A-hello-ack.json; B waits for A-B-hello-ack.json)
```

The ack is a zero-length marker matched by name existence -- no body is read, and there is no read gate (a zero-byte file has no partial-sync window: its name appearing is completion). Legal contents: both hellos and both acks coexist and persist into the message loop (they are not deleted at rendezvous). This is why sites 3 and 4 must rely on the grammar discriminant: the party's own `<id>-hello.json` and ack share its message prefix and would be false positives under a bare `<id>-*` glob.

### Phase 2 -- message loop, delete mode

The sender writes `temp-<uuid>.tmp`, renames it to the final numeric-terminal name (so a watcher never sees a partial file under its final name), and then blocks in `hasOutstandingMessage` until the peer deletes it. The receiver's `poll()` reads one peer message once its on-disk size reaches the declared byte count, emits `data`, and deletes it. Legal contents: at most one outstanding message per direction.

### Phase 2 -- message loop, retain mode

The write path is the same as delete mode, but the receiver never deletes the message file. Instead it writes a zero-length `ack` marker named after the consumed message (`<myId>-<consumedMessageName>-ack.json`) before emitting `data`. This is the same construct as the lockless rendezvous ack, applied to a message instead of a hello: only the receiver writes one (delete mode signals the same thing by deleting the file).

This ack is a **durable-received** acknowledgment, not an application-consumption one. By the time it is written the message is a fully-synced file that retain mode never deletes, so it is already durable. That is what licenses writing the ack before the `data` hand-off, and it keeps the sender's go-ahead independent of whether the local hand-off succeeds. The order must not be flipped to emit-before-ack: an ack-write failure after a successful `emit` would re-deliver an already-consumed message. Conversely, because the ack precedes `emit` while `recvSeq` advances only after a successful `emit`, the ack write is idempotent per `<NNN>` -- a message reprocessed after a transient receive failure is not acked twice. The marker name is also deterministic per message (a pure function of the consumed message's fixed name), so even if the per-`<NNN>` write guard were bypassed the re-write would reuse the identical name and create no duplicate file; the guard only saves a redundant `put`+`rename`.

The sender gates its next send on the existence of the ack of its just-sent message, located by constructing the expected name `<peerId>-<lastSentStem>-ack.json` from the stem it already holds in `lastSentFile` -- not by parsing a marker on disk. The ack is zero-length, so the former byte-count size-gate degenerates to existence. No exchange file is deleted as a protocol step.

Because no message is ever deleted, every consumed message stays on disk on every transport -- not only those that fail to propagate deletions. The receiver therefore tracks the next unprocessed message with an in-memory per-session counter, `recvSeq`: it processes only the peer message whose `<NNN>` equals `recvSeq`, then increments it. That counter stays aligned with the peer's monotonic `<NNN>` sequence only because of the clean-directory precondition ([I0](#invariants)), on which the [I8](#invariants) counter shadow rests.

### Phase 3 -- cleanup and close

`close()` calls `cleanup()`, which removes every file in `responsibleFiles`. In retain mode `cleanup()` removes only an in-flight `temp-*.tmp`; the `*.json` transcript is left in place by design. `close()` and `cleanup()` must not diverge on this.

Before `cleanup()`, in delete mode the side that sent the terminal frame **drains** it: it polls the directory until the peer consumes (deletes) `lastSentFile`, so a clean close never sweeps a frame the peer has not yet read. The drain is bounded by a short fixed budget `TERMINAL_FRAME_DRAIN_TIMEOUT_MS` (60s), applied as `min(TERMINAL_FRAME_DRAIN_TIMEOUT_MS, peer_timeout_ms)` measured from when the drain begins (after the abort-marker decision gate, not the first instruction of `close()`) -- deliberately **not** the full `peer_timeout_ms` (default one hour). At teardown the exchange result is already computed and persisted, and on timeout `cleanup()` deletes the frame as a fallback, so the drain protects nothing durable (durability is decoupled from deletion): a longer bound would only make a clean close against a crashed or departed peer block for up to the hour before exiting, with no gain, while a peer that legitimately needs the frame reads it from `cleanup()`'s fallback delete on its own next listing. The constant is sized to a sync tool's flush latency (the peer's poller listing, consuming, and the deletion propagating back), on the order of the per-operation liveness bounds, and is internal-only (not a config knob) like `joinerRecoveryMs`; it would only need revisiting if a transport's round-trip propagation routinely exceeded it, re-introducing the premature-sweep race the drain exists to avoid. Each drain `list()` is additionally raced against the time remaining to the deadline (not its own fresh `peer_timeout_ms` `boundTransport` budget, which may now be far larger than the drain window), so a list issued late in the drain cannot run past the deadline. Retain mode skips the drain entirely: it never deletes a `*.json`, so the final send persists as transcript and the peer reads it whenever it next lists. The `min()` cap means a `peer_timeout_ms` configured below the fixed budget still bounds the drain at the smaller value, never longer than the operator asked.

## Invariants

These are the testable properties the state machine must preserve. New work should be checkable against this list.

- **I0 -- clean entry (classification with an opt-in override).** At `synchronize()` start (enforcement site 1), with no sweep flag, the directory must be empty except for at most one *peer* hello (a `-hello.json` whose recovered id is non-empty and not this party's own; a bare `-hello.json` slices to an empty id and is an unexpected protocol file, not a peer hello -- see site 5) and any foreign files, which are snapshotted (in-memory) and tolerated in both modes. Any other protocol file -- a second or self hello, a lock file, an ack marker, a `joining` sentinel, or a message (including a message-shaped `<id>-<digits>.json`, which matches the grammar and is therefore a protocol file, never foreign) -- is a terminal `UsageError`. With `--sweep-exchange-files`, the directory is first inspected for a retain signal (local retain mode, a peer hello advertising `retain_files=true`, or a retain-only message ack -- an `-ack.json` whose two trailing dash segments, the NNN and byte count, are both numeric); if none, every protocol-grammar file is deleted and entry proceeds against the resulting clean slate. Foreign files are never deleted by the sweep. When a retain signal is present the bare flag refuses; sweeping then requires `--force-retain-sweep`. This is the precondition the counter arguments in I8 rest on (a swept directory holds no message at entry, and a snapshotted foreign file is never counted -- see [I8](#invariants)); it also code-enforces retain mode's fresh-directory requirement (see [Fresh directory in retain mode](#preconditions-for-a-correct-exchange)), now advisory-unless-the-retain-guard-is-passed. The `ignored` set remains the entry point for the orphaned-temp sweep (a crashed in-flight write is `safeDelete`d and added to `ignored` rather than rejected), which targets only the protocol's own `temp-<uuidv4()>.tmp` shape (`isProtocolTempName`); a foreign `temp-*.tmp` with a non-UUID stem fails the grammar and is a grammar-failing name handled by the foreign snapshot, not deleted (198451188). The foreign snapshot is a sibling tolerance mechanism for grammar-failing names.
- **I1 -- one peer at rendezvous.** More than one peer hello is a terminal usage error ("other sessions using this path?"), in both rendezvous paths. **Carve-out:** `--sweep-exchange-files` overrides this -- and a wider set than just "a second hello". Under lockless rendezvous a live peer the operator wrongly believed dead may have written its hello AND its `-hello-ack.json` AND a first message/ack; the sweep deletes every peer-prefixed protocol file, not only a second hello. The flag is the operator asserting the directory is theirs to clear and no concurrent session is live -- a responsibility the protocol cannot take on, since the directory has no entry-time liveness signal (a stale hello is byte-identical to a fresh one; `modifyTime` is unreliable). The residual is real: a delete-mode sweep of a peer the operator misjudged as dead silently corrupts that peer's exchange. The pre-sweep retain-signal inspection narrows only the cross-mode case (it refuses to destroy a retain transcript without `--force-retain-sweep`); it does not make the sweep safe against a live delete-mode peer.
- **I2 -- grammar routing.** A file is routed as a message iff its terminal `.json` segment is all digits. Control files (`hello`, `ack`, `joining`, `lock`) are never routed. An ack marker is the case that makes this load-bearing: a message ack carries the acked message's `<NNN>` and `<byteCount>` mid-name (both all-digit), but its terminal segment is `ack`, so right-anchored parsing routes it as a control file. *Routing* (whether a file is delivered as a message) is unchanged by `unexpected_files`; what the post-entry policy changes is the *non-routing disposition* of a non-message file during the message loop (enforcement site 3): the former unconditional "non-numeric terminals are ignored, not errors" now holds only for files recognized for the loop, while any other non-message file is subject to `unexpected_files` (`error`/`warn`/`ignore`). The routing discriminant itself is untouched by 195255994 too: the foreign snapshot is a non-routing *tolerate*, not a routing change; I0's own entry relaxation is 195255994's (see [I0](#invariants)).
- **I3 -- own files are not self-messages (a corollary of I2, kept as its own entry).** Not an independent property: I2 already entails it, since a party's own control files have non-numeric terminals. It is called out because it is the specific drift lockless makes load-bearing -- own hello and ack persist into Phase 2, so sites 3 and 4 must exclude them via the grammar discriminant (`parseMessageByteCount(...) !== undefined`), never a bare `<id>-*` glob or a per-type suffix list.
- **I4a -- delete-mode crash safety.** In delete mode, every committed file a party creates is tracked in `responsibleFiles` so a mid-write fault still leaves it swept by `cleanup()` at the next `close()`. *When* the `add` happens depends on the write discipline, and the two are not interchangeable: a **direct-write** to the final name (`hello`, `lock` via `createExclusive`) is pre-tracked *before* the call, because the final name can be left on disk by a call that then throws (e.g. `createExclusive` creates the file on the server but the handle-close fails); a **temp-then-rename** write (`message` in `send()`, `ack` in `writeAck()`) is tracked *immediately after* the rename, because the final name only appears at that atomic step and the `add` follows it with no throwable statement between, so there is no reachable orphan window short of a rename that throws-but-succeeded -- a pathological mode the message write has always accepted identically. (`responsibleFiles` is in-memory and `cleanup()` only runs in-process at `close()`, so a hard process kill loses the set and sweeps nothing regardless of add timing; pre-tracking guards the *caught-exception* path, not a kill.) All four `add`s are guarded by `!retainFiles`, because retain mode never sweeps (I4b) so tracking would serve no purpose.
- **I4b -- retain mode never sweeps (except a retain-guarded entry sweep).** In retain mode `cleanup()` is a global no-op: the directory is the durable transcript, so no `*.json` is removed by `cleanup()` and `responsibleFiles` membership is irrelevant. `close()` and `cleanup()` must not diverge on this. The literal statement (about `cleanup()`) survives, but its spirit ("retain never deletes a `*.json`") does not: a `--sweep-exchange-files --force-retain-sweep` entry sweep deletes `*.json` transcript files at *entry*, wiping a prior transcript before a fresh exchange. That is an entry-time operation under explicit operator escalation, not a `cleanup()` deletion. Corollary the inspection relies on: because retain never sweeps its hellos (this invariant), a retain directory always advertises itself via a `retain_files=true` hello, which is what makes the pre-sweep retain-signal check reliable for a transcript already on disk at inspection time. In both modes the in-flight `temp-*.tmp` is never tracked in `responsibleFiles`; it is cleaned inline (best-effort) in `send()`/`writeAck()`, independent of `cleanup()`.
- **I5a -- partial-sync read safety.** A file whose body is read for content must not be read before the sync tool has finished writing it. The message is the only payload-bearing file that is byte-count gated: it self-describes its full size in the filename and the receiver waits until the on-disk size reaches it. The hello has no byte-count segment, so it uses a read gate (`readControlFileWithGate` in `fileSyncConnection.ts`) that retries on a short or unparseable read until the peer timeout, then validates `HelloEnvelope`. The ack marker needs neither: it is zero-length, so its name appearing *is* completion -- there is no body to gate or read. (This is what licensed dropping the former empty `{}` ack body and its read gate.) The hello is therefore the sole payload-bearing control file. **Control-file payload fields are camelCase on disk** (`locklessRendezvous`, `retainFiles`) -- a control file is a protocol message, not user-facing schema, so the snake_case-in-YAML config convention does not apply and there is no `camelizeKeys` conversion on the serialize or parse path. A later envelope-field addition must stay camelCase to match. **The gate is now also reached at *entry*** (195255994): the pre-sweep retain-signal inspection reads the peer hello body through `readControlFileWithGate` to check `retain_files`, validating against the hello schema (not a strip-unknowns base, which would read an absent flag as a false "no signal"). That read MUST be bounded -- a near-future `timeToLive` of a small multiple of the polling frequency, never the full `peer_timeout_ms` -- so a non-resolving body cannot stall the sweep; an unresolved read is treated as retain-uncertain and refuses the bare flag. I5b still holds at this site: a fully-synced hello body that fails the schema is a terminal `UsageError`, same classification as 193901017. (The message-ack half of the inspection is filename-only and reads no body.)
- **I5b -- a completed body that fails validation is terminal (an instance of I6).** Once a body has reached its declared size the partial-sync excuse is exhausted: a parse or schema failure is genuine corruption, not a retryable short read, and is a terminal `UsageError`. For the hello this is the `HelloEnvelope` schema check after the gate. For messages in `poll()` it is a `JSON.parse` or `Message` validation failure, a body-`seq`/filename-`NNN` mismatch, or a duplicate `NNN`; each stops the poller. Transient transport failures (and the unreachable emit-handler throw, see I8) are *not* `UsageError`s and reschedule instead.
- **I6 -- terminal failure, no caller retry.** `synchronize()` and `send()` always present a terminal failure to the caller; the caller does not retry. Any bounded retry is internal. The I5b classification and the I8 counter discipline are instances of this contract at specific read/write sites. The "fail, then re-run with `--sweep-exchange-files`" recovery is an *operator re-invocation of the CLI with different flags*, NOT an in-process caller retry: `synchronize()` stays terminal per invocation, so I6 is preserved -- "on a re-run" means a new process, not a new in-process retry path.
- **I7 -- role from filename order.** Lock-path role assignment derives from hello filename order, never from a fresh id comparison.
- **I8 -- in-memory counters are a shadow of on-disk NNN state, not an independent source of truth.** `seq` (next NNN to write), `recvSeq` (retain: next NNN to read), `lastAckedNNN` (retain: ack idempotency guard), and `lastSentFile` (the name of the last message this party sent -- read by the delete-mode drain and the retain-mode ack construction) are all in-memory projections of the shared directory's NNN sequence. Three rules keep the shadow aligned; breaking any one silently desynchronizes the two parties:
  - `seq` advances only AFTER a message's durable `rename` in `send()`. A failed write must not leave the counter past an unwritten NNN. If it did, the retain-mode ack gate on the next send would wait for the ack of a message that was never written to disk.
  - In retain mode, `recvSeq` advances only after a successful `emit` in `poll()`. The ack marker (and `lastAckedNNN`) is written before `emit`. So a transient (non-`UsageError`) failure during receive -- e.g. an ack-write transport hiccup -- reprocesses the never-deleted message on the next poll and acks it exactly once: never skipped, never double-acked. (The ack name is a pure function of the message's fixed name, so even a bypassed `lastAckedNNN` guard re-derives the identical name and writes no duplicate file; the guard only saves a redundant `put`+`rename`.) A `UsageError` reaching `poll()`'s catch is terminal and stops the poller instead (I5b). This reprocess path covers a failure *during* receive, not a throw from the `data` consumer. The sole production consumer (`deliver()` in `messageConnection.ts`) latches an overflow as a non-throwing failure and never throws synchronously, so an emit-handler throw is unreachable defense-in-depth.
  - All four counters reset together via `resetSessionState()` at every session-boundary path: the rendezvous outer catch, the joiner prefix-at-dash error, and `close()`. A partial reset that left any counter non-zero would corrupt the next session on the same instance.

  The shadow is only safe because of I0 (clean entry): both counters begin at 0 against an empty directory and therefore stay aligned with the peer's monotonic NNN sequence, which also starts at 0. The I0 relaxation (195255994) preserves this by rejection, not by exclusion from message selection. The grammar discriminant is digits-terminal, not prefix-gated, so a `<peerId>-<digits>.json` does pass `parseMessageByteCount` and is selected at `poll()` -- the foreign snapshot does not make it inert there (that is 194800733's concern). Alignment holds for two reasons: a swept directory holds no message at entry (counters start at 0 against a clean slate), and a message-shaped name is never snapshotted -- with no flag it is rejected at the entry guard, so no pre-existing message reaches the loop. Should such a name reach `poll()` by another path (appearing after entry, or surviving a sweep), it is selected and then terminally rejected as a non-conforming `Message` body (I5b) without advancing `recvSeq`: alignment holds because the file is *rejected, not counted*.

  Enforcement sites: `resetSessionState()` (session-boundary resets), the post-rename `this.seq = seq + 1` in `send()` (I8 write rule), and the post-emit `this.recvSeq++` guarded by the pre-emit ack write in `poll()` (I8 receive rule).

  **Production composition note.** The reschedule-on-transient half of the second rule is a property of `poll()` *in isolation*. In the CLI a `FileSyncConnection` is never consumed directly: `apps/cli/src/protocol.ts` always bridges it through `fromEventConnection` (`messageConnection.ts`), whose `error` listener routes every emitted `poll()` error into `QueuedMessageConnection.fail()`. `fail()` synchronously calls `hooks.close()` -> `conn.close()`, and `conn.close()`'s first statement -- before any `await` -- is `conn.stop()`, so `pollerActive` is cleared within that synchronous prefix, before `poll()`'s `finally` runs (`conn.close()` is itself `async`, but only its leading `stop()` must run synchronously for this to hold, and it does) -- so the reschedule that `finally` would otherwise perform never happens. Under composition, therefore, the *first* emitted `poll()` error is terminal: the parked `receive()` rejects exactly once (naming the underlying cause, which `asConnectionError` copies into the surfaced error's message and attaches as `cause`, rather than the bridge's generic peer-silence inactivity message), the connection stops, and the never-deleted retain message is not reprocessed. A hypothetical retry counter on `poll()` could never reach 2 in production for this reason. So the rule above (and the isolation unit tests that drive `poll()` without the bridge -- e.g. the transient-`list()`-recovery test in `packages/core/test/fileSyncConnection.test.ts`) describes the module standalone; the composed contract is pinned by the bridged test that lives alongside them.
- **I9 -- one outstanding message per direction (delete mode).** The delete-mode sender blocks in `hasOutstandingMessage` until the peer deletes its last message, so at most one message per direction is ever in flight; this is the channel's flow control and the property that bounds legal directory contents in Phase 2 delete mode. Retain mode imposes no such bound -- messages accumulate as the transcript and the receiver selects the next by `recvSeq` (I8).
- **Untouched by 195255994.** I2 (routing discriminant -- the foreign snapshot is a non-routing tolerate; only the post-entry classification was extended, by 194800733), I3 (the snapshot adds no `<id>-`-numeric file), I4a (swept files predate the session and never entered `responsibleFiles`, which `synchronize()` prunes against the live listing before the guard), I7 (sweep and inspection run before role logic), and I9 (entry precedes Phase 2). I5a is no longer in this untouched list -- the pre-sweep inspection now reads the hello body through its gate (see I5a above).

## Preconditions for a correct exchange

These operator-level requirements are part of the model: violating any of them silently corrupts or stalls the exchange, and several are not otherwise enforced in code. They are collected here so they live in one place rather than scattered across feature docs.

- **Dedicated directory.** One active exchange, exactly two parties, per directory, for *protocol* use. Foreign (non-protocol) files sharing the directory are tolerated: snapshotted at entry, warned or ignored mid-loop per `unexpected_files`, and never deleted by the protocol -- so the exclusivity requirement is on the protocol-grammar namespace, not the directory as a whole. See [EXCHANGE_REFERENCE.md "Directory exclusivity"](../EXCHANGE_REFERENCE.md#directory-exclusivity).
- **Matching builds.** There is no `protocol_version` field; both parties run compatible builds and the payload schema is a hard contract. A payload-schema change (e.g. adding hello flags) is a flag-day that both sides must deploy together.
- **Identical bilateral flags.** `lockless_rendezvous` and `retain_files` must match on both sides.
- **Fresh directory in retain mode.** Retain mode never deletes, so a reused directory leaves a prior exchange's files: a stale message would be mis-consumed against the `recvSeq` counter and a stale ack marker could prematurely release the sender's gate, corrupting or stalling the exchange with no error. The entry guard (site 1) rejects a non-clean start in both modes, which is what protects retain specifically -- so this is advisory-unless-the-retain-guard-is-passed: `--sweep-exchange-files --force-retain-sweep` lets a non-clean retain start proceed by deliberately wiping the prior transcript first. Without that explicit escalation the guard still rejects, so the precondition holds for any normal invocation. In a [split configuration](#split-inboundoutbound-directories) the check runs on **both** the inbound and the outbound directory.
- **Distinct, non-prefix peer ids.** When `peer_id` is configured, the two ids must be distinct, and neither may be the other extended by `-` (e.g. `site` / `site-2`), which would break message prefix-routing. UUIDs (the default) satisfy this automatically.

## Bilateral configuration: detect and fail, never negotiate

`lockless_rendezvous` and `retain_files` are bilateral agreements with no negotiation step. The deliberate stance is **mismatch detection and clear failure, not capability negotiation**: a party advertises its own flags in its hello (`HelloEnvelope`, the two required fields) and fails fast with a both-sides-named error when the peer's differ. It never adapts to the peer's mode. This boundary is intentional and load-bearing -- advertising flags is one step toward negotiation, and the design explicitly stops there. Do not let mismatch detection grow into capability negotiation without revisiting this decision.

The same bilateral-vs-local split governs the CLI surface: `lockless_rendezvous`, `retain_files`, and `timestamp_in_filename` get command-line flags (`--lockless-rendezvous`, `--retain-files`, `--timestamp-in-filename`) because an operator must coordinate them with the peer at invocation, whereas a purely-local tuning preference such as `unexpected_files` -- which the two parties may set differently -- is configured in YAML only. CLI flags are otherwise reserved for bilateral settings that must match across parties; local tuning lives in the connection config.

`--outbound-path` is the deliberate exception to that rule: a local setting (each party chooses its own inbound and outbound directories, which the deployment bridges) that is nonetheless a CLI flag. The justification is that it is a *locator*, not a tuning preference -- the companion to the inbound directory, which is already a command-line argument (the server URL or positional path) -- so pairing it with that argument on the command line is the natural surface, where a per-party tuning flag would not be. The CLI maps the URL/positional path (or, for `psilink exchange`, the loaded config's directory) to `inbound_path` and `--outbound-path` to `outbound_path`, for both `sftp` and `filedrop`, in the single `applyConnectionOverrides` chokepoint every bootstrap command routes its connection through (so the four commands share one mapping). It then validates the assembled connection through the same `ConnectionConfigSchema` a hand-written `psilink.yaml` is parsed with, so the split-mode preconditions -- the pair set together, the two directories distinct ([`pathsResolveToSameDir`](#split-inboundoutbound-directories)), the filedrop-absolute rule, and the retain-mode requirement -- reject identically whether a split arrived from flags or from YAML. Only the retain-mode requirement is additionally fast-failed at the CLI, with a `--retain-files`-named message, before the schema runs; the rest surface with the schema's own messages. The invitation-borne role swap for a split exchange (one party's outbound seeding the other's inbound) is a separate mechanism from this local CLI surface: it rides the invitation connection-endpoint wire format and is applied once at accept-seed time through `connectionFromEndpoint`, not through `applyConnectionOverrides`. Both accept modes consume it -- offline seeds the whole block from it, online grafts only its swapped pair onto the URL-built connection -- and an explicit `--outbound-path` (which does route through `applyConnectionOverrides`) overrides it on the online path. See [Invitation-borne role swap](#split-inboundoutbound-directories) above. (The inviter-side complement -- emitting that endpoint from online `psilink invite` -- is `endpointFromConnection` in apps/cli, the producer inverse of `connectionFromEndpoint`: it derives the endpoint from the connection the invite is using and records the inviter's pair verbatim, with no pre-swap at emit, so the single accept-side swap runs end-to-end. The acceptor mirror-swap is in place regardless of which inviter produced the endpoint. WebRTC endpoint emission from the CLI is a separate follow-up, gated on the CLI gaining a WebRTC transport.)

The entry-sweep flags (`--sweep-exchange-files`, `--force-retain-sweep`) are a third category that does not fit that split: they are **CLI-only, unilateral, and non-persistable**. CLI-only not because they are bilateral but because "always sweep" must never be a persisted state -- they thread to the connection's runtime options on a path separate from config construction, never as `FileSyncOptions`/`psilink.yaml` fields. Unilateral: one party sweeps its own view of the directory, no peer must pass them, and they carry none of the bilateral-mismatch *stall* risk of `retain_files`/`lockless_rendezvous`. They do, however, interact destructively with the bilateral `retain_files` agreement: because the sweep runs before rendezvous, a *blind* sweep would delete the peer hello that 193901017 reads to detect a `retain_files` mismatch -- destroying a retain peer's transcript AND bypassing the fast-fail that would have caught the misconfiguration. The pre-sweep retain-signal inspection (site 1) is what closes this; it must not regress to a blind sweep.

**Detection is symmetric (two-sided), via durable hellos.** The flag check sits at *every* peer-hello read site, not just one, and a detecting party leaves its own advertised hello in the directory (writing it first if it had not yet done so) and does not delete the peer's hello. The two rendezvous protocols are not symmetric in *who reads whom* -- a `lockless_rendezvous` mismatch pairs a lockless ack-barrier against a delete-based lock rendezvous -- so a check at one site alone would fast-fail only the party that happens to read the peer hello, leaving the other to hit the peer timeout. Because the advertisement is durable on both sides, the peer (whatever protocol it runs) reaches a peer-hello read site, reads the flags, and fails the same way. This holds in both arrival orders and for both flags, including a `retain_files` mismatch where both parties run the lockless ack-barrier. Leaving an already-written hello so the peer can read it is **not** adaptation: the boundary the bilateral stance forbids is adaptation, not durability, so symmetric detection stays inside it.

**A mismatch leaves both hellos as the directory's terminal state.** The mismatch path does not sweep the hellos: a party that advertised a mode has its advertisement respected as the terminal state rather than silently swept, and the peer needs the file on disk to reach the same conclusion. In-memory session state is still reset (`resetSessionState()`) so the instance is not wedged; only the on-disk sweep is skipped. The cost is that a rerun against the same directory is rejected by the entry guard ([I0](#invariants), "directory not clean", a terminal usage error, not a silent stall); the operator clears the directory and reruns with corrected flags. This is consistent with retain mode's fresh-directory precondition. The mismatch is carried as a distinct typed error (`BilateralModeMismatchError`, a `UsageError` subclass) so the two cleanup sites -- the rendezvous outer catch and the joiner fast-path, which has no enclosing catch -- branch on it deterministically rather than by string-matching the message.

**The symmetric guarantee is best-effort, contingent on the advertising write landing.** Leaving a durable hello presumes the `put` that writes it succeeds. The joiner fast-path -- the one mismatch site that must write a NEW hello at detection time -- retries that write up to a bounded budget (`ADVERTISE_HELLO_RETRY_ATTEMPTS`, at the polling cadence) so a transient failure has more chances to land; the peer polls concurrently, so the advertisement need not arrive on the first try, only before the peer would otherwise time out, and the budget stays far under `peerTimeoutMs`. If the write still fails every attempt (a transport error sustained across that whole window), no write ordering can recover -- there is no durable file for the peer to read, so the peer degrades to the legacy peer-timeout on that one side. The detecting party still fails locally and correctly: it swallows the write failure and throws the typed `BilateralModeMismatchError` (CLI exit 64) rather than letting a transport rejection mask the actionable cause as a generic error (exit 69). This degradation is the floor, never the target -- it is no worse than the pre-advertisement behavior (a silent stall until the peer timeout), and it only occurs when shared storage is failing writes, in which case the exchange could not have proceeded regardless.

## Error taxonomy

Errors out of `synchronize()` and `send()` fall into two classes, and the distinction is part of the contract with the CLI:

- **Usage / configuration error** -> CLI exit `64` (`EX_USAGE`). Wrong directory state (preexisting files), multiple concurrent sessions, a stale lockless ack, a message-consumption timeout, a bilateral-mode mismatch (`BilateralModeMismatchError`), a malformed control payload (a fully-synced hello missing a required flag or carrying an out-of-type value), a `--sweep-exchange-files` refusal on a retain signal (local retain mode, a peer `retain_files=true` hello, a retain-only message ack, or a retain-uncertain hello body that did not resolve within the inspection budget -- the error names the file(s) and the signal and points at `--force-retain-sweep`), and `--force-retain-sweep` passed without `--sweep-exchange-files`.
- **Transport failure** -> CLI exit `69` (`EX_UNAVAILABLE`). The peer went silent past `peer_timeout_ms`, `list`/`get`/`put` rejected, the listing will not converge, or a `--sweep-exchange-files` delete rejected (the sweep uses `client.delete`, which rejects -- not `safeDelete`, which swallows -- so a transport that cannot delete surfaces here instead of falsely reporting a clean slate).

This distinction is carried by a typed `UsageError` (exported from `@psilink/core`, defined in [errors.ts](../../packages/core/src/errors.ts)): the usage class is anything `instanceof UsageError`, and everything else is a transport failure. The CLI inspects it at its catch sites -- the blocks in [exchange.ts](../../apps/cli/src/commands/exchange.ts) and [zeroSetup.ts](../../apps/cli/src/commands/zeroSetup.ts) map `err instanceof UsageError ? 64 : 69`. New throw sites in `synchronize()` or `send()` should throw `UsageError` for the usage bucket rather than a plain `Error`, classifying into one of these two buckets from the start.

## SFTP rename retry

The SFTP adapter retries a transient `rename` failure, because the transport publishes every atomic write by renaming a temp or sentinel file to its final name (the joiner's `<id>-joining.json` -> `<id>-hello.json` in Phase 1, and a message's `temp-<uuid>.tmp` -> its numeric-terminal name and an ack's temp -> `<writerId>-<originalName>-ack.json` in Phase 2). The retry is bounded the same as `put` (one initial attempt plus up to five re-issues, 100 ms apart) but, because `rename` is not idempotent, fires only on the generic `SSH_FX_FAILURE` (status 4) -- the status that reports the rename did not take effect, so the source still exists and a re-issue is safe. Every other status is terminal and surfaces at once; in particular `SSH_FX_NO_SUCH_FILE` is not retried, because a second attempt after a lost success reply would manufacture a spurious error from an operation that actually succeeded.

## Sender-side peer-silence attribution

The peer-silence timeout is asymmetric, by necessity rather than by oversight. When the peer dies mid-exchange -- the canonical case being its exchange directory going read-only so it can never write the next message or its ack -- the *receiver* (the failing party) names its own cause locally: `poll()` emits the underlying transport error, the `fromEventConnection` bridge makes it terminal on first emit, and `asConnectionError` copies that cause into the surfaced error (see the I8 [production-composition note](#invariants)). The *sender* (the waiting party) has no equivalent signal. It observes only that nothing arrived within the inactivity budget (`peer_timeout_ms`) and fails with the generic peer-silence `transport` error.

A cross-party signal -- the failing party writing a marker the waiting party reads -- cannot cover the headline case: an exchange directory that has become unwritable is precisely what stops the peer from writing *any* file, so for that mode (and a hard kill) the channel is absent in exactly the scenario it would serve. For the modes where the directory is still writable (a transport-op fault, a terminal parse error, a graceful crash) the marker now ships: a `<writerId>-abort.json` the failing party writes best-effort on teardown. It deliberately carries **no cause** -- only the fact "the peer aborted." The specific cause delivers no actionable value to the waiting party (the two orgs coordinate out-of-band and the partner already holds its own full-fidelity error), and a plaintext cause on the shared directory would only worsen the disclosure posture; the win is killing the hedge's "maybe it's slow, raise the timeout" branch and exonerating the local side. It is additive to the filename grammar: the non-numeric `abort` terminal token routes it as a control file (the `parseMessageByteCount` discriminant), never mis-consumed as a message (see [the filename grammar](#the-single-load-bearing-invariant-the-filename-grammar)). And it is unforgeable: the body carries a 32-byte per-direction token derived from the post-handshake session key both parties hold (`HKDF(sessionKey, "psilink-abort-token-v1:" + role)`, beside `deriveAeadKey`), so the untrusted directory admin or any non-handshake party -- holding no key -- cannot produce a token the reader accepts; a captured marker does not validate in any other session (the session key is fresh-ephemeral per handshake) or under the other party's name (the per-role binding). The waiting party's poll loop reads a present `<peerId>-abort.json` size-bounded, constant-time-compares its token against the locally derived peer token, and on a match fails fast with a definitive `PeerAbortError` (CLI exit 69) rather than waiting out the full timeout; a non-match, absent, unreadable, or oversized marker is ignored and falls back to the hedge, so its absence stays strictly uninformative.

The operator guidance is the baseline beneath the marker, not a claimed ceiling, and remains the floor whenever the marker is absent. The sender's timeout error carries a transport-supplied hint (`PEER_SILENCE_GUIDANCE` in [protocol.ts](../../apps/cli/src/protocol.ts), threaded through `fromEventConnection`'s `inactivityHint` option) that names the likely receiver-side causes -- the peer exited, or its directory became unwritable -- directs the operator to the peer's own logs where the real cause was recorded, and notes the slow-large-dataset case so the timeout is not misread as a death. It deliberately hedges rather than asserting a specific cause. The hint is file-sync-specific and supplied by the caller, so the WebRTC transport, which shares the same core inactivity message but passes no hint, is unaffected. The authenticated marker above *upgrades* this hedge to a definitive peer-abort error for the writable-directory modes only; it complements the guidance rather than replacing it, so the read-only-directory headline, a hard kill, and a lost best-effort write all still degrade to this floor.
