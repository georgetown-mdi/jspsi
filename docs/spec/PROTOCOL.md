# PSI-Link protocol

This document describes the PSI and PSI-C algorithms, how they are composed to produce a privacy-preserving record linkage, and the wire-level X25519 authenticated key exchange. It does not cover the exchange agreement format that parameterizes the protocol (see [EXCHANGE_REFERENCE.md](../EXCHANGE_REFERENCE.md)), the threat model and authentication design (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md)), or the network layer over which the protocol runs (see [COMMUNICATION.md](../COMMUNICATION.md)). Intended readers are security auditors, external implementors, and developers.

# Privacy preserving record linkage

The PPRL protocol utilizes a base PSI function to repeatedly reveal the size of the sets of shared statistical linkage keys. This reveals to the parties an association map between their shared members and nothing about elements they do not have in common.

## PSI base function

The PSI base function is a lightly modified version of OpenMined's [PSI](https://github.com/OpenMined/PSI). That package implements private set intersection layering over the encryption in Google's [Private Join and Compute](https://github.com/Google/private-join-and-compute) (itself using OpenSSL), written in C++ that is compiled into WebAssembly. The base function divides the two participants into "server" and a "client" roles. At a high level, the steps of the protocol are:

1. Both client and server generate their own private keys which live only in memory for the duration of the exchange. Keys are random scalars in the P-256 elliptic curve group and are generated using OpenSSL.
2. The client initializes the exchange by encrypting their own data with their own private key using a commutative encryption algorithm and then sends it to the server.
3. The server commutatively encrypts both their own data and the client's data with their own private key and then sends both datasets to the client.
4. The client can then remove their own key from their own data, leaving them with client and server datasets encrypted only by the server.
5. A straightforward string comparison allows the client to see which elements they have in common. They can then choose to share the association table back with the server.

The terminology of "server" and "client" derives from OpenMined's PSI implementation and is used here to be consistent with their documentation. However, "server" and "client" are disfavored throughout the rest of this project as there are many other instances of servers and clients, and OpenMined's PSI includes no networking coding. Most often we will want to execute the protocol so that both parties learn the outcome. When it is necessary to distinguish between the two roles, we will instead use *receiver* and *sender* respectively. Both parties exchange their record counts (raw row counts) at the start of every exchange, and the role assignment is a deterministic, work-minimizing function of those counts. When only one party receives output, that party's role is fixed as the receiver regardless of the counts -- it is the only party that learns the result. When both parties receive output the assignment is free, so it is chosen to minimize total work: the party with the smaller row count becomes the receiver, and an equal-count tie is broken in favour of the initiator. The count exchange and the derivation of that single rule -- why it is optimal for both linkage strategies -- are in [Role resolution and work minimization](#role-resolution-and-work-minimization) below.

## Linkage keys

Statistical linkage keys are data elements that combine several other inputs into a single value that uniquely represents an individual with an extremely high probability. In this application, the most common data types for linkage keys are social security number, first name, last name, and date of birth. An example linkage key is the last four digits of the social security number concatenated with last name and date-of-birth as a character string.

Linkage keys can be designed to produce links even in the place of data quality errors, for instance by exhaustively generating all transpositions of two digits in a Social Security Number (SSN) or comparing all single-character edits of strings of a fixed length. By repeatedly executing the PSI base function on such keys, two parties execute a fuzzy PPRL. 

In order to preserve the guarantee that no information is revealed about individuals not in the intersection, linkage keys must be designed to be precise enough (high positive-predictive value) that any match is definitive.

## Matching Algorithms

### PSI

The way in which links are decided depends on whether or not both parties will receive output. If so, then they can communicate directly with each other and optimize the procedure. For one-to-one mappings, linkage keys are applied in sequence forming a cascade of deterministic matches: keys are ordered from most to least precise, and at each round only records that match uniquely on that key are accepted as pairs and removed from the candidate set - the pool of records not yet definitively matched. Inputs without a match or without a unique linkage key carry forward to the next round.

If only one party receives the output, then a compromise will be necessary. A deterministic cascade as described above can be executed with only one party resolving the final set of association between each party's elements. In this case, the other party learns how many records match each linkage key and how many records match overall. They also learn which of their records match, but not the identifiers of the records to which they match. As an alternative, the deterministic cascade can be enacted entirely by a single party without the other receiving a single output. In this case, all of both datasets must be processed and sent to the receiver every round. This discloses to that party the existence of matches on less precise keys that the cascade would have filtered out.

In linkages that involve multiple links - either many-to-one or many-to-many - the multiplicity is resolved into single entity clusters by applying a transitive closure algorithm. Transitive closure may create scenarios where two members are linked through a third record without a rule linking them directly, so careful consideration of linkage keys and their consequences is required. Having an output that includes multiple links per input implies that some meaning is imparted to the data holder through entity resolution; as such, these exchanges require that the "many" parties receive the output. In order to communicate this effectively to users, rather than describe the multiplicity of the exchange they are asked if they want to *deduplicate* their data, as they effectively use their partner's data to group their own.

In a many-to-one exchange where both parties receive the output, the "many" party can filter its candidate set to remove linked elements after each round similar to the deterministic cascade used in a one-to-one linkage. If the "one" party is not allowed to receive the output, the "many" party must ensure the uniqueness constraint.

Crucially, unlike traditional PPRL, blocking when using PSI is neither necessary nor appropriate. The PSI base function's computational complexity is O(n log n) in the total number of elements rather than quadratic in their product, so there is no cross-product comparison to reduce. Blocking would also compromise the privacy guarantee by revealing to each party how many of the other's records fall into each partition.

The practical upper limit on the number of records for browser-based execution is determined by available memory rather than computation. Note that an encrypted element's ~35-byte serialized size (one compressed curve point on the wire) understates its resident footprint during the comparison step, which is materially larger; for the non-streaming single-pass strategy that resident cost -- not the wire size -- sets the binding dataset ceiling (see the single-pass dataset ceiling below). The only part of the algorithm that requires WebAssembly is the application of the commutative encryption algorithm, which can be streamed and parallelized over the data.

Ahead of that algorithmic ceiling, the web app applies a CSV intake cap of 100 MB (`MAX_CSV_FILE_BYTES`, `apps/web/src/components/csvIntake.ts`) - roughly one to two million identifier rows. This is a per-tab *intake* memory budget, distinct from the comparison-step figure above and not headroom added to it: the two are separate costs that coexist during an exchange. Intake's dominant cost is the parsed row array, which is retained by reference for the whole exchange (it backs `rawRows` through preparation and the output table), so at typical object-overhead inflation a 100 MB file resolves to a few hundred MB resident, plus a transient parse-time spike - a profile the modern-workstation execution target absorbs comfortably, but one that does not scale down to a constrained tab. The cap is therefore a deliberate, tunable choice against the intake budget, not a protocol or wire-format limit; it can be raised as that intake profile is measured. The file is parsed inline (on the main thread) and accumulated across PapaParse chunks, so a file that spans more than one `Papa.LocalChunkSize` chunk yields its complete row set rather than the silently truncated final chunk (which, on a record-linkage input, would be a wrong intersection with no error). PapaParse's `worker: true` is deliberately not used: it spawns its worker from its own source by reading the running script's URL, which a Vite production build breaks (the bundled worker mis-applies `header: true`, leaking the first data row into the header), so the off-main-thread parse it offered is traded for a correct one; an off-main-thread parse for very large files would have to go through a Vite-native worker instead. The resulting payload then rides the WebRTC data channel under PeerJS's own message chunking, which already fragments and reassembles payloads far larger than a single frame at any cap in this range - so the cap carries no transport or wire-format consequence.

#### Linkage strategies: cascade and single-pass

The cascade above can be realized on the wire in two ways, chosen by the mandatory-consistency [`linkage_terms.linkage_strategy`](../EXCHANGE_REFERENCE.md#linkage_termslinkage_strategy) term (both parties must agree, like `algorithm`; the schema default is `cascade`). Both produce the byte-identical association table; they differ only in how the per-key PSI exchanges are sequenced, and in what one party discloses to the other.

`cascade` runs one dependent PSI base-function exchange per linkage key, in sequence: each round's candidate set is the records not yet matched by an earlier key, so a round cannot begin until the previous round's matches are known. The number of network round trips therefore grows with the number of keys. This is the established default, and the strategy the disclosure description above assumes.

`single-pass` batches every key into a single PSI exchange and has the receiver reconstruct the cascade locally, so the number of round trips does not grow with the number of keys -- which is what makes a multi-key linkage practical over a high-latency channel (file-sync), where a chain of dependent rounds is prohibitively slow. The receiver is the [PSI base function](#psi-base-function) client; the sender is the server. Each party first forms the union of its distinct linkage-key values across all keys and encrypts each distinct value once. Because the reconstruction only ever compares values within a single key's round, a value that appears under more than one key maps to the same encrypted form with no ill effect, and values are not tagged by which key they came from (no per-key domain separation); this would only need revisiting if the reconstruction were changed to compare a single encrypted value across keys.

The exchange is exactly three frames, one per message, regardless of the key count. Message 2 packs everything the sender owes into a **single binary frame**: on a high-latency channel each frame costs a round trip (file-sync gates each send on the peer consuming the previous one), so combining the sender's reply into one frame keeps the whole exchange at three round trips rather than paying one per piece. (The transport carries one binary blob or one JSON object per frame, never the two mixed, so the record count is packed into the binary frame rather than sent as its own JSON frame.)

1. **receiver -> sender** -- the receiver's PSI request over its distinct-value union (encrypted once under the receiver's key).
2. **sender -> receiver** -- a single binary frame packing four parts in order (all integers little-endian):
   - (a) a `uint32` setup length, then the **setup message** -- the sender's distinct-value union, each value encrypted once under the sender's key;
   - (b) a `uint32` response length, then the **response** -- the receiver's request re-encrypted under the sender's key;
   - (c) a `uint32` carrying the sender's record count;
   - (d) the **distinct-value index table**, the remaining bytes: an `Int32` array in key-major order (one row per key) of length `keyCount * senderRecordCount`. Each entry, for a (key, record), is the index of the value that record holds for that key -- an index into the sender's distinct values -- or `-1` if the record has no value for that key. The indices are in the setup message's sorted order (see the remapping in the next paragraph).
3. **receiver -> sender** -- the resolved association table for the sender's side.

The sender remaps its distinct-value index table into the setup message's sorted order before sending, so the sorting permutation the setup carries internally never crosses the wire and the receiver's match result indexes the table directly. From the setup and response the receiver works out which of its values equal which of the sender's values. It then replays the cascade locally -- over the sender's index table and the equivalent table it builds from its own records, applying the same within-round uniqueness rule and key order the step-by-step rounds use -- to produce the same matched pairs, and returns the sender's half in message 3.

The index table is transmitted in full -- every record, not a pre-filtered set of within-key-unique values -- because whether a value is unique depends on which records earlier keys already matched: a value shared by two records over the whole dataset can become unique once an earlier, more precise key matches one of them, so the receiver needs the complete per-record structure to replay the key order. This is the key disclosure difference between the two strategies. Single-pass hands the receiver the sender's full per-key duplicate structure and complete (encrypted) value set, so the receiver's reconstruction observes matches on less precise keys that the lockstep cascade would have filtered out before they were ever exchanged -- the same disclosure as the single-party variant above ("all of both datasets must be processed and sent to the receiver"). Only the cascade-equivalent table is emitted as the result; the weaker-key matches are visible to the receiver's process but are never surfaced. Single-pass therefore trades this additional disclosure -- and the extra bandwidth of the full index table, and extra encryption work, since it cannot skip records an earlier round would have removed -- for the constant round-trip count. Because `linkage_strategy` is mandatory-consistency, that trade is one both parties agree to before any data is exchanged.

Single-pass cannot stream, and its single frame imposes a much tighter dataset ceiling than the streaming cascade. The cascade processes one key per round, so its working set is a single key's column at a time; single-pass holds every key's values in memory at once and ships message 2 as one frame -- the encrypted setup and response plus the index table's `keyCount * senderRecordCount * 4` bytes. The frame travels as raw bytes -- the AEAD and file-sync envelopes do not base64-expand it -- so the on-wire frame is essentially its raw size plus a small fixed envelope. That frame is no longer bounded by the static transport cap: the single-pass reply is bounded by a per-exchange cap **derived from the exchanged record counts and the agreed key count**, identical on both parties, up to a fixed maximum dataset size -- specified in [The single-pass dataset ceiling](#the-single-pass-dataset-ceiling-receiver-memory-and-masking-compute) below. The receiver enforces that derived cap at its transport read gate (replacing `MAX_FRAME_SIZE_BYTES` for that one read on file-sync; the WebRTC data channel keeps its fixed browser-tab envelope, see [CHANNEL_SECURITY.md](CHANNEL_SECURITY.md)), and the sender checks its built reply against the same value before sending.

The dominant term is the encrypted sets, not the index table. Each distinct key-value either party holds serializes to about 35 bytes (one encrypted curve point), versus 4 bytes per (key, record) cell in the index table -- so for near-unique keys the encrypted sets outweigh the index table by roughly nine to one, and the practical ceiling is set by distinct-value count, not raw rows. With `D` the distinct key-values pooled across all keys, the reply is about `35 * (D_sender + D_receiver) + 4 * keyCount * senderRecordCount` bytes, so for near-unique keys it grows with the distinct-value count, not raw rows; the derived byte cap below (`singlePassReplyByteCap`) upper-bounds it from `keyCount * recordCount`. The frame the single-pass ceiling admits is well below the tens-of-millions of rows the cascade sustains. Peak memory scales the same way, with the full index table and both encrypted sets resident at once, so single-pass's memory ceiling is also tighter than the cascade's per-round working set. Neither the wire size nor the derived frame cap is the limit that binds first, though: the receiver's peak memory -- it must hold both encrypted sets resident to match -- is what bounds a single-pass exchange in practice, with masking compute the wall-clock cost on top. The operative ceiling, and the cap it sets (board item 206154573), are below.

An exchange that exceeds the single-pass ceiling aborts cleanly on both parties with actionable guidance that does **not** recommend cascade (reduce the key or record count, or split the dataset). Because the ceiling is computed from the exchanged record counts and the agreed key count -- authenticated session state both parties hold after role resolution -- both reach the same verdict and abort in lockstep, before any single-pass frame moves, so neither side sends nor waits and neither hangs to the inactivity timeout. The pre-flight gate (`prepareForExchange`) is a coarse one-party shadow of this, checked before connecting from this party's own row count alone; the authoritative two-party check runs in `linkViaSinglePassPSI` once both counts are exchanged. Choose single-pass when the round-trip saving outweighs that ceiling and the additional disclosure; otherwise cascade.

The reference implementation is `linkViaSinglePassPSI` (`packages/core/src/link.ts`); the four PSI base-function operations it composes are on `PSIParticipant` (`packages/core/src/participant.ts`).

##### The single-pass dataset ceiling: receiver memory and masking compute

Single-pass exists to collapse a many-round-trip exchange over a minute-scale-latency channel (a 15-minute directory poll, times one dependent round per key, is hours of waiting) into one sweep, at the cost of more disclosure (the receiver sees the sender's full per-key duplicate structure and weaker-key matches the cascade filters out, above). It is therefore an operator opt-in, agreed up front in the mandatory-consistency `linkage_strategy` term -- never a default, never selected automatically. Board item 206245686 measured what bounds it, with the harness at `scripts/single-pass-bench.mjs` (its `sweep` mode runs a real `linkViaSinglePassPSI` over a relayed pipe with each side in its own process, so peak resident set is isolated per side; its `rates` mode times each masking operation directly), driving near-unique keys -- every cell a distinct value, so the pooled distinct count `D = keyCount * rows`, the worst case the ceiling must hold for. The per-value constants below were measured on one workstation-class core (aarch64, single-threaded); the row ceilings derived from them are linear projections of a fit whose measured range tops out near 16k rows (224k distinct values), not measured points, and the constants themselves move as the implementation's allocation churn is reduced (below). The implemented cap (206154573) therefore does not project receiver RSS at runtime from these constants; it sets a single fixed, conservative budget on the per-party (key, record) cell count derived from them (`MAX_SINGLE_PASS_CELLS`, below), which is non-configurable -- a configurable maximum reintroduces the memory DoS -- and revisited as a one-line change when the per-value peak moves.

**Receiver memory binds single-pass, but as a transient peak, not live data.** The receiver must hold both parties' full encrypted sets resident to run the match -- it decodes the reply, builds its own and the sender's distinct-value index tables, and replays the cascade -- so single-pass is `O(total distinct values)` resident and cannot stream. Two quantities differ by more than an order of magnitude here and must not be conflated. *Live* retained memory is small: about `~130 bytes per distinct value` (`D ~= keyCount * rows`) -- roughly `~60 B` of psilink JS plus `~74 B` of WebAssembly linear heap and wire -- the ~35-byte encrypted point resident in the linear heap together with its wire-buffer copy and per-element allocator overhead (about two copies' worth). At that rate even 1.4M distinct values (100k rows at the ~14-key default template) is only ~180 MB live, far inside a 16 GB machine; because the bound is on distinct values, the same memory reaches millions of rows at 1-2 keys. What climbs at `~2-4 KB per distinct value` is the process's *lifetime peak RSS* -- transient allocation churn, not retained state (the next paragraph explains where it comes from and why it is reducible). The practical single-pass ceiling is set by that transient peak, so it moves as the churn is reduced; the live-data wall is far higher.

**Why the peak is kilobytes per value while live data is ~130 bytes.** The `~2-4 KB/value` is the high-water mark of transient garbage churned during the per-element work, captured because neither V8 nor the WebAssembly allocator (`emmalloc`, grow-only under `-s ALLOW_MEMORY_GROWTH=1`) returns freed pages to the OS. Direct measurement (forked sender and receiver, GC forced at each phase boundary, `D` of 14k and 56k) locates that churn in the JavaScript layer, not the PSI backend: the WebAssembly linear heap and wire buffers grow only ~74 B/value, while the lifetime peak grows ~2-4 KB/value. The dominant transient term is psilink JS -- marshalling every element across the JS-to-WASM boundary in `createRequest`/`computeValueMatches` and building the per-key reconstruction maps in `linkViaSinglePassPSI` -- so it sits in V8's heap and answers to GC pressure (`--max-semi-space-size`, or an explicit `gc()` in the hot loops), not in native WASM memory outside `--max-old-space-size`. It is collectable, not retained: forcing GC at just the four phase boundaries cut the `D` = 14,000 receiver peak from 161 MB to 117 MB with no algorithmic change. Lowering this transient peak (board item 206377899, now a `packages/core` JS-churn and GC-pressure task) is what raises the ceiling. The live floor it cannot drop below is small and `O(total distinct values)` -- the receiver's match is a full sort-merge join over both sets -- but at ~130 B/value that floor is only hundreds of MB even at millions of values, well under the memory wall.

**Masking compute is a wall-clock expectation, not a cap.** Masking is one elliptic-curve scalar multiplication per distinct value, single-threaded, exactly linear in `D` (confirmed `D` from 14k to 224k): ~0.45 ms/value to first-encrypt a set, ~0.36 ms to re-encrypt, ~0.37 ms to match, so the two-party critical path is about `1.2 ms * keyCount * rows` (~28 minutes for 100k rows at 14 keys; hours at a million). For a batch or scheduled exchange this is tolerable, so it should not *refuse* a dataset -- it sets the operator's expectation and sizes the run window. With receiver memory no longer the binding wall at target scale (above), this masking compute is the practical limiter past ~100k rows. The cost is shared with the cascade (same primitive; the cascade does *less* curve work when match rates are high, since it drops matched records each round, and about the same at low overlap), so it is not a single-pass-versus-cascade discriminator. It is, separately, the target of the native multithreaded backend (board item 199653275) and of a possible curve/library replacement (board item 206378144).

**The implemented cap (206154573): a (key, record) cell-count budget plus a derived byte cap.** The single-pass ceiling is gated on the receiver's projected memory, which the exchanged record counts and the agreed key count bound -- never on a static frame size. (A "cell" is one (key, record) pair; a party's cell count `keyCount * recordCount` is the worst-case upper bound on its count of distinct (deduplicated) linkage-key values `D`, which is itself never computed or exchanged.) Two quantities, both in `packages/core/src/connection/frameSize.ts`, computed identically on both parties:

- **`MAX_SINGLE_PASS_CELLS` = 2,000,000** -- the per-party budget on the cell count `keyCount * recordCount`. It is **the real ceiling**. An exchange exceeds it (and aborts, below) when EITHER party's `keyCount * recordCount` exceeds the budget. It is a cell-count budget, not a bare row count, because a rows-only cap is off by ~14x between a 1-key and a 14-key linkage at the same memory; it admits ~143k rows at the ~14-key default template and ~2M rows at a single key. **Methodology and value.** It is set against the receiver's peak RSS, NOT against what the frame format allows: a party's cell count upper-bounds its `D`, so 2M cells bounds `D` to 2M distinct values. At the ~2-4 KB/value *transient* peak slope (above) that projects to roughly 6-8 GB peak RSS, which is what the current value holds headroom against on a 16 GB target. That projection is the transient peak, not live data -- 2M distinct values is only ~260 MB live -- so the cap is conservative against the true memory wall and is slated to rise once board item 206377899 lowers the transient peak and re-derives the budget from a fresh measurement. The true `D` is lower for sparse or repeating keys and is deliberately never exchanged (it would leak each key's duplication and sparsity structure -- see role resolution below), so gating on the cell-count upper bound is conservative by construction. It is a **fixed constant, not operator-configurable** -- a configurable maximum reintroduces the memory-exhaustion denial of service -- and is a one-line change as the backend's per-value memory is reduced (board items 199653275, 206377899).

- **`singlePassReplyByteCap(keyCount, senderRecordCount, receiverRecordCount)`** -- the accepted byte size of the reply frame, a defense-in-depth tightening of the read gate derived from the same authenticated counts so a hostile transport admin cannot pad a small exchange's frame up to the static ceiling. Operation order and rounding, fixed so two independent implementations produce the bit-identical integer: with `sCells = keyCount * senderRecordCount` and `rCells = keyCount * receiverRecordCount`, the cap is `(40 + 4) * sCells + 40 * rCells + 256` in exact integer arithmetic. The sender contributes one masked value (40 bytes -- a deliberate upper bound on the ~34-byte serialized encrypted point plus protobuf framing) and one index-table cell (4 bytes, one `Int32`) per (key, record); the receiver contributes one masked value per (key, record); the `256`-byte overhead covers the reply's three `uint32` length/count prefixes plus the AEAD and file-sync message envelopes the read gate measures around it. Each coefficient is an upper bound, so the cap never rejects a legitimate frame. The receiver enforces it at its transport read gate (`fileSyncConnection`'s `get()` `maxBytes`, threaded via `setInboundFrameCap`), and the sender checks its built reply against the same value before sending -- one computation. It is **transport-aware via per-transport clamps**: file-sync uses the derived cap up to `MAX_FRAME_SIZE_BYTES`; the WebRTC data channel keeps its fixed browser-tab envelope `MAX_WEBRTC_FRAME_BYTES` (256 MiB) at its reassembly read gate (see [CHANNEL_SECURITY.md](CHANNEL_SECURITY.md)). At the cell-count ceiling the derived reply frame is about 168 MB -- below both envelopes -- so the per-transport clamp does not bind there, and the cell-count budget is what gates first.

The receiver additionally validates every count it decodes from the reply against authenticated state before it drives any allocation: the sender's record count packed into the reply must equal the count the sender exchanged during role resolution, and the index table the frame actually carries must have exactly `keyCount * senderRecordCount` entries.

**Exceeding the ceiling aborts; it does not route to cascade.** `linkage_strategy` is a mandatory-consistency agreed term, so it cannot change unilaterally mid-exchange. An over-ceiling exchange therefore fails closed with actionable guidance that does **not** recommend cascade (reduce the key or record count, or split the dataset); re-agreeing on cascade is a separate out-of-band step. Because the ceiling is computed from authenticated counts, both parties abort in lockstep before any single-pass frame moves, so neither hangs. (Cascade also discloses less, so a silent downgrade would not be a privacy hazard in itself -- the blocker is the consent invariant, that both parties agreed to single-pass specifically, plus the added round trips.) An in-protocol consented fallback remains possible future work, not the default.

#### Role resolution and work minimization

Before the linkage rounds the two parties exchange their record counts and resolve which is the [PSI base function](#psi-base-function) receiver and which is the sender. Both the exchange and the assignment are specified here. The reference implementation is `resolveRole` (`packages/core/src/protocolSetup.ts`).

**The count exchange is unconditional.** Both parties put their record count on the wire at the start of every exchange, in lockstep: the initiator sends `{ recordCount }` first, the responder receives then sends. The frame is byte- and order-identical whether one party or both expect output; the two cases differ only in the role decision taken from the counts, never in what crosses the wire (the one-sided path drains the count frame, it does not skip it). This pre-linkage exchange is independent of the linkage rounds and is not one of the single-pass exchange's three frames -- in particular it is a separate quantity from the sender's record count packed into the single-pass reply (part (c) above), which is that party's own row count embedded so the receiver can size the index table. The integer exchanged is the raw row count: the quantity the role assignment below uses, the single-pass pre-flight checks locally, and the data-dependent single-pass frame cap derives its bound from (see the single-pass dataset ceiling above). (The quantity that actually drives that frame's size is the distinct-value count, which is not exchanged; the cap bounds it from the row and key counts.)

**The assignment.** When exactly one party expects output, that party is the receiver regardless of the counts -- it is the only party that learns the result. When both parties expect output the assignment is free, and it is chosen to minimize total work -- total communication bytes plus total per-party compute -- for the densely-populated near-unique keys linkage typically uses. The rule is: the party with the smaller row count becomes the receiver; an equal-count tie is work-neutral and broken deterministically in favour of the initiator. This is one rule for both linkage strategies; the rest of this section derives it, including the regime where a row-count rule cannot be optimal.

**Why the receiver is the heavier side.** In the PSI base function the receiver (client) encrypts its distinct-value set under its own key and sends it; the sender (server) encrypts its own set under its key and *re-encrypts the receiver's set*, returning both. So the receiver's set crosses the wire twice -- once as its request, once re-encrypted in the reply -- while the sender's crosses once, and the receiver's set drives more of the expensive curve operations (its own encryption, the sender's re-encryption, and the receiver later stripping its own key from the reply) than the sender's does. Writing `D_recv` and `D_send` for the two parties' pooled distinct-value counts and ~35 bytes per encrypted element, the dominant communication cost is `35 * (2*D_recv + D_send)` bytes; both that and the curve-operation count are larger for whoever is the receiver, by an amount that grows with the receiver's distinct count. Total work is therefore minimized by putting the smaller set on the receiver.

**Cascade.** The cascade is one base-function round per key over a shrinking candidate set; every round carries the `2*D_recv + D_send` asymmetry above and nothing that opposes it. Smaller-as-receiver is optimal, and the win over the other assignment is up to about `35 * (D_large - D_small)` bytes (plus the matching curve operations), where `D_large` and `D_small` are the two parties' distinct-value counts; the per-round candidate set shrinks as the cascade proceeds, so the realized gap is at most that bound.

**Single-pass.** The single-pass reply adds one term the cascade has no analogue for: the distinct-value index table, `4 * keyCount * senderRecordCount` bytes (defined above), which scales with the *sender's row count*. It is the one term that favours the opposite assignment -- making the smaller-row party the sender shrinks it. So single-pass has two asymmetric terms pulling against each other:

- the masked-set / re-encryption term, about `35 * (D_large - D_small)` bytes, favouring smaller-as-**receiver**;
- the index table, `4 * keyCount * (rows_large - rows_small)` bytes, favouring smaller-as-**sender**.

A linkage key earns its place by being near-unique where present (high positive-predictive value; see [Linkage keys](#linkage-keys)), and a value repeated within a dataset is dropped from its round, so where a key is also densely populated -- present on most records, with comparable density on both sides -- the participating distinct values track the row count: `D ~= keyCount * rows`. Under that assumption both terms carry the same `keyCount * (rows_large - rows_small)` factor and reduce to the ratio of their coefficients, **35 to 4** -- the masked-set term outweighs the index table by about nine to one. Smaller-as-receiver wins, by a net of up to about `31 * keyCount * (rows_large - rows_small)` bytes, reinforced by the curve-operation saving (which the index table, being integer work, does not offset). So in the typical case single-pass agrees with the cascade: one rule serves both.

**When the rule is not optimal, and why it still stands.** The index table can dominate whenever distinct participating values fall well below row counts (`D` much less than `keyCount * rows`), because the index table counts every record (a `-1` for each absent value) and so scales with total rows, while the masked sets scale only with distinct *participating* values. That happens not only for a heavily-repeating low-precision key (bad design), but also for a precise key that is *sparsely populated* -- present on only a fraction of records, such as an SSN available for a minority of rows -- which is legitimate and exactly the kind of key the cascade combines with denser ones, and for two parties whose per-key density differs markedly. In those cases the cheaper assignment can flip to smaller-rows-as-sender; the crossover is roughly when populated `(key, record)` cells average below `4 / 35` (about 11%) of the table. The decisive point is that all of these are invisible to a row-count rule: the quantity that distinguishes them is the distinct-value count -- and that count is deliberately not exchanged, because unlike the raw row count (mere dataset-size metadata) it would disclose data content: how much a party's key values repeat, and how sparsely a key is populated. Given only the row counts, smaller-rows-as-receiver is the best fixed rule for the typical case (near-unique, densely and comparably populated keys) and is the one implemented; it is not optimal on every input, and the residual gap on sparse or asymmetrically-dense keys is accepted, not closed: closing it would mean exchanging distinct-value counts, trading away the privacy of each key's duplication and sparsity structure for a modest bandwidth saving -- a trade this protocol deliberately does not make. (This supersedes an earlier statement of the rule that justified smaller-as-receiver purely as "minimising data transmitted"; the conclusion -- keep the rule -- is unchanged, but the reasoning now accounts for the single-pass index table and is explicit about the regime where a row-count rule cannot be optimal.)

### PSI-C

> **Not yet implemented:** PSI-C is not yet fully implemented. It is targeted for a release after 1.0; see [ROADMAP.md](../ROADMAP.md). The description below is the intended design.

PSI-C is also executed by sequentially executing deterministic linkages. Membership anonymity is granted by the sender permuting the receiver's doubly-encrypted data before returning it to them. The results of multiple linkage keys can be combined so long as the sender uses a consistent permutation algorithm for each round. The association map in the permuted space has the same size as one in the original space. This allows the cardinality to be measured without revealing which specific members are in common.

# Datasets

## Raw data

Data is input as a csv or other tabular data format. These files include rough metadata such as column names and storage types, which can be augmented with [metadata](../EXCHANGE_REFERENCE.md#input-metadata) from the exchange specification. Thus raw data consists of one or more columns, metadata for each column, and a record count.

## Standardized data

A standardized dataset bridges between raw data and the fields expected as inputs to linkage keys. It is an abstraction over the application of [data standardizing](../EXCHANGE_REFERENCE.md#data-standardizing-transformations) transformations. As noted in that section, there can be zero, one, or more than one standardized record for any input, so a standardized data element is essentially a mapping between an index into the raw dataset and a set of strings. These mappings are lazily evaluated and their results are cached. A standardized dataset is a collection of these mappings and the names of the linkage key fields they provide.

## Linkage participation: the role axis

A column participates in matching -- its standardized value is hashed into a PSI set element -- only when the operator roled it `linkage`. There is one binding rule, shared by the dataset builder, the satisfiability pre-flight, and the default-standardization derivation so they cannot drift: a declared [linkage field](../EXCHANGE_REFERENCE.md#linkage-terms) binds to a column when either (1) an explicit data standardizing transformation names the field as its `output` and a present `role: linkage` column as its `input`, or (2) absent such a transformation, the first `role: linkage` column whose semantic `type` equals the field's type. A column roled `identifier`, `payload`, or `ignored` is never bound -- even as the only column of its type, and even when an explicit transformation names it: the `role` wins over a contradictory transform. A field that binds to no column produces no key input and is surfaced as unsatisfiable by the satisfiability pre-flight; it is never silently dropped.

This keeps matching participation a separate axis from transmission. Transmission is governed by `is_payload` (a column is sent to the partner when `is_payload` is true and its role is not `ignored`); matching is governed by `role: linkage`. Neither implies the other: a column that both matches and is transmitted is `role: linkage` with `is_payload: true`. Keeping the axes disjoint is what prevents a column the operator marked sent-to-partner, or as a local row identifier, from also being hashed into a cross-party match key without an explicit linkage designation.

This rule would only need revisiting if a disclosure model wanted a column to participate in matching without an explicit `linkage` role -- for example re-deriving participation from semantic `type` alone, which was the prior behavior. It was narrowed because a matchable `type` is not consent to match: a column roled for another purpose could otherwise be hashed into a key as a side effect of its type, on every linkage path and in the from-scratch authoring defaults, not only on a crafted import.

## Transform regular-expression dialect

Some standardizing transformations carry a partner-authored regular expression: the `pattern` of `replace_regex`, `extract_regex`, and `filter_regex`, the `delimiter` of `split_on`, and the pattern `parse_date` builds from its `input_format` (each `MM`/`DD` token expands to `(\d{1,2})`, each `YYYY` to `(\d{4})`, every other character a literal, anchored `^...$`). These patterns are compiled and run per row over the other party's data. Because both parties MUST derive byte-identical keys, and the linkage terms are canonically hashed into the exchange-agreement receipt, both parties MUST evaluate every such pattern under one normative dialect; a pattern is otherwise meaningless across implementations (the protocol would inherit "whatever the running engine does"). The dialect is also a security boundary: it admits only patterns that match in time linear in the input, eliminating the catastrophic-backtracking denial of service a backtracking engine allows (see [CHANNEL_SECURITY.md](CHANNEL_SECURITY.md#transform-regex-linear-time-dialect)).

**Engine and semantics.** Patterns MUST be evaluated under an RE2-style engine: a Thompson NFA / lazy DFA that performs leftmost-first matching in time linear in the input length, with no backtracking. The reference implementation uses `re2js` (a pure-JavaScript RE2 port), the same build in the CLI (Node) and the web (browser). Matching is over Unicode code points of the NFC-normalized value. No flags are set: matching is case-sensitive, `.` matches any code point except newline (`\n`), and `^` and `$` anchor the start and end of the whole value (not lines); inline flag groups (`(?i)`, `(?s)`, `(?m)`) within a pattern are permitted. The predefined classes `\d` and `\w` use their ASCII definitions, as JavaScript's do. `\s` is ASCII-only (`[\t\n\f\r ]`) -- note this is NARROWER than JavaScript's `\s`, which matches Unicode whitespace (a no-break space, a vertical tab, a line/paragraph separator) regardless of the `u` flag; likewise RE2's `.` excludes only `\n`, where JavaScript's `.` also excludes `\r` and the Unicode line separators. These are deliberate consequences of the engine, normative for both parties (who run the same engine, so they agree); an operator carrying a pattern over from a JavaScript engine should account for the narrower `\s` and broader `.`.

**Permitted syntax.** The dialect is the RE2 syntax this engine accepts with no flags: literal characters and escapes, character classes `[...]` (and negated `[^...]`), the predefined and Unicode-property classes (`\d \w \s` and `\p{...}`), anchors (`^ $ \b \B \A \z`), greedy and lazy quantifiers (`* + ? {n} {n,} {n,m}` and their `?` forms), grouping (`(...)`, non-capturing `(?:...)`, named `(?P<name>...)` or `(?<name>...)`), and alternation `|`. The product of the nested counted-repetition bounds applied to any subexpression MUST NOT exceed 1000 (RE2's repeat-count limit): a single `{n}` may range up to 1000, but a nested repetition such as `(x{100}){11}` is rejected because the product of its bounds (1100) exceeds 1000 even though each individual bound is under it. A flat (non-nested) concatenation of repetitions such as `a{1000}b{1000}` is not bounded by this rule -- the engine caps repeat counts, not total program size, so overall pattern size is bounded only by the source-length cap (see the per-pattern length bound in the operation semantics).

**Excluded.** The dialect MUST NOT include the features RE2 omits, because they require backtracking: backreferences (`\1`, `\k<name>`) and lookaround (`(?=)`, `(?!)`, `(?<=)`, `(?<!)`). The JavaScript `\uHHHH` / `\u{...}` escape is also excluded; use RE2's `\x{...}`. A pattern using any excluded feature, or otherwise not compiling under the engine, MUST be rejected at terms validation, before any row is processed (fail closed); an implementation MUST NOT fall back to a backtracking engine.

**Operation semantics.** `replace_regex` replaces every match (global); the replacement string follows JavaScript `String.prototype.replace` substitution semantics in full -- `$n` and `$nn` (numbered groups), `$<name>` (named groups), `$&` (the whole match), `` $` `` (the text preceding the match), `$'` (the text following the match), and `$$` (a literal `$`) -- and an unrecognized `$`-sequence is emitted literally. `extract_regex` yields the first capturing group when the pattern has one and it participated in the match; otherwise it yields the whole match -- both when the pattern has no capturing group and when its first group did not participate (an unmatched optional group such as `a(z)?` on input `a`); a selected value that is the empty string, or no match at all, yields `null`. `filter_regex` tests for a match anywhere in the value. `split_on` splits on the delimiter; unlike JavaScript `String.prototype.split`, capturing groups in the delimiter are NOT emitted as output elements, and the function discards empty parts. `parse_date` matches the NFC-normalized value against the anchored pattern built from `input_format`; on no match it yields `null`. From a match it takes the `YYYY` group verbatim and zero-pads each `MM` and `DD` group to two digits; if any of `YYYY`, `MM`, or `DD` is absent from `input_format` it yields `null`. It then applies a coarse range check through the platform date parser: the assembled `YYYY-MM-DD` is parsed as an ISO 8601 date-only (UTC) value, and a string the parser cannot represent -- a month outside 1-12 or a day outside 1-31 (e.g. month 13, day 32, month or day 0) -- yields `null`. This check does NOT reject a day that is merely out of range for its specific month: `02/29` in a non-leap year, `02/30`, and `04/31` are ACCEPTED and pass through (the parser rolls such a date over internally, but the output is assembled from the captured digits, not the rolled-over date). A reimplementation MUST match this exactly -- reject month not in 1-12 and day not in 1-31, accept every other digit combination -- to derive identical keys. The result is `output_format` with the first occurrence of each of `YYYY`, `MM`, and `DD` replaced by its (padded) value; any later occurrence of a token, and every other character, is left literal.

**Versioning.** A change to the permitted syntax, the flag set, the Unicode mode, or the engine's match/replace/split semantics is a wire-compatibility event: it changes what a pattern means or whether it is valid, and two parties running different dialects would silently derive different keys. The linkage-terms [`version`](../EXCHANGE_REFERENCE.md#linkage-terms) field does NOT by itself prevent this -- it is an operator-authored schema version, compared only for equality between the two parties' authored values, and is not bound to the engine or dialect a party actually runs (two parties can declare the same `version` while running different psilink releases). A dialect change therefore MUST be accompanied by a `version` bump in this specification AND requires the two parties to run releases that implement the same dialect; the protocol does not currently negotiate or hash an engine/dialect version at runtime, so this is a process obligation rather than a guarantee the handshake enforces. Binding the dialect to a runtime-negotiated version is tracked as follow-up work.

## Key input data

As defined in [linkage terms](../EXCHANGE_REFERENCE.md#linkage-terms), linkage fields are the inputs that are typically transformed to create the character strings that are concatenated to form statistical linkage keys. The linkage algorithm attempts to connect individuals - not just character strings - represented by their indices in the original dataset. Consequently, key inputs are realizations of linkage fields and are, like standardized data elements, maps between indices and sets of character strings of arbitrary size. In order to realize a complete linkage key, the set of all combinations of all key inputs is computed each and each combination is concatenated together into a string.

When that set has zero elements, i.e. one of the elements is `NULL`, it is omitted from the linkage protocol. When that set has more than one element and the algorithm is using the optimization implied by the deterministic cascade detailed in [Matching algorithms](#psi), communication is required in order to determine if the link is unique. For example, if one party has a member named "Mary Shaye-Smith" and another two members named "Mary Thorne" and "Mary Smith", unless the mapping was many-to-one it would be incorrect to accept both matches as valid and remove the individuals from the candidate set.

This extra communication step violates the threat model which guarantees that nothing can be learned about members that are not in common. That said, the existence of a match on a key would be sufficient evidence that two records represent the same person and it is only by revealing contradictory evidence that the link is not made. Users who employ these transformations are warned about their consequences.

The `NULL`/absent omission above is distinct from a realized key that is the *empty string* (`""`). A `NULL`/absent key (the realized set has zero elements) is the "no key" sentinel and the record is omitted from the round; a record whose realized set is the singleton `{""}` carries a present, matchable key value and participates in matching like any other single-valued key. It is subject to the same within-dataset uniqueness rule, so a column in which every value standardizes to `""` matches nothing (every `""` is a within-dataset duplicate and is dropped from the round), while a lone `""` on each side matches. This distinction would only need revisiting if `""` were to regain a special meaning at the linkage layer: historically the matching layer dropped `""` together with `NULL`, from before standardization was operator-authored and exported, because doing so silently prevented a lone blank cell on each side from spuriously matching. Now that operators author and export their own standardization, `""` is a first-class key value in matching, and that blank-cell footgun is prevented explicitly at the standardization layer (the default per-type cleaning chains map an empty value to `NULL`) rather than silently in the matching code.

# Post-linkage steps

## Non-repudiation

> **Partially implemented:** The signing identity that certificate-backed receipts rest on -- key generation, the self-signed certificate, fingerprint pinning, and the identity binding -- is implemented today and specified under [Signing identity and certificate pinning](#signing-identity-and-certificate-pinning) below. Receipt assembly, signing, and the receipt swap are not yet wired up; they are targeted for the 1.0 release (see [ROADMAP.md](../ROADMAP.md)). The receipt-content description in this section is the intended design.

A receipt is produced at the conclusion of a successful exchange, after the result - and any payload columns - already exist on both sides. The receiver learns the intersection during the PSI rounds themselves: the association map is computed and shared as a step of the matching algorithm, and payload columns are transmitted immediately afterward. By the time a receipt could be assembled both parties already hold the result, so the receipt is a post-exchange audit artifact recording that the exchange occurred. It does not gate, withhold, or otherwise condition delivery of the result; the result is already in hand on both sides.

Each receipt records the timestamp, a hash of the exchange agreement, each party's `identity` (the self-asserted [`linkage_terms.identity`](../EXCHANGE_REFERENCE.md#linkage-terms) field), and the size of the result if that information was learned by both parties. The hash of the agreement and the signature over the receipt are computed over a canonical byte serialization (RFC 8785, JSON Canonicalization Scheme) so that both parties - and an independent third party using a different implementation - derive byte-identical input and the hash and signature verify across implementations. That serialization is specified in [CANONICAL_ENCODING.md](CANONICAL_ENCODING.md). The two parties then swap signed receipts and each retains the other's. A receipt can be signed in one of two modes, which differ sharply in what they prove:

- **Session-derived (symmetric).** The signature is a MAC under the shared key-exchange session key. Both parties hold the same key, so either can forge a MAC the other could have produced. A session-derived receipt is therefore not evidence against a denying counterparty and is not verifiable by any third party; it yields only a tamper-evident local record - a party can later detect that its own stored copy was altered, but cannot use it to hold the partner to the exchange. This mode does not provide non-repudiation.
- **Certificate-backed (asymmetric).** The signature is made with a party's long-lived signing private key, whose self-signed certificate the partner has pinned by fingerprint out-of-band at setup. Only the holder of that key could have produced the signature, and the pin ties the key to the partner, so the signature is verifiable by anyone who holds (or is given) the pinned fingerprint -- it does not depend on a certificate-authority chain. This is the only mode that provides non-repudiation, and the only mode in which the `identity` field carries evidentiary weight: the certificate binds the asserted identity to the pinned key, whereas a session-derived receipt leaves it an unverified label. The signing identity, certificate format, fingerprint pinning, and the identity-binding rule are specified under [Signing identity and certificate pinning](#signing-identity-and-certificate-pinning) below.

Because the receipt is collected after the result already exists on both sides, the signature swap is best-effort evidence, not a fairness or atomicity guarantee. Two limitations follow directly. First, the swap is not a fair exchange: a party may receive the partner's signature and then decline to send its own, and no protocol step can compel the second signature once the first has been handed over. Second, aborting does not undo the data exchange: because both parties already hold the result, terminating the program and restarting does not roll the data back. An aborting party can capture the partner's signature, withhold its own, and still keep the intersection. Restarting the exchange is not a remedy for data already transferred; it only repeats the work.

Retention, access controls, and log integrity beyond the receipt remain each party's internal compliance obligation.

## Signing identity and certificate pinning

> **Implemented.** Identity generation, persistence, fingerprint display, fingerprint pinning, and the identity binding are produced and enforced today (`psilink fingerprint`; the `signing` block in [EXCHANGE_REFERENCE.md](../EXCHANGE_REFERENCE.md#signing)). Signing receipts with this identity and swapping them is the remaining, not-yet-wired work above.

Certificate-backed receipts rest on a long-lived **signing identity**: an asymmetric keypair and a self-signed certificate that carries the party's `identity`. Each party generates one, persists it, and reuses it across every exchange and every partner -- it is not a per-exchange credential. It is therefore a distinct credential from the shared secret in the key file, which rotates every exchange: the signing key must be stable for its whole life so that a fingerprint a partner pinned once keeps verifying later receipts. Regenerating the identity is a deliberate act (`psilink fingerprint --force`) that produces a new key and a new fingerprint and invalidates any fingerprint a partner has already pinned.

The trust model is **pinned self-signed**: there is no certificate-authority chain and no revocation. A self-signed certificate proves only possession of its own key, not who the holder is; the binding to a real partner is established out-of-band, the same way the parties already exchange the invitation and shared secret. The certificate's **fingerprint** -- an unpadded base64url `SHA-256` over the domain-separated canonical encoding (RFC 8785) of the certificate body -- is shared with the partner over a trusted channel and pinned in the partner's `signing.partner_fingerprint`. The fingerprint is not secret (it is a hash of a public certificate), so the channel must be authentic, not confidential. This is the SSH-host-key / certificate-pinning model, not the web-PKI model. The rationale for choosing it over X.509 + CA for v1, and the seam by which an authority-backed mode could layer on later, are in [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#receipt-signing-identities).

The certificate is a small canonical-JSON document rather than an X.509 structure, so the bytes that are signed and fingerprinted reuse the project's single canonicalization primitive (the same one the agreed-terms hash and the record commitments use) and reproduce across implementations; that serialization is specified in [CANONICAL_ENCODING.md](CANONICAL_ENCODING.md). The body carries a format `version` (a single recognized literal for v1, rejected rather than migrated if unrecognized -- also the format discriminant a future authority-backed representation would extend), the signature `algorithm` (`ed25519` in v1), the party's `identity`, and the public key as a JWK (RFC 8037 OKP). The self-signature is an Ed25519 signature over the domain-separated canonical bytes of that body, under the body's own public key; the fingerprint is a SHA-256 over a body encoded under a distinct domain label, so a signature and a fingerprint can never be confused for one another. Ed25519 is used for its deterministic, non-malleable signatures and single canonical encoding; the public key is rejected on load if it is the wrong length, not a valid curve point, or a small-order (degenerate) point.

Two checks gate accepting a certificate-backed receipt, and both must pass before the receipt is accepted:

- **Pinning.** A presented partner certificate is trusted only if its self-signature verifies and its fingerprint matches the configured `signing.partner_fingerprint`. A certificate presented when no fingerprint is pinned, one whose self-signature does not verify, and one whose fingerprint does not match the pin are each rejected with a clear error. The fingerprint comparison is constant-time.
- **Identity binding.** A receipt is accepted only if its asserted `identity` is the one the presenting certificate authorizes. Because `identity` is self-asserted free text (`Consistency: none`, often multi-line), the rule is an exact match of the full identity, compared over the same canonical `identity` bytes the record commits to and the receipt signs -- so the certificate check and the signed record agree on what "identity" is. A party that uses a different identity string needs a different certificate (a deliberate regeneration); the certificate authorizes exactly the identity it carries.

## Self-attested record

> **Implemented.** Produced today, at the conclusion of every successful exchange. Unlike the signed receipt above, this record is deliberately unsigned: a session-derived MAC is not non-repudiation (see the two signing modes above), so the unsigned record is an honest, local audit artifact rather than a receipt that overstates what it proves. Signing it is the separate, deferred Signed Exchange Receipts work.

At the conclusion of every successful exchange, each party produces a self-attested record of what it disclosed -- a local, unsigned audit artifact that stands on its own as a disclosure-log entry, so an operator can populate a HIPAA accounting of disclosures or a FERPA disclosure record without separately retaining and re-matching the original linkage-terms config. It is built from data both sides already hold, so it needs no private key and adds no protocol round-trip. It is explicitly **not** a signed or non-repudiable receipt and **not** evidence against the partner.

The record format itself -- the record and opening file shapes, the format version, the agreed-terms hash, the HMAC commitment scheme, the readable governance metadata, the result-size and per-direction disclosure framing, and the record's privacy properties -- has its own specification: [EXCHANGE_RECORD.md](EXCHANGE_RECORD.md). The deferred signing work reuses that record's commitment scheme and on-disk format.

## Output

The basic output is an association table between each party's element. As noted above, if parties supplied identifier columns with their inputs and flagged them in their metadata, the association table will be between each party's identifiers. Otherwise, the table references the row indices of each dataset.

If parties elected to transmit payload data, the relevant columns for the appropriate rows will be transmitted in-band over the secure connection and appended to the output in-the-clear.

# X25519 authenticated key exchange

PSI-Link authenticates every recurring exchange with an X25519 authenticated key exchange: the key-agreement primitive that produces the exchange session key (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#key-agreement-design) for the decision and its rationale; it replaced an earlier SPAKE2 design). It performs an ephemeral X25519 Diffie-Hellman keyed together with the pre-shared secret, plus an explicit, role-asymmetric mutual key confirmation, and yields a 32-byte session key with forward secrecy and mutual authentication. The handshake also carries a per-party request-encryption flag and returns a transcript-bound `applyEncryption` decision (`own request OR peer request`) that tells the caller whether to wrap the connection in the application-layer AEAD; see the **Request-encryption flag** discussion below. `authenticateConnection` (in `auth.ts`) runs it immediately after the connection opens and uses the resulting session key to key the application-layer AEAD and the shared-secret rotation. It is implemented as `runKex` in `packages/core/src/kex.ts` and exported from `@psilink/core`. This section specifies the primitive and its wire format.

**Library selection.** The governing rule is to prefer a well-respected, ideally audited, dual-platform library and to hand-roll only the minimal glue. No production-ready, audited Noise Protocol Framework library with dual Node.js/browser support exists in the JavaScript ecosystem, so none qualifies. The construction therefore reuses [`@noble/curves`](https://github.com/paulmillr/noble-curves) -- already a dependency, independently audited, and used for the Ed25519 signing identity -- for the X25519 primitive (`x25519.getSharedSecret`) and the ephemeral key generation (`x25519.keygen`). No new dependency is added. The only hand-written layer is the minimal NNpsk0 glue: the Noise key-schedule mixing and the two confirmation tags. The full Noise framework is deliberately not implemented. The Diffie-Hellman is a library call, never hand-rolled curve math.

**Construction.** The handshake is pinned to the Noise NNpsk0 pattern over X25519 -- no static keys, the pre-shared secret mixed in at position 0, and ephemeral-ephemeral Diffie-Hellman -- followed by an explicit key-confirmation round, following [NIST SP 800-56A Rev. 3](https://doi.org/10.6028/NIST.SP.800-56Ar3) (section 5.8.1 for the key-derivation step over the shared secret plus fixed info, sections 5.9 / 6.2.1.5 for key confirmation). All symmetric operations (SHA-256, HMAC-SHA-256, HKDF) use the platform `crypto.subtle` API, so they are identical on Node and in the browser.

**Protocol-version tag.** The Noise "protocol name" is the version tag and is hashed into the initial handshake hash, binding every derived key and confirmation tag to it:

```
protocol_name = "psilink-kex-v1:NNpsk0_25519_SHA256"
```

A future authenticated mode (for example a certificate-chain variant) is an additive new version selected out of band; bumping this string makes a mismatched peer derive a different transcript and fail closed, so the door stays open without any pluggable-auth framework today. The version tag is reserved for a genuine cryptographic-suite change and is not bumped for an additive transcript payload such as the request-encryption flag described below: that flag is a required field on the wire schemas, so a flag-aware and a flag-unaware peer already fail closed at parse -- the flag-aware side rejects a message that omits the required field, and the strict (`.strict()`) schema rejects a message that carries the unrecognized field -- before any transcript is computed.

**Key schedule** (a minimal subset of the Noise `SymmetricState`: a chaining key `ck` and a handshake hash `h`). `protocol_name` is longer than the 32-byte hash output, so the Noise `InitializeSymmetric` reduces to `h0 = SHA-256(protocol_name)`, and `ck0 = h0`. The NNpsk0 tokens (`-> psk, e` then `<- e, ee`, with an empty prologue) are then processed in order:

```
MixHash("")                       # empty prologue
MixKeyAndHash(psk)                # psk token at position 0
MixHash(e_initiator_public)       # initiator's e: MixHash ...
MixKey(e_initiator_public)        #                ... and MixKey (PSK mode)
MixHash(initiator_flag_byte)      # msg1 payload: initiator's request-encryption flag
MixHash(e_responder_public)       # responder's e: MixHash ...
MixKey(e_responder_public)        #                ... and MixKey (PSK mode)
MixHash(responder_flag_byte)      # msg2 payload: responder's request-encryption flag
MixKey(dh)                        # ee: dh = X25519(e_self_private, e_peer_public)
```

Each `*_flag_byte` is a single byte, `0x01` if that party requests the additional application-encryption layer and `0x00` otherwise, MixHash'd as that message's handshake payload right after the party's `e` token (the standard Noise handshake-payload position). It is MixHash only -- the flag need not be confidential and enters `h` alone, so the chaining key (and the confirmation key derived from it) stays flag-independent while `h`, the session key, and both confirmation tags depend on both flags. See the **Request-encryption flag** discussion below.

In a PSK handshake, Noise (rev 34 section 9.2) processes each `e` token with `MixKey(e.public)` in addition to `MixHash(e.public)`, folding the ephemeral public keys into the chaining key as well as the hash; this implementation does both, so the mix chain is faithful NNpsk0. (`MixHash` touches only `h` and `MixKey` only `ck`, so their order within an `e` token does not affect the result, and the flag MixHashes likewise commute with the surrounding MixKey calls; the chaining-key order `psk -> e_initiator -> e_responder -> ee` and the hash order `e_initiator -> initiator_flag -> e_responder -> responder_flag` are what matter, and are preserved.) Here `MixHash(x)` sets `h = SHA-256(h || x)`; `MixKey(ikm)` sets `ck = HKDF-Noise(ck, ikm, 2)[0]`; and `MixKeyAndHash(ikm)` sets `(ck, temp_h) = HKDF-Noise(ck, ikm, 3)[0..1]` then `MixHash(temp_h)`. `HKDF-Noise(ck, ikm, n)` is the Noise chaining HKDF (Noise rev 34 section 4.3): `temp_key = HMAC(ck, ikm)`, then `output_i = HMAC(temp_key, output_{i-1} || byte(i))` for `i` in `1..n` (with `output_0` empty). This equals RFC 5869 HKDF with `salt = ck` and an empty info string.

After the tokens, the session key and a distinct confirmation key are derived, and the two confirmation tags computed, using the application HKDF (`hkdfDerive`: HKDF-SHA-256, 32-byte zero salt, named info):

```
session_key       = HKDF(ck || h, "psilink-kex-v1:session", 32)
confirm_key        = HKDF(ck,      "psilink-kex-v1:confirm", 32)
initiator_confirm  = HMAC(confirm_key, "psilink-kex-v1:initiator-confirm" || h)
responder_confirm  = HMAC(confirm_key, "psilink-kex-v1:responder-confirm" || h)
```

Deriving `session_key` over both `ck` (which carries the pre-shared secret and the X25519 DH output) and `h` (which carries the protocol-version tag, both ephemeral public keys in role order, and both request-encryption flags) is the load-bearing invariant: the session key depends on the ephemeral DH output, so it is neither the pre-shared secret alone -- which would have no forward secrecy -- nor the raw X25519 output alone -- which would not be transcript-bound. The confirmation key is independent (distinct input keying material and a distinct label), and the two confirmation tags are role-asymmetric (distinct labels) and each binds the full transcript hash `h`. All labels are namespaced `psilink-kex-v1:` and are disjoint from every other domain-separation label in the system (`psilink-aead-v1:*`, `psilink-shared-secret-rotation-v1`, and the `psilink-signing-*` labels).

**Message flow** (initiator sends first; all messages are JSON objects; `e` is a base64url-encoded 32-byte X25519 public key, `confirm` is a base64url-encoded 32-byte HMAC-SHA-256 tag, and `reqEnc` is the party's request-encryption boolean):

1. Initiator -> Responder: `{ kexMsg: "1", e: e_I, reqEnc: req_I }`
2. Responder -> Initiator: `{ kexMsg: "2", e: e_R, confirm: responder_confirm, reqEnc: req_R }`
3. Initiator -> Responder: `{ kexMsg: "3", confirm: initiator_confirm }` or `{ kexMsg: "abort" }`

The responder confirms first (in msg2); the initiator verifies `responder_confirm` before sending its own confirmation in msg3, and the responder verifies `initiator_confirm` on msg3. A mismatched pre-shared secret therefore fails closed on both sides before any non-handshake frame is sent. Each side sends the tag for its own role and verifies the tag for the opposite role, so a reflected or echoed confirmation does not verify.

**Request-encryption flag.** Each party declares, on its own handshake message (`req_I` on msg1, `req_R` on msg2), whether it wants an additional application-encryption layer over the post-handshake connection. The flag is a required field on the strict message schemas. Both parties therefore hold both flags before the key schedule runs, so both compute the same negotiated decision `apply_encryption = req_I OR req_R` and bind both flags into the transcript hash `h` (as the per-message handshake payload, above) with no extra message or round-trip. Because `h` feeds both confirmation tags, a flag flipped on the wire makes the two sides derive different transcripts and the confirmation fails closed: the handshake cannot be downgraded to an encryption decision the parties did not agree on. The negotiation is a single fixed-meaning boolean, deliberately not an extensible options block -- open-ended in-handshake negotiation is the historical home of downgrade bugs. `runKex` returns `apply_encryption` as `KexResult.applyEncryption`; the caller applies the AEAD layer (see [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#channel-security)) when it is true and a session key is in hand. File-sync transports (SFTP, file-drop) request it unconditionally; a transport already end-to-end confidential against any in-path party (for example WebRTC over DTLS, except under a DTLS-terminating WebSocket relay) declines it.

**Contributory check.** The peer's X25519 share is rejected if the computed shared secret is all-zeros. `@noble/curves` enforces this in `getSharedSecret`, which rejects low-order and non-canonical peer shares by throwing (RFC 7748), so the check comes from the audited library; an explicit all-zeros guard is retained as defense in depth against a future primitive swap.

**Failure handling.** All wire schemas use strict (`.strict()`) Zod parsing, so an unexpected field fails the parse. Tag comparison is constant-time. Every authentication failure throws a single generic, non-oracular error (`"key exchange authentication failed"`) that does not reveal which check failed; a silent peer surfaces the distinct `"key exchange handshake timed out"` after a 30 s per-message bound. Every receive-side failure before the final message sends a best-effort `{ kexMsg: "abort" }` so the peer stops waiting immediately rather than blocking until the timeout.

**Test vectors and interoperability.** The key-schedule mix chain above is faithful Noise NNpsk0, but the overall handshake is pinned by psilink's own checked-in known-answer vector (`packages/core/test/vectors/kex-vectors.json`) rather than being wire-compatible with generic Noise. Two things diverge from a stock Noise NNpsk0 session: the protocol name differs (so the initial `h` differs), and instead of Noise `Split()` the session key uses a custom KDF over `ck || h` with an added explicit confirmation round. So no end-to-end Noise vector corresponds. What does correspond is the underlying machinery, cross-checked against its published references in `packages/core/test/kex.test.ts`: the X25519 Diffie-Hellman against [RFC 7748](https://www.rfc-editor.org/rfc/rfc7748) section 6.1, and the Noise chaining HKDF against [RFC 5869](https://www.rfc-editor.org/rfc/rfc5869) test case 3. The vectors fix the pre-shared secret and both ephemeral private keys and record, for all four `(initiator, responder)` request-encryption flag combinations, the handshake hash, session key, and both confirmation tags (the chaining key and confirmation key are flag-independent and recorded once); they are reproduced by the independent generator at `packages/core/test/vectors/generate-kex-vectors.mjs`.

## Shared-secret rotation

The pre-shared secret that authenticates a recurring exchange is a 32-byte value encoded as unpadded base64url (43 characters). Two mechanisms mint one, differing only in how the 32 bytes are produced:

- **Invitation tokens** are generated directly from a CSPRNG (`crypto.getRandomValues`), giving 256 bits of entropy.
- **Rotation tokens** are derived from the 32-byte session key (above) with the application HKDF (`hkdfDerive`: HKDF-SHA-256, 32-byte zero salt), so a rotated secret carries the same 256-bit security level as the session key it derives from:

```
rotated_secret = HKDF(session_key, "psilink-shared-secret-rotation-v1", 32)
```

The rotation runs after every successful handshake and before the data exchange begins: both parties recompute it independently from the shared `session_key`, so it adds no message or round-trip, and each persists the result as the secret for the next exchange. The `psilink-shared-secret-rotation-v1` info string is a single fixed label (not a per-party or per-context family) and is disjoint from every other domain-separation label in the system (`psilink-kex-v1:*`, `psilink-aead-v1:*`, `psilink-abort-token-v1:*`, `psilink-webrtc-peerid-v1:*`, and the `psilink-signing-*` labels), so a rotated secret can never collide with a session, AEAD, abort-marker, peer-id, or signing derivation.

The operator-facing rotation policy -- that the secret is never reused, that a rotated token carries no expiry by default, and the recovery path when the two parties' tokens desynchronize -- is in [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#recurring-exchange-authentication).

### `expires` enforcement and the 'expiring soon' advisory

A `.psilink.key` token may carry an `expires` instant written by either of two mechanisms -- an invitation setup lifetime or the `token_max_age_days` policy -- and enforcement is kind-agnostic by construction: it reads only the stored `expires`, never its origin, and the same recovery (re-invite) applies to both, so no consumer branches on origin and no provenance marker is stored (a marker would add state to the key file without changing any decision taken from it, and could be present on one party's key file and absent on the other's, in a model whose recoverability depends on both parties holding uniformly-interpreted key files). The operator-facing policy is in [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#two-sources-one-expires).

**Enforcement sites.** Four checks read `expires`. Three are kind-agnostic and abort on any `expires` at or before the current time, directing both parties to re-invite: the load-time check before an exchange, the pre-handshake check, and the post-handshake check (which catches a token that lapses during the key-exchange round-trip). The fourth, the accept-time invitation check, is the only single-kind site -- unavoidably, since it validates the encoded invitation, which carries only an invitation lifetime. There is no expired state in which the two kinds call for different recovery, so the expired-side message is uniform by intent rather than branched on origin.

**'Expiring soon' threshold.** The advisory fires when the loaded `expires` is within `token_max_age_days / 3` days of the current time -- a fraction of the configured age rather than a fixed number of days, so it scales with the policy (a 90-day age warns with 30 days remaining, a 30-day age with 10) and is silent unless a policy is configured. It is suppressed when the exchange that surfaces it succeeds (rotation then stamps a farther-out `expires`) and shown only when no successful rotation refreshed the token -- a failed exchange.

**Invitation token flagged 'expiring soon'.** Because the advisory does not consult origin, a freshly minted invitation token can be flagged. This is reachable only on the offline re-invitation path (the inviter's key file is the only one that persists an invitation `expires`; the acceptor's offline key file carries none, and the online flow persists only the rotated token), before the first exchange rotates the token, and it survives only past a *failed* first exchange. The notice is accurate and its guidance is correct for either kind, so it is accepted rather than suppressed; suppressing it for invitation tokens alone would require persisting a per-file origin marker -- the invisible cross-party state the unified representation deliberately avoids.

**Design rationale.** The unification rests on the two sources sharing one recovery. No migration arises: an existing key file holds a bare `expires`, which is exactly what the uniform interpretation reads. It would need revisiting only if an expired max-age token were ever made recoverable without re-invitation, so that the expired-side action diverged by origin -- not the case today.

## WebRTC rendezvous peer-id derivation

The WebRTC transport meets its peer with no coordination backend: both web parties derive their PeerJS rendezvous peer ids deterministically from the invitation's 32-byte shared secret, so each side computes the other's id locally and they rendezvous on the PeerJS signaling broker without exchanging an id out of band. This precedes and is independent of the X25519 handshake above -- it places the two parties on the same WebRTC connection, over which the handshake then runs.

The derivation is HKDF-SHA-256 over the decoded 32-byte shared secret as input keying material, with a 32-byte zero salt (matching the application `hkdfDerive` above) and a role-separated info string:

```
inviter_info  = "psilink-webrtc-peerid-v1:inviter"
acceptor_info = "psilink-webrtc-peerid-v1:acceptor"
peer_id       = hex(HKDF-SHA-256(ikm = secret, salt = 0x00 * 32, info = role_info, 32)[0..15])
```

The peer id is the first 16 bytes of the HKDF output, encoded as lowercase hex. Hex (not base64url) is used because the PeerJS client rejects an id with a leading, trailing, or doubled `-` or `_`, which base64url output can contain. The inviter listens on its derived id (`inviter_info`) and the acceptor dials it.

This is a cross-implementation contract: a future CLI WebRTC transport must compute the same ids from the same shared secret to rendezvous with a web peer (see [`cli-webrtc-stack.md`](../notes/cli-webrtc-stack.md)). The `psilink-webrtc-peerid-v1:` labels are disjoint from every other domain-separation label in the system.
