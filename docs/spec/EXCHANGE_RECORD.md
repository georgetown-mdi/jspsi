# Exchange record format

This document specifies the self-attested exchange record: the local audit artifact each party produces at the conclusion of every successful exchange. It covers the record and verification-keys file shapes, the format version and reader-rejects-unknown rule, the commitment scheme and the agreed-terms hash, the readable governance metadata, the result-size and per-direction disclosure framing, and the record's privacy properties. It does not cover the PSI protocol that produces the exchange (see [PROTOCOL.md](PROTOCOL.md)), the exchange-agreement format that supplies the record's governance fields (see [EXCHANGE_REFERENCE.md](../EXCHANGE_REFERENCE.md)), or the canonical byte encoding the commitments and the agreed-terms hash are computed over (see [CANONICAL_ENCODING.md](CANONICAL_ENCODING.md)). The deferred Signed Exchange Receipts work (see [ROADMAP.md](../ROADMAP.md)) reuses this record's commitment scheme and on-disk format, so the commitment-scheme and file-shape sections below are written to stand on their own. Intended readers are security auditors, external implementors, compliance reviewers, and developers.

> **Implemented.** The self-attested record described here is produced at the conclusion of every successful exchange, on every path. It is a standalone disclosure-log entry, deliberately unsigned: a session-derived MAC is not non-repudiation, so the unsigned record is an honest, local audit artifact rather than a receipt that overstates what it proves. The *separate* certificate-backed [signed receipt](#signed-receipt) -- a mutually-verifiable dual-signed record -- is produced alongside it on the CLI's authenticated exchanges when a signing identity is configured; it reuses this document's canonical encoding and committed-payload shape but is its own artifact.

Each party produces a self-attested record of what it disclosed. The record is designed to stand on its own as a disclosure-log entry: an operator can use it to populate a HIPAA accounting of disclosures or a FERPA disclosure record without separately retaining and re-matching the original linkage-terms config. It is **not** evidence against the partner, and -- being built from data both sides already hold -- it needs no private key and adds no protocol round-trip.

## Record fields

The record captures: a format `version` (a single recognized literal for v1, which a reader rejects rather than migrates if unrecognized); the local timestamp (an ISO 8601 datetime in UTC, ending in `Z`; a timestamp carrying a numeric offset is rejected, so the format keeps one canonical timestamp form across implementations); a hash, over the canonical encoding, of the agreed exchange terms (both parties' linkage terms in a fixed, canonical-sorted order, so both parties and an independent third party derive the same hash); both parties' self-asserted `identity` strings; the number of records this party contributed to the exchange (`recordsExposed`) -- its own input row count, recorded on every record as a per-direction statement of what it put in; the result size, but only in the both-output case; an optional self-facing retention/disposition pointer recording where this party filed its copy of the result and under what retention schedule (see below); and a per-exchange binding nonce (CSPRNG-generated, at least 128 bits) so two runs with identical terms still produce distinct records. The binding nonce is the unsigned record's per-exchange binder and is distinct from the per-commitment salts below; do not conflate them. The deferred signing work will add a stronger session-key-derived binder.

## Result size and per-direction disclosure

The two disclosure figures -- `recordsExposed` and the result size -- are gated differently, and the difference is deliberate. `recordsExposed` is the count of records this party contributed to the exchange -- its input row count, counting every contributed record rather than only those that resolve to a usable linkage key, so it is an honest upper bound on what the party exposed rather than a derived match figure; it is known from the party's own input alone, so it does not depend on the result or on the partner, and it is recorded for every party regardless of role. The result size (the intersection cardinality) is recorded only when both parties' agreed terms have them both receive output, so it is stored only when both sides are entitled to the result, and omitted when only one party does. The gate is the terms agreement (entitlement), not what a party happens to learn during the exchange: in a single-output exchange the helper that does not receive output can still observe its match count during the [clean cascade](PROTOCOL.md#psi), but the record deliberately does not surface it. Privacy here is enforced by what the tool writes down, not by what is theoretically discoverable; the helper's own outbound disclosure is carried by `recordsExposed` instead. Framing the helper's figure as "records I exposed" rather than "the result size" is also what keeps the record meaningful under a future matching algorithm that discloses neither the result size nor the partner's set size -- the figure comes only from the party's own input, which any such algorithm still leaves it knowing.

Reading the disclosure direction off a record: a PSI exchange discloses the membership of matched records mutually -- each party reveals its matched membership to the other to make the match -- so direction is not a flag but a reading of the fields together. `recordsExposed` is the magnitude of what this party put in (its outbound exposure); the `algorithm` says what kind of disclosure occurred (`psi` revealed matched identifiers, `psi-c` only a count); `payloadSent` and `payloadReceived` give the payload categories per direction; and the presence of the association-table commitment (below) marks whether this party itself received the matched pairing. So "what this party disclosed" reads off `algorithm` + `recordsExposed` + `payloadSent`, and "what it received" off `payloadReceived` and the association-table commitment -- without consulting the output configuration, which the record does not carry.

The `algorithm` field is bound to what the run actually did, not to an unvalidated terms value. Only `psi` has a run path today; core refuses a `psi-c` (count-only) algorithm before any matched identifier is revealed (`assertAlgorithmImplemented` in `packages/core/src/exchange.ts`, at both the pre-connection prepare step and the run boundary), so a written record's `algorithm` is always `psi` and can never attest count-only over a run that revealed identifiers. This is the run-side half of the guarantee; a pure record-side clamp is rejected because it would keep the record honest while silently ignoring an operator's stated intent to disclose only a count. Would only need revisiting when a count-only run path lands, at which point a `psi-c` record legitimately attests a count-only run and the refusal is replaced by that path (product board item 208371871).

## Governance metadata

Alongside those fields the record carries readable **governance metadata** so it is self-describing as a disclosure record. This is: the governing data-sharing agreement -- its reference (for example `MOU-2025-0042`), the purpose of the disclosure under it (a brief readable statement of why the disclosure was made, as a HIPAA 164.528 accounting and a FERPA 99.32 record both require; a single agreement can authorize several purposes, and this names the one this exchange was for), and its expiration date -- identifying the authority for, and the purpose of, the disclosure; the `algorithm` (`psi`, which revealed matched identifiers, or `psi-c`, which revealed only a count), so the record shows whether identifiers or only a count were disclosed; the categories of data exchanged -- the payload columns this party committed as sent and received, each recorded as its name (the column actually transmitted by the metadata disclosure gate, not a name from the optional declared data dictionary) plus any data-dictionary description that dictionary supplies for it, with the no-payload case (no columns disclosed, or no matched records) recorded explicitly as an empty list rather than by ambiguous omission; and the matching basis -- the linkage fields the match keyed on, each recorded as its standardized field name and semantic type (for example `last_name`, `date_of_birth`, `ssn4`), documenting the PII on which shared membership was determined. The standardized name (not the raw source column) is the identifier the linkage keys and the standardization config both reference, so it is the anchor for tracing the basis back through standardization to the data; the basis covers only the fields the linkage keys actually reference, so a declared-but-unused field is not listed (it was not matched on). The legal agreement reference, purpose, and expiration, the algorithm, and the linkage-field set come from terms both parties validated at exchange time, so both parties' records carry consistent values for them. The payload column-name sets are not read from the (cross-checked) declared dictionary at all but from the committed payloads -- the columns the `isDisclosedToPartner` metadata gate actually transmitted -- so the readable list cannot drift from the committed bytes (a record can never read "sent nothing" while real columns were committed, the failure mode when no dictionary is authored). They are still consistent across the two parties, by a stronger route than cross-checking: the two directions' payload commitments cover byte-identical data, so one party's `payloadSent` equals the other's `payloadReceived` by construction (see [Commitment scheme](#commitment-scheme)). The data-dictionary description, looked up by committed column name, is the one governance field not cross-party-validated -- the two parties may annotate the same column differently, and a committed column the dictionary does not describe carries a bare name. The metadata is names, categories, descriptions, and references only -- never a payload value, a linkage-field value, or a matched identifier -- so the record stays free of protected data at rest, with the data exchanged bound by the commitments below rather than embedded.

## Retention and disposition pointer

Distinct from that governance metadata, the record can also carry an optional **retention/disposition pointer**: a free-text note recording where this party filed its copy of the result (the association table and any received payload) and under what retention schedule it is held or disposed of, so the record answers an auditor's first operational question -- where did the result go -- without a separate lookup. Unlike the governance fields, it is **not** drawn from the agreed terms: it is sourced from this party's own local exchange config, so it is per-party (the two parties' pointers are independent and need not match), is never exchanged with the partner, and is not folded into the agreed-terms hash. It is metadata only -- never a payload value, a linkage-field value, or a matched identifier -- and is recorded only when configured; its absence is explicit (the field is omitted, never an empty string). Configured via [`retention_disposition`](../EXCHANGE_REFERENCE.md#retention-and-disposition) in the exchange spec.

## Commitment scheme

The record commits to the data exchanged rather than embedding it: the payload columns this party sent, the payload columns it received, and -- when this party holds it (it received output) -- the association table. A bare `SHA-256` of a low-entropy result (for example an association table over identifiers; cf. the SSN brute-force warning in [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#threat-model)) would be brute-forceable by anyone holding the record, leaking the intersection. Instead each commitment is

```
commitment = HMAC-SHA-256(key = salt, message = canonical({ domain, data }))
```

with a fresh per-commitment `salt` (CSPRNG-generated, at least 128 bits) and a per-kind `domain` label that separates the three commitment kinds. Keying HMAC with the secret salt gives computational hiding (the commitment reveals nothing about the data, and a low-entropy data set cannot be brute-forced from the commitment without the salt); binding follows from the collision resistance SHA-256 lends HMAC, so the committer cannot open one commitment to two different data sets. The committed `message` is the canonical encoding (RFC 8785) of `{ domain, data }`, so both parties commit over byte-identical serializations of the same logical data (row order, encoding, and null handling all fixed) and an independent implementation reproduces the bytes; that serialization is specified in [CANONICAL_ENCODING.md](CANONICAL_ENCODING.md). The two payloads are committed in a canonical representation owned by the record format rather than the transport encoding (the wire-level "has data" discriminant is dropped; a no-data payload is the empty value). A committed payload binds the disclosed column names and the row values only -- **not** any party's internal row indices. The received-payload commitment would otherwise bind the partner's row numbers, which the holder does not retain in a reproducible form (its result file records the received values, not the partner's row numbers), leaving the holder unable to reopen its own commitment; which records matched is bound separately, by the association-table commitment -- an ordered `[our row indices, partner row indices]` pair -- whose partner side the result file retains (see the partner row-index column in [PROTOCOL.md](PROTOCOL.md#output)) so the holder can reproduce it. Because both payloads normalize to this one shape, a sender's payload-sent commitment and the receiver's payload-received commitment cover byte-identical data for the same logical payload. That byte-identical-data property lets a verifier re-supply the shared payload when it opens a commitment. It does not, on its own, let one party recompute the *other* party's commitment: the sent and received kinds use different domain labels and independent per-commitment salts, so cross-verifying a partner's record additionally requires that partner's per-commitment salt (its verification keys, which are otherwise kept private) or a purpose-built cross-commitment step -- a mechanism the deferred signing/verify work must define, not something the format delivers by itself. Today the record is local and self-attested, so commitments are never compared between parties.

## Signed receipt

The certificate-backed **signed receipt** reuses this document's canonical encoding and committed-payload shape but is a *separate* artifact from the self-attested record: it is a single **dual-signed record** carrying facts *both parties derive byte-identically*, over which each party makes an Ed25519 signature that a third party can verify against either party's view. Where the self-attested record is a per-party, salted, hiding artifact, the signed receipt is a mutually-verifiable attestation of what flowed, whose directional payload values are session-keyed MACs (see below) rather than salted commitments. The signing/exchange step that produces it, its trust model, and the fingerprint-pin-before-signature check are specified in [PROTOCOL.md](PROTOCOL.md#the-signed-receipt-step); this section fixes the byte layout so an independent implementation reproduces it.

### Receipt content

The **receipt content** is the object both parties sign. Its canonical bytes are folded into the per-signer signed bytes below; the content object has exactly these keys (the canonical encoder sorts them):

- `termsHash` -- the agreed-terms hash (unpadded base64url `SHA-256`), computed exactly as the self-attested record's `termsHash` above, so both parties produce the same value.
- `initiatorToResponderPayload` and `responderToInitiatorPayload` -- the **directional payload MACs**, each an unpadded base64url `HMAC-SHA-256` over the canonical encoding of the `<committedPayload>` that flowed in that direction, under a per-direction key derived from the session key. The MAC key is `HKDF-SHA-256` of the session key with a zero salt and the info string `psilink-signed-receipt-payload-v1:<direction>`, where `<direction>` is `initiator-to-responder` or `responder-to-initiator`; the label is prefix-free against every other session-key label. `<committedPayload>` is the record's [committed-payload shape](#commitment-scheme) (`{ columns, rows }`, the transport `has_data` discriminant dropped, a no-data payload the empty value `{ "columns": [], "rows": [] }`). Both parties hold both directions (each sends one payload and receives the other, and a sender's committed payload is byte-identical to the receiver's) *and* both hold the session key, so both compute the same two MACs; keying them by the fixed handshake direction (not by local/partner) makes the two content objects byte-identical. The MAC key is derived at sign/verify time and **never stored** in the receipt file -- the file carries only the resulting MAC value. Keying off the session key (not a bare hash) is what stops a third-party receipt holder, who has neither the session key nor the flowing data, from recomputing a directional MAC or brute-forcing a low-entropy payload from it; the no-data direction is likewise a session-keyed MAC, not a public constant, so it does not leak payload presence. The salted record commitments remain the record's hiding artifact.
- `binder` -- the per-exchange replay binder (unpadded base64url over 32 bytes): `HKDF-SHA-256` of the session key with a zero salt and the info string `psilink-signed-receipt-binder-v1:initiator`. Both parties fold in the *initiator-role* label, so both derive the same binder (see [PROTOCOL.md](PROTOCOL.md#the-signed-receipt-step)).

The salted record commitments and the association-table pairing are **not** part of the receipt content; the rationale (they are not byte-identical across parties, or not held by both) is in PROTOCOL.md.

### Receipt signature

Each party signs its own **signer-bound bytes**: the RFC 8785 canonical encoding of `{ "domain": "psilink-signed-receipt-content/v2", "content": <content>, "signer": { "fingerprint": <signer fingerprint>, "role": <signer role> } }`, where `<content>` is the shared receipt content above, `<signer fingerprint>` is the signer's OWN certificate fingerprint ([the same value a partner pins](PROTOCOL.md#signing-identity-and-certificate-pinning)), and `<signer role>` is `initiator` or `responder`. The signature is an Ed25519 signature (unpadded base64url) over those bytes under that party's signing key, verified strictly (RFC 8032, `zip215: false`) exactly as a certificate self-signature is. Binding the signer's fingerprint and role into the signed bytes makes the two `{ certificate, signature }` blocks **non-interchangeable**: a holder cannot swap them to re-attribute a direction's payload to the wrong party, since each signature binds its own signer (the shared content alone would otherwise verify under either key). A verifier reconstructs the partner's signed bytes from the shared content, the partner's *pinned* fingerprint, and the partner's role (the opposite of the local role). The `content` domain (`psilink-signed-receipt-content/v2`) is domain-separated from the certificate signature and fingerprint domains, so a receipt signature can never be confused with a certificate self-signature.

### Dual-signed record file

The dual-signed record carries a format `version` (`psilink-signed-receipt/v1`, a single recognized literal a reader rejects rather than migrates), the shared `content` above, and one `{ certificate, signature }` block per party keyed by the fixed handshake role (`initiator`, `responder`) -- so both parties write a byte-identical artifact. It holds **no payload contents and no private keys**: only public certificates, signatures, the agreed-terms hash, and the two session-keyed payload MACs (from which a holder without the session key can recover neither the payload nor its presence). Both parties' identities and the terms are bound into it -- that is its purpose, a mutually non-repudiable attestation -- so it is not anonymous; see [Receipt privacy properties](#receipt-privacy-properties) for exactly what a holder learns. The CLI writes it owner-only by default (an operator shares it by copying the file, matching the record's conservative default) and atomically (temp file then rename), to a timestamped `psilink-receipt-<stamp>.json` path sharing the exchange's `createdAt` stamp with the record, overridable via `signing.receipt_output`. Serialized as ordinary pretty JSON with a trailing newline (the human-readable persisted form, not the canonical encoding, which is only the bytes that are signed), like the self-attested record.

### Receipt privacy properties

A third-party holder of the dual-signed record (without the session key and without either party's retained input or result) learns:

- the two parties' `identity` strings (each carried in a certificate the record embeds);
- that an exchange occurred between them under a given agreed-terms hash;
- the mutual, non-repudiable attestation itself -- each party's Ed25519 signature binding it to the terms and this exchange.

It does **not** learn:

- payload contents -- the directional payload values are session-keyed MACs, not the data;
- payload presence -- the no-data direction is a session-keyed MAC, not a public constant, so a holder cannot tell an empty direction from a non-empty one, nor recover a low-entropy payload by brute force (both require the session key, which the record never carries);
- any private key -- only public certificates travel in the record.

The session key, the per-direction MAC keys, and the flowing data all stay off the receipt, so recovering a payload from it is not possible for anyone but the two data holders. Retention and access control of the receipt remain the holder's responsibility.

## Record and verification-keys files

The artifact is split across two files, on the CLI and the web app alike. The **record** holds the commitments and the non-secret summary above; it does not contain the matched data or the salts, so it does not reveal -- or allow brute-force recovery of -- the intersection, and it is the part safe to retain or hand to an auditor (and, once signing is built, to sign and swap). The **verification keys** hold, per commitment, only the salt (base64url, at least 128 bits) needed to open that commitment later; the file carries its own format version (`psilink-exchange-keys/v1`), rejected-if-unrecognized like the record's. The keys carry **no snapshot of the committed data**: unlike an earlier self-contained design, they are not a second at-rest copy of the matched data (see [No data snapshot in the keys](#no-data-snapshot-in-the-keys-data-minimization) below). They are still sensitive and must be kept private: a salt together with the record's commitment can open -- and brute-force a low-entropy -- committed value, notably the intersection, so anyone holding both the keys and the record could open the commitments.

Both files are written owner-only. The CLI writes them atomically (temp file then rename; see [CREDENTIAL_STORAGE.md](CREDENTIAL_STORAGE.md#posix-write-discipline)) to a timestamped default path (overridable, or skipped with `--no-record`): the record as `psilink-record-<stamp>.json` and the keys as `<name>.keys.json` alongside it. The web app offers the two as two separate downloads -- the shareable record and the private verification keys -- mirroring the CLI split, so the record can be shared with an auditor without the keys. The record is written owner-only even though it is shareable: "shareable" means it carries no secret material (no salts, no matched data), not that it is world-readable on disk. It still names both parties, the governing agreement, the data categories, this party's exposed-record count, and (when present) the result size in cleartext (see below), so the conservative default keeps it private to the owner; share it by copying the file, not by loosening its permissions. The verification keys are not shareable at all -- keep them private and separate from the record.

### No data snapshot in the keys (data minimization)

The verification keys deliberately hold only salts. An earlier design stored, per commitment, a snapshot of the exact committed data alongside the salt: that made verification self-contained but persisted a **second copy of the matched data at rest** -- a copy an operator would not expect from a tool built for a transient exchange, and outside any accounting of where data is stored. Dropping the snapshot removes that copy; the matched data is then persisted only in the result file the operator chose to write. The cost is that verification is no longer self-contained: it must **re-supply** the committed data (from the holder's own retained input and result) and re-canonicalize it. Because a commitment is `HMAC-SHA-256(key = salt, message = canonical({ domain, data }))`, the re-supply path MUST reproduce the exact RFC 8785 canonical bytes (see [CANONICAL_ENCODING.md](CANONICAL_ENCODING.md)) the original commit used -- the committed-payload shape (column names and row values) for a payload, or the `[our row indices, partner row indices]` pair for the association table -- or every opening fails. The committed payload holds no row indices, so it re-supplies from the retained input, config, and result; the association table's partner side re-supplies from the partner row-index column the result file carries for every matched record. Reproducing from those files requires them **unmodified**: the commitments bind the matched-row order the result was written in, so a re-sorted or re-exported result no longer reproduces them (the "hash a file, check it later" constraint any file verifier has). Two reproduction edges are left for the verify consumer to handle, and are flagged for its security review: an identifier column with duplicate values makes the our-side row index ambiguous when a result row is mapped back to an input row, and the order-dependence could be lifted by committing the pairing as a canonically ordered set -- deferred, since it changes the committed bytes and warrants its own review. That re-supply path is owned by the verification consumer (the deferred verify-receipt work); this document fixes only the artifact shape it consumes.

A corollary of re-supply: opening a commitment is a data-holder capability. A third party who lacks the holders' retained input and result can still check the record's signatures and identity bindings (once signing exists) but cannot open the commitments itself. This is the intended consequence of dropping the snapshot -- previously an opening-file holder could open commitments (the very second-copy leak this removes); now only a party that already has the data can. The salts-only keys hand no one the data.

### Keys written by default (with the record)

Writing the verification keys is default-on: whenever recording is enabled the keys are written alongside the record under the single `--no-record` control (the web app offers both downloads together the same way), with no separate suppressor for the keys alone. This is a deliberate default, worth recording because the keys are private, non-shareable material provisioned for a consumer that does not exist yet -- the deferred verify-receipt work is the only thing that opens them.

The default turns on an asymmetry in recoverability. Each salt is CSPRNG-generated at commit time and stored nowhere but the keys file, so if the keys are not written at the exchange the salts cannot be reconstructed and the commitments can never be opened -- and the exchange generally cannot be re-run to mint fresh ones. An operator cannot know at exchange time whether they will later want to open a commitment, because the verifier that would consume the keys is not built; writing the keys now is the only way to keep that capability open, while not writing them forecloses it irrecoverably. Default-on costs one extra owner-only file the operator can delete at any time; default-off costs a capability that cannot be recovered after the fact.

The keys being cheap and low-risk to retain is what makes that asymmetry decisive. After the snapshot drop (above) they hold only salts, no matched data, so default-on no longer persists a second copy of protected data; the record is itself written owner-only and default-on, so a second owner-only, salts-only file is a marginal at-rest increase, not a new category of exposure. The keys' one real risk -- that with the record they open the commitments (the brute-force concern above) -- materializes only if both files leak, and both are 0600; an operator who wants neither uses `--no-record`, which suppresses the pair. Keeping them under one control also avoids a record whose commitments can never be opened -- a partial state with no use today and a trap for the future verifier. This default would only need revisiting if the keys came to hold more than salts, or a concrete need emerged to retain the shareable record while deliberately withholding its opening material.

## Privacy properties

Privacy of the record itself: it stores the terms hash, both `identity` strings, this party's exposed-record count, the result size (when present), any retention/disposition pointer, and the governance metadata above (the agreement reference, purpose, and expiration, the algorithm, and the data categories) in cleartext, so anyone who reads it learns that an exchange with that partner occurred, under which agreement, over what categories of data, how many records this party exposed, where it filed the result (when a pointer is configured), and -- when both sides were entitled to it -- its size, but no protected values. As with the receipt, retention and access control of the record (and the strict protection of the verification keys) are the holder's responsibility.
