# Exchange record format

This document specifies the self-attested exchange record: the local audit artifact each party produces at the conclusion of every successful exchange. It covers the record and opening file shapes, the format version and reader-rejects-unknown rule, the commitment scheme and the agreed-terms hash, the readable governance metadata, the result-size and per-direction disclosure framing, and the record's privacy properties. It does not cover the PSI protocol that produces the exchange (see [PROTOCOL.md](PROTOCOL.md)), the exchange-agreement format that supplies the record's governance fields (see [EXCHANGE_REFERENCE.md](../EXCHANGE_REFERENCE.md)), or the canonical byte encoding the commitments and the agreed-terms hash are computed over (see [CANONICAL_ENCODING.md](CANONICAL_ENCODING.md)). The deferred Signed Exchange Receipts work (see [ROADMAP.md](../ROADMAP.md)) reuses this record's commitment scheme and on-disk format, so the commitment-scheme and file-shape sections below are written to stand on their own. Intended readers are security auditors, external implementors, compliance reviewers, and developers.

> **Implemented.** Unlike the deferred signed receipt (see [Non-repudiation](PROTOCOL.md#non-repudiation)), the self-attested record described here is produced today, at the conclusion of every successful exchange. It is a standalone disclosure-log entry, deliberately unsigned: a session-derived MAC is not non-repudiation (the two signing modes are specified there), so the unsigned record is an honest, local audit artifact rather than a receipt that overstates what it proves. Signing this record is the separate, deferred Signed Exchange Receipts work, which reuses its commitment scheme and on-disk format.

At the conclusion of every successful exchange, each party produces a self-attested record of what it disclosed. The record is designed to stand on its own as a disclosure-log entry: an operator can use it to populate a HIPAA accounting of disclosures or a FERPA disclosure record without separately retaining and re-matching the original linkage-terms config. It is a local audit artifact, explicitly **not** a signed or non-repudiable receipt and **not** evidence against the partner. It is built from data both sides already hold, so it needs no private key and adds no protocol round-trip.

## Record fields

The record captures: a format `version` (a single recognized literal for v1, which a reader rejects rather than migrates if unrecognized); the local timestamp (an ISO 8601 datetime in UTC, ending in `Z`; a timestamp carrying a numeric offset is rejected, so the format keeps one canonical timestamp form across implementations); a hash, over the canonical encoding, of the agreed exchange terms (both parties' linkage terms in a fixed, canonical-sorted order, so both parties and an independent third party derive the same hash); both parties' self-asserted `identity` strings; the number of records this party contributed to the exchange (`recordsExposed`) -- its own input row count, recorded on every record as a per-direction statement of what it put in; the result size, but only in the both-output case; an optional self-facing retention/disposition pointer recording where this party filed its copy of the result and under what retention schedule (see below); and a per-exchange binding nonce (CSPRNG-generated, at least 128 bits) so two runs with identical terms still produce distinct records. The binding nonce is the unsigned record's per-exchange binder and is distinct from the per-commitment salts below; do not conflate them. The deferred signing work will add a stronger session-key-derived binder.

## Result size and per-direction disclosure

The two disclosure figures -- `recordsExposed` and the result size -- are gated differently, and the difference is deliberate. `recordsExposed` is the count of records this party contributed to the exchange -- its input row count, counting every contributed record rather than only those that resolve to a usable linkage key, so it is an honest upper bound on what the party exposed rather than a derived match figure; it is known from the party's own input alone, so it does not depend on the result or on the partner, and it is recorded for every party regardless of role. The result size (the intersection cardinality) is recorded only when both parties' agreed terms have them both receive output, so it is stored only when both sides are entitled to the result, and omitted when only one party does. The gate is the terms agreement (entitlement), not what a party happens to learn during the exchange: in a single-output exchange the helper that does not receive output can still observe its match count during the [clean cascade](PROTOCOL.md#psi), but the record deliberately does not surface it. Privacy here is enforced by what the tool writes down, not by what is theoretically discoverable; the helper's own outbound disclosure is carried by `recordsExposed` instead. Framing the helper's figure as "records I exposed" rather than "the result size" is also what keeps the record meaningful under a future matching algorithm that discloses neither the result size nor the partner's set size -- the figure comes only from the party's own input, which any such algorithm still leaves it knowing.

Reading the disclosure direction off a record: a PSI exchange discloses the membership of matched records mutually -- each party reveals its matched membership to the other to make the match -- so direction is not a flag but a reading of the fields together. `recordsExposed` is the magnitude of what this party put in (its outbound exposure); the `algorithm` says what kind of disclosure occurred (`psi` revealed matched identifiers, `psi-c` only a count); `payloadSent` and `payloadReceived` give the payload categories per direction; and the presence of the association-table commitment (below) marks whether this party itself received the matched pairing. So "what this party disclosed" reads off `algorithm` + `recordsExposed` + `payloadSent`, and "what it received" off `payloadReceived` and the association-table commitment -- without consulting the output configuration, which the record does not carry.

## Governance metadata

Alongside those fields the record carries readable **governance metadata** so it is self-describing as a disclosure record. This is: the governing data-sharing agreement -- its reference (for example `MOU-2025-0042`), the purpose of the disclosure under it (a brief readable statement of why the disclosure was made, as a HIPAA 164.528 accounting and a FERPA 99.32 record both require; a single agreement can authorize several purposes, and this names the one this exchange was for), and its expiration date -- identifying the authority for, and the purpose of, the disclosure; the `algorithm` (`psi`, which revealed matched identifiers, or `psi-c`, which revealed only a count), so the record shows whether identifiers or only a count were disclosed; the categories of data exchanged -- the payload columns this party committed as sent and received, each recorded as its name (the column actually transmitted by the metadata disclosure gate, not a name from the optional declared data dictionary) plus any data-dictionary description that dictionary supplies for it, with the no-payload case (no columns disclosed, or no matched records) recorded explicitly as an empty list rather than by ambiguous omission; and the matching basis -- the linkage fields the match keyed on, each recorded as its standardized field name and semantic type (for example `last_name`, `date_of_birth`, `ssn4`), documenting the PII on which shared membership was determined. The standardized name (not the raw source column) is the identifier the linkage keys and the standardization config both reference, so it is the anchor for tracing the basis back through standardization to the data; the basis covers only the fields the linkage keys actually reference, so a declared-but-unused field is not listed (it was not matched on). The legal agreement reference, purpose, and expiration, the algorithm, and the linkage-field set come from terms both parties validated at exchange time, so both parties' records carry consistent values for them. The payload column-name sets are not read from the (cross-checked) declared dictionary at all but from the committed payloads -- the columns the `isDisclosedToPartner` metadata gate actually transmitted -- so the readable list cannot drift from the committed bytes (a record can never read "sent nothing" while real columns were committed, the failure mode when no dictionary is authored). They are still consistent across the two parties, by a stronger route than cross-checking: a sender's sent-payload commitment and the receiver's received-payload commitment cover byte-identical data, so one party's `payloadSent` equals the other's `payloadReceived` for the same exchange by construction. The data-dictionary description, looked up by committed column name, is the one governance field not cross-party-validated -- the two parties may annotate the same column differently, and a committed column the dictionary does not describe carries a bare name. The metadata is names, categories, descriptions, and references only -- never a payload value, a linkage-field value, or a matched identifier -- so the record stays free of protected data at rest, with the data exchanged bound by the commitments below rather than embedded.

## Retention and disposition pointer

Distinct from that governance metadata, the record can also carry an optional **retention/disposition pointer**: a free-text note recording where this party filed its copy of the result (the association table and any received payload) and under what retention schedule it is held or disposed of, so the record answers an auditor's first operational question -- where did the result go -- without a separate lookup. Unlike the governance fields, it is **not** drawn from the agreed terms: it is sourced from this party's own local exchange config, so it is per-party (the two parties' pointers are independent and need not match), is never exchanged with the partner, and is not folded into the agreed-terms hash. It is metadata only -- never a payload value, a linkage-field value, or a matched identifier -- and is recorded only when configured; its absence is explicit (the field is omitted, never an empty string). Configured via [`retention_disposition`](../EXCHANGE_REFERENCE.md#retention-and-disposition) in the exchange spec.

## Commitment scheme

The record commits to the data exchanged rather than embedding it: the payload columns this party sent, the payload columns it received, and -- when this party holds it (it received output) -- the association table. A bare `SHA-256` of a low-entropy result (for example an association table over identifiers; cf. the SSN brute-force warning in [SECURITY_DESIGN.md](../SECURITY_DESIGN.md#threat-model)) would be brute-forceable by anyone holding the record, leaking the intersection. Instead each commitment is

```
commitment = HMAC-SHA-256(key = salt, message = canonical({ domain, data }))
```

with a fresh per-commitment `salt` (CSPRNG-generated, at least 128 bits) and a per-kind `domain` label that separates the three commitment kinds. Keying HMAC with the secret salt gives computational hiding (the commitment reveals nothing about the data, and a low-entropy data set cannot be brute-forced from the commitment without the salt); binding follows from the collision resistance SHA-256 lends HMAC, so the committer cannot open one commitment to two different data sets. The committed `message` is the canonical encoding (RFC 8785) of `{ domain, data }`, so both parties commit over byte-identical serializations of the same logical data (row order, encoding, and null handling all fixed) and an independent implementation reproduces the bytes; that serialization is specified in [CANONICAL_ENCODING.md](CANONICAL_ENCODING.md). The two payloads are committed in a canonical representation owned by the record format rather than the transport encoding (the wire-level "has data" discriminant is dropped; a no-data payload is the empty value), so a sender's payload-sent commitment and the receiver's payload-received commitment cover byte-identical data for the same logical payload. That symmetry is what later lets the deferred signing work have the two parties cross-check each other's records; today the record is local and self-attested, so the commitments are never compared between parties.

## Record and opening files

The artifact is split across two files. The **record** holds the commitments and the non-secret summary above; it does not contain the matched data or the salts, so it does not reveal -- or allow brute-force recovery of -- the intersection, and it is the part safe to retain or hand to an auditor (and, once signing is built, to sign and swap). The **opening data** holds, per commitment, the salt and a snapshot of the exact committed data -- the material needed to reveal (open) a commitment later. Because anyone holding the opening data can recompute the commitments, it is as sensitive as the matched data itself and must be kept private. The CLI writes both files atomically (temp file then rename; see [CREDENTIAL_STORAGE.md](CREDENTIAL_STORAGE.md#posix-write-discipline)) and owner-only, to a timestamped default path (overridable, or skipped with `--no-record`). The web app instead offers them as a single combined download: one JSON file with a `public` part (the record) and a `private` part (the opening). Because that combined file embeds the opening, the whole file is as sensitive as the opening -- it is not the safe-to-share record on its own. The CLI's two-file split and the web's single-file download are not yet unified. The record is written owner-only too, even though it is shareable: "shareable" means it carries no secret material (no salts, no matched data), not that it is world-readable on disk. It still names both parties, the governing agreement, the data categories, this party's exposed-record count, and (when present) the result size in cleartext (see below), so the conservative default keeps it private to the owner; share it by copying the file, not by loosening its permissions.

## Privacy properties

Privacy of the record itself: it stores the terms hash, both `identity` strings, this party's exposed-record count, the result size (when present), any retention/disposition pointer, and the governance metadata above (the agreement reference, purpose, and expiration, the algorithm, and the data categories) in cleartext, so anyone who reads it learns that an exchange with that partner occurred, under which agreement, over what categories of data, how many records this party exposed, where it filed the result (when a pointer is configured), and -- when both sides were entitled to it -- its size, but no protected values. As with the receipt, retention and access control of the record (and the strict protection of the opening data) are the holder's responsibility.
