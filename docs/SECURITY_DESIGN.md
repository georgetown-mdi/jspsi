# PSI-Link security

This document covers the threat model, authentication design, and channel security for PSI-Link exchanges, and introduces the private set intersection (PSI) protocol on which the privacy guarantee rests at a conceptual level. It does not specify the PSI and PSI-C algorithms or the wire-level key-exchange protocol in detail (see [PROTOCOL.md](spec/PROTOCOL.md)), the network channels over which exchanges run (see [COMMUNICATION.md](COMMUNICATION.md)), or CLI configuration for authentication (see [CLI.md](CLI.md)). It covers the channel-security controls only at the level of what each protects against and why the design is sufficient; their construction, derived constant values, and enforcement points are specified in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md). Intended readers are security teams and compliance officers.

# Overview

PSI-Link protects an exchange with three independent layers:

1. **The PSI protocol protects the data itself.** Before any record leaves a participant's machine it is encrypted under that party's own ephemeral key, and the protocol then layers the partner's key on top, so each side only ever handles the other's records in a form it cannot decrypt. Neither side can recover the other's underlying values; both learn only which records the two of them hold in common. This is what makes the approach privacy-preserving regardless of which channel carries the traffic. See [Private set intersection](#private-set-intersection) below.

2. **Key-exchange authentication proves you are talking to the right partner.** For recurring exchanges the two parties hold a shared secret, established once out of band, and prove to each other that they both hold it without ever sending it over the wire. The secret is rotated after every successful exchange, so it is never reused. Authentication is a hard gate: if it fails, no data is exchanged. See [Authentication](#authentication).

3. **Transport encryption protects the data in transit.** The underlying channel - an SSH/SFTP connection, a WebRTC link, or a network-mounted share - encrypts traffic as it crosses the network. For zero-setup exchanges, which carry no shared secret, this transport layer is the sole protection, which is why trust there rests on whoever administers the server or share. See [Channel security](#channel-security).

A fourth layer - application-layer message encryption keyed from the key-exchange session, so that even the operator of an SFTP server or shared drive sees only opaque ciphertext - is wired into recurring (authenticated) CLI exchanges: immediately after authentication the CLI wraps the connection in an AEAD decorator, so every PSI frame is encrypted on the wire end to end. It does not cover zero-setup exchanges (which carry no session key) or the web application's application layer. Its mechanism and wire format, the inbound frame-size bound now enforced on the transport read, and the one remaining channel-hardening follow-up that gates the first release, are summarized under [Channel security](#channel-security) and specified in full in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md).

The remainder of this document treats each layer in turn, beginning with the PSI protocol the other two are built to protect.

# Private set intersection

Private set intersection (PSI) is the cryptographic primitive the privacy guarantee rests on. It lets two parties compute the overlap between their datasets - the records they have in common - while revealing nothing about the records they do not.

The intuition is a layered, order-independent ("commutative") encryption. Each party holds an ephemeral key, generated fresh for the exchange and never shared. Each side encrypts its own linkage keys under its own key, and the protocol applies the partner's key as a second layer, so a record is only ever seen by the other party in a form that party cannot decrypt. Because the scheme is commutative, two values that started out equal remain equal once both keys have been applied, which is what lets the parties recognise shared records by comparing encrypted forms alone. Neither side can strip the other's key to recover a plaintext value, and records outside the intersection are never revealed.

PSI-Link uses a lightly modified build of OpenMined's [PSI](https://github.com/OpenMined/PSI) - which in turn layers over Google's Private Join and Compute - as this primitive. The base function is run repeatedly over a sequence of linkage keys to build the association map between matched records. A cardinality-only variant, PSI-C, reveals the size of the overlap without revealing which members are shared; it is designed but not yet implemented.

Two properties of the primitive are deliberately not hidden, and both are load-bearing for the threat model that follows:

- **Set sizes are revealed.** The base PSI function inherently leaks how many records each party holds. This is treated as acceptable for linking administrative data, where membership and identity are sensitive but the size of a database is not.
- **The result is only as private as the linkage keys.** Because each party learns the linkage key for every shared member, a weak key - for example one built from a single identifier - can leak membership through a differencing or brute-force attack. Combining several PII elements into each key is what keeps "you learn only the overlap" a meaningful guarantee.

The algorithm itself - role assignment, the encryption and key-removal steps, the matching cascade, and the PSI-C design - is specified in [PROTOCOL.md](spec/PROTOCOL.md#psi-base-function). The two properties above are developed further in the [Threat model](#threat-model).

# Threat model

The system is designed to be utilized by partner agencies with signed data sharing agreements, and the primary goal of the security design is to prevent parties from learning anything about each other's data beyond the mapping between shared members. With that in mind, an honest-but-curious threat model has been adopted which assumes that partners are not actively tampering with inputs, but that it is still beneficial to minimize what is disclosed.

For each successive use of the base PSI function, information is revealed to each party. For PSI, this includes the key that links individual members. For PSI-C, the cardinality of that key can be learned. This implies that linkage keys could be chosen to reveal sensitive information through a differencing attack in order to reveal membership, so it is crucial that both parties review the linkage keys before agreeing to use them.

A malicious adversary cannot learn anything beyond what the PSI protocol reveals, but they can attempt a membership attack using specific inputs. For instance, if a statistical linkage key included only social security numbers, it would be easy to brute-force. To protect against this, it is recommended that keys combine multiple elements of personally identifiable information (PII). Even with complex linkage keys, membership attacks are still possible, but only if a target's PII is already known.

Separately from adversarial attacks, note that the PSI base function used inherently leaks the size of each parties' sets. This is considered acceptable for the use-case of linking administrative data, as it is individual membership and identifying information that is considered sensitive and not the size of the database.

When using public services to facilitate scheduled exchanges, some metadata around the exchange is leaked such as who is conducting the exchange and when. Parties are encouraged to stand up their own services when necessary and resources to facilitate this are available.

A separate trust boundary lies in the configuration file itself. A `psilink.yaml` is designed to be shared -- it records `@path` references to credentials rather than the credentials themselves, so it carries no secrets and is safe to commit or send to a partner -- but that safety protects the file's author, not whoever later runs it. Loading a configuration you did not author is equivalent to running its referenced files as your own credentials: when an exchange runs, each `@path` credential reference is read from local disk with your privileges, and a resolved credential -- in the CLI today, the SFTP password -- is sent to the host the configuration names, so a substituted configuration can turn such a reference into a read of an arbitrary local file, a private key for instance, delivered to a host of the author's choosing. Reaching this requires loading a wholesale attacker-authored configuration and running an exchange against it; it cannot be introduced through an invitation, which carries no credential by construction (see [Invitation contents and confidentiality](#invitation-contents-and-confidentiality)). The operational specifics -- which fields are read, and how to treat a configuration from an untrusted source -- are in [CLI.md](CLI.md#configuration).

# Authentication

Before establishing connections, parties need to ensure that they are communicating with the correct partner. Recurring exchanges (see [User journey](DESIGN.md#user-journey)) use a pre-shared secret and an authenticated key exchange for application-layer authentication. Zero-setup exchanges rely instead on transport-layer authentication.

## Recurring exchange authentication

In order to share secrets, one party generates a random cryptographic token using an available cryptography library and shares it with their partner using a trusted, existing communication channel such as secure email. At the start of the exchange, both parties must execute an authenticated key exchange -- an ephemeral X25519 Diffie-Hellman keyed together with the shared secret and an explicit mutual confirmation (see [Key-agreement design](#key-agreement-design)).

The shared secret is automatically rotated after each successful authentication handshake, before the data exchange begins. Both parties independently derive the same replacement token from the key-exchange session key via HKDF - no extra round-trip is required. Each party is responsible for persisting the new token immediately after the handshake completes (the CLI writes it to the key file). If that write fails (for example, due to a disk error or clock skew that causes the post-handshake expiry check to fail on one side only), the two parties may hold different tokens; see [Out-of-sync tokens](CLI.md#out-of-sync-tokens) for recovery steps. If the data exchange subsequently fails, both parties already hold the rotated token and can retry without re-inviting. If a secret is lost after rotation, a new invitation can be generated from the existing configuration to re-establish a shared secret (see [Recovery](CLI.md#recovery)).

Tokens generated for invitations carry a bounded lifetime. A default expiration window of 1 hour is used if none is provided by the inviting party. The inviting party may override this window (the CLI's `--expires-in`), up to a maximum of one year. The ceiling is deliberately generous rather than tight: recurring exchanges often run only monthly, and an invitation may need to outlast several months of operational breakage before a re-invite, against agency credential-rotation schedules that in practice often go unenforced. It is nonetheless a hard bound, so an erroneous override cannot make the setup secret effectively permanent and defeat the bounded-lifetime property. Because the invitation token is rotated on first use, it functions as a one-time setup credential: its window of validity is the period between generation and acceptance. The inviting party can close that window early rather than wait out the expiry. Online, this is automatic: the pending secret lives only in the inviter's memory during the wait and is discarded when the inviter cancels, the connection times out, or the accept-timeout is reached (see [Online invitation](CLI.md#online-invitation)). Offline, where the inviter instead persists the pending secret to a key file, deleting that key file before its expiry withdraws the invitation -- the offline key exchange cannot complete unless the inviting party still holds the secret, so the copy carried in the forwarded invitation becomes inert (see [Abandoning a pending offline invitation](CLI.md#abandoning-a-pending-offline-invitation)). This deliberate withdrawal is distinct from the [Compromise response](#compromise-response) for a leaked token: a withdrawn invitation need not have been exposed at all, only no longer wanted, and it removes the pending invitation's key material only -- not a configuration or a persistent key a recurring exchange relies on. Replacement tokens generated by rotation carry no expiration by default, making them suitable for recurring scheduled exchanges without further coordination; an operator who needs a maximum token age regardless of exchange cadence sets [`token_max_age_days`](EXCHANGE_REFERENCE.md#authenticationtoken_max_age_days), described under [Token age and rotation policy](#token-age-and-rotation-policy).

The wire-level key-exchange specification -- construction, key schedule, message encoding, and interoperability notes -- is in [PROTOCOL.md](spec/PROTOCOL.md#x25519-authenticated-key-exchange).

In the exchange specification (`psilink.yaml`), this mechanism is configured in a top-level `authentication` block -- channel-agnostic, and a sibling of the `signing` block rather than nested inside `connection` (see [EXCHANGE_REFERENCE.md](EXCHANGE_REFERENCE.md#authentication)). `authentication` and `signing` are the two partner-trust mechanisms, and they are kept as separate blocks deliberately, because they have opposed lifetimes and trust models. The shared secret here is symmetric, low-value, rotated every exchange, and cheaply recovered by re-invite; the pinned signing certificate (see [Receipt signing identities](#receipt-signing-identities)) is asymmetric, stable for its whole life, high-value, and required for third-party non-repudiation of receipts. Holding the stable, high-value signing key apart from the rotating, low-value auth secret keeps each mechanism's blast radius contained and matches their opposed lifecycles -- they already live in separate files (`signing-identity.json` vs `.psilink.key`) on separate rotation schedules. A future protocol version could let the signing identity subsume authentication -- static-key mutual authentication that drops the pre-shared secret, part of why `signing` earns its own top-level place -- but that is a protocol/handshake change and is out of scope for the current shared-secret design.

### Token format and entropy

Tokens are base64url-encoded 32-byte values. Invitation tokens are generated using `crypto.getRandomValues`, giving 256 bits of entropy. Rotation tokens are derived from the 32-byte key-exchange session key via HKDF-SHA-256 (info string `"psilink-shared-secret-rotation-v1"`, output 32 bytes) and carry the same 256-bit security level as the underlying session key.

### Invitation contents and confidentiality

An invitation carries the linkage terms and the short-lived setup secret, and it may also carry a connection endpoint: a public locator (a PeerJS signaling URL, an SFTP host and port, or a file-drop directory) that tells the acceptor where to rendezvous. A locator of this kind is not a secret -- it names a meeting point, not a means of authenticating at it -- so embedding it does not weaken the invitation. The credential-free property is enforced by the schema: the endpoint has no field for a password, private key, key file, or PeerJS API key, and an invitation carrying any such field is rejected when decoded. Connection credentials are therefore never transmitted in an invitation; each party configures its own. By the same token, the endpoint carries no server-identity material (an SSH host-key fingerprint, for example): how a party authenticates the server it reaches is unchanged by the hint and continues to rest on the transport layer and the server administrator, as described under [Transport-layer authentication](#transport-layer-authentication).

On the web, the inviter derives those linkage terms from its own data file rather than authoring them by hand (see [COMMUNICATION.md](COMMUNICATION.md#web-invitation)): the default terms, narrowed to the keys the file's columns can satisfy. The embedded terms therefore reflect the inviter's column *shape* -- which of the default field types (name, date of birth, SSN, and so on) the file carries -- but never any row *value*; the file is read in the browser and never uploaded. This is presence-only disclosure to the invited recipient, who must hold matching columns to link at all, and it stays within the out-of-band confidential-invitation model this section describes.

The decoder also bounds the *size* of every attacker-influenceable field, the invitation-surface sibling of the transport frame-size and directory-listing bounds described under [Channel security](#channel-security). The token's 4-byte checksum detects transcription errors only and is computable by anyone, so it is no barrier to a crafted payload; without a size bound a checksum-valid token could carry a multi-megabyte `identity`, tens of thousands of linkage keys, or an oversized endpoint host, accepted and rendered to briefly jank the recipient (and, for a WebRTC host, to aim the recipient's own browser's signaling WebSocket at an arbitrary host -- SSRF-shaped only, since the inviter is already the rendezvous peer and no secret is sent there). The decoder refuses an over-long encoded string at the boundary, before it is base64-decoded or parsed, and caps each free-text and array field (`identity`, the linkage-key/field counts and names, the endpoint host, and the other partner-controlled names) with a generous maximum. These are defense-in-depth, not semantic limits: every bound is set far above any real invitation, so legitimate tokens decode unchanged, and the impact they close is a low-severity, self-inflicted nuisance on a recipient who opened the attacker's link, bounded already by the browser's URL-length limit.

Beyond size, terms validation bounds the *complexity* of one class of partner-supplied content: the regular-expression patterns in linkage-term element transforms (`replace_regex`, `extract_regex`, `filter_regex`, `split_on`, and the pattern `parse_date` expands from its format). Such a pattern is compiled and run per row over the acceptor's full dataset, so on a backtracking engine a crafted catastrophic-backtracking (ReDoS) pattern would hang the single JavaScript thread -- freezing the web acceptor's browser tab or hanging the CLI acceptor's process. The impact is a denial of service, and it is not meaningfully self-inflicted: consent is no defense, because a human cannot vet a pattern string for catastrophic backtracking, and authentication does not help either, because the handshake proves the counterparty's identity, not that the terms they authored are benign. The pattern cannot be rewritten -- PSI requires both parties to derive byte-identical keys and the terms are hashed into the exchange-agreement receipt, so a unilateral rewrite both breaks matching and is detected as a terms mismatch. The fix is by construction: these patterns execute under a linear-time regex engine (RE2 semantics) rather than the backtracking `RegExp`, so catastrophic backtracking is impossible, not merely screened for. The permitted pattern dialect is pinned normatively in the protocol so both parties evaluate every pattern identically; a pattern outside it -- one the engine cannot compile, such as a backreference or lookaround -- is rejected at terms validation, before any row is processed (fail closed), with no fallback to the backtracking engine. The engine is pure JavaScript, so the CLI and web builds run the same engine and derive byte-identical keys. This replaces an earlier best-effort static screen; the engine, the dialect, the conformance gate, and the removed heuristic are specified in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md#transform-regex-linear-time-dialect) and the dialect itself in [PROTOCOL.md](spec/PROTOCOL.md#transform-regular-expression-dialect).

Terms validation bounds *magnitude* the same way, for the partner-supplied transform-parameter values whose size drives unbounded per-row work. A `pad_left` `length` left-pads each row's value and so allocates a string of that size per row, so a crafted `1e9` would exhaust the acceptor's memory the way an unbounded regex would exhaust its CPU. A `parse_date` `inputFormat` is expanded into a regex compiled per row, so a long format burns CPU per row, and its `outputFormat` allocates a per-row output of the format's size -- both length-capped; the raw regex patterns are likewise length-capped to bound their per-row compile cost. The shape risk that expanded regex once posed (adjacent variable-width groups that backtrack) is now closed by the linear-time engine above, not a separate screen. Each magnitude vector is the same denial of service reachable from the same counterparty, equally not self-inflicted, since a human cannot vet a numeric pad length or a format string. Each is capped at a generous ceiling on every parse path, rejecting an out-of-range value before any row is processed; the constants and their enforcement point are specified in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md#unbounded-transform-parameter-rejection).

The invitation must nonetheless be treated as confidential, because it carries the setup secret -- the established secret an attacker needs to authenticate as you. This is sharpened by the web rendezvous flow, in which both parties derive the coordination-server rendezvous id deterministically from that shared token. An attacker who obtains the invitation therefore learns both the authentication secret and the rendezvous id, and could reach the meeting point first to mount a man-in-the-middle attack before the intended partner arrives. In the web flow the inviter's browser additionally holds the secret in memory -- in page state, to derive its own rendezvous id and run the exchange -- for the lifetime of the tab. This is the same in-memory exposure, and within the same trust boundary, as the encoded invitation the page already displays for copying: a same-origin script (an XSS) able to read one could already read the other, so the discrete field adds no exposure the displayed invitation does not. The secret is never sent to a backend. This is the standard out-of-band-channel trust model: the security of the exchange rests on the invitation reaching only its intended recipient. Forward invitations only over a trusted channel (for example, secure email), and treat a leaked invitation as a compromise -- generate a fresh one (see [Recovery](CLI.md#recovery)) rather than reuse it.

## Transport-layer authentication

Transport-layer authentication is the defined path for zero-setup exchanges only. It is not an option for recurring exchanges: the `exchange` command requires a key file (`.psilink.key`) and will abort if one is absent.

For zero-setup exchanges, parties rely on transport-layer authentication (DTLS for WebRTC, SSH for SFTP, and commonly SMB or krb5p for file-drops). For an SFTP connection or a network-mounted file-drop, user and path management on the server-side limit who is able to listen in. This offloads trust to the server's administrator to ensure that the directory is specific to the exchange and cannot be accessed by other users. This method may be preferable if managing an additional encryption key is perceived as too burdensome, even though it is less secure overall. Users who wish to establish a persistent shared secret are encouraged to bootstrap one (see [Bootstrapping a shared secret](#bootstrapping-a-shared-secret)).

On the SFTP channel the server's identity can additionally be pinned. When an operator sets `connection.server.host_key_fingerprint` to the server's OpenSSH SHA256 host-key fingerprint, every SFTP connection verifies the presented host key against the pin before authentication and aborts on a mismatch, so a man-in-the-middle or a substituted server presenting its own host key is detected rather than silently trusted. The fingerprint is non-secret. It can be pinned out-of-band, exactly as the shared secret and the signing fingerprint are -- the invitation endpoint deliberately cannot carry it (the endpoint schema rejects `host_key_fingerprint`; see [Invitation contents and confidentiality](#invitation-contents-and-confidentiality)), so an attacker who seeds a configuration cannot pin their own key. An operator who cannot obtain the fingerprint out-of-band can instead establish it on a deliberate first connection, ssh-style: the first interactive run of any command that opens the SFTP connection (`exchange`, an online `invite`/`accept`, or a zero-setup exchange) shows the presented fingerprint, asks the operator to confirm, and pins it, after which every subsequent run verifies it silently. The no-pin default is fail-closed: a connection with no pin and no first-use confirmation is refused rather than trusted, and an automated (non-interactive) run with no pin fails closed rather than prompting or silently accepting. A legitimately rotated key is never auto-accepted -- the operator verifies the new fingerprint out-of-band and re-pins deliberately (clearing the stored pin and re-confirming, as one edits `known_hosts` for ssh). This applies to the CLI `sftp` channel only -- a file-drop makes no SSH connection, and the web path does not run the SSH host-key check. The construction -- the fingerprint format, the pre-authentication enforcement point, the fail-closed behavior, and the provider-options interaction -- is specified in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md#sftp-host-key-verification).

Pinning is each party's local decision, so on its own it cannot catch a one-sided interception, where one party pins an attacker's key and the other pins the real key and neither view is ever compared, nor a server rekey that lands between the two parties' setups. For recurring (authenticated) exchanges the two parties therefore reconcile their views: each advertises the host-key fingerprint it observed inside the authenticated post-handshake exchange, and a divergence raises a warning that names both observed values and explains that it may be a one-sided interception or a between-setups rekey for the operators to disambiguate out-of-band. The check rides the authenticated channel, so the advertised fingerprint cannot be forged by an unauthenticated party; it is meaningful only where each party connects via SSH and observes a host key, so a party with none (a file-drop or proxy path) reconciles to no divergence rather than a false alarm. This is the post-handshake complement to the endpoint decision above: the invitation endpoint still carries no server-identity material, and the reconciliation adds its check strictly after the authenticated handshake. The construction is specified alongside the pinning control in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md#sftp-host-key-verification).

## Bootstrapping a shared secret

Parties wishing to transition from a zero-setup exchange to a recurring exchange may save the parameters from the zero-setup invocation (see [Zero-setup exchange](CLI.md#zero-setup-exchange)). Because this intent affects key generation, each party advertises it to the other at the start of the exchange.

| Party A | Party B | Outcome                                                                                                                                                                     |
| ------- | ------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| _save_  | _save_  | Initiator generates a fresh shared secret and transmits it to both; both save it as the basis for future recurring exchanges. Both also save their configuration files.     |
| _save_  | _none_  | No secret generated. A's configuration is saved; A is notified that their partner did not also choose to save and instructed to invite B to establish a recurring exchange. |
| _none_  | _none_  | Standard zero-setup exchange; no configuration is saved.                                                                                                                    |

The party that did not signal an intent to save the exchange parameters is notified that their partner is trying to establish a recurring exchange, that nothing is being saved on their end, and that they can either wait for an invitation from their partner or coordinate to run the exchange again and save the parameters.

The save intent rides the linkage-terms exchange -- the one round-trip both parties always perform -- so each side learns the other's intent even when it advertised nothing itself, and a one-sided save never stalls. When both parties save, the initiator generates a fresh base64url 32-byte secret (the same format and entropy as a rotation token; see [Token format and entropy](#token-format-and-entropy)) and transmits it to the responder on a dedicated in-band message.

That secret is protected only by the transport layer (SSH for SFTP, DTLS for WebRTC, or operator access controls for a file-drop), because a zero-setup exchange carries no key-exchange session key and so has no application-layer AEAD to wrap it. This is a deliberate, accepted trust model: bootstrapping with `--save` rests on the same transport trust -- and the same trust in the server administrator -- as the zero-setup exchange that carries it, for that one initial exchange only. The convenience is the point; a party who is not willing to trust the transport with the secret should use `psilink invite` instead, which never sends the secret through the exchange server (the invitation token travels out of band over a trusted channel such as secure email). After this first exchange the secret is rotated on every subsequent recurring exchange and is never reused.

In `--retain-files` mode the receiver does not delete exchange files after consuming them (it writes an acknowledgment marker instead), so the dedicated message carrying the secret persists as a cleartext file in the shared directory until it is removed by hand. This does not cross the trust boundary above -- the operator who holds that directory is already the trust anchor and already sees the secret in transit -- but it extends the exposure from in-transit to at-rest. A party bootstrapping a secret with `--save` should avoid `--retain-files` for that exchange, or clear the shared directory once the secret has been saved on both ends.

# Key-agreement design

Recurring exchanges derive their session key from an ephemeral X25519 key exchange, keyed together with the pre-shared secret and an explicit mutual confirmation, rather than the customized SPAKE2 PAKE used in earlier builds. The secret is high-entropy -- a 256-bit token, not a human password -- so a PAKE's one distinctive property, resistance to offline guessing, is unused; the requirements that actually matter, forward secrecy and mutual authentication, are preserved by ephemeral X25519 plus a secret-keyed confirmation, and the change sheds the largest hand-rolled-crypto surface in the system. The honest limit is the same as for any secret-authenticated design: a leaked secret still permits active impersonation until it is rotated out, and forward secrecy buys only that a recorded transcript stays confidential if the secret later leaks.

The construction is pinned to the Noise NNpsk0 pattern over X25519 with an added explicit, role-asymmetric key confirmation, following NIST SP 800-56A; like the SPAKE2 it replaces, it is modeled on that pattern and pinned by our own test vectors rather than wire-compatible with generic Noise, and its primitive comes from a well-respected, ideally audited library -- the full Noise framework is never hand-rolled. The wire-level specification is in [PROTOCOL.md](spec/PROTOCOL.md#x25519-authenticated-key-exchange) with the implementation. The CLI cut over first; the web application has since adopted the same handshake, running it over its WebRTC `MessageConnection` before each exchange (see the web discussion under [Channel security](#channel-security)). The cutover is a breaking change between builds, taken before 1.0 while that window is open, with future interop kept open by the handshake's version discriminant. Secret rotation is unchanged, and the grace-window mitigation for a rotation desync stays a deferred-not-foreclosed local change.

The pre-shared secret remains the baseline authentication anchor -- low setup friction, no organizational PKI for the bilateral-agency case -- with a certificate-chain, authority-backed mode left open by that same version discriminant, the deferred seam the signing design also leaves (see [Pinned self-signed trust model](#pinned-self-signed-trust-model)). The switch also carries a compliance upside: X25519 and Ed25519 are NIST-approved (FIPS 186-5, SP 800-186) and can be FIPS-validated, which the bespoke SPAKE2 never could -- material for a FIPS-validated build and agency ATO review. This is the principle behind the kept signing certificate, inverted: adopt the standard when it is lighter than the problem (ECDH with a confirmation, over a hand-rolled PAKE whose distinctive property was unused), and go bespoke when the standard is heavier (a canonical-JSON certificate, over X.509).

# Key file security

`.psilink.key` contains the shared secret that authenticates recurring exchanges. It must be treated as a credential with the same care as a private key or password.

## Required permissions

**Unix**: The CLI writes `.psilink.key` with mode `0600` (owner-read-only). The write goes to a sibling temp file created on an exclusive, non-following descriptor (`O_CREAT | O_EXCL | O_WRONLY | O_NOFOLLOW`) whose mode is set on the descriptor before any content is written, then atomically renamed into place, mirroring the Windows create-then-restrict discipline below: a symlink planted at the temp path cannot redirect the write to another file. The write is also durable across a power loss: the temp file's data is `fsync`'d before the rename and the parent directory is `fsync`'d after it, so a crash cannot surface the rename while losing the file's contents. Because each write flushes its own directory entry before returning, two sequential writes are crash-ordered -- if the second's rename is durable, the first's is too -- the guarantee the self-attested exchange record relies on (it writes the private opening file before the summary record, so a crash between the two preserves the proof material) and the one that keeps a freshly rotated token from being lost. Node's `fs` exposes `fsync` (`fsync(2)`), not the macOS `F_FULLFSYNC`, so on macOS the flush moves the data from the OS to the drive but does not force the drive to commit its volatile cache to stable media and does not stop the drive reordering writes; databases such as SQLite and Postgres use `F_FULLFSYNC` on macOS precisely for that stronger guarantee. So on macOS the crash-ordering above holds against process death but not necessarily a true power loss -- a power loss may surface a later write while losing an earlier one -- which is recoverable by re-running. Linux, the CLI's production target (the Docker image), flushes durably with `fsync`, so the guarantee holds there in full. If an existing key file has any group or world permission bits set (i.e. `mode & 0o077` is non-zero), the CLI emits a warning on load. Correct the permissions before running further exchanges:

```sh
chmod 0600 .psilink.key
```

On Unix the owner-only guarantee is enforced through the POSIX mode bits (`0600`), which is sufficient on Linux -- the production/Docker target, where `chmod` also collapses any POSIX ACL mask. On macOS an extended (NFSv4) ACL is governed separately from the mode bits, so an ACL entry a file inherits from its parent directory's inheritable ACEs can grant another principal access that a `0600` mode does not remove. This affects every owner-only artifact written into such a directory (the key file, signing identity, exchange records, and the result CSV), since each lands either in place or on a fresh inode that still inherits the directory's ACEs. On macOS, check with `ls -le` and clear an unexpected entry with `chmod -N <file>` when writing into a directory that carries inheritable ACLs.

Before a recurring exchange's handshake, psilink validates that the key file can be written, because a write that fails after the handshake has rotated the shared secret can desynchronize the two parties' tokens and force a re-invitation. On POSIX the key file's parent directory must be both writable and readable by the owner: the post-write parent-directory fsync opens the parent for reading, so a write-only (mode `0o300`) parent that passes a naive writability check would still fail the durability flush after rotation. The pre-flight therefore rejects a writable-but-not-readable parent up front.

**Windows**: The CLI enforces ACLs on write: it creates an empty placeholder file, narrows its ACL with `icacls /inheritance:r /grant:r` to grant Modify (`M`) to the current user only, then writes the token into the already-protected file. This ensures the token is never on disk while the file still carries inherited ACEs (e.g. the default `BUILTIN\Users` read). If the `icacls` call fails (for example in a restricted container environment), the placeholder is deleted and an error is raised; no key material is written. On Windows the token's data is flushed the same way as on Unix -- the writer reopens the ACL-narrowed file to write the content and `FlushFileBuffers` it through a handle before the rename -- but the parent-directory flush is not reachable: Node's `fs` exposes no way to open a directory handle and `FlushFileBuffers` it (the directory `fsync` the Unix path performs). So the cross-write crash-ordering guarantee above is POSIX-only and NTFS metadata journaling governs the durability of the directory entry here. The operation is recoverable in any case -- a lost rotated token or exchange record is re-produced by re-running -- so the residual Windows gap is a durability one, not a confidentiality one.

On load, the CLI first attempts to use PowerShell's `Get-Acl` with SID translation, which checks both inherited and explicit ACEs in a locale-independent way; SYSTEM (`S-1-5-18`) and Administrators (`S-1-5-32-544`) are not flagged. If PowerShell is unavailable -- for example in Nano Server containers or environments with strict application control policies -- the CLI falls back to `icacls`, which checks only explicit (non-inherited) non-owner ACEs. `fs.statSync` is not used for either check because it returns simulated POSIX mode bits that do not reflect the actual ACL. To correct over-permissive ACLs:

```cmd
icacls .psilink.key /inheritance:r /grant:r "%USERDOMAIN%\%USERNAME%":(M)
```

`%USERDOMAIN%\%USERNAME%` produces the domain-qualified name (e.g. `CORP\alice` or `COMPUTER\alice`) that `icacls` requires to resolve domain accounts unambiguously. This matches the value the CLI obtains internally via `whoami`. On a standalone (non-domain) machine `%USERDOMAIN%` equals the computer name, which is correct.

**Result CSV output**: The matched-records CSV that `psilink exchange` writes to an output path -- the most sensitive artifact the tool produces -- is created owner-only on the same principle as the key file: `0600` on Unix and an `icacls`-narrowed ACL on Windows, applied before any rows are written, so the output is not left world- or group-readable by an inherited umask. On Windows the ACL is recreated free of inherited and foreign ACEs; on Unix the mode-bits caveat above applies to it exactly as to every other owner-only artifact (a pre-existing or directory-inherited extended ACL on macOS is not stripped by the `0600` mode). It is streamed directly to the output path (the result set may be large) rather than written through the temp-file-and-rename the credential writers use, and the operator-supplied output path is not symlink-hardened; the owner-only guarantee is the same. Writing the result to stdout (no output path given) applies no permission handling, unchanged -- in particular, redirecting stdout to a file with a shell `>` leaves that file at the shell's umask, since the shell, not the CLI, creates it; pass an output path to get the owner-only treatment.

## What not to do

- Never commit `.psilink.key` to version control. Add it to `.gitignore`:
  ```
  /.psilink.key
  ```
- Never transmit the token over an unencrypted channel. If the token must be
  copied between machines, use an encrypted transfer (for example, `scp`,
  SFTP, or a secrets manager API).
- Never log or display the token value in plaintext. Avoid including
  `--key-file` content in bug reports or support tickets.
- Never store the token in environment variables that are visible to child
  processes or appear in process listings.

## Backup

Tokens may be backed up to a secrets manager or encrypted store (for example, HashiCorp Vault, AWS Secrets Manager, or an encrypted filesystem). Any backup must carry the same access restrictions as the original file: owner-read-only or the equivalent ACL. If no backup exists and a token is lost, re-invitation is the correct recovery path (see [Recovery](CLI.md#recovery)).

## Compromise response

If a token is believed compromised -- for example, it was observed in a log, a process listing, a shared filesystem, or in transit on an unencrypted channel -- treat it as invalid immediately and take the following steps:

1. Notify the partner out-of-band before any further exchanges are attempted; the partner's copy may also have been exposed.
2. Both parties delete their key files (`.psilink.key` on each side).
3. Re-invite over a channel known to be uncompromised. An observed token remains valid until the next successful exchange rotates it out, so the window between observation and deletion must be closed as quickly as possible.

## Token age and rotation policy

Tokens are rotated automatically on every successful exchange. By default there is no system-enforced maximum age: organizations with their own key-rotation policies can treat each exchange as the rotation event, and a partnership that exchanges frequently never accumulates a stale token.

The optional `token_max_age_days` field (in the [`authentication`](EXCHANGE_REFERENCE.md#authenticationtoken_max_age_days) block) makes that age bound system-enforced. When set, each successful exchange stamps an `expires` of that many days onto the rotated token in `.psilink.key`, and the next `psilink exchange` enforces it at load time: an expired token aborts the run before any connection or key exchange, with an error that names the expired time and directs both parties to re-invite; a token within `token_max_age_days / 3` days of expiry produces a pre-exchange warning. The threshold is a fraction of the configured age rather than a fixed number of days so that it scales with the policy -- a 90-day age warns with 30 days remaining, a 30-day age with 10. The warning is suppressed when the exchange that surfaces it succeeds, because rotation then stamps a fresh, farther-out `expires`; it is shown only when no successful rotation refreshed the token (a failed exchange), where re-inviting before the lapse is the actionable response.

This bound matters most for a dormant partnership: a monthly cadence or a holiday gap could otherwise hold a valid token for months with no mechanism to enforce a maximum age independent of exchange frequency. It is an operator policy, set per deployment, and is independent of `expires` enforcement in general -- a token whose `expires` is already set, by this policy or by an invitation's bounded lifetime, is honored regardless, and changing or removing the field affects only tokens stamped by later rotations. A misspelled key is rejected at config-parse time rather than silently dropped, so a typo cannot quietly disable the control (see [EXCHANGE_REFERENCE.md](EXCHANGE_REFERENCE.md#authentication)).

Both sides set the policy independently; it is not negotiated. If one party enforces a maximum age and the other does not, the enforcing party's token expires on its own schedule and that party refuses to exchange once it lapses, exiting with the re-invite error. From the non-enforcing party's perspective this can look like a peer that has gone silent or a key-exchange failure; the same recovery (re-invite) applies, which is why the expired-side error names re-invitation explicitly rather than emitting the generic out-of-sync guidance.

A deployment may still layer its own idle-interval policy on top (for example, a procedure that triggers re-invitation after 90 days without a successful exchange); `token_max_age_days` is the in-CLI mechanism for the same intent. As net-new security behavior, the enforcement is subject to the explicit security review required by [CONTRIBUTING](../CONTRIBUTING.md#dependency-policy) before release.

### Two sources, one `expires`

Two distinct mechanisms write the single `expires` field of `.psilink.key`: an invitation's bounded lifetime (the short-lived setup window under [Recurring exchange authentication](#recurring-exchange-authentication), default 1 hour) and the `token_max_age_days` policy above (a maximum age stamped onto a rotated token). psilink stores both in the one field and deliberately does not record which mechanism produced a given value. The field has a single meaning to every consumer -- the instant after which the token must not be used -- and a single recovery when it lapses: re-invite. A provenance marker distinguishing the two would add state to the key file without changing any decision taken from it, so the representation stays unified and the interpretation uniform.

Enforcement is kind-agnostic by construction. The load-time check before an exchange, the pre-handshake check, and the post-handshake check (which catches a token that lapses during the key-exchange round-trip) all abort on any `expires` at or before the current time and direct both parties to re-invite -- the correct response whether the value is an invitation lifetime or a max-age stamp. The accept-time invitation check is the only single-kind site, unavoidably so: it validates the encoded invitation, which carries only an invitation lifetime. There is no expired state in which the two kinds call for different recovery, so the expired-side message is uniform by intent rather than branched on origin.

The "expiring soon" advisory is the one surface tied to the max-age policy: its threshold is a fraction of `token_max_age_days`, so it is silent unless a policy is configured. It keys off the loaded `expires` without consulting its origin, so a freshly minted invitation token loaded while a policy is in force can be flagged "expiring soon." This is reachable only on the offline re-invitation path -- the inviter's key file is the only one that persists an invitation `expires`, the acceptor's offline key file carries none, and the online flow persists only the rotated token -- before the first exchange rotates the token, and it survives only past a *failed* first exchange, since a successful exchange rotates the token and clears the notice. The notice is accurate (a one-hour setup token is genuinely near expiry) and its guidance is the right next step for either kind ("run a successful exchange to refresh it; once it lapses, re-invite"), so it is accepted rather than suppressed. Suppressing it for invitation tokens alone would require persisting a per-file marker of each `expires`'s origin -- invisible state that could be present on one party's key file and absent on the other's, in a model whose recoverability depends on both parties holding uniformly interpreted key files. That is a worse trade than a rare, true, already-actionable notice.

No migration arises from this decision: an existing `.psilink.key` holds a bare `expires`, which is exactly what the uniform interpretation has always read, and setting, changing, or removing `token_max_age_days` continues to affect only tokens stamped by later rotations. The unification rests on the two sources sharing one recovery; it would need revisiting only if an expired max-age token were ever made recoverable without re-invitation, so that the expired-side action diverged by origin -- not the case today.

# Receipt signing identities

Certificate-backed, non-repudiable receipts (see [PROTOCOL.md](spec/PROTOCOL.md#signing-identity-and-certificate-pinning)) require a long-lived **signing identity** per party: an Ed25519 keypair and a self-signed certificate carrying the party's `identity`. This is a separate credential from the shared secret. The shared secret rotates on every exchange and lives in `.psilink.key`; the signing key must instead be stable for its whole life, because a partner pins its certificate fingerprint once and every later receipt must verify against the same key. The two are therefore kept in separate files with separate lifecycles.

## Custody and persistence

The signing identity is persisted in its own file (default `~/.psilink/signing-identity.json`), separate from the per-directory `.psilink.key` and `psilink.yaml`, because one identity is reused across every exchange and every partner -- a per-working-directory file would force a different identity (and fingerprint) per partnership and defeat the reuse the pinning model depends on. The file holds the private key and must be protected exactly as the key file is: it is written owner-read-only via the same atomic write path (`0600` on Unix; an `icacls`-narrowed ACL on Windows, with inheritance stripped), and on load the CLI emits the same over-permissive-permissions warning if other users can read it. See [Required permissions](#required-permissions) for the mechanism; the signing private key is a longer-lived, higher-value secret than the rotating shared secret, so it warrants at least the same care -- do not commit it, transmit it in cleartext, or log it.

`psilink fingerprint` creates the identity lazily on first use (the point at which a party must display its fingerprint to share it, which necessarily precedes any exchange) and loads it thereafter; creation is announced, never silent. Regenerating the identity is a deliberate, separate action (`--force`) that warns it invalidates any fingerprint a partner has already pinned.

## Pinned self-signed trust model

Trust in a partner's signing key is established by **pinning its certificate fingerprint out-of-band**, exactly as the parties already exchange the invitation and shared secret -- not by validating a certificate-authority chain. A self-signed certificate vouches only for possession of its own key, so the out-of-band fingerprint is the actual trust anchor: a presented partner certificate is trusted only if its self-signature verifies and its fingerprint matches the pinned `signing.partner_fingerprint`, and a receipt is accepted only if its asserted identity exactly matches the one the certificate binds. The fingerprint is a hash of a public certificate, so it is not secret; the out-of-band channel must be authentic (the partner is who you think) but need not be confidential. This is the SSH-host-key / certificate-pinning model.

This model was chosen for v1 over full X.509 + a certificate authority deliberately. The use case is bilateral and mutually coordinating -- the parties already exchange a secret out-of-band, so they can exchange a fingerprint the same way -- which is exactly where pinning fits and where the large PKI surface (chain building, name constraints, revocation) would be cost without benefit: none of that machinery is exercised when trust comes from a pin. Representing the certificate as a small canonical-JSON document signed over its RFC 8785 bytes also reuses the project's single canonicalization primitive and its cross-implementation reproducibility, rather than adding an ASN.1/DER parser whose byte-level canonicalization is, in practice, a less reliable basis for a fingerprint. The signature uses Ed25519 for its deterministic, non-malleable signatures and single canonical encoding.

This is a deferred-not-foreclosed decision. An authority-backed mode (an agency using certificates issued by its own CA) is a recognized future need and is not ruled out; the design leaves a seam for it. The certificate's `version` is a format discriminant, the `algorithm` field admits other signature schemes (a CA-issued certificate is typically RSA or ECDSA, not Ed25519), and the receipt-verification path is written around "is this key trusted, and does it authorize this identity?" -- where "trusted" is satisfied by a fingerprint pin today and could be satisfied by chain validation later. Adding that mode is additive work (X.509 parsing, chain validation, and the dependency and security review they entail), tracked for if and when a concrete partner requires it, rather than built speculatively now.

# Canonical encoding

The self-attested record's commitments, the agreed-terms hash both parties compute, and a signing certificate's fingerprint are each hashed or signed over a byte string; the planned certificate-backed non-repudiation receipt (see [ROADMAP.md](ROADMAP.md)) will sign an exchange record over the same bytes. For any of these to verify, every party that produces or checks one -- including an independent third party re-deriving them later from a different implementation, in a possibly different language -- must derive exactly the same bytes from the same logical object. Ordinary JSON does not guarantee that: two correct serializers can differ in key order, number formatting, or type handling, and any such difference makes a signature or commitment fail to verify against content that is in fact identical.

PSI-Link closes that gap with a single canonical encoding, reused for every hashed or signed artifact rather than re-derived at each call site, so that "these hash equal" reliably means "these are the same object" -- the property the pinned certificate fingerprint (see [Receipt signing identities](#receipt-signing-identities)) and the cross-party check of linkage terms both rest on. What the encoding is for and what it protects against is the whole of the overview here; the normative byte definition -- the standard it follows, the value domain it accepts, the encoding rules, and the cross-implementation test vectors an auditor reproduces -- is in [CANONICAL_ENCODING.md](spec/CANONICAL_ENCODING.md).

# Channel security

As noted in [Transport-layer authentication](#transport-layer-authentication), server administrators have visibility into exchanges conducted over SFTP and file-drops. The baseline protection on these channels is the transport layer itself (SSH for SFTP; the operator's access controls for file-drops). For authenticated, recurring exchanges an additional application-layer of encryption closes that visibility. Whether it is applied is negotiated in the key exchange rather than keyed off the bare authentication state: each party declares a request-encryption flag in the handshake and both wrap the connection when the negotiated decision (`own request OR peer request`, transcript-bound -- see [PROTOCOL.md](spec/PROTOCOL.md#x25519-authenticated-key-exchange)) is set and a session key was derived. The applicability rule is that the application AEAD applies exactly when the transport is not already end-to-end confidential against the in-path party and a session key exists: file-sync transports (SFTP, file-drop) request it unconditionally, because the server admin can snoop the transport; a transport already end-to-end confidential against any in-path party (WebRTC over DTLS) requests it only when that confidentiality is broken, namely under a DTLS-terminating WebSocket relay (see the WebRTC relay discussion below). The CLI requests encryption on every file-sync exchange and wraps the connection in the `EncryptedMessageConnection` AEAD decorator, so the SFTP/file-drop admin sees only opaque ciphertext. The AEAD construction -- the per-direction HKDF key derivation, the `{ enc }` envelope and sequence-number nonce format, and the rationale for binding no Associated Data -- is specified in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md). Zero-setup exchanges carry no session key and so run under transport encryption alone; the web application's application layer is likewise still cleartext, relying on the WebRTC DTLS transport encryption described below.

Beyond confidentiality, the file-sync channels are hardened against a hostile server admin -- the in-scope adversary of the [threat model](#threat-model) -- along two axes, memory exhaustion and liveness, plus a crash-safety and a parked-peer concern. The overview here is what each control protects against and why the set is sufficient; each is specified in full, with its derived constant values, enforcement points, and the dependency premises it rests on, in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md).

- **Application-layer integrity, replay, and gap detection.** The AEAD decorator validates and decrypts each inbound frame inside `receive()`, rejecting any integrity, replay, reordering, or mid-stream-gap failure as a `security` error at the protocol layer rather than as a transport-level latch behind a buffered frame. End-of-stream / truncated-tail detection is a deliberate accepted deferral, resting on the matching loop's self-driven lockstep.
- **Inbound frame-size bound.** A hostile server could otherwise write an arbitrarily large file that the transport reads and parses before any integrity check, exhausting memory; the transport refuses an oversized inbound frame before reading it.
- **Directory-listing bound.** The same memory-exhaustion impact, reached by directory enumeration -- a very large number of files, or very long filenames -- is bounded by caps on entry count and name length, enforced as the listing streams in.
- **Liveness bounds.** A hostile server can hang an operation rather than exhaust memory -- withhold a callback, flood empty listing batches, or trickle bytes forever. Per-operation deadlines and idle windows fast-fail each read and write; a whole-exchange budget backstops every transport await (including the local-filesystem adapter, closing the stalled-mount gap); and a connect-probe bound covers the one call outside that budget. All surface as a `UsageError` subclass that every transport consumer must treat as terminal.
- **SFTP fatal-packet crash safety.** A malformed SFTP reply packet would otherwise reach a listener-free wrapper and crash the CLI before its orderly cleanup and typed exit-code mapping run; a guarded `'error'` listener turns the fatal error into a handled, orderly failure.
- **Authenticated abort marker.** A party that fails terminally writes an authenticated `<id>-abort.json` so a parked peer fails fast instead of waiting out the full peer-inactivity budget. The token is keyed from the session key, so a hostile admin cannot forge an abort against a live peer, and a missing or forged marker only degrades to the pre-existing peer-silence timeout.

All of these controls are net-new security behavior subject to the explicit security review required by [CONTRIBUTING](../CONTRIBUTING.md#dependency-policy) before release. Their construction, the SFTP-stack internal premises they depend on, and the upgrade checklist for those premises are in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md) and [CONTRIBUTING.md](../CONTRIBUTING.md#upgrading-the-sftp-stack-ssh2--ssh2-sftp-client).

WebRTC connections use DTLS which provides end-to-end encryption, so the peer-coordination server never sees data-channel traffic. A TURN relay, when used to traverse NAT or firewall restrictions, preserves this property: it forwards encrypted DTLS packets without terminating the session.

A WebSocket relay is a distinct and rarer fallback that arises when a firewall blocks TURN through deep-packet inspection of the DTLS handshake. Unlike a TURN relay, a WebSocket relay terminates DTLS and sees plaintext, so it is exactly the in-path party against which the WebRTC transport stops being end-to-end confidential. This is what the application-encryption applicability rule above keys off: WebRTC declines the application AEAD in the ordinary case (the DTLS transport already protects the data against the peer-coordination server and any TURN relay), and requests it only when a DTLS-terminating WebSocket relay is forced into the path. The decision is then negotiated through the same request-encryption flag the CLI's file-sync channels use, not inferred from the authentication state. When a WebSocket relay is in use:

- For a recurring exchange (a session key was derived), the web path declares the request-encryption flag, so both parties negotiate the wrap and proceed with the application-layer AEAD encryption specified in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md) (the same mechanism the CLI's SFTP and file-drop channels apply). The web application does not yet implement this; it would be enabled together with WebSocket relay support.
- For a zero-setup exchange (no session key), the exchange aborts. Application-layer encryption cannot be applied without a shared session key, and proceeding without transport protection would expose data to the relay.

The web application no longer offers a secret-less (zero-setup) WebRTC rendezvous. Every web exchange now carries an invitation secret: both parties derive their peer-coordination rendezvous ids deterministically from it (see [Invitation contents and confidentiality](#invitation-contents-and-confidentiality)), so there is no secret-less web rendezvous to coordinate. The reason is deployment, not only hardening - no agency runs a standalone PeerJS peer-coordination server the way it runs an SFTP server, so a secret-less web rendezvous had no realistic deployment. The web exchange now runs the X25519 authenticated key exchange (the same `authenticateConnection` the CLI uses) over its WebRTC `MessageConnection` immediately after the data channel opens and before any PSI frame, so every web exchange authenticates the peer and derives a 32-byte session key. A failed handshake -- a wrong secret, a tampered or malformed frame, or a credential expired by the time the handshake runs -- fails closed before the exchange begins and is surfaced to the user as an authentication failure rather than a retryable transport error. The handshake threads the invitation's bounded `expires` into `authenticateConnection`, so a credential already past expiry is rejected before any frame is sent and one that lapses during the key-exchange round-trip is rejected once the handshake completes; this complements the earlier expiry check the web acceptor runs at the `/accept` consent boundary, which cannot cover the round-trip window. It still runs under DTLS transport encryption alone: like a recurring exchange over an already-confidential transport, the web path declines the application AEAD in the ordinary case (its request-encryption flag is false), so the session key currently serves only to authenticate. Wrapping the connection in the application AEAD -- which a DTLS-terminating WebSocket relay would require, negotiated through that same request-encryption flag and keyed from the session key the handshake now provides -- is the remaining step, enabled together with WebSocket relay support. (This relay policy will be enforced when WebSocket relay support is added; relays are not yet a supported transport option.)

Distinct from that data-path confidentiality, the bundled peer-coordination (signaling) server carries its own availability hardening, and against a broader adversary than the file-sync bounds. Where those defend against a hostile *server admin*, the web app ships the PeerJS signaling server itself -- internet-facing in production and gated only by the well-known default key -- so its inbound message parse is reachable by any unauthenticated client. A single oversized inbound frame would otherwise crash the shared rendezvous process (dropping signaling for every connected peer) or pin a large block of its memory; the server bounds the size of every inbound signaling frame and refuses an over-cap one before it is parsed. The bound sits far above any legitimate control message -- the SDP and ICE of connection setup are kilobyte-scale, while the PSI payload travels peer-to-peer over the WebRTC data channel and never crosses this socket -- and is a fixed constant rather than a configurable option, like the file-sync frame-size bound. It bounds a single inbound frame; like the file-sync controls it is net-new security behavior subject to the explicit security review required by [CONTRIBUTING](../CONTRIBUTING.md#dependency-policy) before release, and its derived value and enforcement point are in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md).

The web client keeps those derived rendezvous ids out of the browser console as well. PeerJS interpolates the remote peer id into its warning-level logs, so the client runs PeerJS at errors-only by default (warnings suppressed) and logs the ids in its own output only at the `debug` level, which production does not enable. A per-session diagnostic toggle (a `psilink:diagnostics` localStorage flag, documented for testers in [DEPLOYMENT.md](DEPLOYMENT.md#diagnosing-web-connection-failures)) raises PeerJS verbosity for one browser without a redeploy; it is wired through a custom PeerJS `logFunction` that redacts the session's derived ids before printing, so raised verbosity surfaces the connection-establishment and protocol-anomaly detail without reintroducing a rendezvous id to the console. The toggle raises only PeerJS verbosity, not the application's own log level, so engaging it cannot surface the `debug`-level id lines. PeerJS also surfaces a derived id a second way -- not only in the messages it prints, but inside the `Error` objects it *emits* on a connection failure (`ID "<id>" is taken` when the derived id is already registered; a failed negotiation to a dialed id), which propagate to the app's own error sinks (the exchange-failure `console.error` and alert) that the `logFunction` never sees. The app strips the derived ids from those errors' `message` and `stack` at each seam where a PeerJS error crosses into app code -- the rendezvous setup path and the live data-connection's error handler -- before the error is wrapped and propagated, so an ordinary mid-exchange connection failure does not surface a rendezvous id to those sinks either. This is best-effort console/display hygiene over the known PeerJS error shapes -- exact-id redaction, not a categorical guarantee against an id interpolated in some unforeseen form -- and it is defense in depth, not a primary control, since an attacker who already holds the invitation already holds the secret from which the id derives. Raised verbosity additionally surfaces PeerJS's connection-establishment detail (SDP and ICE candidates), which carries the local machine's private IP addresses and network topology; this is accepted as the cost of WebRTC connection diagnostics and mitigated by a tester caution in [DEPLOYMENT.md](DEPLOYMENT.md#diagnosing-web-connection-failures), rather than suppressed in code, which would remove the very signal the toggle exists to surface.

Distinct from that rendezvous-id redaction, the web client also keeps raw *partner-/server-influenced* bytes out of a production browser console. Two first-party console sinks would otherwise dump charset-unconstrained partner text: the exchange-failure handler's raw `console.error` of the caught `Error`, whose message and `.cause` chain can embed a hostile message-file path or other partner-/server-controlled transport-error text, and the acceptor's "dialing the inviter" log, whose host and port come from the partner's invitation endpoint. Both are gated behind the same per-session diagnostic predicate the verbosity toggle uses (a development build, or the `psilink:diagnostics` flag), so an ordinary production console carries none of these bytes while a developer or a diagnosing tester keeps the full structured `Error` object -- the gate runs the live object, not a stringified copy, preserving the devtools stack and `.cause` affordance that a console dump exists to provide. This is developer-surface defense in depth, not a primary control: the operator-facing rendered UI is separately escaped at its own display boundary (the exchange-failure alert and the accept-page summary route partner-supplied values through the shared display-sanitization helpers), so nothing partner-controlled reaches the rendered DOM unescaped regardless of this console gating. One operator-facing sink is escaped at the source rather than at this display boundary, and is worth calling out: the accept page's decode-*error* alert (distinct from the accept-page summary above) renders the decoder's failure message through `describeDecodeError` with no further sanitize pass, so its safety against partner bytes rests on `decodeInvitation` throwing only fixed strings for its non-Zod failures. The decoder deliberately swallows the V8 `JSON.parse` and `atob` error messages -- which can quote partner-supplied input -- and rethrows constant text, while `describeDecodeError` escapes the one partner-controlled component it interpolates from a Zod error (the issue path). Regression tests in `packages/core/test/invitation.test.ts` pin those swallows; the protection for this sink therefore lives at the decoder source, not at the render boundary.

**Diagnostics hardening**: psilink escapes untrusted partner- and server-controlled strings at every operator-facing display boundary -- CLI log lines and error output, and the web console and alerts -- through shared display-sanitization helpers (one for values, one that renders an error's message and cause chain). The threat is that a hostile or misconfigured partner or server controls strings that reach an operator's terminal or log: filenames and the rendezvous host and path (which an offline-accept invitation endpoint lets the partner seed), the partner's self-asserted identity, legal-agreement and column-name fields, abort reasons, and the spans some libraries quote inside an error message. Rendered raw, these allow terminal/ANSI control-sequence injection, bidi-override, zero-width, and homoglyph spoofing, and forged log lines. The rule is escape-at-display, compare-raw: only the displayed copy is escaped, while every value used to build a path or for an equality or hash comparison is kept byte-exact, so routing and match behavior are unchanged. The error renderer reads only an error's message and cause chain (never the stack or arbitrary properties), is cycle-safe, and never throws. Schema-validation errors are rendered without this pass because they carry no partner value -- except the invitation decoder, which deliberately rethrows fixed strings on a JSON or base64 failure so partner bytes never reach the message. The same self-leak discipline covers the operator's own secret-bearing files on the CLI: the config, key file, and signing identity are parsed through a single hardened seam that reports the file path only on a parse failure (and suppresses the YAML library's source-quoting stderr warnings), so a malformed credential file never echoes an inline credential, the shared secret, or a private key into an error or log; the error renderer additionally redacts any embedded PEM/OpenSSH private-key block as a last-resort backstop. Web devtools-console sinks that previously carried raw partner bytes are gated behind the per-session diagnostic predicate, so an ordinary production console carries none. The byte-level escape format is specified in [CHANNEL_SECURITY.md](spec/CHANNEL_SECURITY.md).

# Data handling

PSI-Link does not transmit, log, or retain any personally identifiable information (PII) outside of the configured output file.

**Protocol messages**: All data exchanged between parties during the PSI protocol consists of cryptographic protocol messages - elliptic-curve points. On recurring (authenticated) CLI exchanges these are wrapped in AEAD ciphertext by the application-layer encryption (see [Channel security](#channel-security)); on zero-setup exchanges and on the web application's application layer they cross the wire under transport encryption only. Raw PII values are never sent to a partner in any form. The PSI protocol's privacy guarantee ensures that each party learns only the existence of shared members; records not in the intersection are not revealed to the other party.

**Third parties**: No PII is transmitted to any third party. The peer-coordination server used by the web application's WebRTC channel sees only connection metadata (peer IDs); it has no visibility into data-channel traffic (see [Channel security](#channel-security)). SFTP and filedrop channels use operator-managed infrastructure.

**Logging**: PSI-Link does not write PII to log output. Any operational log output is limited to non-sensitive metadata: exchange timing, transport errors, and protocol state transitions. Log output should be reviewed before forwarding to a third-party logging service.

**Output**: The exchange output is an association table of row indices mapping each party's matched records. It does not contain raw PII fields from the input file. Parties are responsible for joining those indices against their own datasets and handling the resulting joined data under their applicable data governance policies.
